# spec-first 公众号文章运营 playbook（格式/篇幅/标题/文风/CTA/运营优化）

## Context

spec-first 微信公众号是面向 AI coding 开发者的垂直深度技术内容品牌，分三季 20+ 篇长文 + 贯穿内容流。运营约束散落在 `SKILL.md`（自动化 gate）、`references/style-guide.md`（视觉/排版）、`references/polish-prompt.md`（文风）、运营规划 `006/008/009`（战略/节奏）多处，且**运营视角的优化原则（打开率/完读率/转化/传播）在现有文档里完全空白**。op-00 一轮"资深运营视角审查"暴露了这个缺口：内容质量高但运营设计弱（标题作者视角、开头劝退、结尾佛系）。本文把散落规则整合成一份可复制的运营 playbook，让 op-02~07、第三季及未来文章由不同会话写作时质量下限一致。

**2026-06-15 增量（外部权威交叉验证）：** op-02~07 大纲评审时，跑了一轮 deep-research workflow 检索技术内容营销的外部权威最佳实践（NN/g 眼动·可用性研究、Backlinko、HubSpot、Buffer、Orbit Media 2025、Psychology Today 蔡格尼克效应），与内部经验交叉验证。结果**印证了内部经验的方向**（痛点前置、读者视角标题、CTA 不佛系），并**补齐了内部空白的可量化维度**：标题字符数/具体化、NN/g 倒金字塔与"只读 20%"完读模型、单一主 CTA（371% 差异）、系列连载的蔡格尼克追更机制。这些增量已同步进技术 reference `operations-playbook.md`，本文是 source-of-truth 沉淀。

## Guidance

### 一、标题规范（硬性 + 运营）

- **硬性**：主干三季正文标题必须以 `Spec-First：`（全角冒号）开头——品牌一致性；frontmatter 必须有 `title:`；正文不出现 H1（否则编辑器标题栏与正文重复显示）。
- **运营（打开率）**：`Spec-First：` 后面那段写**读者痛点/收益/悬念**，不写作者成就。前缀不可省（辨识），但冒号后是拉打开率的主力。
- **冒号后控制在 13-20 字**：移动端订阅列表只显前 37-48 字符，加 `Spec-First：` 前缀后更易截断；核心词前置、强词放两端。
- **收益/悬念要具体**：用数字/技术名/方法（"延迟降 40%""从 REST 迁到 gRPC"）替代泛泛"更快/更好"；悬念点出问题但不剧透解法，不做纯疑问句标题党。
- **对技术受众不用夸大式紧迫**（"最后机会""不看后悔"）——触发不信任、抬高取关；紧迫感要落在可兑现的真实技术事实上。
- **定稿前准备 2-3 个不同框架（痛点/收益/悬念）的候选标题对照择优**，不一稿定——读标题的人是读正文的 5-8 倍，损耗主要发生在"打开前"。
- **短钩子例外**：豁免 `Spec-First：` 前缀，用口语化钩子标题（目的是泛 AI 读者拉新）。
> 外部依据（2026-06-15 deep-research）：Backlinko / Invesp / Buffer(Ogilvy·Copyblogger) / NN/g。

### 二、篇幅与配图（硬性 gate）

| 内容流 | 正文字数（去空白） | 配图 |
|---|---|---|
| 主干三季正文 | ≥ 1.5 万字 | 6-10 张正文图（不含封面） |
| Building in Public | 5k-1 万字 | 按需 |
| 短钩子 | 800-1500 字 | 1 张主图 |

- 字数不足先扩写再配图；配图 <6 补图、>10 精简。
- 图片必须全部 PNG（微信不接受 SVG）；插图前后有过渡文案，不突兀。
- 由 `SKILL.md` Step 5.5 版式 gate 自动拦截：`bodyCharCount < 15000` 或 `imagePlaceholders < 6` 硬退出。

### 三、摘要 frontmatter description（硬门槛）

- **必须有** `description`，**≤ 120 字**（微信摘要上限）。
- 缺失时微信会用正文开头自动填充（污染、半句截断）。Step 4.7 校验：缺失或 >120 字 `DESC_OK=no` 阻断发布。

### 四、正文版式规范

- **章节编号**：H2 用 `## 01 标题`，H3 用 `### 01.1 标题`；H2 ≥ 5 个，H3 编号 ≥ 4 处。
- tech-blog 渲染必须给元素写 **inline style**（微信粘贴剥离 CSS class）。
- 段落短、留白稳定，移动端单段 ≤ 3 行；只用真实 Markdown 列表不手写 `•`；blockquote 只承载关键判断不堆长段。
- 代码块 + 文档块交替呈现 spec-first 产物。

### 五、文风模板（从成稿抽取）

**开篇钩子（首屏必须完成）**：① 加粗金句（一句话张力）→ ② 1-2 个真实痛点故事（有人有事有卡壳现场）→ ③ 点破张力 → ④ 正面体感故事（站对位置后的画面）。

**首屏完读率的可量化约束（NN/g 眼动/可用性研究）**：
- **倒金字塔**：首段就给核心结论/价值，不铺垫背景。自检——只读首段是否就知道这篇解决什么、给什么答案。
- **假设读者只读全文约 20%**（实测 18-28%）：核心价值压进最初几百字；读完开头 20% 离开也应已拿到主要收获。
- **钩子进前两屏**：74% 浏览时间在前两屏、42% 在顶部 20%，最有价值的钩子必须出现在这。
- **段首前两词测试**：79% 用户只扫描，句首/段首/标题头几个词必须直接表意——遮住每段首句前两个词仍能猜出段意。
- **去营销腔**：删"史上最强/颠覆性"等无法被事实支撑的最高级（仅去营销腔即提升可用性 27%），前几屏放真实数字/来源作"值得继续读"的信号。

**导读 blockquote**：`> **导读**` 格式，说清本篇解决什么、声明不依赖前序阅读、预告结构。

**加粗金句**：全文用 `**加粗**` 标注可独立传播的判断句——要可截图、脱离上下文仍有力。

**其他**：表格对比（before/after、节点总结）；超短段落（叙事 1-3 句）。

### 六、结尾 CTA 三件套（转化规范）

必须包含：
1. **关注引导**（不佛系）：「怕错过的话，关注一下不迷路。」
2. **转发钩子**（给读者转发理由 = 社交货币/实用价值）：「如果你身边也有人……，把这张地图转给他。」
3. **阅读原文话术 + GitHub/官网链接**：「想直接看代码或装来试，文末『阅读原文』直达 GitHub。」
4. **下一篇预告**（降门槛引导先读）：blockquote 给下篇标题 + 一句话钩子。

**主次分明（外部：单一主 CTA 比多 CTA 点击高 371%，3 个以上点击率下降）**：公众号语境下"关注/转发/阅读原文"是平台固定动作，但**四件套不要平铺等权**——按本篇目标定一个**主 CTA**（拉新期→关注追更；导流期→阅读原文去 GitHub），其余作次级。主 CTA 用更强的**动词+收益**措辞（外部：动词+收益比笼统措辞转化高一倍以上），并紧跟一句降风险/建信任的话（开源/免费/已能用）。

### 六·五、系列连载承接与追更（追更率）

连载靠"未闭合的认知张力"驱动追更（蔡格尼克效应）+ 可预期节奏养成习惯：

- **下一篇预告留悬念不剧透**：点名下一篇要回答的**那个具体问题/要揭晓的具体结果**（不是泛泛"敬请期待"），但不给结论或完整路径——说全了张力消散，追更动机没了。
- **开头一句"上集回顾"**：一两句承接上一篇，同时服务返场老读者和首次进来的新读者；配合系列命名/编号/前后篇互链显性化系列结构。
- **单篇自成价值 + 长文承载纵深**：每篇让新读者独立读懂（不依赖前序也能拿到收获），靠 1.5 万字深度留住老读者，避免中段稀薄掉队（外部 Orbit Media：2000+ 字长文是与"强效果"相关性最强的单项）。
- **固定可预期的发布节奏**：把追更变成约定式习惯；用每篇留存/承接数据复盘驱动下一篇选题与钩子。
> 外部依据：Psychology Today(Zeigarnik) / Orbit Media 2025。

### 七、运营优化原则（漏斗四层）

- **打开率**：标题前半段命中当下痛点/承诺收益，写"你正踩什么坑"不写"我做了什么"；≤20 字、收益具体化、不用夸大紧迫。
- **完读率**：倒金字塔首段给价值；钩子进前两屏；痛点故事前置首屏（不用导读+专名劝退）；情感弹药（画面感故事）前置别囤结尾；段落短、一段一观点、关键词在段首；可枚举内容改要点列表（外部：提升可用性 47%）；每隔几屏有列表/配图/代码块当视觉落点。
- **转化**：CTA 完整且**主次分明**（一个主 CTA + 动词收益措辞 + 降风险句），给具体行动指令，不佛系。
- **传播**：做"全景一图流"可截图总图当传播资产（「建议截图存下」）；金句可独立传播；表格/对比图/结论卡适合截图（可备竖版 1080×1350 / 方版 1080×1080 / 预览图 1200×630）。
- **追更**：下一篇预告留悬念不剧透 + 开头"上集回顾" + 固定发布节奏（见六·五）。

### 八、读者分层（一篇锁定一类主受众）

- A 型（认同方法论）→ 第一季；B 型（想确认能用）→ 第二季；进阶/团队（想用深用稳）→ 第三季。
- **一篇只锁定一类主受众**，不要同时服务三类（焦点模糊、篇幅膨胀）。

### 九、文章 DNA 与防重叠

- **骨架篇**：痛点 → 选地图 → 走完整链路 → 节点输入/输出 → 产物 → 可带走判断。
- **切面篇**：痛点 → 切链路一段 → 放大 → 判断 + 引向第三季机制篇。
- 共同点：**真实场景痛点开场 + 可带走判断收尾**。
- **防重叠三铁律**：op 篇不复述机制定义（给直觉 + 引向 s3 篇）；op 篇主轴"场景+判断"、s3 主轴"机制+契约"；高风险三对（op-05↔s3-08、op-06↔sessions/s3-10、op-07↔s3-09）写前交叉检查。

### 十、内容流并行与发布节奏

- 四流并行：正文（留存，周更）/ BIP（信任，每 3-4 篇穿插）/ 短钩子（拉新，周更错开）/ 资产物料（收藏转发，每约 4 篇沉淀 1 个）。
- 策略：**最小实操季过桥 + 库存变现**——成品优先发，未写不阻塞已写。
- 发布顺序锁定：`op-00 → op-01 → op-05 → op-02 → op-06 → op-03 → op-04 → op-07`（写作顺序 ≠ 发布顺序，op-05 传播最强提前）。

## Why This Matters

不遵守的后果（按严重度）：

- **缺 description / 正文 <1.5万字 / 配图 <6**：发布流水线 Step 4.7/5.5 直接阻断，根本发不出去；摘要缺失还会被正文开头污染成半句。
- **标题无 `Spec-First：` 前缀**：系列品牌碎裂，读者订阅列表里认不出同系列，建立不了追读心智。
- **正文有 H1**：编辑器标题栏与正文第一行重复显示。
- **首屏导读+专名起手**：新读者划走，完读率低。
- **结尾佛系 CTA**：读完认同但不关注、不转发、不试用，转化归零（公众号传播主靠转发，不给理由就不转）。
- **一篇服务三类读者**：谁都没被精准击中。
- **op 篇复述机制**：与 s3 篇重叠，读者弃读。
- **不按锁定发布顺序**：递进逻辑断裂（如 op-06 的 docs 价值兑现依赖前面 op 篇攒的产物）。

## When to Apply

- 所有主干三季正文的写作与发布。
- BIP 篇（前缀/字数/配图放宽，开篇钩子 + CTA 三件套仍适用）。
- 短钩子（前缀豁免、字数/配图独立标准，传播资产原则适用）。
- 新文章启动时：先定位主受众、选 DNA 类型、确认与已有篇目不重叠。
- 发布流水线校对、跨会话接手半成品时。

不适用：仓库内部规划文档、README/台账/导航、非公众号渠道内容。

## Examples

**标题改写（作者视角 → 读者痛点）：**

| 改前 | 改后 |
|---|---|
| `Spec-First：我用 spec-first 跑完了整个流程` | `Spec-First：同样用 AI 写代码，为什么有人越改越顺、有人越改越崩` |
| `Spec-First：如何用 AI 调试 bug` | `Spec-First：AI 改崩了怎么收拾——不是"你再检查一下"能解决的` |

**开头改写（导读劝退 → 痛点前置）：**

改前：「导读：本文是 spec-first 第二季第 0 篇，讲解三种需求模式和三种仓库拓扑……」

改后：「**同样用 AI 写代码，有人越改越顺，有人越改越崩。差别往往不在 AI，在你站在了哪个位置。** 先讲两个我真实见过的卡壳现场……」（先让读者在场景里看到自己，再放导读）

**CTA 改写（佛系 → 三件套）：**

改前：「spec-first 是开源项目，欢迎试用、提 issue。GitHub：…」

改后：「**怕错过的话，关注一下不迷路。** 如果你身边也有人'用 AI 写代码越改越乱'，把这张地图转给他。想上手的话，最低成本第一步是先读下一篇 op-01……文末『阅读原文』直达 GitHub。」+ 下一篇 blockquote 预告。
