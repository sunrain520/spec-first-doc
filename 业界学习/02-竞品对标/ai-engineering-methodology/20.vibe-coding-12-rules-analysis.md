# Vibe Coding 12 条"天条"深度分析报告

> **分析对象**：Karpathy-Inspired Claude Code Guidelines — 12 Rules（andrej-karpathy-skills-12）
> **源材料**：微信公众号文章《Vibe Coding 错误率从 85% 降到 8%，我只做了一件事》（主文章）、GitHub 项目 README、搜索结果（5 篇相关报道）
> **分析视角**：以 spec-first 方法论为基准框架，对照 12 条规则体系进行评估
> **关联报告**：#15 pm-skills、#16 OpenSpec+Superpowers、#17 AI PRD 方法论、#18 Gstack 三层架构、#19 组织级 Coding Agent
> **生成日期**：2026-06-14

## 一、项目概述

### 1.1 背景

Andrej Karpathy 在长期使用 LLM 编程工具的过程中，观察到 AI 编码助手存在三类高频失败模式：**瞎猜需求**（沉默假设）、**乱改代码**（附带修改）、**堆代码**（过度工程化）。开发者 Forrest Cheung 将 Karpathy 的观察提炼为 4 条可执行规则，放入一个 CLAUDE.md 文件，项目随即以 140K+ Star 登顶 GitHub 热榜。

随后，社区发现仅靠 4 条"编码阶段规则"无法覆盖现代 AI Agent 工作流中的新失败模式。twj515895394 在此基础上扩展出 12 条规则，新增 8 条"Agent 执行阶段规则"，形成了一个覆盖"理解需求 → 编写代码 → 自我验证"完整链路的 AI 行为契约（Behavior Contract）。

### 1.2 核心问题域

12 条规则精准命中 AI 编码 Agent 的两类失败模式：

| 阶段 | 规则范围 | 典型失效模式 |
|------|---------|-------------|
| **编码阶段**（Rules 1-4） | 单文件/单函数级别的代码生成行为 | 沉默假设、过度工程化、附带修改、验证缺失 |
| **Agent 执行阶段**（Rules 5-12） | 跨文件/多步骤/长 Session 的 Agent 工作流行为 | 上下文漂移、模型越界判断、冲突模式折中、浅层测试、状态丢失、约定分叉、静默失败 |

### 1.3 实测效果

主文章（来源：微信公众号）中作者披露的量化数据：

- **规则前**：约 85% 的轮次至少遇到一种上述失败模式
- **规则后**：不到 8% 的轮次出现同类问题
- **核心机制**：不是让模型"更聪明"，而是通过显式行为边界消除其自由发挥空间——将隐含的工程师直觉转化为机器可执行的具体约束

## 二、架构分析

### 2.1 12 条规则的层次结构

```
┌─────────────────────────────────────────────────────┐
│                  AI 编码行为契约                      │
├─────────────────────┬───────────────────────────────┤
│   编码阶段规则 (1-4)  │   Agent 执行阶段规则 (5-12)     │
│                     │                               │
│  1. Think Before    │  5. 模型仅用于判断             │
│     Coding          │  6. Token 预算不是建议          │
│  2. Simplicity      │  7. 暴露冲突，不要折中          │
│     First           │  8. 写之前先读                  │
│  3. Surgical        │  9. 测试验证意图                │
│     Changes         │ 10. 重要步骤后设置检查点        │
│  4. Goal-Driven     │ 11. 匹配代码库约定              │
│     Execution       │ 12. 大声失败                    │
│                     │                               │
│  来源：Karpathy      │  来源：社区实践经验              │
│  原始 4 条           │  面向 Agent 工作流的 8 条扩展    │
└─────────────────────┴───────────────────────────────┘
```

### 2.2 设计原则

1. **每条规则对应一个真实翻车模式**：不写"保持代码整洁"这种正确但无法执行的废话，每条规则都是具体可操作的边界定义
2. **否定式约束优先**：12 条规则的核心表述方式不是"要做什么"，而是"不要做什么"——用反例定义边界比正例更清晰
3. **优先级序位**：Rules 1-4 是地基（编码阶段的总纲），Rules 5-12 是上层结构（Agent 工作流的护栏），缺失前 4 条则后 8 条失去基础上下文
4. **项目级持久化**：规则通过 CLAUDE.md / .cursor/rules 机制在项目范围内持久生效，不依赖 session 级别的提示词注入

### 2.3 与 spec-first 方法论的架构对照

12 条规则体系与 spec-first 的 PRD → Plan → Work → Review 四阶段存在天然映射关系：

| spec-first 阶段 | 对应规则 | 映射逻辑 |
|----------------|---------|---------|
| **PRD（需求定义）** | Rule 1, 4 | Think Before Coding 暴露假设 → 对应需求澄清；Goal-Driven Execution 定义成功标准 → 对应验收条件 |
| **Plan（方案设计）** | Rule 2, 7, 8 | Simplicity First 约束方案膨胀；Surface Conflicts 要求显式选择；Read Before You Write 强制上下文理解 |
| **Work（编码执行）** | Rule 3, 5, 11 | Surgical Changes 限制修改范围；模型仅用于判断限定决策边界；Match Conventions 维护一致性 |
| **Review（验证审查）** | Rule 6, 9, 10, 12 | Token 预算防止上下文崩溃；测试验证意图；检查点机制防止步骤漂移；Fail Loud 暴露不确定性 |

## 三、逐维度对照矩阵

以 spec-first 方法论为基准框架，对照 12 条规则体系进行逐维度分析。

### 3.1 行为边界管理

| 维度子项 | spec-first 实践 | 12 条规则对应 | 契合度 | 差异说明 |
|---------|----------------|--------------|--------|---------|
| 需求理解边界 | PRD 阶段强制需求拆解与验收标准定义 | Rule 1: Think Before Coding；Rule 4: Goal-Driven Execution | **高** | spec-first 在需求阶段就通过结构化 PRD 完成假设显式化；12 条规则将这一约束前移到编码入口 |
| 修改范围边界 | Plan 阶段定义变更范围，Work 阶段严格按 Plan 执行 | Rule 3: Surgical Changes | **高** | 两者核心理念一致——修改范围由上游决策明确定义，不允许"顺手优化" |
| 技术决策边界 | 架构决策记录（ADR）显式记录技术选型 | Rule 7: Surface Conflicts；Rule 11: Match Codebase Conventions | **中** | spec-first 通过 ADR 完成正式的决策管理；12 条规则仅在代码层面约束"不要折中"和"遵循约定"，缺少决策追溯机制 |

### 3.2 上下文控制

| 维度子项 | spec-first 实践 | 12 条规则对应 | 契合度 | 差异说明 |
|---------|----------------|--------------|--------|---------|
| Session 长度控制 | 通过 PRD-Plan 分层降低单次会话复杂度 | Rule 6: Token Budgets Are Not Advisory | **高** | spec-first 从任务架构层面避免长 Session（每个阶段独立上下文）；12 条规则从执行层面设置硬性预算 |
| 上下文有效性 | Plan 文档作为跨 Session 的持久化上下文锚点 | Rule 10: Checkpoint After Every Significant Step | **中** | spec-first 通过文档传递上下文；12 条规则通过检查点"强制同步"，两者互补但机制不同 |
| 知识复用 | PRD/Plan 作为项目知识库沉淀 | 无直接对应 | **低** | 12 条规则关注执行行为约束，不涉及知识管理；spec-first 的文档体系在此维度上有明显优势 |

### 3.3 代码变更原子性

| 维度子项 | spec-first 实践 | 12 条规则对应 | 契合度 | 差异说明 |
|---------|----------------|--------------|--------|---------|
| 变更粒度控制 | Plan 阶段拆解为独立任务单元，逐项执行 | Rule 3: Surgical Changes | **高** | 两者均要求最小化变更范围，spec-first 通过 Plan 的拆解提供更结构化的粒度定义 |
| 附带修改防护 | Work 阶段要求变更只涉及 Plan 中标记的文件 | Rule 3: Surgical Changes；Rule 11: Match Conventions | **高** | 12 条规则中"不动 B 文件的格式"与 spec-first 的按 Plan 执行理念一致 |
| 回滚机制 | 按任务单元独立提交，失败时精确回滚 | Rule 10: Checkpoint | **中** | spec-first 通过 Git 提交策略实现原子回滚；12 条规则的检查点仅提供状态同步，不直接约束版本控制 |

### 3.4 验证机制

| 维度子项 | spec-first 实践 | 12 条规则对应 | 契合度 | 差异说明 |
|---------|----------------|--------------|--------|---------|
| 成功标准定义 | PRD 阶段定义验收条件（Acceptance Criteria） | Rule 4: Goal-Driven Execution | **高** | 理念完全一致——"优化性能"不是目标，"P99 延迟降到 100ms"才是 |
| 测试深度 | Review 阶段要求测试覆盖业务意图而非仅代码路径 | Rule 9: Tests Verify Intent, Not Just Behavior | **高** | 两者均反对"覆盖率 100% 但没测业务逻辑"的假通过 |
| 失败可见性 | Review 阶段要求暴露所有不通过项 | Rule 12: Fail Loud | **高** | 核心诉求一致——禁止静默跳过、隐藏不确定性 |

### 3.5 多步任务稳定性

| 维度子项 | spec-first 实践 | 12 条规则对应 | 契合度 | 差异说明 |
|---------|----------------|--------------|--------|---------|
| 步骤状态管理 | Plan 阶段定义任务 DAG，按依赖关系执行 | Rule 10: Checkpoint After Every Significant Step | **中** | spec-first 通过 Plan 的 DAG 提供全局状态视图；12 条规则通过检查点做局部状态同步 |
| 漂移防护 | Work 阶段对每个子任务执行前验证前置条件 | Rule 10: Checkpoint；Rule 6: Token Budget | **中** | spec-first 的前置条件验证更结构化；12 条规则的检查点+Token 预算是运行时兜底 |
| Agent 自主决策边界 | Work 阶段 Agent 在 Plan 边界内自治 | Rule 5: Use the Model Only for Judgment Calls | **高** | 两者都明确划分"确定性逻辑"与"需要判断力"的边界 |

### 3.6 约定一致性

|---------|----------------|--------------|--------|---------|

| 架构约定 | ADR + Plan 定义架构约束 | Rule 7: Surface Conflicts；Rule 8: Read Before You Write | **中** | 12 条规则侧重"不要破坏一致性"和"先读再写"；spec-first 通过 ADR 提供正式的架构治理 |

### 3.7 失败可见性

|---------|----------------|--------------|--------|---------|
| 不确定性的处理 | Review 阶段要求标记所有不确定项为 Blocked | Rule 12: Fail Loud | **高** | 完全一致——禁止将"可能没问题"伪装成"已完成" |
| 验证失败处理 | Review 阶段要求记录失败原因和重试策略 | Rule 12: Fail Loud | **中** | 12 条规则的"大声失败"更侧重暴露而非处理；spec-first 的 Review 阶段有更完整的失败处理流程 |

## 四、关键差异深度分析

### 4.1 从 4 条到 12 条的演进逻辑

Karpathy 原始 4 条规则解决的是**编码阶段**的失败模式——本质上是对"写代码"这个单一步骤的行为约束。当 AI 编码从"辅助写函数"演进到"Agent 自主执行多步骤任务"时，新出现的失败模式已经超出了编码阶段的范畴。

**演进路径分析**：

| 演进驱动力 | 新失败模式 | 新增规则 | 关键洞见 |
|-----------|-----------|---------|---------|
| Agent 工作流引入多步骤执行 | 上下文漂移（长 Session 中遗忘早期决策） | Rule 6: Token Budget | Token 不是成本问题，是质量保障——超过预算的上下文是不可靠的 |
| Agent 具备工具调用能力 | 模型越界判断（用 LLM 替代确定性代码） | Rule 5: 模型仅用于判断 | 最隐蔽的错误：AI 在用概率输出替代确定性逻辑 |
| 多 Session 协作 | 冲突模式被"平均"而不是被选择 | Rule 7: 暴露冲突 | AI 面对不一致时的本能是折中，结果继承了双方的缺点 |
| Agent 自主决定操作范围 | 不读上下文就写（重复造轮子） | Rule 8: 写之前先读 | 这是对人类工程师 Code Review 习惯的程序化——先理解再动手 |
| 自动化测试生成 | 浅层测试通过但不验证业务意图 | Rule 9: 测试验证意图 | 覆盖率是手段不是目的——测 getter/setter 的测试是噪音 |
| 长任务链路执行 | 步骤间状态丢失，错误级联扩散 | Rule 10: 检查点 | 强制同步机制——在每一步结束时把 Agent 拉回现实 |
| 多项目/多团队 | 风格分叉（AI 按自己偏好改约定） | Rule 11: 匹配约定 | 一致性 > 正确性——不好的统一约定优于好的不一致 |
| Agent 自我评估 | 静默失败（跳过、隐藏、假装成功） | Rule 12: 大声失败 | 翻转默认值：从"说做好了"变为"暴露不确定" |

### 4.2 12 条规则的"否决性"设计

一个值得注意的设计选择：12 条规则中，绝大多数采用了**否定式约束**（"不要 X"而非"要 Y"）。这是针对 LLM 行为特征的刻意设计：

- LLM 对"要做什么"的理解存在极大的解释空间（"要简洁"可以理解成从 1 行到 200 行的任意粒度）
- 对"不要做什么"的执行更加可靠——因为边界是离散的、可验证的
- 否定式约束天然适合"护栏"角色：护栏不需要告诉你往哪开，只需要告诉你哪些方向会翻车

这种设计思路与 spec-first 中的 Plan 阶段约束存在内在一致性：spec-first 的 Plan 同样通过定义"不在本次范围内"来划定边界。

### 4.3 12 条规则可直接落地于 spec-first 的部分

以下规则与 spec-first 方法论天然兼容，几乎不需要适配：

| 规则 | 直接落地方式 | 落地阶段 |
|------|------------|---------|
| Rule 1: Think Before Coding | 在 Work 阶段入口前置：AI 开始编码前必须输出对 PRD/Plan 的理解确认 | Work 入口 |
| Rule 2: Simplicity First | Plan 阶段已通过任务拆解约束范围，Rule 2 作为 Work 阶段的补充护栏 | Work |
| Rule 3: Surgical Changes | 映射到 Plan 中定义的文件范围——AI 只操作 Plan 中标记的文件 | Work |
| Rule 4: Goal-Driven Execution | 直接对应 PRD 中的验收条件（Acceptance Criteria） | PRD → Review |
| Rule 7: Surface Conflicts | 在 Plan 阶段遇到架构选型冲突时，要求显式选择并记录到 ADR | Plan |
| Rule 12: Fail Loud | Review 阶段的标准动作——不确定项标记为 Blocked 而非忽略 | Review |

### 4.4 需要适配的部分

以下规则与 spec-first 的现有机制存在重叠或冲突，需要针对性适配：

| 规则 | 适配挑战 | 适配建议 |
|------|---------|---------|
| Rule 5: 模型仅用于判断 | spec-first 的 Agent 工作流中已有明确的工具路由规则，Rule 5 可能冗余 | 建议将 Rule 5 作为工具路由补充约束，嵌入到 Work 阶段的 Agent 指令中 |
| Rule 6: Token Budget | spec-first 通过分阶段执行天然控制 Token，不需要硬性数字预算 | 保留 Rule 6 作为兜底机制（当单一子任务过大时触发），但不作为日常约束 |
| Rule 8: Read Before You Write | spec-first 的 Plan 阶段已要求上下文分析，Work 阶段 Agent 默认持有 Plan 信息 | 适配为"Work 阶段执行前验证 Plan 中的上下文信息是否仍然有效" |
| Rule 9: 测试验证意图 | spec-first 的 Review 阶段已有业务验收，Rule 9 更侧重单元测试层面 | 将 Rule 9 拆分为两个层面：Review 阶段验收业务意图（已有），Work 阶段的单元测试增加意图检查 |
| Rule 10: 检查点 | spec-first 的 Plan 已将任务按 DAG 拆解，检查点在 Plan 层面已天然存在 | 适配为"每个 Plan 子任务完成后，Agent 必须输出执行摘要（已完成/已验证/待处理）" |
| Rule 11: 匹配约定 | spec-first 的约定定义在 Plan 阶段，但缺少运行时校验 | 建议在 Work 阶段增加约定一致性检查环节 |

### 4.5 12 条规则的适用范围限制

主文章作者在末尾给出了务实的判断，以下场景中全套 12 条规则反而会拖后腿：

1. **快速原型开发**：Surgical Changes 过于保守，原型阶段需要快速试错
2. **单文件小修改**（改 typo、加一行注释）：Think Before Coding 的完整流程开销过大
3. **探索性编程**（Spike）：Goal-Driven Execution 要求先定义成功标准，但探索性编程的目标正是发现成功标准
4. **已有严格 CI/CD 管控的项目**：部分规则（如 Rule 9、Rule 11）可能与现有 CI 规则重复

作者建议的底线是：**Rules 1-4 + Rule 12（大声失败）作为最小必选集**，其余按项目阶段和复杂度按需启用。

## 五、知识链收束

### 5.1 关联 #15 pm-skills：职业级 Agent 的工程化要求

#15 报告中提出的核心论点——"职业级 Agent 需要工程化约束而非自由发挥"——在 12 条规则中得到了精确具象化。

- **#15 的"行为可预测性"** 直接对应 Rule 1（Think Before Coding）和 Rule 12（Fail Loud）：Agent 的行为必须可预测，假设必须暴露，失败必须可见
- **#15 的"修改范围约束"** 对应 Rule 3（Surgical Changes）：职业级 Agent 不会自作主张扩大修改范围
