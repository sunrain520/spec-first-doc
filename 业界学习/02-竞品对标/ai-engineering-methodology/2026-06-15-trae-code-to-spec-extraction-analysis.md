# Trae 解构项目业务与代码流程：从代码中提取规范的方法论复盘

> 归档日期：2026-06-15
> 归档主题：code-to-spec / 代码理解 / 规范提取 / Knowledge Wiki / spec-first 反向校验

## 1. 项目概述

这篇文章以开源项目 Akali（阿卡丽）为案例，演示如何使用 Trae IDE、AI 智能体、提示词和 PlantUML，从已有代码中反向梳理项目架构、业务流程、术语、数据模型、业务规则、最佳实践和组件说明文档。

它表面上是在介绍 Trae 如何“读项目、写文档”，但对 spec-first 更关键的价值是：**它展示了一条 code-to-spec 路径，即从已有代码中提取规范资产。**

文章中的产物不是单一 README，而是一组面向项目维护的知识资产：

- 架构文档；
- 术语文档；
- 数据模型手册；
- 业务规则文档；
- 开发最佳实践；
- 项目结构与模块划分；
- PlantUML 架构图 / 流程图 / 类图；
- 综合指南；
- HTML 组件介绍页。

核心判断：**从代码中提取规范，本质上不是“把代码翻译成文档”，而是从代码中提炼稳定的概念、流程、规则、边界和验证线索。** 这些内容经过人工校验后，可以反向补齐历史项目的 Spec，并进一步进入 spec-first 的双向一致性闭环。

## 2. 架构分析

### 2.1 文章中的 code-to-spec 流程

根据文章，可以抽象出如下流程：

```text
已有代码仓库
  ↓
AI IDE / Agent 读取项目结构
  ↓
分析模块、类、接口、配置、注解、规则逻辑
  ↓
生成 Markdown + PlantUML + HTML 文档
  ↓
人工校验与修订
  ↓
沉淀为 guide / wiki / spec 资产
```

这个流程的关键不是“AI 自动写文档”，而是把代码中的隐性知识外化为可维护资产。

### 2.2 可提取的规范类型

从代码中可提取的规范大致分六类：

| 规范类型 | 代码来源 | 典型产物 |
| --- | --- | --- |
| 架构规范 | 目录结构、模块依赖、核心组件 | 架构说明、模块图、组件职责 |
| 流程规范 | Controller / Service / Listener / Job 调用链 | 流程图、时序说明、关键路径 |
| 术语规范 | 类名、枚举、注解、领域对象、README | 术语表、概念定义、别名表 |
| 数据规范 | Entity / DTO / Config / Rule / Enum | 数据模型手册、字段含义、枚举值 |
| 业务规则规范 | if/else、策略类、规则类、计算逻辑 | 判定规则、公式、边界条件 |
| 工程实践规范 | 设计模式、配置方式、异常处理、性能路径 | 最佳实践、排障指南、调优建议 |

这些规范的稳定性不同：术语、职责、主流程较稳定；具体分支、配置值、SQL、异常条件变化更快，不能无脑写入长期文档。

### 2.3 与 Knowledge Wiki 的关系

这篇文章与已有的 `Knowledge Wiki：面向 AI 的项目知识层建设实践` 形成互补。

`Knowledge Wiki` 的核心判断是：不要做全量 Code Wiki，不要逐行翻译代码，而是提炼 AI 从代码中难以自动发现、但后续理解和排障有价值的知识。其推荐流程是“先识别流程，再穷举概念，再归并流程，再划分上下文，最后收敛概念并写 wiki”。

Trae 这篇文章更偏工具实践，证明 AI IDE 可以实际生成多类项目文档；Knowledge Wiki 则提供了更强的筛选原则：

```text
该沉淀：稳定概念、关键流程、关键逻辑、导航指引、排障经验
不沉淀：逐行代码说明、易变实现细节、一次性过程、证据不足猜测
```

因此，spec-first 吸收这篇文章时，不能走向“把整个代码库翻译成 guide”。正确方向是：**把代码反向提取为可审查、可回源、可持续维护的规范候选。**

### 2.4 与 spec-first 的闭环关系

spec-first 正向链路是：

```text
Spec → Plan → Tasks → Code → Review → Knowledge
```

这篇文章提供的是反向链路：

```text
Code → Guide / Wiki / Reverse Spec → Review → Spec 修订 / Knowledge 沉淀
```

两者组合后形成闭环：

```text
正向：规范指导代码实现
反向：代码反查规范是否完整、是否过期、是否偏离
闭环：Spec 与 Code 持续双向校验
```

这对历史项目尤其重要：如果项目并非一开始就是 spec-first，可以先用 code-to-spec 做逆向初始化，再逐步进入 spec-first 管理。

## 3. 对照矩阵

| 维度 | 文章做法 | spec-first 应吸收的做法 | 风险控制 |
| --- | --- | --- | --- |
| 输入 | 已有代码仓库 Akali | 目标项目源码 + README + 配置 + 测试 + 历史文档 | 明确扫描范围，避免跨仓误读 |
| Agent 角色 | Trae 智能体执行项目分析 | 专门的 code-to-spec / knowledge initializer workflow | 角色提示词必须要求出处和不确定性标注 |
| 输出目录 | 项目根目录 `guide` | 可映射为 `.wiki/`、`docs/solutions/`、`docs/spec-reverse/` 或项目约定目录 | 不应直接覆盖现有 README / Spec |
| 图表 | PlantUML 架构图、流程图、类图 | 推荐保留文本化图表，便于版本管理 | 图表节点必须能回链到源码路径 |
| 术语表 | AI 提取项目术语 | 可作为 Spec 术语层和 Knowledge Wiki seed | 术语需人工确认，避免 AI 乱定义 |
| 数据模型 | 从实体、配置、枚举中提取 | 可作为数据模型规范候选 | 字段含义必须标注来源文件 |
| 业务规则 | 从流控、热点检测、降级逻辑中提取 | 可作为 reverse spec 的重点 | 幻觉风险最高，必须强制 source refs |
| 最佳实践 | AI 总结设计模式、配置优化、排障建议 | 可进入 knowledge / playbook | 区分“项目事实”和“通用建议” |
| HTML 展示 | 生成漂亮组件介绍页 | 可选展示层，不是规范核心 | 不让展示层替代可审查 Markdown |
| 人工校验 | 作者提醒可能有不存在内容 | 必须成为流程门禁 | 无人工校验不得提升为 confirmed spec |

## 4. 关键差异

### 4.1 文章偏“一次性生成 guide”，spec-first 需要“可治理的 reverse spec”

文章中的实践是：打开项目，让 AI 生成一组 guide 文档。这个方式适合快速上手和演示，但在 spec-first 中不能直接变成“生成完就算规范”。

spec-first 需要多几个治理层：

- 每条规则要有源码出处；
- 每个流程要有入口、出口、异常分支和调用链证据；
- 每个术语要标注来自代码、文档还是人工补充；
- 每个不确定判断要标注 confidence / limitations；
- 每个被排除候选要有原因；
- 产物需要 Review 后才能从 candidate 变成 confirmed knowledge。

也就是说，文章给的是可行性，spec-first 需要补上证据合同。

### 4.2 文章没有完整提示词，spec-first 不能依赖“万能提示词”

本次抓取中，文章最关键的提示词截图没有完整解析，只看到“角色核心身份”等片段。因此不能照抄提示词。

但可以抽象出提示词应包含的约束：

```text
角色：资深项目架构分析师 / 业务规则逆向分析师
目标：从代码提取项目规范候选，不虚构
步骤：结构扫描 → 入口识别 → 流程归并 → 概念提取 → 规则提取 → 图表生成 → 风险标注
输出：Markdown + PlantUML + 索引
证据：每个结论附源码路径、类名、方法名或配置路径
纪律：无法确认则标注待确认，不得编造
```

这比依赖某个 IDE 的一次性提示词更适合 spec-first。

### 4.3 文章强调文档生成，spec-first 更应强调“规范差异检测”

从代码生成文档只是第一步。对 spec-first 更有价值的是把反向生成的规范与原始 Spec 对比：

| 对比项 | 可发现的问题 |
| --- | --- |
| 原 Spec 有、代码没有 | 未实现、遗漏、回归 |
| 代码有、原 Spec 没有 | 偷偷加功能、隐式规则、历史遗留 |
| 两者都有但描述不同 | 语义偏差、边界条件不一致 |
| 代码规则更细 | Spec 粗糙，需要补充 |
| Spec 规则更严格 | 代码实现不完整 |

因此，code-to-spec 的终点不是 guide，而是 **Spec-Code Drift Review**。

### 4.4 文章使用 PlantUML，spec-first 应保留文本化图表但避免图表过载

PlantUML 的优点是可版本化、可 diff、可嵌入 Markdown，非常适合规范体系。但图表也有风险：

- 图太大后不可读；
- AI 可能生成不存在的关系；
- 类图容易退化成代码清单；
- 流程图如果不标证据，会变成想象流程。

建议在 spec-first 中只生成三类图：

1. 核心业务流程图；
2. 模块 / 上下文边界图；
3. 关键类 / 策略 / 状态机图。

每张图都应附“证据表”：节点、边、来源文件、置信度。

## 5. 知识链收束

### 5.1 对“渐进式 Spec”的补强

已有 `2026 年 AI 编码的“渐进式 Spec”实战指南` 提到三条铁律：

```text
No Spec, No Code
Spec is Truth
Reverse Sync
```

这篇 Trae 文章提供了 Reverse Sync 的落地补充：当历史代码或实现偏离文档时，可以让 AI 从代码反向提取规范候选，再由人审查并修订 Spec。

但要注意：反向提取出来的是 candidate，不是自动成为 truth。只有经过源码证据校验和人工确认，才能进入 Spec。

### 5.2 对 Knowledge Wiki 的补强

Trae 文章证明 AI IDE 能生成架构、术语、数据模型、业务规则和最佳实践等文档；Knowledge Wiki 则提醒不能沉淀所有代码细节。

两者合起来给出原则：

```text
AI 可以从代码中提取很多东西，
但长期知识库只沉淀稳定、可复用、能降低后续上下文成本的部分。
```

### 5.3 对 CodeGraph / Graphify 链路的补强

前一篇 CodeGraph 文章强调用代码图谱降低探索成本。本篇 Trae 文章强调把代码理解结果文档化。二者可以形成分工：

- CodeGraph：战术索引，快速定位调用链和影响面；
- Graphify / project-graph：宏观项目图和模块边界；
- Trae 式 code-to-spec：把分析结果转成术语、规则、流程、模型和 guide；
- spec-first：负责证据合同、Review、promotion 和后续一致性维护。

### 5.4 推荐的 spec-first code-to-spec 子流程

建议将“从代码中提取规范”固化为一个可审查流程：

```text
Phase 0：确定范围
  - 目标仓库 / 目标模块 / 目标业务域
  - 排除目录 / 生成目录 / 证据要求

Phase 1：结构扫描
  - 目录、入口、配置、测试、文档
  - 产出 project surface inventory

Phase 2：流程与概念穷举
  - 入口流程、消息流、核心服务、实体、规则类
  - 所有候选必须保留、合并或排除

Phase 3：规范候选生成
  - 术语表、数据模型、业务规则、流程图、架构图
  - 每条结论附 source refs

Phase 4：Review 与漂移检测
  - 与现有 Spec / README / wiki 对照
  - 标注一致、缺失、冲突、待确认

Phase 5：Promotion
  - confirmed 内容进入 Spec / Knowledge Wiki
  - 不确定内容进入 pending-confirmations
  - 临时过程产物归档或删除
```

## 6. 分级建议

### P0：立即采纳的规则

1. **从代码提取规范必须带 source refs**
   每条业务规则、流程、术语、数据模型都要标注文件路径、类名、方法名、配置路径或测试路径。没有出处的内容只能标为“待确认”。

2. **业务规则是最高风险输出，必须人工校验**
   文章作者已经提醒 AI 可能生成项目中不存在的规则。spec-first 中，业务规则文档不得无审查直接 promotion。

3. **区分 reverse spec candidate 与 confirmed spec**
   AI 从代码提取的规范是候选，不是事实源。confirmed spec 必须经过 Review。

4. **不要把全量代码翻译成文档**
   只提炼稳定概念、关键流程、关键规则、导航入口和复用经验。逐行实现细节回到代码本身。

### P1：建议落地的能力

1. **新增 code-to-spec / reverse-spec 工作流**
   用于历史项目初始化、文档缺失项目补规范、重构前建立理解基线。

2. **建立 reverse spec 输出模板**
   建议包含：项目概述、术语表、架构图、流程图、数据模型、业务规则、证据索引、待确认事项、与现有 Spec 差异。

3. **引入 Spec-Code Drift Review**
   将反向生成的规范与现有 Spec 对比，产出：缺失、冲突、隐式规则、待确认、建议修订。

4. **图表使用 PlantUML / Mermaid 等文本格式**
   图表可版本管理，但必须附证据表，避免漂亮但不可审计。

### P2：后续增强

1. **结合 CodeGraph / Graphify 提升提取质量**
   CodeGraph 用于调用链和影响面，Graphify 用于模块边界和项目图，再由 code-to-spec 生成规范候选。

2. **将 pending-confirmations 作为一等产物**
   无法确认的业务规则、术语和流程不要混入正文，应集中进入待确认清单。

3. **加入覆盖率指标**
   例如：多少入口已覆盖、多少流程已归并、多少业务规则有 source refs、多少待确认项未处理。

4. **支持双向同步节奏**
   开发前 Spec 指导实现；开发后 Code-to-Spec 校验偏差；偏差先修 Spec，再修代码或确认代码为新事实。

### 总结判断

这篇文章对 spec-first 的最大启发是：**规范不只能在开发前写，也可以从历史代码中反向提取；但反向提取出来的规范必须经过证据化和审查，不能直接等同于真相。**

一句话收束：

> 从代码中提取规范，不是让 AI 把代码讲一遍，而是让 AI 提炼可复用、可审查、可回源的规则资产；Spec-first 要吸收的是这条反向证据链，而不是一次性生成文档的表演。
