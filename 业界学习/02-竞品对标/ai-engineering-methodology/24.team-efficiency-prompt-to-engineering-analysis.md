## 1. 项目概述

### 核心主张

本文提出一组层层递进的核心论断：

> **AI 不是进来接管流程的，它是进来降低熵、把人的精力抬到决策层的。**

支撑这一主张的三根支柱：

| 支柱 | 内涵 |
|---|---|
| **工具不重要，工作流才重要** | Skill / MCP / Command 都是零件，零件背后是工程化思维——这才是终极护城河 |
| **信息每经过一次人，就损耗一次** | 传统研发流程最大的熵是上下文丢失，AI 的价值是让上下文结构化、可机读、可跨阶段消费 |
| **你不是在跟 AI 比谁写得快** | 你是在设计一套系统，让 AI 的聪明被你的纪律放大，而不是被你的混乱放大 |

### 三代 AI 协作界面演进

```
第一代 Web Chat（ChatGPT/Gemini/DeepSeek）
  └── 核心：Prompt
  └── 边界：单轮对话，无工具调用

第二代 Agent（Claude Code CLI / Cursor）  ← 本文聚焦
  └── 核心：Skills / MCP / Command
  └── 边界：可调用工具，进入文件系统
  └── 定位：当下投入产出比最高的一代

第三代 Gateway（OpenClaw / Hermes）
  └── 核心：多 Agent 协同
  └── 边界：安全 + 复杂任务质量待解
```

## 2. 架构分析

### 2.1 七概念关系图谱

7 个概念在 Agent 执行循环（Perception → Reasoning → Action → Observation → Refining）中的定位：

```
                    ┌──────────────────────────────────────┐
                    │          Agent 执行循环               │
                    │                                      │
    Prompt ────────►│  Perception ──► Reasoning            │
                    │       │              │               │
    Rules ─────────►│       │              ▼               │
                    │       │          Action ◄── Skill    │
                    │       │              │     (自动匹配)  │
    Command ────────►│       │              │               │
    (手动触发)       │       │              ▼               │
                    │       │         Observation          │
                    │       │              │               │
    MCP ◄───────────┼───────┼──────────────┘               │
    (外部感知/操作)   │       │                              │
                    │       ▼                              │
    Hook ──────────►│  Refining（强制执行逻辑）              │
                    └──────────────────────────────────────┘
```

| 概念 | 在循环中的定位 | 类比 |
|---|---|---|
| **Prompt** | 输入层——"你说的话" | 对讲机 |
| **Agent** | 执行体——"说话之外还能动手的家伙" | 操作员 |
| **Rules** | 约束层——在每次 Action 前校验合规 | 法律条文 |
| **Command** | 触发器——你手动触发的流程 | 遥控器按钮 |
| **Skill** | 能力层——AI 按需自动调用的方法论 | 肌肉记忆 |
| **MCP** | 感知/操作层——让 AI 看到/改到外部世界 | 感官 + 手脚 |
| **Hook** | 强制点——在某一步强制执行逻辑，不给 AI 犹豫空间 | 断路器 |

**三个常被混淆的区别**：

1. **Prompt vs Agent**：Prompt 是输入，Agent 是执行体
2. **Skill vs Command**：Skill 是 AI 自动匹配的"肌肉记忆"，Command 是人手动触发的"按钮"
3. **MCP vs Hook**：MCP 是"让 AI 能感知/操作外部世界"，Hook 是"在某一步强制执行逻辑"

### 2.2 四阶段 × 四角色新协作流程

```

├──────────────────────┤    ├──────────────────────┤
│      🔴 人工闸门     │    │ 测试 + AI            │
│ 产品拍板业务取舍     │    │ AI 生成用例矩阵      │
│ 需求评审            │    │ 产出：3 类用例        │
└──────────────────────┘    ├──────────────────────┤
                            │      🔴 人工闸门     │
                            │ 架构决策 + 用例优先级 │
测试期                      │ 评审                │
┌──────────────────────┐    └──────────────────────┘
│ 研发 + AI            │
│ AI 提测文档 + 冒烟   │    发布期

├──────────────────────┤    │ 研发 + AI            │
│ 测试 + AI            │    │ AI changelog/上线文档 │
│ AI 黑盒巡检/性能基线 │    │ 产出：发布包 + 文档   │

│ 上线风险评估        │    ├──────────────────────┤
```

**三个关键变化**（对比传统流程）：

1. 每阶段 AI 与角色**并排**而非替代
2. 每阶段保留 🔴 人工闸门——评审/架构/探索测试/授权回滚
3. 跨阶段交付物结构化——PRD → 用户故事 → 测试矩阵 → 缺陷单可双向追溯

## 3. 逐维度对照矩阵（以 spec-first 方法论为基准）

### 3.1 需求澄清

| 维度 | spec-first 方法论 | 本文方案 | 差异评价 |
|---|---|---|---|
| **需求挖掘** | 用户故事 + 验收标准 | `/office-hours` Startup 模式（6 追问）+ Builder 模式（设计思维头脑风暴） | 本文更结构化：6 个追问问题覆盖 demand reality / status quo / desperate specificity / narrowest wedge / observation / future-fit，比被动接收需求更主动 |
| **评审机制** | PRD Review 会议 | `/plan-ceo-review` 四种模式（SCOPE EXPANSION / SELECTIVE / HOLD / REDUCTION） | 本文多了一层战略视角的自动模式判定，但缺少多角色同步评审的协作机制 |
| **PRD 结构** | 8 大章节（背景、目标、用户故事、功能范围、非功能需求、验收标准、风险、附录） | PRD 模板 6 项（用户故事、边界场景、异常流程、验收标准、非目标声明、可点击原型） | 本文更强调"非目标声明"和"可点击原型"，前者防止范围蔓延，后者让需求可感知 |
| **边界穷举** | 依赖产品经理经验 | AI 穷举边界场景（并发/空值/极限/权限/跨时区跨月） | 本文显著提升覆盖度——"AI 2 分钟穷举的边界，比人坐一下午还多" |
| **需求可追溯** | 用户故事 → 验收标准 | 用户故事 ID ↔ 验收标准 ↔ 原型页面标注 | 本文新增原型页面对应用户故事 ID 的标注，形成三层追溯 |

### 3.2 方案设计

| 维度 | spec-first 方法论 | 本文方案 | 差异评价 |
|---|---|---|---|
| **方案评审** | Plan Review | `/plan-eng-review`（锁定架构、数据流、图表、边界、测试覆盖、性能） | 本文将架构锁定作为一个显式闸门，而非可选步骤 |
| **决策记录** | 技术方案文档 | ADR（Architecture Decision Record）形式记录关键决策 + 被放弃方案的权衡 | 本文更强调"为什么放弃"的显式记录，利于二期上下文消费 |
| **契约定义** | API 文档 | OpenAPI / JSON schema / TypeScript 类型导出（AI 可读契约） | 本文新增 AI 可读性要求，契约不仅是给人看的 |
| **风格对齐** | 代码规范文档 | `CLAUDE.md` / `AGENTS.md` AI 工作手册 + `examples/` 范例代码 | 本文将规范从"人读"升级到"人+AI 共读"，范例代码作为 AI 模仿基准 |
| **设计还原** | 设计稿 → 前端代码 | Figma MCP 自动读取节点 + 设计 token → 生成组件 | 本文打通设计到代码的 MCP 通道 |

### 3.3 实现质量

| 维度 | spec-first 方法论 | 本文方案 | 差异评价 |
|---|---|---|---|
| **开发方法** | 按需 TDD | TDD 作为强制 Skill（`test-driven-development`），先写测试再写实现 | 本文将 TDD 从方法论提升为 Agent 自动执行的 Skill |
| **代码审查** | Code Review（人工） | `/codex challenge`（多模型对抗）+ `/review`（预合并结构化扫描）双层 | 本文新增"多模型对抗"——不同模型盲区不同，交叉火力逼出真问题 |
| **审查维度** | 逻辑 + 风格 | 结构化扫描：SQL 注入/安全边界/LLM 信任边界/条件副作用/隐藏状态变更 | 本文新增 LLM 信任边界、条件副作用等 AI 时代特有的审查维度 |

### 3.4 验收验证

## 4. 关键差异分析

### 4.1 熵增视角 vs 文件交付视角

spec-first 方法论以**文件交付**为核心关注点——PRD、技术方案、测试报告等各阶段产出物。本文引入了一个更底层的视角：**熵增**。

> "信息每经过一次人，就损耗一次。一次完整流程少说过七八手。"

这揭示了文件交付模型的根本局限：即使每个阶段产出了合格的文件，信息在跨阶段传递时仍然在损耗——PRD 的"应该支持批量操作"被不同研发理解出三种实现，不是因为 PRD 写得不好，而是因为文字本身有歧义空间。

本文的解决方案比 spec-first 更进一步：

| spec-first 思路 | 本文思路 | 本质差异 |
|---|---|---|
| 把文件写得更详细、更结构化 | 让 AI 消费文件、穷举边界、暴露歧义 | spec-first 降低歧义，本文让 AI 检测歧义 |
| 各阶段文件归档到 docs/ | Context Pack 让 AI 每次自动消费全部上下文 | spec-first 是人读归档，本文是人+AI 共读 |
| 二期靠人回忆/查文档 | AI 直接消费一期 Context Pack + 历史决策 | 本文消除了"接手的人只能从头再来"的困境 |

### 4.2 Context Pack 的设计哲学

> "一次性建立，持续复利。"

Context Pack 对 spec-first `docs/` 的启发：

| 维度 | spec-first docs/ | Context Pack | 启发 |
|---|---|---|---|
| **消费对象** | 人 | 人 + AI | 每个文档都要同时考虑 AI 能否解析和消费 |
| **组织原则** | 按类型分类（需求/设计/测试） | 按 AI 消费路径组织（入口 → 领域 → 决策 → 模板） | 不只是"把文档放对位置"，而是"让 AI 能找到并按正确顺序消费" |
| **更新机制** | 阶段性归档 | 每阶段即时固化 → AI 下一阶段自动读取 | 文档不是终点，是下一阶段的起点 |
| **最小集合** | 无明确最小集 | 16 项必选（AI 工作手册 + 术语表 + 规则 + 模板 + 决策 + 契约 + 范例） | 给出了明确的"团队 AI 基础设施"清单 |

### 4.3 gstack 生命周期命令链对 spec-first 的补全

spec-first 工作流覆盖了需求 → 设计 → 实现 → 验证四个阶段，但对**发布后环节**缺少自动化的工具链覆盖。gstack 的 Command 链条恰好补全了这一缺口：

```

  需求 ──► 设计 ──► 实现 ──► 验证 ──► [空白]

gstack 补全范围：
                                      ┌──────────────────────────────────┐
                                      │ /ship → /land-and-deploy          │
                                      │ /canary（金丝雀监控）             │
                                      │ /retro（周度复盘）                │
                                      │ /health（质量仪表盘）             │
                                      │ /document-release（文档同步）     │
                                      │ /investigate（根因调查）          │

```

此外，gstack 还对 spec-first 的前四个阶段做了工具化增强：

| spec-first 阶段 | gstack 增强 |
|---|---|
| 需求 | `/office-hours` 结构化需求挖掘（6 追问） |

### 4.4 元规则的启发

> "每个交付物都要问自己——下一个环节的人和下一个环节的 AI，能不能不问我直接消费它？"

这条元规则是整个方法论中**最具穿透力**的一条。它在 spec-first 的"文件完整性"基础上增加了一个新维度：

| 传统标准 | 元规则标准 |
|---|---|
| 交付物是否完整 | 交付物是否可被**无人工介入**消费 |
| 文档是否清晰 | 文档是否可被**AI 解析和执行** |
| 信息是否准确 | 信息是否**自包含**（不依赖口头补充） |

这意味着每个角色的产出物质量要同时接受**人类消费者**和**AI 消费者**的双重检验。一条 PRD 不仅要让研发看得懂，还要让 AI 能从中提取验收标准生成测试用例。

### 4.5 人工闸门设计

四个 🔴 人工闸门与 spec-first 人工确认点的映射：

| 本文闸门 | 定位 | spec-first 对应点 | 差异 |
|---|---|---|---|
| 🔴 需求闸门 | 产品拍板业务取舍 + 需求评审 | PRD Review | 本文明确"业务取舍"是 AI 不可替代的——老板要 A、销售想 B、客户要 C，AI 给不出"这次就不做 B" |
| 🔴 研发闸门 | 架构决策 + 用例优先级 + 评审 | 技术方案评审 | 本文新增"用例优先级"作为闸门内容——不是所有用例都需要自动化 |
| 🔴 测试闸门 | 探索式测试 + 上线风险评估 | 测试报告 Review | 本文保留探索式测试为人的专属领域——"连点三次会不会崩"靠直觉 |
| 🔴 发布闸门 | 发布授权 + 回滚决策 | 上线审批 | 本文新增"回滚决策"作为闸门——故障时先回滚还是先加机器，结合业务重要性人判 |

### 4.6 多模型对抗 Review 的创新

`/codex challenge` 的设计思想是本文最具原创性的工程实践之一：

> "不同模型盲区不一样，交叉火力逼出真问题。"

这在 spec-first 的 Code Review 环节引入了两个新维度：

1. **多模型视角**：Claude 和 Codex 各自盲区不同，放在一起互相挑刺，覆盖盲区交集
2. **对抗模式**：不是被动审查，而是主动尝试"打破"代码——仿照安全领域的红蓝对抗思路

这一实践有两个前提条件：① 团队已有能力接入多个模型；② 代码有足够清晰的规范让模型判断"对错"。前者是工程问题，后者是规范问题——两者恰好是 Context Pack（`CLAUDE.md` + `examples/`）的前置产出。
