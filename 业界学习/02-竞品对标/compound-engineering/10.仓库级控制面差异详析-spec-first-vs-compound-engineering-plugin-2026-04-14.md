# 仓库级控制面差异详析：spec-first vs compound-engineering-plugin（2026-04-14）

> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 结论先行

先明确一个分析前提：

- `spec-first` 当前产品边界只覆盖 **Claude** 与 **Codex**
- 不以扩展到更多宿主平台为目标

“仓库级控制面文件数”这一项之所以明显不同，不是因为某一边少同步了几个文件，而是因为**两边控制面的产品职责已经发生结构性分叉**。

- `compound-engineering-plugin` 的仓库级控制面，本质上是一个 **Claude plugin -> 多宿主平台** 的转换、安装、同步、发布平台。
- `spec-first` 的仓库级控制面，本质上是一个 **面向 Claude + Codex 的本地项目初始化 / runtime 资产托管 / 语言治理 / 图谱分析（CRG）** 的工程工作流 CLI。

因此：

- 上游的文件数主要堆在 `src/sync`、`src/targets`、`src/converters`，因为它要对接多个外部宿主。
- 当前仓库的文件数主要堆在 `src/crg` 与 `src/cli`，因为它把“安装工作流资产”之外的图谱构建、变更分析、评审上下文生成、语言治理、managed-state 清理都内建到了 CLI。

换句话说，**文件数差异反映的是架构职责差异，不是简单的同步缺漏。**

## 为什么文件数会不同

### 一句话解释

上游把复杂度花在“**跨平台分发**”，当前仓库把复杂度花在“**本地工程运行时 + 图谱能力**”。

### 代码级解释

### 上游的复杂度来源

从代码入口可以直接看出，上游控制面围绕三个动作组织：

1. `convert`
   - 入口：`src/commands/convert.ts`
   - 作用：把 Claude plugin 转成 `opencode`、`codex`、`droid`、`pi`、`copilot`、`gemini`、`kiro`、`windsurf`、`openclaw`、`qwen`
2. `install`
   - 入口：`src/commands/install.ts`
   - 作用：解析本地 / bundled / GitHub plugin 源，再执行目标平台转换与落盘
3. `sync`
   - 入口：`src/commands/sync.ts`
   - 作用：把 `~/.claude` 的 home config 同步到多个 provider 配置目录

这会自然拉出三层实现：

- `src/converters/*`
  - 负责 Claude 语义模型 -> 目标宿主 bundle 的内容转换
- `src/targets/*`
  - 负责目标宿主 bundle 如何写入磁盘、如何 merge 配置、如何处理 managed block
- `src/sync/*`
  - 负责 Claude home config 如何同步成各平台的实际用户目录结构

所以，上游 44 个文件里，相当大比例不是“工作流业务能力”，而是“多宿主平台适配层”。

### 当前仓库的复杂度来源

`spec-first` 的入口则明显不同：

- `bin/spec-first.js`
  - 把 CLI 分成两条主线：
    - `src/cli/index.js`：`init / doctor / clean`
    - `src/crg/cli/router.js`：`crg` 图谱子命令路由

这意味着当前仓库的控制面同时承担了两件事：

1. **workflow runtime 安装器**
   - `src/cli/index.js`
   - `src/cli/commands/init.js`
   - `src/cli/plugin.js`
   - `src/cli/adapters/*`
   - `src/cli/state.js`
   - `src/cli/lang-policy.js`
2. **CRG 图谱分析引擎**
   - `src/crg/parser.js`
   - `src/crg/graph.js`
   - `src/crg/flows.js`
   - `src/crg/changes.js`
   - `src/crg/cli/build.js`
   - `src/crg/commands/review-context.js`
   - 以及其余 20+ 个分析/查询命令

所以，`spec-first` 比上游多出来的文件，不是多在“跨平台同步”，而是多在“图谱构建、变更感知、评审上下文生成、治理注入、状态托管”。

## 顶层架构差异

| 维度 | compound-engineering-plugin | spec-first | 代码证据 |
|---|---|---|---|
| 核心定位 | Claude plugin 转换/同步平台 | 面向 Claude + Codex 的工程工作流 CLI + runtime 安装器 + CRG 图谱引擎 | 上游：`src/commands/convert.ts`, `src/commands/install.ts`, `src/commands/sync.ts`；当前：`bin/spec-first.js`, `src/cli/index.js`, `src/crg/cli/router.js` |
| 主控制面焦点 | 多目标平台输出 | 项目内 runtime 生成与检查 | 上游：`src/targets/*`, `src/converters/*`, `src/sync/*`；当前：`src/cli/plugin.js`, `src/cli/commands/init.js`, `src/cli/commands/doctor.js` |
| 配置写入模型 | 按目标宿主分别 merge / write | 按 `claude` / `codex` adapter 安装统一资产 | 上游：`src/targets/index.ts`；当前：`src/cli/adapters/base.js`, `src/cli/adapters/*.js` |
| 状态管理 | 面向目标平台输出目录 | 面向项目内 managed state 与资产清理 | 当前：`src/cli/state.js` |
| 图谱 / 静态分析 | 无等价控制面 | 有完整 CRG 子系统 | 当前：`src/crg/**/*.js` |
| 发布控制面 | 有 release 脚本 | 无对等 release 子系统 | 上游：`scripts/release/*.ts` |

## 关键结构差异

### 1. 上游是“多目标宿主平台”，当前是“单产品 CLI + 双平台 runtime”

上游的核心抽象是：

- `Claude plugin`
- `converter`
- `target writer`
- `sync target`

当前仓库的核心抽象是：

- `bundled workflow assets`
- `platform adapter`
- `managed state`
- `developer/lang governance`
- `CRG graph`

这两种抽象不是同构关系。

### 2. 上游解决“输出到哪里”，当前解决“项目里如何安装、治理、检查”

上游最重要的问题是：

- 如何把同一个 Claude plugin 输出到 `codex / gemini / qwen / windsurf / ...`
- 如何 merge 各平台自己的配置文件
- 如何把 `~/.claude` 的 home config 同步出去

当前仓库最重要的问题是：

- 如何把 `spec-first` 自带 commands/skills/agents 安装到当前项目
- 如何记录哪些文件是自己托管的
- 如何在升级时清理过期资产
- 如何把语言治理写回 `CLAUDE.md / AGENTS.md`
- 如何构建代码图，产出 review-context / impact / flow / search 等分析能力

### 3. `src/crg` 是当前仓库文件数差异的决定性来源

如果只看“runtime 安装器”这一层：

- 上游：`src/commands` + `src/targets` + `src/converters` + `src/sync`
- 当前：`src/cli` + `bin`

虽然职责仍不同，但差异还不至于那么大。

真正把当前仓库文件数进一步拉开的，是完整的 `src/crg`：

- `src/crg/cli`：7
- `src/crg/commands`：13
- `src/crg` 核心库：13
- 合计：33

而上游没有任何等价子系统。

### 4. `version-reminder.js` 不是 `ce-update` 的等价替身

这是一个容易误判的点。

- `ce-update`
  - 检查 `compound-engineering` 插件版本
  - 依赖 Claude Code plugin harness cache 布局
  - 能清理 stale marketplace/cache
- `src/cli/version-reminder.js`
  - 做 npm 包版本比较与提醒
  - 不理解 Claude plugin cache
  - 不执行 plugin cache 修复

所以这里不能因为两边都“和版本有关”就判成同构。

### 5. `spec-first` 的 adapter 不是上游 `target` 的直接镜像

两者都带有“平台”概念，但作用完全不同：

- 上游 `targets/*`
  - 处理的是“目标宿主格式输出”
  - 面向 `codex / gemini / qwen / windsurf / ...`
- 当前 `adapters/*`
  - 处理的是“当前项目内 runtime 布局与文本改写”
  - 只面向 `claude / codex`

所以 `adapter` 更接近“项目安装层抽象”，不是“外部平台 bundle writer 抽象”。

## 可建立的弱映射

下面这些关系可以帮助理解两边的控制面，但都不是严格 1:1：

| 上游控制面 | 当前控制面 | 关系类型 | 说明 |
|---|---|---|---|
| `src/commands/sync.ts` + `src/sync/registry.ts` | `src/cli/adapters/*.js` | 弱近邻 | 都有“平台分派”概念，但一个是 home config 同步，一个是项目 runtime 安装 |
| `src/targets/index.ts` | `src/cli/adapters/base.js` | 弱近邻 | 都是平台抽象入口，但输出模型不同 |
| `src/commands/install.ts` | `src/cli/commands/init.js` | 弱近邻 | 都会把能力安装到本地，但上游是跨宿主 plugin install，当前是项目 runtime init |
| `src/commands/convert.ts` + `src/converters/*` | `src/cli/plugin.js` + `src/cli/adapters/codex.js` | 弱近邻 | 都涉及内容改写，但上游是 bundle 转换，当前是 runtime 文本改写 |
| `scripts/release/*.ts` | `src/cli/changelog.js` | 很弱近邻 | 都和版本/发布边缘流程有关，但当前没有 release 子系统 |
| 无 | `src/crg/**/*.js` | 当前独有 | 上游不存在图谱构建、影响分析、评审上下文生成控制面 |

## 最终判断

“仓库级控制面文件数”的差异，不能解读为“spec-first 比上游少同步了若干控制面文件”，也不能解读为“spec-first 只是上游的轻量变体”。

更准确的代码级判断是：

1. `compound-engineering-plugin` 的仓库级控制面是 **多宿主 plugin 转换/同步/发布平台**。
2. `spec-first` 的仓库级控制面是 **项目 runtime 安装器 + 治理写入器 + CRG 图谱分析引擎**。
3. 两边在 `skill/agent` 层可以高同步，但在仓库级控制面上已经 **非同构**。
4. 因此，“仓库级控制面文件数”这一项不能用于证明“两个仓库已经完全同步一致”，它恰恰证明了两边已经走向不同产品形态。

## 建议的后续动作

如果目标是继续做“功能同步”，建议不要再用“文件数接近”作为目标，而应该改成以下三类目标分别审视：

1. **核心 workflow 一致性**
   - 继续对齐 `skill / agent / prompt contract`
2. **控制面能力补齐**
   - 明确哪些上游控制面值得引入，例如 `ce-update`、`ce-debug`、`ce-sessions`
3. **差异保留清单**
   - 明确保留 `CRG`、`lang-policy`、`managed state`、`spec-graph-bootstrap` 这类当前仓库独有基础设施
