# 仓库级控制面同步策略：spec-first vs compound-engineering-plugin（2026-04-14）

> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 结论

先明确一个产品边界前提：

- `spec-first` **不需要扩展多平台能力**
- `spec-first` 的宿主范围只维持 **Claude** 与 **Codex**

针对“仓库级控制面”这一层，不建议把 `spec-first` 的目标定义成“与 `compound-engineering-plugin` 做 1:1 文件级同步”。

更合理的策略是：

1. **补齐会直接提升当前产品闭环能力的上游控制面缺口**
2. **明确保留 `spec-first` 已经长出来且上游没有的基础设施**
3. **拒绝把上游的多宿主插件平台整套搬进来**

也就是说，后续同步应采用：

- **能力选择性同步**
- **产品形态有意识分叉**
- **核心工作流继续高同步，仓库级控制面保持非同构但可解释**

## 同步目标定义

先把目标说清楚，否则后面会反复误判。

### 不建议采用的目标

- “目录数量接近”
- “文件数量接近”
- “把上游控制面逐个搬过来”
- “让 `spec-first` 也变成多目标 plugin converter”
- “把 Claude / Codex 之外的宿主平台继续纳入产品范围”

### 建议采用的目标

- `spec-first` 是否补齐了对当前用户有价值的缺失控制面
- `spec-first` 是否保留了自己已经形成优势的控制面
- `spec-first` 的控制面边界是否清晰、稳定、可维护
- `skill/agent` 核心链路是否继续跟上游保持高同步
- `Claude + Codex` 这两个宿主的体验是否足够完整

## 决策原则

后续所有“要不要同步”的判断，建议统一用下面四条原则。

### 原则 1：优先同步闭环能力，不优先同步平台铺设能力

如果某个上游能力能直接补足当前产品闭环，例如：

- 更新检查 + cache 修复
- bug 调查闭环
- session 历史检索

这类应优先考虑。

如果某个上游能力主要服务于“把 Claude plugin 输出到更多宿主平台”，而当前仓库产品边界已经明确只支持 `Claude + Codex`，那么这类能力应直接排除，不进入同步范围。

### 原则 2：优先同步用户可感知能力，不优先同步中间转换层

用户可直接感知的是：

- 有没有明确 update / repair 入口
- 有没有 session / slack / debug 这类能力入口
- init / doctor / clean / graph 能不能形成闭环

用户并不直接关心：

- `claude-to-qwen.ts` 是否存在
- `targets/openclaw.ts` 是否也实现
- 是否支持 10 个宿主 bundle writer

### 原则 3：保留 `spec-first` 已形成优势的控制面

这些不是“偏离上游”，而是当前仓库已经形成的产品壁垒：

- `src/crg/**/*.js`
- `src/cli/state.js`
- `src/cli/lang-policy.js`
- `src/cli/commands/init.js`
- `src/cli/commands/doctor.js`

这些不应该因为上游没有，就被视为“待删除差异”。

### 原则 4：把“工作流同步”和“控制面同步”分开治理

当前最大的风险，是把两类问题混在一起：

- `skill/agent/prompt` 同步
- 仓库级控制面同步

前者可以追求高同步。

后者不应追求同构，只应追求：

- 战略上对齐
- 能力上补洞
- 架构上可解释

## 分层同步决策

## A. 必须补齐的控制面能力

这些能力一旦不补，用户会持续感知到“当前仓库在关键操作闭环上弱于上游”。

| 能力 | 当前状态 | 建议 | 优先级 | 理由 | 上游代码证据 | 当前建议落点 |
|---|---|---|---|---|---|---|
| 更新检查 + 缓存修复 | 只有版本提醒，无 cache repair | 补齐显式 update / repair 入口 | P0 | 这是最小成本、最高感知价值的控制面缺口 | `plugins/compound-engineering/skills/ce-update/SKILL.md` | 新增 `skills/spec-update/` 或 `skills/update/`，并在 [`src/cli/version-reminder.js`](/Users/kuang/xiaobu/spec-first/src/cli/version-reminder.js) / [`src/cli/index.js`](/Users/kuang/xiaobu/spec-first/src/cli/index.js) 提供触发入口 |
| 更完整的 debug-and-fix 闭环 | 当前偏 issue reproduction | 补齐从复现 -> 根因 -> 修复建议的工作流 | P0 | 用户会把“能否系统排障”视为核心能力，不只是复现 issue | `plugins/compound-engineering/skills/ce-debug/` | 以 [`skills/reproduce-bug/`](/Users/kuang/xiaobu/spec-first/skills/reproduce-bug/SKILL.md) 为基础，新增 `spec-debug` 或升级 `reproduce-bug` 为完整 debug workflow |
| 会话历史检索控制面 | 当前无 | 补齐 session 检索与追问入口 | P1 | 对长期复杂任务非常关键，能直接提升多轮调试、知识追踪与复盘效率 | `plugins/compound-engineering/skills/ce-sessions/`、`agents/research/session-historian.md` | 新增 `skills/spec-sessions/`，必要时引入独立 agent；不要并入 CRG |

### 为什么这三项优先级最高

因为它们都满足三个条件：

1. 用户可直接感知
2. 不要求把上游平台层整套搬进来
3. 能和当前 `spec-first` 主产品形态兼容

其中：

- `update` 解决的是“升级 / 修复闭环”
- `debug` 解决的是“排障闭环”
- `sessions` 解决的是“历史上下文闭环”

这三者都属于控制面“入口级能力”。

## B. 建议补齐，但不是第一阶段

这些能力有价值，但不是当前最先要补的。

| 能力 | 当前状态 | 建议 | 优先级 | 理由 | 上游代码证据 | 当前建议落点 |
|---|---|---|---|---|---|---|
| Slack 组织知识研究 | 当前无 | 作为 research 能力补入 | P2 | 很有价值，但依赖外部组织环境，不是所有用户都需要 | `plugins/compound-engineering/skills/ce-slack-research/`、`agents/research/slack-researcher.md` | 新增 `skills/spec-slack-research/` + `agents/research/slack-researcher.md` |
| Demo evidence capture | 当前有 `feature-video`，但不等于证据采集 | 补齐 screenshot / GIF / terminal demo 证据模式 | P2 | 对 PR 交付与验证有帮助，但不属于最底层闭环 | `plugins/compound-engineering/skills/ce-demo-reel/` | 优先增强 [`skills/feature-video/`](/Users/kuang/xiaobu/spec-first/skills/feature-video/SKILL.md)，而不是新开完全平行 skill |
| 实验驱动优化闭环 | 当前无 | 作为高级能力引入，不进入核心 CLI | P3 | 有价值，但建设成本高，且适用面较窄 | `plugins/compound-engineering/skills/ce-optimize/`、相关脚本 | 新增独立 `skills/spec-optimize/`，不要污染 `src/cli` 主控制面 |

## C. 建议明确保留分叉，不做 1:1 同步

这些是最容易“为了同步而同步”的误区。建议明确写入架构结论：**不做 1:1 同步**。

| 上游能力 | 是否建议同步 | 结论 | 理由 | 代码证据 |
|---|---|---|---|---|
| 多目标 plugin convert 平台 | 否 | 明确保留分叉 | `spec-first` 的产品边界已明确只支持 `Claude + Codex`，不需要生成更多宿主 bundle | 上游：`src/commands/convert.ts`, `src/converters/*`, `src/targets/*` |
| 多目标 plugin install 平台 | 否 | 明确保留分叉 | 当前仓库不承担“面向更多宿主的 plugin 安装与格式转换”职责 | 上游：`src/commands/install.ts` |
| Claude home -> 多 provider sync 平台 | 否 | 明确保留分叉 | 当前仓库只服务 `Claude + Codex` 项目 runtime，不做多 provider home config 同步 | 上游：`src/commands/sync.ts`, `src/sync/*` |
| 发布脚本子系统 | 否 | 明确保留分叉 | 当前仓库没有必要复制上游的 plugin marketplace 发布模型 | 上游：`scripts/release/*.ts` |

### 为什么这些能力不应同步

因为一旦同步，会把当前仓库从“只聚焦 Claude + Codex 的工程工作流 CLI”拉向另一个方向：

- 从“工程工作流 CLI”变成“多宿主 plugin 平台”
- 从“项目内 runtime 管理器”变成“平台兼容层集合”

这会直接造成三类问题：

1. 控制面变重
2. 维护成本激增
3. 产品叙事模糊

## D. 必须保留的 spec-first 独有控制面

这些能力应明确标记为“当前仓库长期保留差异”，不应该因为上游没有而被弱化。

| 当前独有控制面 | 建议 | 理由 | 代码证据 |
|---|---|---|---|
| CRG 图谱引擎 | 保留并持续强化 | 这是当前仓库最明显的基础设施优势 | [`src/crg/`](/Users/kuang/xiaobu/spec-first/src/crg) |
| managed state | 保留 | 解决初始化、升级、清理过程中的可控性问题 | [`src/cli/state.js`](/Users/kuang/xiaobu/spec-first/src/cli/state.js) |
| 语言治理注入 | 保留 | 这是 `spec-first` 工作流质量控制的一部分 | [`src/cli/lang-policy.js`](/Users/kuang/xiaobu/spec-first/src/cli/lang-policy.js) |
| 项目级 `init/doctor/clean` 闭环 | 保留 | 这是当前产品最核心的 CLI 入口 | [`src/cli/commands/init.js`](/Users/kuang/xiaobu/spec-first/src/cli/commands/init.js), [`src/cli/commands/doctor.js`](/Users/kuang/xiaobu/spec-first/src/cli/commands/doctor.js), [`src/cli/commands/clean.js`](/Users/kuang/xiaobu/spec-first/src/cli/commands/clean.js) |
| bootstrap / graph-bootstrap 路线 | 保留 | 与 CRG 构成前置分析能力闭环 | `skills/spec-graph-bootstrap/`, `skills/spec-graph-bootstrap/` |

## 推荐路线图

## Phase 1：补闭环，不扩平台

目标：先把用户最容易感知的上游缺口补掉。

### 建议纳入

1. `update / repair`
2. `debug-and-fix`
3. `sessions`

### 不纳入

1. `convert/install/sync` 多宿主平台
2. `release` 子系统
3. `optimize` 实验平台
4. `Claude + Codex` 之外的宿主适配

### Phase 1 产出标准

- 用户可以在当前仓库里明确触发更新修复能力
- 用户可以从“复现问题”走到“定位根因 / 修复建议”
- 用户可以查询历史会话而不是依赖人工回忆

## Phase 2：补研究能力，不污染主 CLI

目标：把外围但高价值的研究型能力补进来，同时不把主控制面做得过重。

### 建议纳入

1. `slack-research`
2. `demo-reel` 能力增强

### 设计约束

- 优先以 skill/agent 方式引入
- 不优先在 `src/cli/index.js` 上增加新一级命令
- 不把这些能力耦合进 `init/doctor/clean`

## Phase 3：评估 optimize 是否值得独立产品化

`ce-optimize` 不应作为“同步动作”直接搬入，而应先回答两个问题：

1. `spec-first` 是否真的需要 metric-driven experiment loop
2. 这项能力是面向内部工程团队，还是面向通用用户

如果答案不明确，建议先不做。

## 具体实现建议

## 1. update / repair

### 推荐实现方式

- 新增独立 skill
- 不急着新增 CLI 一级命令
- 先复用现有版本提醒能力

### 推荐落点

- `skills/spec-update/` 或 `skills/update/`
- 复用 [`src/cli/version-reminder.js`](/Users/kuang/xiaobu/spec-first/src/cli/version-reminder.js) 的版本比较逻辑
- 如有必要，在 [`src/cli/commands/doctor.js`](/Users/kuang/xiaobu/spec-first/src/cli/commands/doctor.js) 增加“检测到缓存异常时给出 update/repair 指引”

### 不建议的做法

- 不要把它实现成上游 `ce-update` 的 literal copy
- 不要假设当前仓库也存在 Claude plugin harness cache 结构

## 2. debug-and-fix

### 推荐实现方式

- 不新起一个完全独立的仓库级命令
- 以当前 [`skills/reproduce-bug/`](/Users/kuang/xiaobu/spec-first/skills/reproduce-bug/SKILL.md) 为基础升级
- 让它从“复现”扩展到“根因分析 + 修复策略”

### 推荐结构

- `reproduce`
- `investigate`
- `isolate`
- `root-cause`
- `fix-plan`
- `verify`

### 这样做的原因

因为从用户心智看，“debug”不是另一个产品，而是“调查 bug 的完整闭环”。

## 3. sessions

### 推荐实现方式

- 作为独立 skill/agent 能力实现
- 不并入 CRG

### 原因

`CRG` 处理的是代码图与静态结构，`sessions` 处理的是代理会话历史。

两者的数据模型不同：

- CRG：代码、符号、调用关系、变更风险
- sessions：操作历史、尝试路径、失败原因、上下文链条

如果硬并，会让 CRG 边界被污染。

## 4. slack-research

### 推荐实现方式

- 保持 skill + research agent 形态
- 不放进主 CLI

### 原因

这是一个典型的外围研究能力，不属于 runtime 安装器，也不属于 graph engine。

## 不建议做的事情

下面这些看起来“同步度更高”，但实际上会伤害当前仓库：

| 动作 | 不建议原因 |
|---|---|
| 把 `src/converters/*` 全量搬进来 | 会把产品重心拉到多宿主格式兼容 |
| 把 `src/targets/*` 全量搬进来 | 会让当前仓库承担大量无关宿主配置 merge 逻辑 |
| 把 `src/sync/*` 全量搬进来 | 会把项目级 CLI 扭曲成个人 home config 平台 |
| 把 `scripts/release/*.ts` 原样移植 | 当前仓库没有同样的发布问题域 |
| 为了同步而压缩 `src/crg` | 会主动丢掉当前仓库最有价值的基础设施优势 |
| 继续新增 `Claude / Codex` 之外的宿主 adapter | 会直接破坏当前已经明确的产品边界 |

## 最终建议

后续仓库级控制面同步，应以这四句话作为执行准则：

1. **补闭环，不补平台。**
2. **补用户感知能力，不补 `Claude / Codex` 之外的宿主兼容层。**
3. **保留 CRG、治理、managed state 这些独有基础设施。**
4. **把仓库级控制面同步定义为“能力选择”，而不是“文件复制”。**

## 下一步建议

如果继续往下推进，最值得马上产出的不是代码，而是两份执行性文档：

1. **P0 控制面补齐实施计划**
   - 只覆盖 `update / debug / sessions`
2. **控制面边界原则文档**
   - 明确哪些目录永远不追求与上游同构

这样后面你再做任何同步动作，就不会重新回到“要不要把上游整套搬过来”的争论里。
