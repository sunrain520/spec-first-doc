# spec-first 相对 compound-engineering-plugin 的 Claude 运行层能力缺口分析

> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、`CRG Stage-0`、`ECC`、`compound-engineering-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 结论先行

本文只看 **Claude Code 运行层**，不看平台与生态。

`spec-first` 保留了五步闭环，但运行时控制面更轻。它和 `compound-engineering-plugin` 的主要差距，集中在：

- 官方入口是否足够完整
- 任务编排是否足够厚
- 并行依赖是否能被系统化管理
- 断点恢复和中间态持久化是否足够强
- review 之后到 compound 的收敛链是否足够长

在这个前提下，`spec-first` 相对 `compound-engineering-plugin` 的差距不是“有没有五步闭环”，而是：

**`spec-first` 保留了 Claude 侧的工作方法，但缺少更完整的运行时控制面。**

从目录规模看，两边的 `skills` 和 `agents` 数量几乎一致：

| 项目 | skills | agents |
|---|---:|---:|
| `compound-engineering-plugin` | 41 | 47 |
| `spec-first` | 41 | 47 |

所以问题不在于“技能文件少很多”，而在于**同样的能力资产，在 Claude 运行层被组织成了更轻的控制面**。

## 一、对比范围

本次只比较 Claude 运行层的四个面向：

1. **入口与流程组织**：brainstorm -> plan -> work -> review -> compound 是否完整，入口是否丰富
2. **任务编排**：任务如何拆解、串联、阻塞、推进、并行
3. **运行时稳定性**：断点续跑、恢复、持久化、长会话管理
4. **质量门禁**：review / compound 的校验深度、收敛能力、产物约束

## 二、先说已经对齐的部分

### 1. 五步闭环还在

`spec-first` 的主流程仍然是：

- `/spec:brainstorm`
- `/spec:plan`
- `/spec:work`
- `/spec:code-review`
- `/spec:compound`

这意味着它没有丢掉方法论骨架。和 `compound-engineering-plugin` 一样，它仍然承认：

- 先发散
- 再收敛
- 再执行
- 再评审
- 再沉淀

### 2. Review 和 compound 不是空壳

`spec-first` 的 `spec:code-review` / `spec:compound` 不是纯文字模板，仍然保留了：

- 多 persona review
- 置信度门禁
- diff / plan 约束
- 产物落盘
- overlap 检测
- 相关文档检索
- refresh 提示

这说明 `spec-first` 在“质量意识”上并不弱，至少不是把 review 变成一句口号。

### 3. 技能资产本身并没有明显缩水

从目录层看，`spec-first` 依然保留了：

- review agents
- spec-doc-review agents
- research agents
- design agents
- workflow agents
- browser / testing / git helpers

因此，它不是“Claude 版缩水仓库”，而是**把同一批资产放进了更轻的 Claude 运行框架**。

---

## 三、`spec-first` 真正少掉的 Claude 运行层能力

### 1. 官方入口更少，流程起点更窄

`compound-engineering-plugin` 的 Claude 主叙事更完整，除了五步闭环，还把 `ideate`、`compound-refresh` 这类前后端流程都做成了正式入口。

`spec-first` 目前的官方入口更窄，主命令基本集中在：

- brainstorm
- plan
- work
- review
- compound

**这意味着：**

- 它保留了执行链路，但发散前端和复盘后端的门面更少
- 用户在 Claude 里能直接感知到的“标准动作”更少
- 产品叙事更像一条线，而不是一套闭环体系

**影响：**

- 在需求尚不清晰时，入口支撑不够丰富
- 在已经完成一次闭环后，复盘和刷新知识的官方动作不够显眼

**判断：**

这是入口层的缺口，不是核心方法论缺口。

---

### 2. 完整任务引擎 / 工作流编排器缺失

这是最关键的运行时缺口。

`spec-first` 目前更像“命令式闭环”，而不是一个完整的任务引擎。它依赖的是：

- `task.yaml`
- 阶段脚本
- `stage-gate.sh`
- `review-judge.sh`
- 轻量化的命令编排

而 `compound-engineering-plugin` 的 Claude 流程，已经更接近一个带控制面的工作流体系：

- 流程阶段更明确
- 交接点更强
- review / compound 之间的反馈链更细
- 运行态控制更完整

**`spec-first` 缺少的不是“能不能跑”，而是“能不能稳定地管复杂任务”。**

具体来说，少的是：

- 明确的任务生命周期管理
- 更强的阶段状态机
- 任务级依赖表达
- 任务推进的显式控制点

**影响：**

- 复杂任务更依赖人工把控
- 长链路流程容易退化成脚本串联
- 状态解释能力不够强

---

### 3. 并行任务与依赖管理明显偏弱

`spec-first` 的路线图里已经明确承认：并行任务依赖管理不是一期重点。

这会直接带来一个结果：它更擅长顺序型闭环，不擅长复杂 DAG 式任务组织。

`compound-engineering-plugin` 的 Claude 流程虽然也不是重型调度系统，但它在任务推进、review 门禁、compound 收敛这几段之间，有更强的编排感。

**`spec-first` 这里少掉的能力包括：**

- 任务拆分后的依赖表达
- 并行子任务的协调
- 多分支结果的收敛
- 跨阶段重排能力

**影响：**

- 多工作流并发时容易靠经验而不是系统管控
- 任务数一多，用户必须自己记住依赖关系
- 长任务不容易做成“系统可恢复的编排”

---

### 4. 运行态持久化和断点续跑能力更弱

`compound-engineering-plugin` 在 `compound` 和相关流程里，已经把：

- auto memory scan
- related docs search
- overlap detection
- refresh recommendation

做成了流程的一部分。

`spec-first` 也吸收了类似思想，但它的运行态更轻，缺少更完整的：

- session recovery
- orphaned worktree recovery
- compaction-safe task tracking
- 更强的 disk-persisted 中间态管理

**这不是一个小细节，而是长会话可靠性的核心。**

**影响：**

- 断线、切换、重启后的恢复成本更高
- 中间状态更依赖用户记忆
- 长任务更容易变成“半系统化、半手工化”

---

### 5. Review 之后的收敛链条更短

`spec-first` 有 review，但 `compound-engineering-plugin` 的 review / compound 不是单点动作，而是更强的收敛链条。

它的特点是：

- review 不是只给结论，而是会把问题往后续动作传
- compound 不是只做总结，而是会继续检查 overlap、相关资料、知识缺口
- 流程里有更明确的“复盘 -> 收敛 -> 再使用”链路

`spec-first` 这部分保留了核心，但自动化程度更低，链条更短。

**少掉的能力：**

- review 结果到 compound 的自动接力更弱
- review 失败后的推进动作更少
- 知识刷新不够制度化

**影响：**

- review 更像检查点，不像控制面的一部分
- compound 更像文档沉淀，不像知识闭环

---

### 6. 流程控制更轻，适合简单闭环，不适合重型编排

从文档结构看，`spec-first` 更强调：

- 最小流程
- 最少入口
- Claude 内的轻量闭环

这带来的直接结果是，它保留了方法论，但把控制面压薄了。

`compound-engineering-plugin` 则更像把 Claude 运行过程中的“流程”本身做厚了：

- 哪一步何时执行
- 哪一步要先看什么产物
- 哪一步失败后如何回滚或重试
- 哪一步需要额外门禁

**`spec-first` 的短板是：**

- 流程更像指南
- 控制更像约定
- 运行更像轻量脚本

而不是一套强约束 runtime。

---

### 7. 深度角色适配不够充分

`spec-first` 虽然有 review、design、research 等角色，但它的产品表述和流程组织仍然偏通用，针对不同工作场景的运行差异化不够强。

这会表现为：

- 角色边界不够硬
- 不同工作类型的入口策略不够分化
- Claude 内部的任务分派没有更强的专门化

**影响：**

- 研发、测试、设计、分析类任务虽然都能跑，但运行体验趋同
- 复杂工作需要用户自己补“该走哪个角色”的判断

---

## 四、明确属于 `spec-first` 设计上暂不展开的能力

下面这些不是本文重点，也不是当前 Claude 运行层分析要追的东西：

| 能力 | spec-first 现状 | 备注 |
|---|---|---|
| 完整任务引擎 | 暂不实现 | 轻量闭环优先 |
| 并行任务依赖管理 | 暂不实现 | 一期不做复杂 DAG |
| 可视化看板 | 暂不实现 | 不在当前 Claude 范围 |
| 复杂自动化编排状态机 | 暂不实现 | 控制面保持轻量 |
| 多模型协同编排 | 暂不实现 | 本文不讨论 |
| 深度角色适配扩展 | 暂不展开 | 先保证主闭环稳定 |

这部分的意义是：**`spec-first` 的不少“缺口”其实是有意收缩，而不是遗漏。**

## 六、如果只问“spec-first 少了什么”

可以压缩成五条：

1. **少了更完整的 Claude 入口体系**
   - 入口更少，流程起点和复盘起点都更窄

2. **少了完整任务引擎**
   - 任务生命周期、状态机、显式控制点不够强

3. **少了并行任务依赖管理**
   - 顺序闭环可以，复杂 DAG 不强

4. **少了更强的运行态耐久性**
   - 断点续跑、恢复、中间态持久化更弱

5. **少了 review -> compound 的更长收敛链**
   - review 和 compound 都在，但自动化接力不够厚

---

## 七、补齐优先级建议

如果目标是把 `spec-first` 继续推进成一个更强的 Claude 运行框架，而不是停留在“轻量闭环”，建议优先补这些：

| 优先级 | 建议补齐项 | 原因 |
|---|---|---|
| P0 | 更完整的任务生命周期管理 | 这是运行时控制面的基础 |
| P0 | 更明确的阶段状态机 | 让流程不只是一串命令 |
| P0 | 断点续跑 / 持久化中间态 | 长任务必须具备 |
| P1 | 并行任务依赖管理 | 复杂工作会直接受益 |
| P1 | review -> compound 自动接力 | 让质量闭环更完整 |
| P2 | 扩展官方入口到前后两端 | 让 Claude 叙事更完整 |
| P2 | 更细的角色分工策略 | 提升复杂任务的可读性和可控性 |

## 八、最终判断

`spec-first` 没有丢掉 Claude 的五步闭环，也没有丢掉 review / compound 的基本质量意识。

它真正少掉的是：

**更完整的 Claude 运行时控制面。**

具体来说，就是：

- 更强的任务编排
- 更强的并行依赖管理
- 更强的断点恢复
- 更强的中间态持久化
- 更强的 review 到 compound 收敛链
- 更完整的官方入口体系

所以如果把两者都限定在 Claude 运行层来看：

- `compound-engineering-plugin` 更像一个“厚控制面”的 Claude 工作系统
- `spec-first` 更像一个“轻控制面”的 Claude 闭环框架

本文不讨论的平台与生态能力，留到单独文档再展开。
