# spec-first 相对 compound-engineering-plugin 的平台与生态能力缺口分析

> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、`CRG Stage-0`、`ECC`、`compound-engineering-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 结论先行

本文只看 **平台与生态层**，不看 Claude 运行层。

`spec-first` 是可用的 Claude Code 框架，但还不是平台化产品。它和 `compound-engineering-plugin` 的主要差距，集中在：

- 能不能跨宿主分发
- 能不能做安装 / 转换 / 同步
- 能不能做分支化安装和路径治理
- 能不能做发布治理和元数据校验
- 能不能把外部工具和 MCP/browser 这些能力封成更厚的生态入口

在这个前提下，`spec-first` 更像单宿主的本地 CLI + 项目级闭环工具包，而 `compound-engineering-plugin` 更像一个可分发、可治理的插件平台。

## 一、对比范围

本次只比较平台与生态相关能力：

1. **多宿主分发**
2. **安装 / 转换 / 同步**
3. **分支化安装与路径治理**
4. **发布治理与元数据一致性**
5. **生态入口与配套工具**

不比较的内容：

- Claude 运行层的五步闭环
- review / compound 的内部流程质量
- 任务引擎、断点恢复、并行依赖管理

这些内容已经在另一份 Claude-only 文档中单独分析。

## 二、先说两边的定位差异

### 1. `compound-engineering-plugin` 是“插件平台”

它的公开能力不是只服务 Claude Code，而是服务一整套宿主和工作流目标。

从 README 公开信息看，它具备：

- 多个核心 workflow skill
- 多个 workflow utility skill
- 多个 review / research / design agent
- 一个可用的 Context7 MCP server
- browser automation 支持
- 面向不同宿主的转换和安装路径

这说明它不是单纯的“Claude 里的一套命令”，而是一个**可以被分发、被复用、被迁移的产品层**。

### 2. `spec-first` 是“Claude Code 框架”

`spec-first` 的公开入口非常克制：

- `spec-first doctor`
- `spec-first init --claude`

README 里也明确说明：

- npm 发布模型以 `spec-first` CLI 为入口
- `/spec:*` 命令来自 `spec-first init --claude` 生成的项目级 `.claude/commands/spec/`
- `skills/` 和 `agents/` 会同步到项目级 `.claude/skills/` 和 `.claude/agents/`

这意味着它的目标不是“做一个平台”，而是“把 Claude 里的一个工作框架跑稳”。

## 三、`spec-first` 真正少掉的平台与生态能力

### 1. 多宿主分发能力缺失

`compound-engineering-plugin` 的最显著差异，是它把同一套工作资产面向多个宿主公开出来。

它公开支持的目标包括：

- OpenCode
- Codex
- Droid
- Pi
- Gemini
- Copilot
- Kiro
- OpenClaw
- Windsurf
- Qwen

这类能力的本质，不是“多几个命令”，而是：

- 能把一个工作方法打包成多个宿主可消费的形式
- 能让同一套能力跨环境复用
- 能让产品不依赖单一宿主的生命周期

`spec-first` 没有这层能力。它的能力边界基本收缩在 Claude Code。

**影响：**

- 不能把同一套能力同步到多个 agent 平台
- 不能把方法论作为跨宿主资产复用
- 不能把一个仓库自然地变成多宿主发行物

---

### 2. 安装 / 转换 / 同步链路缺失

`compound-engineering-plugin` 不只是“能装”，而是具备完整的安装和转换链路。

这一类能力通常包含：

- 选择目标宿主
- 转换生成对应目标结构
- 安装到目标侧
- 同步更新
- 保持多目标一致性

而 `spec-first` 的入口非常简单：

- `npm install -g spec-first`
- `spec-first doctor`
- `spec-first init --claude`

它没有等价的多目标转换层，也没有等价的同步层。

**影响：**

- 只能围绕 Claude Code 做本地初始化
- 没有统一的跨目标转换语义
- 不能把发布动作抽象成“目标选择 + 结构转换 + 资产同步”

---

### 3. 分支化安装与路径治理缺失

`compound-engineering-plugin` 还提供更细的安装治理能力，比如：

- branch 级安装
- cache 路径解析
- `plugin-path` 这类便于脚本和别名组合的输出

这类能力看上去“偏工具”，但实际很关键，因为它决定了：

- PR 分支能不能方便验证
- worktree 场景是否好用
- 你的安装位置是否稳定可预测

`spec-first` 没有这套治理层。

它更像“安装到当前 Claude 项目里”，而不是“围绕分支、缓存、路径做治理”。

**影响：**

- 分支测试更原始
- worktree 验证成本更高
- 迁移、回切、重装时更容易依赖人工记忆

---

### 4. 发布治理与元数据一致性缺失

`compound-engineering-plugin` 公开存在更完整的 release / validate / sync 体系。

这类体系的价值在于：

- 发布前先验证
- 发布时保持元数据一致
- 发布后保证各目标同步
- 避免一个目标更新、另一个目标落后

`spec-first` 没有对应的发布治理层。

它目前更像：

- 一个 npm CLI 项目
- 一套 Claude 项目初始化器

而不是一个“有发布管线、有一致性校验、有元数据治理”的产品平台。

**影响：**

- 没有统一的发布前检查链路
- 没有跨目标一致性约束
- 没有平台级的版本治理语义

---

### 5. 生态入口较少，外部能力封装不够厚

`compound-engineering-plugin` 的公开入口很厚：

- workflow skills
- review agents
- research agents
- design agents
- browser automation
- MCP server

这意味着它不只是一个“写命令的地方”，而是一个**可被探索、可被组合、可被扩展的能力目录**。

`spec-first` 的生态入口则明显更窄，重点仍然是：

- `/spec:*` 生成命令
- 项目级 skills 和 agents
- Claude Code 内部工作流

它没有把 browser、MCP、目标转换、发布治理一起包成同等级的生态叙事。

**影响：**

- 可发现能力更少
- 扩展路径更窄
- 外部协作接口更弱

---

### 6. 资产分发方式更单一

`spec-first` 的资产分发方式基本是：

- npm 安装
- `spec-first init --claude`
- 写入项目级 `.claude/commands/spec/`
- 同步 `skills/` 和 `agents/`

这很稳定，但也意味着分发方式单一。

`compound-engineering-plugin` 的分发方式更偏平台化：

- 面向多个宿主
- 面向多个 target
- 面向统一转换链路
- 面向更强的发布和同步控制

**影响：**

- `spec-first` 更容易被理解成一个“项目模板工具”
- `compound-engineering-plugin` 更容易被理解成一个“可扩展生态层”

## 四、能力缺口总表

| 能力域 | compound-engineering-plugin | spec-first | 差距性质 | 业务影响 |
|---|---|---|---|---|
| 多宿主分发 | 支持多个目标宿主 | 仅聚焦 Claude | 明显缺口 | 不能跨宿主复用 |
| 安装 / 转换 | 有安装和转换链路 | 只有 npm CLI + Claude init | 明显缺口 | 无法做统一发布 |
| 同步能力 | 有 sync 语义 | 没有等价层 | 明显缺口 | 多环境一致性弱 |
| 分支化安装 | 支持 branch / cache / 路径治理 | 无等价能力 | 明显缺口 | PR/worktree 验证成本高 |
| 发布治理 | 有 validate / metadata sync | 没有治理层 | 明显缺口 | 版本一致性弱 |
| 生态目录厚度 | skill / agent / MCP / browser 更完整 | 主要是 Claude 资产同步 | 中到明显缺口 | 能力发现性弱 |
| 工具化外壳 | 有多种辅助命令和平台化包装 | 入口收缩到 doctor/init | 中等缺口 | 产品面更窄 |
| 目标输出稳定性 | 有 target / writer 思维 | 输出偏项目级初始化 | 中等缺口 | 交付面更单一 |

## 五、如果只问“spec-first 少了什么”

可以压缩成六条：

1. **少了多宿主分发能力**
   - 不能把同一套能力发到多个 agent 平台

2. **少了安装 / 转换 / 同步链路**
   - 没有统一的跨目标发布语义

3. **少了分支化安装和路径治理**
   - worktree / PR 验证不够顺手

4. **少了发布治理**
   - 没有一致性校验、元数据同步、发布前后控制

5. **少了更厚的生态目录**
   - MCP、browser、workflow utility 的组合厚度不够

6. **少了平台化产品心智**
   - 更像 CLI 框架，而不是生态平台

---

## 六、补齐优先级建议

如果目标不是立刻做成完整平台，而是先把 `spec-first` 往“可扩展生态”推进，建议按这个顺序补：

| 优先级 | 建议补齐项 | 原因 |
|---|---|---|
| P0 | 多宿主分发抽象 | 这是平台化的第一层 |
| P0 | install / convert / sync 语义 | 这是跨目标复用的基础 |
| P1 | 分支化安装 / 路径治理 | 这是验证和迭代的底座 |
| P1 | 发布治理 / 元数据校验 | 没有治理，生态会失控 |
| P2 | 更完整的 target / writer 层 | 让输出不只停留在 Claude |
| P2 | MCP / browser / 外部工具封装 | 扩生态厚度和可发现性 |
| P3 | 更完整的官方入口目录 | 让产品叙事更像平台 |

---

## 七、最终判断

`spec-first` 在 Claude 运行层已经足够完整，五步闭环和质量门禁都成立。

但从平台与生态角度看，它还明显少了 `compound-engineering-plugin` 那层能力：

**把工作方法打包成可分发、可转换、可同步、可治理的平台资产。**

所以两者的定位差异可以直接概括为：

- `compound-engineering-plugin` 更像“平台”
- `spec-first` 更像“Claude 框架”

这也是为什么这份文档要单独写出来：它讨论的不是“流程做得好不好”，而是“能不能把流程产品化并持续扩展”。
