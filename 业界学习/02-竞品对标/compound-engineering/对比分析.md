# `adamfeldman/compound-workflows` vs `EveryInc/compound-engineering-plugin`

## 最新公开实现的详细差异分析

先给结论：

**两者表面上都在做同一条五步闭环，但现在已经不是“同一个东西换了包装”。**
`EveryInc/compound-engineering-plugin` 更像**官方参考架构 / 通用能力平台**；
`adamfeldman/compound-workflows` 更像**围绕长任务、上下文耗尽、会话恢复、状态持久化做重构的增强版运行时**。([GitHub][1])

一句话概括：

* **Every**：定义“compound engineering”的标准工作流与能力版图。([GitHub][1])
* **Adam**：把这套工作流改造成更耐久、更可恢复、更像“外部状态机”的执行系统。([GitHub][2])

---

## 一、定位差异：平台型上游 vs 自包含强化版

### 1）Every 的定位是“官方插件 + 多宿主分发平台”

Every 根 README 直接把仓库定义为一个 **plugin marketplace**，其中 Compound Engineering Plugin 是核心插件之一，而且仓库自带 Bun/TypeScript CLI，可把插件转换到 OpenCode、Codex、Gemini CLI、Copilot、Kiro、Windsurf、Qwen 等多个目标环境。也就是说，Every 不是单纯做 Claude Code 内部工作流，而是在做**标准能力定义 + 跨宿主分发**。([GitHub][1])

插件级 README 里也把它描述为 **“AI-powered development tools that get smarter with every use”**，并给出 **35+ agents、40+ skills、1 MCP server** 的规模。这个规模说明它不是只围绕五个主命令，而是一个较大的官方能力集合。([GitHub][3])

### 2）Adam 的定位是“self-contained workflow runtime”

Adam 的插件 README 直接写得很清楚：这是 **self-contained compound engineering workflows for Claude Code**，而且强调的不是跨平台，而是 **disk-persisted agent outputs、beads/TodoWrite task tracking、multi-model red-team challenges、subagent dispatch architecture**。它还明确警告**不要和 Every 插件同时安装**，因为它已经把 agents/skills 打包进来，自成一体。([GitHub][2])

这说明 Adam 的重点不是生态分发，而是**在 Claude Code 里把这套流程跑得更稳、更长、更能诊断和恢复**。([GitHub][2])

---

## 二、命令层：看起来相似，但 Adam 已经重做了“命令世界观”

### 1）Every 的核心命令是 `ce:*`

Every 当前公开主入口是：

* `/ce:ideate`
* `/ce:brainstorm`
* `/ce:plan`
* `/ce:work`
* `/ce:review`
* `/ce:compound`

README 对它们的定义相当标准：
`brainstorm` 负责需求探索，`plan` 负责计划，`work` 负责执行，`review` 负责多代理评审，`compound` 负责沉淀经验。并且仓库明确写了：**每个 cycle 都会 compound**，也就是“脑暴提升计划、计划提升未来计划、评审发现更多问题、模式被文档化”。([GitHub][1])

另外，Every 还保留了历史兼容层：CLAUDE.md 里说明 `ce:` 命名是为了避免和 Claude Code 内建 `/plan`、`/review` 冲突，旧的 `workflows:` 前缀仍作为 deprecated alias 转发。([GitHub][4])

### 2）Adam 已经迁移到 `do:*`

Adam 当前主命名空间是：

* `/do:setup`
* `/do:brainstorm`
* `/do:plan`
* `/do:deepen-plan`
* `/do:work`
* `/do:review`

并且 README 中把这些命令都定义成 workflow spine。([GitHub][2])

这不是单纯“改名字”。它意味着 Adam 已经从“沿袭上游命名”进一步转向“把自己当成一套独立操作层”。而 changelog 里还能看到它之前经历过从更长前缀到 `compound:*` 的缩短，再继续演化到现在的命名体系，说明它在**不断优化人机接口和路由语义**。([GitHub][5])

### 3）命令的“表面一致”掩盖了执行模型的“深度分叉”

两边都有 brainstorm / plan / work / review / compound，但 Adam 的命令定义已经内嵌了：

* disk-persist
* phase gates
* red-team relay
* subagent dispatch
* recover
* compact-prep
* worktree lifecycle
* beads/TodoWrite fallback

这意味着它的命令不是“一个 prompt 文件”，而更像一套**有状态、有门禁、有恢复路径的流程控制器**。([GitHub][2])

---

## 三、最核心的分叉：Every 更偏“上下文驱动”，Adam 更偏“磁盘驱动”

这是两边最大的本质差异。

### 1）Every：工作流主干清晰，但仍主要是技能/代理驱动

Every 的 `/ce:work` 公开描述是：
读取 work document（plan/spec/todo），系统化执行，重点是快速理解需求、遵循仓库现有模式、持续保持质量。([GitHub][6])

它当然也有 plan 文件、solutions 文档、worktree、并行 agent，但公开材料显示，其主逻辑仍然是：

* 通过 command/skill 驱动流程
* 通过 docs/plans、docs/solutions 等文档承接流程结果
* 通过 agents 增强研究、评审、执行

这是一个**结构清楚但相对“原生”的 prompt workflow 架构**。([GitHub][3])

### 2）Adam：把 agent 输出外部化，主动对抗上下文耗尽

Adam 的 README 把 **Disk-Persisted Agents** 直接列成关键概念：
agent 输出写入 `.workflows/`，保持 orchestrator 上下文轻量；错误可诊断；工作流可追踪。([GitHub][2])

changelog 里更明确写了：

* 新增 `disk-persist-agents` 技能
* red team 响应包装进 Task subagents 以防止 orchestrator context bloat
* `/compound:plan` 是 **disk-persisted research agents**
* `/compound:review` 是 **multi-agent review with disk-persisted outputs**([GitHub][5])

这说明 Adam 的真正核心不是“换了一组 agent”，而是把工作流从：

> “模型主要靠会话上下文串起来”

改成了：

> “模型把关键中间态落盘，主控只消费必要摘要和路径”

这一步非常关键。
**Every 更像 prompt-native workflow；Adam 更像 prompt-native workflow + 外部耐久化状态层。** ([GitHub][2])

---

## 四、长任务能力：Adam 明显更强调“活得久”

### 1）Adam 把“会话会死”视为一等公民问题

Adam README 明确写了 session recovery，recover skill 的说明更具体：
当 session 死掉、上下文耗尽、忘了 compact 时，它会读取 **JSONL session log**，交叉比对 **git history、beads state、`.workflows/` artifacts、plan files**，重建发生了什么、剩下什么要做，并提取原本会丢失的 memory-worthy 内容。([GitHub][7])

changelog 还说明：

* SessionStart hook 检测 orphaned worktrees
* recover 支持 worktree recovery
* compact-prep 在 session 结束时自动 merge worktree
* 支持 stats、cost snapshot、classification 等持久化辅助能力([GitHub][8])

这表明 Adam 已经把“上下文耗尽 / 崩会话 / 中途中断”当成系统设计的中心约束。

### 2）Every 更重视正常路径，而不是故障恢复路径

Every 的公开材料重点在：

* 工作流闭环
* 更大的 agent/skill 版图
* review / plan / compound 的标准化
* worktrees / task tracking / docs/solutions 沉淀

但我没有在本轮公开材料里看到与 Adam 同等级的“dead session recovery”被当成一级产品能力。([GitHub][3])

所以工程哲学上：

* **Every**：先把流程设计正确
* **Adam**：先承认流程会断，再把恢复做成内建能力([GitHub][7])

---

## 五、计划阶段：Adam 更像“流程编排器”，Every 更像“高质量 planning 系统”

### 1）Every 的 `ce:plan` 强调 repo pattern + confidence checking

Every README 对 `/ce:plan` 的描述是：
把 feature 转成 grounded in repo patterns 的结构化 implementation plan，并带 automatic confidence checking。([GitHub][3])

这说明 Every 的 plan 强项在于：

* 结合代码库已有模式
* 形成结构化技术计划
* 带置信度校验

这是一个高质量 planning 系统的典型定义。([GitHub][3])

### 2）Adam 的 plan 被强化成“多阶段门禁 + 红队 + readiness 复检”

Adam changelog 里对 plan 阶段的升级写得很细：

* **Phase 6.8**：新增 red team gate，支持 3-provider parallel dispatch（Gemini / OpenAI / Claude Opus）
* **Phase 6.9**：如果红队修改了计划，通过 SHA-256 比较判断计划是否变动，若变则重新跑完整 readiness check
* **Phase 7**：把 handoff decision tree 从 5 条扩到 7 条，加入 red-team-specific routing；无严重问题可直接去 work，有 deferred serious findings 则去 deepen-plan([GitHub][5])

这就不是普通的“plan 命令”了，而是一个**带阶段门控、自动重检、回流机制的 planning state machine**。([GitHub][5])

### 3）Adam 还把“rationale capture”做成耐久化要求

changelog 里写到，brainstorm、plan、deepen-plan 都开始记录**为什么**用户做这些决策，而不只记录**做了什么**，因为用户推理会随着对话上下文消失，文档才是 durable record。([GitHub][5])

这个细节很重要。它说明 Adam 对 planning 的理解比上游更“控制论”一点：
不只是输出 plan，还要保存决策因果链，方便后续恢复、审查、再规划。([GitHub][5])

---

## 六、执行阶段：Every 的 `work` 更像执行器，Adam 的 `work` 更像调度器

### 1）Every 的 `ce:work`：系统化执行 work document

Every 的 `work.md` 描述很直接：
给它一个 plan/spec/todo，它就系统化执行，重点是快速吃透需求、遵守现有模式、交付完整特性。根 README 也明确把 `/ce:work` 概括为 **execute plans with worktrees and task tracking**。([GitHub][6])

这说明 Every 的 `work` 更像：

* 执行器
* 质量保持器
* 模式对齐器
* 与 worktree/task tracking 集成的 delivery 命令([GitHub][6])

### 2）Adam 的 `do:work`：subagent dispatch 是默认路径

Adam README 把 `/do:work` 明确描述为：
**Execute plans via subagent dispatch with beads/TodoWrite tracking**。([GitHub][2])

而 changelog 里还特别写到：

* `work-agents.md` 被合并进 `work.md`
* **subagent dispatch is now the default mode**
* “parallel is recommended when steps touch separate files”
* graceful degradation：无 beads 时回退到 TodoWrite，无 PAL 时回退到 Claude subagent，无 bundled agents 时回退到 general-purpose agent([GitHub][5])

这说明 Adam 的执行阶段更像**任务编排系统**：

* 能分派给多个 subagent
* 能按文件/步骤做并行
* 能对依赖缺失进行回退
* 能把任务状态和中间结果写入外部路径

所以在 work 阶段：

* **Every** 更像“一个强执行器”
* **Adam** 更像“一个带路由与回退策略的 orchestrator”([GitHub][6])

---

## 七、评审阶段：两边都很强，但侧重点不同

### 1）Every 的 review 更成熟、更体系化

Every 的 README 里明确写 `/ce:review` 是：
**Structured code review with tiered persona agents, confidence gating, and dedup pipeline**。([GitHub][3])

而 agents 清单里，Every 的 review 阵容明显更厚：

* correctness-reviewer
* maintainability-reviewer
* security-reviewer
* reliability-reviewer
* testing-reviewer
* performance-reviewer
* architecture-strategist
* adversarial-reviewer
* 各语言/框架 reviewer
* 文档类 reviewer：coherence / design-lens / feasibility / product-lens 等([GitHub][3])

这意味着 Every 的 review 能力更偏向**大而全的专业 persona 体系**。

### 2）Adam 的 review 更强调“输出耐久化”和“跨模型挑战”

Adam 当前 README 把 `/do:review` 定义为：
**Multi-agent code review with disk-persisted findings**，并且整个系统强调 **3-provider parallel red-team challenges**。([GitHub][2])

changelog 还说明：

* red team provider 独立并行运行，彼此不读对方 critique，以避免锚定偏差
* provider preference 改为 runtime-detected，避免配置过期
* PAL / clink / Claude-only 有多级降级路径([GitHub][5])

也就是说，Adam 的 review 更注重：

* 独立 critique 的“去锚定”
* review 结果写盘
* triage 流程
* 在长会话下仍能保留审查证据链([GitHub][5])

### 3）谁更强？

如果只看“评审人格与覆盖面”，**Every 更厚**。
如果看“评审过程的耐久化、可恢复性、与长任务兼容性”，**Adam 更激进**。([GitHub][3])

---

## 八、知识沉淀：Every 更强调“标准解法资产化”，Adam 更强调“全过程留痕”

### 1）Every 的 `compound` 非常明确：写 `docs/solutions/`

Every 的 `ce-compound` 技能说明写得很清楚：

* 在上下文还新鲜时记录问题解决方案
* 写入 `docs/solutions/`
* 使用 YAML frontmatter 提高可搜索性和复用性
* 用并行 subagents 提高效率
* 其核心理念是“每次记录方案，下一次同类问题从研究小时级降到分钟级”([GitHub][9])

这说明 Every 的 compound 是一个很标准的**知识资产化契约**。

### 2）Adam 当然继承了 compound，但更强调“前序痕迹也要存活”

Adam 不是不重视结果文档，而是把重点前移了：

* brainstorm 的红队反馈
* plan 的研究输出
* deepen-plan 的多轮目录
* review 的发现
* compact-prep 的状态
* recover 的 manifest

这些都进入 `.workflows/` 或相关外部状态。([GitHub][2])

所以可以这么理解：

* **Every 的复利对象**：主要是“解法”
* **Adam 的复利对象**：既包括“解法”，也包括“研究过程、决策理由、失败痕迹、恢复线索”([GitHub][9])

---

## 九、并发与隔离：Adam 把 worktree 提升到了“session 边界”

### 1）Every：明确支持 worktree，但偏通用工具层

Every 的核心说明里写 `/ce:work` 配 worktrees and task tracking；插件 README 里还有独立的 `git-worktree` skill 用于并行开发。([GitHub][1])

这说明 Every 把 worktree 当成重要能力，但还是偏“并行开发工具”。

### 2）Adam：把每个 session 默认绑到自己的 worktree

Adam release 里明确写：

* 每个 Claude Code session 自动获得自己的 git worktree
* isolated index + working tree
* concurrent sessions no longer cross-contaminate commits
* 新增 `/do:merge` 用于 deferred merge 重试([GitHub][10])

changelog 还写了：

* 禁止直接 `git worktree add`
* 强制经由 `bd worktree` 或 `worktree-manager.sh`
* SessionStart 检测 orphaned worktrees
* session-end compact-prep 自动 merge + ExitWorktree([GitHub][5])

这说明 Adam 已经把 worktree 从“开发技巧”升级成了**会话隔离与故障恢复的底层机制**。

---

## 十、依赖与降级策略：Adam 更像“容错运行时”

Adam changelog 明确写了 graceful degradation：

* 无 beads → TodoWrite fallback
* 无 PAL → Claude subagent fallback
* 无 bundled agents → general-purpose agent fallback([GitHub][5])

同时它还做了：

* provider runtime detection
* setup 时检测 Gemini CLI / Codex CLI / beads 等环境
* 不同机器上配置拆分为 committed config 和 local config([GitHub][5])

这像是一个真正考虑“用户环境并不一致”的运行时系统。

相比之下，Every 当然也有安装与 MCP 配置说明，但其公开材料更强调的是“能力供给”和“跨平台安装”，不是像 Adam 那样把**多级降级**做成主要卖点。([GitHub][1])

---

## 十一、规模与取舍：Every 更广，Adam 更深

当前公开描述里：

* **Every**：35+ agents、40+ skills、1 MCP server。([GitHub][3])
* **Adam**：26 agents、31 skills、9 commands。([GitHub][2])

这组数字很能说明问题。

### Every 的策略：能力面更广

Every 除了核心 workflow，还有：

* `/changelog`
* `/feature-video`
* `/reproduce-bug`
* `/resolve-pr-feedback`
* `/test-browser`
* `/test-xcode`
* `/onboarding`
* `/todo-resolve`
* `/todo-triage`
* 大量 review/document review persona([GitHub][3])

它像一个**官方插件操作系统**。

### Adam 的策略：把最关键主链做厚

Adam 的资源没有 Every 那么广，但在：

* session recovery
* compact-prep
* worktree lifecycle
* disk persistence
* red-team orchestration
* readiness checks
* rationale capture
* stats capture / classification

上明显更深入。([GitHub][5])

它像一个**workflow reliability 专项强化 fork**。

# 最后给你一个“本质判断”

## Every 的核心精神

**把工程闭环标准化，并让每次工作都为下一次工作铺路。**
它的重点是：统一方法论、构建通用 agent/skill 体系、把 plan/review/compound 变成稳定的团队操作系统。([GitHub][1])

## Adam 的核心精神

**承认代理式工程在真实世界会遇到上下文耗尽、任务过长、会话中断、并发污染，于是把“耐久化、恢复、追踪、隔离”做到主流程里。** ([GitHub][2])

---

# 你如果问“到底有什么区别”，最准确的答案是这三句

**第一句：**
Every 主要定义了**“怎么做 compound engineering”**。([GitHub][1])

**第二句：**
Adam 主要解决了**“当 compound engineering 任务很长、很复杂、上下文会炸时，怎么还能继续做”**。([GitHub][2])

**第三句：**
所以 Every 更像**方法论上游 + 官方基线**，Adam 更像**围绕运行时可靠性重构出来的生产化 fork**。([GitHub][11])

**“逐命令对照表”**

* 输入

* 状态存储
* agent 调度方式
* 人机交互点
