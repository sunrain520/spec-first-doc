# `adamfeldman/compound-workflows` 与 `EveryInc/compound-engineering-plugin` 详细对比分析

## 摘要

这两个仓库共享同一条核心闭环：**Brainstorm → Plan → Work → Review → Compound**。Every 的仓库主页把这条工作流直接写成官方哲学和主流程，并强调“80% 在计划和评审，20% 在执行”。([GitHub][1])
但到了最新公开实现，二者已经明显分叉：

* **EveryInc/compound-engineering-plugin** 更像**官方参考架构 / 平台型上游**：定义方法论、提供更广的 agents/skills、支持多宿主分发与跨工具适配。([GitHub][1])
* **adamfeldman/compound-workflows** 更像**围绕真实长任务场景重构出来的增强版运行时**：重点不在扩展更多能力，而在解决上下文耗尽、长会话断裂、并发污染、状态丢失、计划反复重做等问题。它自己在仓库首页就把差异总结为“agent outputs 写盘、3 模型并行 red team、compaction-safe task tracking、session recovery、readiness checks”。([GitHub][2])

最准确的一句话是：

> **Every 定义“怎么做 compound engineering”；Adam 重点解决“当任务很长、上下文会炸、会话会断时，compound engineering 怎么还能继续做”。** ([GitHub][1])

---

## 一、项目定位与目标差异

### 1. Every：官方基线、平台化、跨宿主

Every 根 README 把仓库定位成一个 **plugin marketplace**，其中 Compound Engineering plugin 是核心插件，目标是让“每一单位工程工作都让下一单位更容易”。它明确给出工作流闭环、理念说明、以及“Learn more / Full component reference”等平台式组织方式。([GitHub][1])

从仓库结构看，它不只是一个 Claude Code 内部工作流包，而是包含：

* 插件目录 `plugins/`
* 脚本与发布目录 `scripts/release`
* TypeScript 源码 `src/`
* 测试目录 `tests/`

这说明它承担了**能力定义 + 打包 + 分发 + 质量控制**的上游职责。([GitHub][1])

插件 README 里还列出 **35+ agents、40+ skills、1 MCP server**，显示它追求的是一个覆盖面广的工程能力集合，而不是只围绕五步主链做深挖。([GitHub][3])

### 2. Adam：自包含、长任务优化、以 Claude Code 为中心

Adam 仓库主页直接写明自己是 **“Fork of Every's compound-engineering for Claude Code”**，并列出它新增的核心能力：
“agents that don't exhaust context, session recovery after exhaustion, compaction-safe task tracking, multi-model red team, fewer plan iterations via readiness checks, and tiered memory management”。([GitHub][2])

插件 README 也把它定义成 **self-contained compound engineering workflows for Claude Code**，并警告**不要和 Every 的插件同时安装**，因为它已经把所有 agents/skills 打包进来，会产生名称冲突。([GitHub][2])

这说明 Adam 的目标不是做更大的 marketplace，而是做一套**在 Claude Code 中能更久、更稳、更可恢复地跑下去的 workflow runtime**。([GitHub][2])

---

## 二、方法论相同，但实现哲学不同

### 1. Every 的哲学：复利式工程

Every 在主页上给出的哲学非常明确：

* 传统开发会累积技术债
* compound engineering 反过来做
* **80% 在 planning 和 review，20% 在 execution**
* 要“plan thoroughly, review to catch issues and capture learnings, codify knowledge so it's reusable, keep quality high”([GitHub][1])

它的重点是：
**通过更重的计划、更系统的评审、更结构化的知识沉淀，让后续工作越来越容易。** ([GitHub][1])

### 2. Adam 的哲学：现实主义运行时

Adam 不是推翻这个哲学，而是给它加上“现实约束”：

* agent 输出不应该继续塞满上下文
* 任务跟踪不能在 compaction 后消失
* 计划不能无限反复
* 会话死掉后不能全靠人肉恢复
* 单模型 review 容易漏问题，应该并行多模型挑战([GitHub][2])

所以 Adam 的哲学可以概括为：

> **承认代理式工程在现实里会遇到上下文耗尽、崩会话、并发污染与状态丢失，于是把耐久化、恢复、隔离和回退机制做到主流程里。** ([GitHub][2])

---

## 三、命令体系对比

### 1. Every：`ce:*` 为主的官方工作流

Every 主页把命令直接列成：

* `/ce:ideate`
* `/ce:brainstorm`
* `/ce:plan`
* `/ce:work`
* `/ce:review`
* `/ce:compound` ([GitHub][1])

这套命令是标准闭环，逻辑清晰、层次分明，强调“从想法到实施到审查再到沉淀”的完整工程链路。([GitHub][1])

### 2. Adam：迁移到 `do:*`，更像操作层

Adam 插件 README 显示，v3.0.0 起工作流命令从 `/compound:*` 迁移到了 `/do:*`，旧命名暂时保留为别名。它当前的 workflow cycle 是：

`brainstorm -> plan -> [deepen-plan] -> work -> review -> compound`。([GitHub][4])

这有两个重要信号：

第一，Adam 不只是照搬上游命名，而是在**重构命令接口层**。
第二，它把 **deepen-plan** 单独拉出来，并明确说：对于多文件或新组件变更，deepen-plan 应视为“非可选”。([GitHub][4])

所以 Adam 的命令体系比 Every 多了一个明显的设计倾向：
**更把命令当作状态机阶段，而不是单次 prompt 调用。** ([GitHub][4])

---

## 四、最本质的差异：上下文驱动 vs 磁盘驱动

### 1. Every：文档与技能驱动的 prompt-native workflow

Every 的 `ce:work` 文档展示了它的执行方式：
如果在默认分支，可选择新建 branch，或调用 `git-worktree` 技能；然后创建 TodoWrite 任务，按计划拆解、排序、执行，并保留测试与质量检查任务。([GitHub][5])

这说明 Every 的工作流核心形态是：

* 通过 command/skill 推进流程
* 通过 TodoWrite 与 worktree 辅助执行
* 通过 docs/solutions 等文档沉淀结果([GitHub][5])

这是一个很强的 prompt-native 工程框架，但其主逻辑仍然主要依赖会话上下文与 orchestrator 对过程的持续掌控。([GitHub][5])

### 2. Adam：agent 输出写盘，主控只消费必要摘要

Adam 仓库主页把两者差异写得非常直接：

* Every：**Agent outputs in-context**
* Adam：**agents write to disk, not context**([GitHub][2])

插件 README 还明确说每一步都会产出文档供下一步消费，并规定目录约定，说明它把工作流中间态外部化成了文件系统中的工件，而不是尽量留在会话上下文里。([GitHub][4])

这带来的结果是：

* 会话更轻
* 中间状态更可检查
* 失败后更容易恢复
* 长任务更不容易被上下文窗口压垮([GitHub][2])

这是两套代码最大的架构分水岭。

---

## 五、计划阶段的差异：标准 planning vs 带门禁的 planning state machine

### 1. Every：高质量 planning，强调 grounded in repo patterns

Every 在插件级能力说明里把 `/ce:plan` 描述为：
把 feature ideas 转成 **detailed implementation plans**，并强调 grounded in repo patterns。([GitHub][1])

这说明 Every 的 planning 重点在于：

* 结构化
* 与现有仓库模式对齐
* 给实施阶段提供高质量输入([GitHub][1])

### 2. Adam：planning 变成多阶段门控系统

Adam 的 changelog 展示了大量 planning 强化动作，包括：

* work step 不再按时间估算，而按“单次 subagent dispatch 可完成的 coherent unit”划分
* 通过 readiness checks 和 auto-consolidation 更早发现计划问题
* 通过 red-team challenge 和后续 decision routing 减少计划返工([GitHub][6])

更重要的是，它把 planning 的问题从“写出一个计划”升级成“让计划尽快达到可执行质量，并控制反复迭代成本”。这就是主页里那句：

* Every：**Plan quality unbounded iteration**
* Adam：**Plans reach quality faster — readiness checks and auto-consolidation catch issues early** ([GitHub][2])

所以：

* **Every 的 plan** 更像高质量设计输出
* **Adam 的 plan** 更像有自动质检、自动路由、自动收敛倾向的 planning state machine([GitHub][2])

---

## 六、执行阶段：执行器 vs 调度器

### 1. Every：`ce:work` 是一个强执行器

Every 的 `ce:work` 文档清楚体现出它的执行观：

* 选择 branch/worktree 策略
* 用 TodoWrite 拆任务
* 任务需具体、可完成、含依赖和测试
* 执行计划并保持质量标准([GitHub][5])

也就是说，Every 的 `work` 非常像一个**系统化执行器**：
给你 plan，它帮你把工程工作有条理地做完。([GitHub][5])

### 2. Adam：`do:work` 更像 subagent dispatch orchestrator

Adam 的 README 把执行阶段的不同写得很明确：

* **Task tracking**：Every 是 TodoWrite only；Adam 是 beads preferred, TodoWrite fallback
* **Work step**：强调一个 coherent unit 对应一次 subagent dispatch
* 对无 beads、无 PAL、无其他依赖的场景有回退路径([GitHub][2])

这说明 Adam 的执行阶段不是简单“让主 agent 去做事”，而是：

* 默认按子代理派发
* 把任务状态外部化
* 允许更长链条的任务在不丢上下文的前提下运行
* 对环境缺失进行降级处理([GitHub][2])

因此：

* **Every 的 work** 更像“强执行器”
* **Adam 的 work** 更像“有容错和状态外化的调度器”([GitHub][5])

---

## 七、评审阶段：大而全的 persona 体系 vs 多模型并行挑战

### 1. Every：review 体系更厚

Every 的插件 README 中列出大量 review agents，包括：

* correctness-reviewer
* architecture-strategist
* code-simplicity-reviewer
* 以及其他多种 quality / architecture / API / review persona([GitHub][3])

这说明 Every 的 review 更像一个**大而全的专业人格评审系统**。它的强项是覆盖面广、角色明确、适合系统化审查。([GitHub][3])

### 2. Adam：review 更强调并行 red-team 与写盘

Adam 主页明确写：

* Every：**Single model**
* Adam：**3 models in parallel surface what a single model misses** ([GitHub][2])

插件 README 也说明 PAL MCP 是可选依赖，用于 multi-model red-team；没有时退化为 Claude subagent 单模型路径。([GitHub][4])

这意味着 Adam 把 review 的重点从“扩展更多 reviewer persona”部分转到了：

* 多模型独立质疑
* 避免单模型盲点
* 让 review 结果可持久化、可回看、可用于恢复([GitHub][2])

所以：

* **Every 更强在 review 覆盖面**
* **Adam 更强在 review 过程的去锚定、并行化与耐久化** ([GitHub][3])

---

## 八、知识沉淀：Every 更强调标准化解法库，Adam 更强调全过程留痕

### 1. Every：`ce:compound` 是标准化的 solutions 生产线

Every 的 `ce-compound` 技能文档明确要求：

* 进行并行研究
* 提取上下文与解决过程
* 读取 schema 与 yaml-schema 映射
* 输出到 `docs/solutions/`
* 使用规范命名和 YAML frontmatter([GitHub][7])

这代表 Every 的 compound 不是“随手记笔记”，而是一个**结构化解法资产化流程**。它最看重的是：
**把已经解决的问题沉淀成未来可搜索、可复用、可沿用的知识资产。** ([GitHub][7])

### 2. Adam：不仅沉淀“解法”，还沉淀“过程状态”

Adam 的插件 README 说明“每一步都产出文档供下一步消费”，并且提供 recover、classify-stats 等与过程分析有关的命令。([GitHub][4])

它沉淀的对象不只是最终 solution，还包括：

* 研究输出
* planning 工件
* task tracking 状态
* review 发现
* compact-prep 结果
* recover manifest 与 session cross-reference 信息([GitHub][2])

所以可以概括为：

* **Every 的复利对象主要是“解法知识”**
* **Adam 的复利对象既有“解法”，也有“过程状态与恢复线索”** ([GitHub][7])

---

## 九、会话恢复与长任务生存能力：Adam 显著领先

这里是两者差异最鲜明的地方。

Adam 插件 README 直接提供了：

* `/compound-workflows:recover`：从 dead/exhausted sessions 恢复上下文，解析 JSONL logs，并 cross-reference state([GitHub][4])

主页也明确对比：

* Every：**Session recovery manual**
* Adam：**`/do:compact-prep` + `/recover`** ([GitHub][2])

changelog 进一步写明：

* SessionStart 自动分配独立 worktree
* 检测 orphaned worktrees
* compact-prep 会在 session end 自动 merge worktree
* recover 增加 worktree recovery operations([GitHub][6])

这些都说明 Adam 把“会话会中断”当成一等公民问题来设计系统。([GitHub][6])

相比之下，Every 的公开材料更多聚焦于标准工作流本身，没有把死会话恢复提升到这种核心地位。([GitHub][1])

---

## 十、并发与隔离：Adam 把 worktree 提升为 session 级边界

Every 的 `ce:work` 里已经把 worktree 作为推荐方案之一，尤其适合并行开发和保持默认分支整洁。([GitHub][5])

但 Adam 又往前走了一步。其 changelog 说明：

* **每个 Claude Code session 自动获得自己的 git worktree**
* concurrent sessions 不再 cross-contaminate commits
* 若禁用 worktree，则用 `safe-commit.sh` + 临时 `GIT_INDEX_FILE` 防止提交互相污染
* 新增 `/do:merge` 处理延迟合并([GitHub][6])

这意味着：

* 对 Every 来说，worktree 是强推荐的并行开发工具
* 对 Adam 来说，worktree 已经变成**会话隔离和可靠提交的默认基础设施** ([GitHub][5])

---

## 十一、依赖管理与降级路径：Adam 更像“容错运行时”

Adam 的 README 明确列出依赖与对应能力：

* beads：推荐，用于 compaction-safe task tracking
* PAL MCP：可选，用于 multi-model red-team
* GitHub CLI：可选，用于 PR 创建
* ccusage：可选，用于成本与 token 追踪([GitHub][4])

最关键的是，它给出了明确的回退路径：

* 无 beads → TodoWrite fallback
* 无 PAL → Claude subagent fallback([GitHub][4])

这让 Adam 更像一个考虑真实环境差异的“容错运行时”。
Every 当然也有技能与工具层支持，但其公开强调点更多是能力覆盖，而不是这种显式、多级的降级策略。([GitHub][3])

---

## 十二、规模与取舍：Every 更广，Adam 更深

Every 插件 README 显示的是 **35+ agents、40+ skills、1 MCP server**。([GitHub][3])
Adam 当前主打的是更少但更集中的 workflow 命令和相关能力，围绕：

* plan quality
* task tracking
* red team
* recovery
* process analysis
* memory management([GitHub][2])

这意味着两边的产品策略不同：

* **Every**：能力面更广，适合作为官方基线与方法论分发平台。([GitHub][1])
* **Adam**：主链做得更厚，适合重长任务、重可靠性、重恢复的使用场景。([GitHub][2])

---

## 十三、谁更适合什么场景

### 适合选 Every 的场景

如果你要的是：

* 官方基线
* 更完整的 agent/skill 版图
* 方法论学习与标准化 adoption
* 跨宿主、跨工具迁移潜力
* 更成熟的“知识沉淀为 solution 文档”契约

那 Every 更合适。([GitHub][1])

### 适合选 Adam 的场景

如果你更在意：

* 长任务不爆上下文
* 会话中断后能恢复
* 多模型并行红队
* compaction 后任务状态还在
* 多 session 并发不互相污染
* 计划尽快达到 ready-to-work 状态

那 Adam 更合适。([GitHub][2])

---

## 结论

如果把两者关系说得最透彻一点：

**`EveryInc/compound-engineering-plugin` 是方法论上游与官方参考架构。**
它定义了 compound engineering 的主循环、工程哲学和广覆盖能力图谱。([GitHub][1])

**`adamfeldman/compound-workflows` 则是在这个参考架构上，把“运行时可靠性”推进到前台。**
它通过磁盘持久化、compaction-safe tracking、recover、session-level worktree isolation、multi-model red-team 与 readiness checks，把这套工作流从“强 prompt framework”进一步推向“可在真实复杂环境中长期运行的 workflow runtime”。([GitHub][2])

所以最后一句判断是：

> **Every 更像“官方参考架构”；Adam 更像“面向长任务与故障现实的生产化重构 fork”。** ([GitHub][1])
