# CRG v2：算法升级清单与评测路线

## 一、这份文档解决什么问题

前两份文档已经回答了两件事：

1. `spec-graph-bootstrap` 当前产物为什么还不是终局
2. `CRG + Stage-0 + Workflow + Eval` 的合理终局分层是什么

但如果要真正指导后续演进，还缺一张更硬的路线图：

- `src/crg` 当前实现到底已经做到了什么
- 哪些学术算法值得吸收，哪些暂时不值得
- 哪些能力必须先补，哪些可以后补
- 怎么证明这些升级真的提升了“代码库理解”，而不是只让系统更复杂

这份文档的目标不是罗列论文，而是把：

- 当前代码实现
- 业界方案
- 学术算法
- 评测方法

压成一份可执行的 `CRG v2` 路线图。

## 二、结论先行

`src/crg` 当前最准确的定位是：

**一个工程上合格的、本地优先的、AST-first 代码事实索引器。**

它已经具备：

- 输入收敛
- 文件级增量更新
- AST/符号抽取
- SQLite 持久化
- FTS 检索
- 基于调用图的轻量 blast radius 扩展

但它还不具备顶尖代码库理解系统的三个关键层：

1. **多代际、可回滚的索引发布层**
2. **任务感知的最小上下文检索与打包层**
3. **以 benchmark 证明价值的评测层**

因此，`CRG v2` 的重点不应该是“再多抽一些节点”，而应该是：

**把 `CRG` 从“会建图”推进到“能为任务稳定分发最小证据包”。**

## 三、以代码实现为准：`src/crg` 当前真实能力边界

下面的判断以当前仓库实现为准，而不是按设计愿景推断。

### 3.1 已经成立的能力

#### A. 输入收敛和基础构建链路已经比较扎实

从 [src/crg/cli/build.js](/Users/kuang/xiaobu/spec-first/src/crg/cli/build.js) 可以看到，当前构建链路已经具备：

- `--repo` 显式输入
- `collectInputFiles()` 收敛输入集
- `detectChangedFiles()` 做变更检测
- `parseFile()` 做 AST 抽取
- `upsertNodes()` / `upsertEdges()` 回写图数据库
- `resolveEdges()` / `replaceUnresolvedEdges()` 维护边状态
- `classifyParseQuality()` 暴露 `ok / no_parser / parse_error / module_only`

这说明当前 `CRG` 不是玩具脚本，而是一个已经有明确 build pipeline 的系统。

#### B. 增量机制已经存在，但粒度仍是“文件级”

从 [src/crg/incremental.js](/Users/kuang/xiaobu/spec-first/src/crg/incremental.js) 可以看到，当前增量检测基于：

- 文件内容 `SHA256`
- `fingerprints` 表
- `changed / unchanged / deleted` 分类

这在工程上是合理且稳定的第一步。

#### C. 检索机制已经存在，但仍是轻量 lexical FTS

从 [src/crg/search.js](/Users/kuang/xiaobu/spec-first/src/crg/search.js) 可以看到，当前检索主要是：

- FTS5
- 字段为 `name / file_path / kind`
- `bm25` 排序
- 简单 `kind` 过滤

这说明当前已经有“可查询入口”，但离“代码理解检索层”还很远。

#### D. review 场景已经有一版 blast radius 能力

从 [src/crg/commands/review-context.js](/Users/kuang/xiaobu/spec-first/src/crg/commands/review-context.js) 可以看到，当前已经具备：

- `git diff` 驱动的变更文件识别
- 变更节点提取
- 基于命名约定的候选测试匹配
- `calls` 边的 2-hop 反向 BFS 扩展
- 扩展节点风险排序

这很重要，因为它说明当前系统已经不只是“查某个函数”，而是开始做任务化的上下文扩展。

### 3.2 当前明确缺口

#### A. 还没有 generation-based / snapshot-based 索引发布

当前 build 仍然是对同一份图数据库进行更新，而不是：

- 先构建新 generation
- 验证通过后再 promote
- 保留上一个稳定 generation 供回滚

这会带来三个问题：

1. 构建中断时难以保证消费稳定性
2. 退化解析和部分失败不够容易追溯
3. workflow 很难绑定“某一代已验证索引”

#### B. 还没有 retrieval-ready 语义字段

当前 `search.js` 只索引：

- `name`
- `file_path`
- `kind`

但顶尖检索层通常至少还需要：

- `summary`
- `signature`
- `docstring/comment digest`
- `callee/caller sketch`
- `risk/test/integration tags`
- `retrieval_text`

否则检索系统能找到“名字像什么”，但很难找到“语义上相关什么”。

#### C. `review-context` 还不是统一 retrieval layer

当前 `review-context.js` 有价值，但它仍然更像一个：

- review 专用启发式命令

而不是：

- 统一的任务检索 API

它缺：

- token budget
- evidence packing
- 去重策略
- 统一 rerank
- 任务 profile
- 结构化 fallback reason

#### D. 还没有“理解提升”的正式评测

当前可见指标更多是：

- build 是否成功
- unresolved edges 数量
- parse quality

这些指标只能说明“索引器有没有跑起来”，不能说明：

- agent 是否更快找到入口
- review 是否更能抓到 cross-file 问题
- plan 是否更少漏掉依赖与测试面

## 四、哪些学术算法最值得吸收

不是所有“AI + 图 + 代码”论文都值得直接引入 `CRG`。

对当前仓库最有价值的，不是大而全的端到端模型，而是能直接提升 **retrieval / packing / eval** 的方法。

## 4.1 第一优先级：检索与扩图

### RepoCoder：迭代式 retrieval-generation

来源：

- https://arxiv.org/abs/2303.12570
- https://aclanthology.org/2023.emnlp-main.151/

价值：

- 说明 repository-level 任务不应只做一次检索
- 检索应根据当前生成/推理状态继续迭代

对 `CRG` 的启发：

- `review-context` 不该只有单次扩图
- `plan` / `work` / `review` 应允许 `seed -> expand -> refine` 多轮检索

### RepoHyper：graph-enhanced search-expand-refine

来源：

- https://arxiv.org/abs/2403.06095

价值：

- 明确把图扩展与 refine 排序作为独立步骤
- 说明 graph 不该只用于展示，而应该直接进入 retrieval 主链

对 `CRG` 的启发：

- 当前 2-hop BFS 是正确方向，但还只是最初级形态
- 后续应升级为：
  - lexical seed retrieval
  - graph expansion
  - risk/test prior
  - rerank

### Repoformer：选择性检索

来源：

- https://arxiv.org/abs/2403.10059
- https://www.amazon.science/blog/enhancing-repository-level-code-completion-with-selective-retrieval

价值：

- 强调“不是所有任务都要检索”
- retrieval 本身有成本，应该学会判断何时检索、检索多少

对 `CRG` 的启发：

- 不要把所有 workflow 都升级成“默认大规模扩图”
- 应该让 evaluator 先判断：
  - 当前任务是否需要 graph expansion
  - 需要几跳
  - 需要多少 token 预算

## 4.2 第二优先级：结构化 chunking 与语义索引

### cAST：基于 AST 的结构化 chunking

来源：

- https://aclanthology.org/2025.findings-emnlp.430/
- https://aclanthology.org/2025.findings-emnlp.430.pdf

价值：

- 证明 code retrieval 的效果高度依赖 chunking
- AST-aware chunking 优于简单按行切块

对 `CRG` 的启发：

- 当前 `CRG` 已经有 AST 抽取能力
- 下一步非常适合生成：
  - function/class 级 chunk
  - oversized node 的 AST 子块
  - chunk-level retrieval text

这比“先上向量库”更值得优先做，因为它直接改善检索基本盘。

### ReflectCode / RepoAlign-Bench：面向变更意图的 repository-aware retrieval

来源：

- https://arxiv.org/abs/2510.24749
- https://aclanthology.org/2025.findings-emnlp.1147/
- https://aclanthology.org/2025.findings-emnlp.1147.pdf

价值：

- 表明“按函数搜代码”不够，变更请求往往需要 repository-aware retrieval
- 说明检索系统需要把函数依赖、意图扩展、语义对齐放进统一排序框架

对 `CRG` 的启发：

- 如果后续引入 semantic retrieval / dual-encoder，应优先把它放在：
  - optional rerank
  - optional semantic candidate expansion

而不是一上来就把系统改造成 vector-first。

## 4.3 第三优先级：benchmark 与任务级评测

### RepoBench / RepoEval：检索与补全基本盘

来源：

- https://github.com/Leolty/repobench
- https://iclr.cc/virtual/2024/poster/17776

价值：

- 明确 repository-level 能力不能只看单文件补全
- retrieval 与 generation 需要分开评测

对 `CRG` 的启发：

- 先把 retrieval@k、evidence recall、packing efficiency 做出来
- 不要只看最终 agent 成功率

### CodeRAG-Bench：代码检索增强生成总评测

来源：

- https://arxiv.org/abs/2406.14497
- https://aclanthology.org/2025.findings-naacl.176

价值：

- 说明“是否检索到高质量上下文”与“生成器是否能用好上下文”是两件事
- 当前系统常见失败点是 lexical overlap 不足时召回差

对 `CRG` 的启发：

- `CRG` 应先优化 retrieval quality
- Stage-0 / Workflow 再解决“消费这些上下文”的问题

### CodeRepoQA / StackRepoQA / SWE-QA：代码库问答能力

来源：

- https://arxiv.org/abs/2412.14764
- https://arxiv.org/abs/2603.26567
- https://huggingface.co/posts/YerbaPage/735409135497339

价值：

- 仓库理解不是只看生成，还要看问答
- 多轮 repository QA 是检验“是否真正理解代码库”的直接方法

对 `CRG` 的启发：

- `spec-first` 应建立自己的 repo QA 集
- 用于检验：
  - 入口定位
  - 数据流理解
  - 风险点定位
  - 测试面识别

### AACR-Bench：自动 code review 的 repo-level 评测

来源：

- https://arxiv.org/abs/2601.19494
- https://github.com/alibaba/aacr-bench

价值：

- 说明 code review 的效果高度依赖完整仓库上下文
- 说明检索方法和上下文粒度会显著影响结果

对 `CRG` 的启发：

- `review-context` 的正确升级方向，不是更多启发式规则，而是：
  - 更高质量的证据召回
  - 更可解释的证据排序
  - 更稳定的 context pack

### SWE-CI：从“能修一次”升级到“能长期维护”

来源：

- https://arxiv.org/abs/2603.03823

价值：

- 把评测目标从静态修 bug，推进到长期演进与可维护性

对 `CRG` 的启发：

- 终局评测不应只问“这次答对了吗”
- 还要问：
  - 是否降低了无关扫描
  - 是否提升了测试决策
  - 是否能在长期迭代中保持索引与知识层新鲜度

## 五、`CRG v2` 的正确算法组合，不是单点最强，而是分层最稳

结合当前代码和外部研究，`CRG v2` 最合理的算法组合应该是下面这条链，而不是单押某一个“神算法”。

## 5.1 Index Layer

建议组合：

- AST node extraction
- file-level SHA incremental
- generation snapshot
- unresolved edge audit
- node/chunk summary generation

这里的关键不是“更聪明的模型”，而是：

**先把可审计、可回滚、可复用的事实层做稳定。**

## 5.2 Retrieval Layer

建议组合：

1. lexical seed retrieval
2. graph expansion
3. risk/test/integration priors
4. optional semantic candidate expansion
5. rerank
6. token-budget packing

推荐顺序不是偶然，而是考虑到当前 `CRG` 的真实基础：

- 已经有 AST
- 已经有 edges
- 已经有 risk/test 信号雏形
- 还没有成熟的 semantic layer

所以最合理的工程路径是：

**先把 lexical + graph + heuristic prior 做强，再把 semantic retrieval 作为增强项补上。**

## 5.3 Compiled Context Layer

建议组合：

- one-page overview
- module map
- minimal context cards
- source traceability
- contradictions lint
- freshness report

这一层不需要追求“更多文档”，而要追求：

**更稳定的机器消费入口。**

## 5.4 Eval Layer

建议组合：

- retrieval metrics
- repo QA metrics
- review metrics
- context efficiency metrics
- long-term maintainability metrics

## 六、优先级路线图：什么必须先做，什么应该后做

## P0：必须先做，不做就不该继续扩容

### 1. generation snapshot + atomic promote

目标：

- 每次 build 产出新的 generation
- 通过校验后再切换 active generation
- 保留最近一个稳定 generation 以供回滚

原因：

- 这是让 `CRG` 从“本地工具”升级为“团队底座”的前提

### 2. retrieval-ready 字段

建议新增：

- `summary`
- `signature_text`
- `retrieval_text`
- `tags`
- `last_seen_generation`

原因：

- 没有这些字段，后面的 rerank / packing 都无从谈起

### 3. 统一 retrieval API

建议目标接口：

```json
{
  "task_type": "review",
  "budget": 2000,
  "seed_candidates": [],
  "expanded_candidates": [],
  "ranked_context": [],
  "packing_reason": [],
  "fallback_reason": []
}
```

原因：

- `review-context`、后续 `plan-context`、`work-context` 需要共用一套求值层

### 4. 范围隔离与噪声治理

目标：

- 明确主代码、workflow asset、template、example 的分析边界
- 防止上下文污染主代码理解

原因：

- 这是当前 Stage-0 事实层噪声问题的根因之一

## P1：完成最小上下文分发闭环

### 1. hybrid retrieval

建议先做：

- BM25/FTS
- graph expansion
- risk/test priors
- dedup
- rerank

可选增强：

- vector candidate expansion

### 2. AST-aware chunking

建议做法：

- function/class 默认 chunk
- 超大节点递归切块
- chunk 保留父节点路径与依赖摘要

### 3. task-aware packing

建议区分：

- `review`
- `plan`
- `work`
- `qa`

不同任务不同打包策略，而不是共用一套文件列表。

## P2：把 Stage-0 真正接成 compiled context compiler

### 1. 为 compiled context 提供 machine-first 入口

建议新增：

- `context-routing.json`
- `minimal-context/*.json`
- `freshness.json`
- `contradictions.json`
- `lint-report.json`

### 2. 将 workflow 消费结果纳入回路

建议记录：

- 这次任务读了哪些上下文
- 哪些证据有用
- 哪些证据缺失
- 哪些 pack 被降级

## P3：建立正式评测体系

### 1. Retrieval benchmark

建议指标：

- Recall@k
- MRR
- evidence recall
- blast-radius hit-rate

### 2. Repo QA benchmark

建议问题类型：

- 入口与调用链
- 数据流与状态更新
- 风险与副作用
- 测试与验证面

### 3. Review benchmark

建议问题类型：

- cross-file defect recall
- false positive rate
- high-risk module prioritization quality

### 4. Context efficiency benchmark

建议指标：

- 每类任务平均上下文 token
- 首个有效证据命中率
- 无关上下文比例
- 降级率

### 5. Longitudinal benchmark

建议指标：

- context freshness
- stale claim rate
- rebuild latency
- schema drift incident count

## 七、哪些事现在不该优先做

## 7.1 不要一上来就做 vector-first 大改

原因：

- 当前 lexical / graph / chunking 基础都还没打牢
- 没有 benchmark 时，向量检索很容易只是“看起来高级”

## 7.2 不要过早押注复杂 GNN 或 end-to-end 学习排序

原因：

- 当前数据规模、标注资产、评测体系都还不足
- 会显著提高实现复杂度和维护门槛

## 7.3 不要继续横向膨胀 narrative 文档

原因：

- 当前系统的主要瓶颈不是“文档太少”
- 而是“最小上下文分发和质量证明不足”

## 八、`CRG v2` 的验收标准应该是什么

如果只用一句话定义：

**同样一个仓库任务，`CRG v2` 应该能让 agent 用更少 token、更少无关文件、更高证据召回率，完成更稳的 plan / work / review。**

建议最小验收标准：

1. 相比当前版本，`review` 场景的 evidence recall 明显提升
2. `plan` 场景能更稳定命中入口、依赖、测试面
3. Stage-0 生成产物的 schema drift 明显下降
4. retrieval 消费过程可解释，可复盘
5. 索引构建失败或退化时，消费者仍能绑定到上一个稳定 generation

## 九、最终判断

从当前代码出发，`CRG` 的下一步正确方向不是“更复杂的图”，而是：

**更稳定的代际索引、更强的最小上下文检索、更明确的任务打包、更严格的评测闭环。**

如果按这个方向推进，那么：

- `CRG` 负责可信事实与检索底座
- Stage-0 负责上下文编译与持久知识层
- Workflow 负责任务消费
- Eval 负责证明价值

这四者就会形成真正闭环。

如果不按这个方向推进，而是继续只做：

- 更多节点
- 更多文档
- 更多启发式脚本

那么系统会越来越“看起来完整”，但很难真正成为团队级的代码库理解底座。
