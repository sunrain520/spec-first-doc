## 二、2026 年业界主流路线（已收敛）

### 2.1 从“单检索”到“多策略协同检索”

业界领先产品都不再押注单一路径（仅向量或仅关键词），而是采用组合策略：

- 语义检索（semantic）
- 文本检索（grep / keyword）
- 符号与引用检索（LSP / usages / references）
- 结构关系检索（code graph）

**证据**：

- VS Code Copilot workspace context 文档明确列出 semantic/text/grep/file/usages 多工具协同，并强调 agent 会多轮 follow-up 检索。  
  https://code.visualstudio.com/docs/copilot/reference/workspace-context
- Sourcegraph Cody context 文档明确列出 Keyword Search + Sourcegraph Search + Code Graph 三类来源。  
  https://sourcegraph.com/docs/cody/core-concepts/context

### 2.2 持久化索引 + 增量更新已成基础能力

“每次全仓重扫”已被视为不可扩展。领先方案的共识是：

- 首次全量建索引
- 后续按变更增量更新
- 把 time-to-first-query 作为核心指标

**证据**：

- Cursor（2026-01）公开技术细节：Merkle Tree + 分块 embedding 缓存 + 索引复用（含访问证明）。  
  https://cursor.com/blog/secure-codebase-indexing
- Copilot 文档强调语义索引持续维护，语义索引不可用时仍可退化到 text/grep/usages。  
  https://code.visualstudio.com/docs/copilot/reference/workspace-context

### 2.3 “图”不是替代检索，而是结构推理层

图结构的现实定位已经很清楚：

- 不替代所有检索
- 用于关系推理（依赖、调用链、影响半径、社区/模块边界）
- 与语义检索组合时收益最大

**证据**：

- Aider repo-map：基于 Tree-sitter 的仓库结构图 + graph ranking + token budget。  
  https://aider.chat/docs/repomap.html
- GraphRAG 开源仓库强调“图化组织 + 检索增强”，其定位是图式 RAG 管道。  
  https://github.com/microsoft/graphrag
- Codebase-Memory（2026-03）在代码场景给出“图查询在更低 token 下保持可比质量”的量化结果。  
  https://arxiv.org/abs/2603.27277

### 2.4 协议化集成（MCP）成为生态事实标准之一

MCP 的角色不是“更好检索算法”，而是“统一工具接入与上下文交换协议”。

**证据**：

- MCP 规范（2025-06-18）已形成版本化协议，并明确 JSON-RPC 2.0 约束、生命周期、授权框架。  
  https://modelcontextprotocol.io/specification/2025-06-18/basic/index
- MCP 官方仓库与 servers 仓库呈现大规模生态扩展。  
  https://github.com/modelcontextprotocol/modelcontextprotocol  
  https://github.com/modelcontextprotocol/servers

### 2.5 评测体系进入“防污染、重可解释”阶段

业界在 2025-2026 的一个关键转向：不再盲信单一 benchmark 数字。

**证据**：

- OpenAI 对 SWE-bench Verified 的发布（2024）与后续反思（2026）：明确指出 benchmark contamination 与测试有效性问题。  
  https://openai.com/index/introducing-swe-bench-verified/  
  https://openai.com/index/why-we-no-longer-evaluate-swe-bench-verified/
- SWE-bench 原论文本身定义了真实 GitHub issue 场景的复杂性（跨文件、长上下文、多步推理）。  
  https://arxiv.org/abs/2310.06770

## 五、业界最佳实践（可执行清单）

以下是对 `spec-graph-bootstrap` 最直接可落地的最佳实践：

1. **索引先行，检索后置**  
   先建设持久化索引（结构 + 语义），再做任务查询，避免“每任务重扫”。

2. **增量更新必须是一等公民**  
   使用文件哈希树/指纹依赖图，确保只重算受影响部分。

3. **检索策略必须多路并发**  
   semantic + grep + symbol + graph 并行尝试，按质量与延迟择优。

4. **图用于关系推理，不替代全部检索**  
   图优先承载 blast radius、callers、跨模块依赖、社区结构。

5. **上下文预算要可控**  
   为图摘要设置 token budget，并支持动态扩缩容（参考 `--map-tokens` 思路）。

6. **上下文契约要结构化**  
   输出 envelope/schema 固定；字段、枚举、错误码可测试。

7. **置信度与证据强绑定**  
   所有 `Inferred` 必须附 `inference_reason + evidence`，禁止“无证据高置信断言”。

8. **降级链必须显式且不阻断主任务**  
   图不可用时自动降级到最小上下文集合，并记录 `fallback_reason`。

9. **消费端与生产端解耦**  
   上下文生产（bootstrap）和消费（plan/work/review）通过路由契约连接，避免耦死。

10. **评测体系采用多指标而非单榜单**  
    同时看 token、成功率、误报/漏报、冷启动耗时、返工率。

11. **benchmark 污染要纳入治理**  
    引入真实任务回放与线上样本抽检，不仅依赖公开数据集。

12. **协议化接入优先**  
    对外工具优先走 MCP 这类标准协议，降低后续集成成本。

## 六、Karpathy「LLM Wiki」思想的整合启发（2026-04-04）

参考原文：`llm-wiki.md`（Karpathy Gist）  
https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f

这份思路对本方案的最大启发，不是“再做一个检索器”，而是把上下文工程从 **query-time 拼装** 升级成 **持续维护的知识编译层**。

### 6.1 核心思想（与本方案强相关）

1. **从 RAG 到 Compounding Wiki**  
   传统 RAG 每次问答都重新拼素材；LLM Wiki 强调把新信息持续“编译”进中间层知识页，形成可复用积累。

2. **三类操作面固定化：Ingest / Query / Lint**  
   - Ingest：吸收新来源并更新已有页  
   - Query：基于 wiki 回答问题，并可回写高价值答案  
   - Lint：周期性检查冲突、过期结论、孤岛页面、缺失链接

3. **双索引文件非常关键：`index.md` + `log.md`**  
   - `index.md`：内容导向的全局导航  
   - `log.md`：时间导向的演化轨迹（append-only）

4. **知识回写是复利开关**  
   高质量问答结果不应停留在聊天记录，而应沉淀为后续任务可直接消费的页面。

### 6.2 对 `spec-graph-bootstrap` 的具体映射

已具备的基础：

- Facts First（事实层）
- injection routing（消费层）
- fingerprints（增量层）

建议新增的“复利层”：

1. **在控制面新增知识维护资产**  
   - `index.md`（内容索引）  
   - `log.md`（时间索引）  
   - `contradictions.md`（冲突登记）  
   - `superseded.md`（被新证据覆盖的旧结论）  
   - `wip.md`（进行中假设与待验证结论）

2. **把 Lint 升级为正式阶段（可命令化）**  
   例如：`spec-first graph lint --repo=...`，输出：`orphan_pages`、`stale_claims`、`missing_cross_refs`、`conflict_claims`。

3. **给事实对象加生命周期字段**  
   建议在 fact item 增加：`claim_id`、`status(active|superseded|disputed)`、`last_verified_at`、`superseded_by`。

4. **把“查询结果回写”纳入验收指标**  
   除 token/耗时外，新增：`query_to_memory_writeback_rate`、`superseded_claim_half_life`。

### 6.3 与业界主流方案的差异化价值

如果吸收 LLM Wiki 思想，`spec-graph-bootstrap` 会从“上下文准备系统”进一步升级为“知识复利系统”：

- 不只解决“本次任务读什么”
- 还解决“上一轮高价值结论如何自动成为下一轮输入”

这与当前业界多数只做检索/索引的方案相比，是更高一层的产品壁垒。

### 6.4 反过度设计原则（维护复杂度优先）

在引入 LLM Wiki 机制时，需要明确“渐进式落地”，避免一次性把治理系统做重。

v1 建议只保留最小治理闭环（Must-have）：

1. `schema` 契约测试（产物结构可判定）
2. 每个产物具备 `owner + consumer + refresh_trigger`
3. `safe_crg_call` 降级不阻断主流程
4. 一个核心维护指标：`stale_claim_ratio`

v1 暂缓项（Defer to v2）：

1. `graph prune` 全自动清理
2. `graph doctor` 复杂诊断引擎
3. 细粒度生命周期状态机全量落地
4. 大规模多指标运营看板

升级到 v2 的触发条件（任一满足即可）：

1. 上下文产物数持续超过 20
2. 连续两周 `stale_claim_ratio` 超阈值
3. 出现 2 次以上错注入/漏注入导致返工

核心原则：先保证“可持续维护”，再扩展“可生成范围”。

## 七、对 `spec-graph-bootstrap` 的对标结论

### 7.1 已经与业界一致的点

- Facts First（结构化事实优先）
- 增量刷新（fingerprints）
- 路由注入（injection-index）
- 降级不阻断
- 图能力与 workflow 消费协同

### 7.2 建议进一步增强的点（按优先级）

#### P0（必须）

1. **指标体系落地到 CI 与验收门**  
   至少固定：`token p50/p95`、`fallback 命中率`、`review 漏检率`。

2. **检索编排策略显式化**  
   在文档中明确 semantic/grep/symbol/graph 的优先级与并发策略。

3. **输出契约自动校验**  
   所有关键产物走 schema 校验，失败即阻断发布。

#### P1（应做）

1. **预算调度器**（任务类型 → token budget）
2. **图查询热路径缓存**（impact、callers、test-map）
3. **真实任务回放评测集**（替代只看公开 benchmark）

#### P2（可选）

1. 引入 SCIP/LSIF 类协议做跨工具互操作层
2. 针对多仓库场景扩展跨 repo 检索（当前可后置）

## 八、风险与边界（诚实评估）

1. **厂商博客中的性能数字不可直接等同通用效果**  
   需要你们自己的仓库基准复测。

2. **GraphRAG 经验可借鉴，但其主战场是文档/知识图谱，不完全等于代码图**。

3. **MCP 解决的是“怎么接”，不是“接什么最有效”**  
   上下文质量仍取决于检索策略与消费设计。

4. **论文结论存在任务分布依赖**  
   需结合 `spec-first` 自身任务类型做二次验证。

---

## 九、一句话结论

> 2026 年业界最佳实践已经从“做一个能检索的 RAG”升级为“构建可持续的上下文工程系统”：持久化索引、增量更新、多策略检索、图关系推理、协议化接入、消费闭环验证。`spec-graph-bootstrap` 的方向是正确的，下一步关键是把这些最佳实践固化成可量化、可测试、可回归的工程契约。
