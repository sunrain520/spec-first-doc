## 一、先定义真实竞争维度

如果只看“谁能检索代码”，我们会误判战场。真正的竞争分三层：

| 层级 | 竞争问题 | 代表玩家 | 与 spec-graph-bootstrap 的关系 |
|---|---|---|---|
| **索引层** | 能否高效建立/更新代码索引 | Cursor、Copilot、Cody、Aider、Graph 工具 | 强能力重叠，但不是唯一战场 |
| **编排层** | 如何按任务类型选择最小必要上下文 | Aider（部分）、workflow agents（弱） | **核心差异化战场** |
| **治理层** | 如何验证消费有效、控制维护复杂度 | 业界普遍薄弱 | **潜在护城河战场** |

`spec-graph-bootstrap` 的定位应明确为：

> **“上下文编排与消费治理层”**，而不是另一个“代码检索器”。

---

## 二、业界基线（2026）与方案对齐度

结合调研结论，业界已收敛到以下共识：

1. 多策略检索（semantic + keyword + symbol + graph）优于单一路径。
2. 增量更新是可用性的前提，不是优化项。
3. 图用于关系推理（impact/callers/community），不替代全部检索。
4. 协议化（MCP）是生态接入事实标准之一。
5. 评估从“单 benchmark”转向“可解释、多指标、防污染”。
6. Karpathy 的 LLM Wiki 思想把重心从“生成”推进到“持续知识编译”。

`spec-graph-bootstrap` 与上述共识的匹配状态：

| 业界共识 | 当前方案状态 |
|---|---|
| 增量索引 | ✅ `crg build` + SHA 增量 + `fingerprints` |
| 图关系推理 | ✅ flows / communities / impact / review-context |
| 协议化消费 | ✅ CLI 契约 + 可接入 workflow（可进一步 MCP 化） |
| 可解释输出 | ✅ `confidence/source_tier/evidence/inference_reason` 契约 |
| 维护治理 | ✅ 已引入“反过度设计”最小闭环（v1） |

## 三、核心优势（更新后）

### 优势 1：从“索引工具”升级为“消费编排系统”

业界大多数方案停在“能检索到什么”；本方案额外定义“什么时候该加载什么、怎么降级、怎么验证效果”。

关键机制：

- `injection-index.yaml` 路由规则（stage × task_type × condition）
- `context-packs/*` 的任务导向结构
- `3A -> 3B` 消费固化门控

这不是功能堆叠，是层级上移。

### 优势 2：契约化程度高，减少隐性失败

阶段0方案已经把常见“看起来能用，实际错用”的问题前置治理：

- `symbol_key` + `source_id/target_id`，避免同名符号串线
- `crg query` 参数矩阵化（`--symbol/--module/--subject`），减少歧义调用
- 统一 JSON envelope + schema 约束
- `safe_crg_call` 保证故障可降级、不阻断主流程

这会显著降低 workflow 难以诊断的静默错误。

### 优势 3：新鲜度治理不是“建议”，而是契约

`fingerprints` 明确记录输入、版本、依赖关系，且把 stale 可见化。

竞争意义：

- 大多数方案“索引是否过期”对下游不可见
- 本方案把 freshness 变成可消费的运行时信号

### 优势 4：安装链路从“外部依赖”转向“内嵌能力”

阶段0将 CRG 能力内嵌 Node.js CLI（`spec-first crg ...`），不再依赖 Python + MCP 手工注册。

这降低了采用阻力，也提高了平台一致性。

### 优势 5：把“维护复杂度”纳入架构目标

吸收业界与 LLM Wiki 思路后，方案不再只追求生成能力：

- v1 最小治理闭环（契约测试、产物责任、降级不阻断、核心 stale 指标）
- v2 扩展需有触发条件，不提前引入重治理系统

这是“可持续交付”能力，不是锦上添花。

## 四、核心短板（诚实评估）

### 短板 1：IDE 原生实时感知仍弱

相较 Cursor/Copilot/Cody，当前方案主要是 workflow 级编排，不是编辑器内实时协同。

影响：

- 对“边写边补全”场景不是主解
- 对任务前/任务中上下文治理更强

### 短板 2：语义检索上限受限（v1）

v1 主体依赖结构化查询 + FTS5；embedding 语义检索未纳入主链路。

影响：

- “发现型”问题（模糊语义找相似实现）能力不如强 embedding 产品
- “结构型”问题（impact、entrypoint、risk）优势更明显

### 短板 3：优势仍需实证数据兑现

目前大量优势是“架构正确性优势”，必须通过真实项目数据转化为“实测优势”：

- token 降幅
- 错注入/漏注入率
- review 漏检率
- 返工率变化

### 短板 4：context-pack 质量是决定层

系统上限最终由“内容质量 + workflow prompt 消费质量”决定，而不是图层本身。

## 五、竞争力地图（更新）

将主要方案放到三维比较：

1. **索引与检索成熟度**
2. **消费编排与验证能力**
3. **维护复杂度可控性**

| 方案 | 索引/检索 | 编排/验证 | 维护可控 |
|---|---|---|---|
| Cursor / Copilot / Cody | 高 | 低-中 | 中（平台托管） |
| Aider | 中-高 | 中 | 中 |
| Continue（平台） | 中 | 中 | 中 |
| Graphify / CodePrism / Graph 工具 | 中-高 | 低 | 低-中 |
| **spec-graph-bootstrap（目标态）** | 中-高 | **高** | **高（若治理落实）** |

结论：

- 不是在“谁索引更快”上硬碰硬
- 竞争焦点应放在“编排正确性 + 维护经济性”

## 六、反过度设计视角下的竞争策略

### 6.1 v1 必须做（最小治理闭环）

1. 契约先行：schema 测试作为发布门禁
2. 产物三元信息：`owner + consumer + refresh_trigger`
3. 故障不阻断：`safe_crg_call` 强制降级链
4. 单一核心维护指标：`stale_claim_ratio`

### 6.2 v1 暂缓（避免治理先于价值）

1. 全自动 `prune`
2. 重型 `doctor` 诊断体系
3. 全量生命周期状态机
4. 多指标运营大盘

### 6.3 升级到 v2 的触发条件

任一命中即升级：

1. 产物规模长期 > 20
2. 连续两周 stale 超阈值
3. 错注入/漏注入导致返工 >= 2 次

这保证架构演进“按压力升级”，而不是“按想象升级”。

## 七、90 天竞争执行建议

### 7.1 先把设计优势转成实测优势

必须建立对照实验（至少 2 个真实项目）：

1. 有/无 `injection-index` 的 token 与耗时对比
2. 有/无 `context-pack` 的 review 漏检率对比
3. stale 可见化前后返工率对比

### 7.2 输出一套可复用的行业资产

1. “任务路由即代码”规范样例
2. 3A/3B 验证记录模板
3. 最小治理闭环检查清单

这三项可以成为对外差异化资产，不依赖单一模型能力。

### 7.3 与生态对齐而非封闭竞争

中期建议：

- 保持 `spec-first crg` CLI 主链路
- 同时提供 MCP/平台适配层，让 Continue/OpenHands/Codex 可消费

目标是做“上下文编排中间层”，不是绑定单一宿主。

---

## 八、一句话竞争力定位（更新）

> 业界已基本解决“怎么索引代码”，`spec-graph-bootstrap` 的真实竞争力在于解决“如何把上下文按任务正确路由、按置信度正确消费、按维护成本可持续演进”，并用可验证数据证明它确实提升了 workflow 质量。
