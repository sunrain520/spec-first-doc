# 胶水编程对 spec-graph-bootstrap v2 的设计启发

> 本文基于文章《97.9%采纳率，胶水编程：业务需求出码最佳实践【天猫AI Coding实践系列】》，提炼其对 `spec-graph-bootstrap v2` 设计的直接启发。
> 重点不是复述文章内容，而是回答：这篇文章对 Stage-0 设计意味着什么。

来源：<https://mp.weixin.qq.com/s/G3aKbzdGUyD2h1aVjvbr2g>

---

## 1. 核心结论

这篇文章最重要的观点可以压缩成一句话：

> 别让 AI 写代码，让它抄代码。

它不是否定 SPEC，而是指出：

- SPEC 解决“这次做什么”
- 胶水编程解决“怎么按团队已有方式做出来”

真正高质量的 AI 编码，不是不断优化 prompt 让模型更会原创，而是让模型：

1. 遵守团队规则
2. 命中已有样板
3. 消费内部领域知识
4. 只在业务差异点写最少量胶水代码

这和 `spec-graph-bootstrap v2` 的方向高度一致。

---

## 2. 文章提供的四层物料模型

文章将 AI 编码所需物料拆成四层：

1. `任务规格`
2. `开发规范`
3. `代码模式`
4. `领域知识`

这四层不是相互替代，而是分别堵不同的漏洞：

- 任务规格解决“做什么”
- 开发规范解决“什么不能做错”
- 代码模式解决“最该抄什么”
- 领域知识解决“公开资料学不到的内部规则和经验”

文章的关键判断是：

> 仅有任务规格，远远不够支撑高采纳率和高一致性。

---

## 3. 为什么它对 spec-graph-bootstrap v2 很关键

`spec-graph-bootstrap v1` 的主要强项是项目认知：

- summary
- architecture
- module map
- pitfalls
- layers
- database

这些文档有价值，但如果用胶水编程的标准看，会发现它更接近“四层物料”中的背景认知，而不是执行物料。

也就是说，v1 更擅长回答：

- 项目长什么样
- 模块怎么分
- 哪些地方有风险

但不够强地回答：

- 什么不能写错
- 最该抄什么
- 哪些内部模式是团队真正稳定使用的

这正是 v2 需要补的部分。

---

## 4. 对 v2 的直接设计启发

## 4.1 `rules/` 必须独立成层

文章里“开发规范”是静态注入、始终在场的底线规则。

这对 `spec-graph-bootstrap v2` 的启发很直接：

- 规则不能散在 `architecture/*` 或 `pitfalls/*` 里
- 规则必须独立成一级资产
- 后续节点应该能直接定位“这个项目的硬约束”

因此，v2 新增 `rules/` 是必要的，不是锦上添花。

## 4.2 `patterns/` 必须独立成层

文章最有价值的观点之一是：

> 代码模式比抽象规则更能稳定提升采纳率。

它强调“样板间”必须是真实代码，不是泛化描述。

这对 v2 的启发是：

- `patterns/` 不应该只是“推荐看看某目录”
- 应该尽量沉淀：
  - 最像的参考实现
  - 标准骨架
  - 哪些部分可替换
  - 哪些部分不要改

## 4.3 文档必须从介绍型升级为执行型

文章反复强调，好的上下文不是“更多信息”，而是“最有用的信息”。

所以 v2 文档应该优先回答：

- 遇到什么任务先看我
- 最像的实现在哪
- 先读哪些规则
- 先改哪些文件

这比单纯解释架构更能提升研发质量。

## 4.4 研发质量目标应显式包含“降低原创比例”

文章很重要的一点是，它把“低原创率”视为高质量信号之一。

对 v2 来说，这意味着成功标准不应只看：

- 文档覆盖度
- 文档结构完整度

还应看：

- 是否帮助后续节点减少无必要原创
- 是否帮助后续节点更容易命中参考实现

## 4.5 领域知识不应只被理解为“坑”

文章指出，领域知识的大头并不是踩坑经验，而是：

- 内部组件文档
- 平台约束
- 业务字段和状态语义
- 只有内部人才知道的使用方式

这对 v2 的启发是：

- `pitfalls/` 不能承载所有领域知识
- 领域知识应拆散进入：
  - `rules/`
  - `patterns/`
  - `decisions/`
  - 部分 `pitfalls/`

## 4.6 需求规格与 Stage-0 不是替代关系

文章没有否定 SPEC，而是指出 SPEC 不是银弹。

这和 v2 的边界一致：

- Stage-0 不替代后续 `plan`
- Stage-0 负责提前提供规则、样板、知识、风险、决策
- 后续 `plan` 负责把“这次差异”说清楚

因此，v2 的定位应始终是：

> 为 SPEC 和后续研发节点提供高质量上游资产，而不是替代它们。

## 5. 对 v2 产物模型的补强建议

基于这篇文章，我认为 v2 产物模型至少应强化以下几点：

### 5.1 `patterns/` 的要求应更具体

每个 patterns 文档都应尽量包含：

- `Applicable Scenarios`
- `Recommended References`
- `Standard Skeleton`
- `Replaceable Parts`
- `Stable Parts`
- `Common Mistakes`

### 5.2 `rules/` 应区分强弱

建议在规则文档中显式区分：

- `Must`
- `Should`
- `Avoid`

这样后续节点更容易判断优先级。

### 5.3 `guides/task-playbooks.md` 应更重要

如果胶水编程的核心是“按任务类型命中合适物料”，那么 `task-playbooks` 就不是可有可无的补充，而是把：

- `rules`
- `patterns`
- `risks`
- `layers`

真正串起来的入口。

### 5.4 `decisions/` 应承担一部分“规格持久化”价值

文章提到 Track 和规格同步的重要性，本质上是在强调：

- 每次需求的决策不要在交付后消失

对 v2 来说，这意味着 `decisions/` 不应只是架构史，而应逐步承接高价值历史取舍。

---

## 6. 对 v2 worker 与 PRD 的补强建议

这篇文章也会影响控制面设计。

## 6.1 `patterns-context` 的地位应再提高

如果“最该抄什么”是质量提升的核心，那 `patterns-context` 就不应只是一个普通 worker，而应是 v2 固定 worker 中最重要的之一。

## 6.2 PRD 必须要求产出“最近似参考实现”

仅要求 worker 写章节还不够。
PRD 应显式要求：

- 至少给出高价值参考路径
- 说明为什么它是最近似实现
- 说明哪些部分可复用、哪些部分不能照抄

## 6.3 PRD 必须要求“任务入口”

这和文章里的四层物料协作方式直接对应。

worker 最终产物必须帮助后续节点回答：

- 我遇到这种任务先看哪里
- 然后读哪条规则
- 然后抄哪个样板

---

## 7. 对成功指标的启发

文章对 v2 成功指标有两个很重要的提醒：

## 7.1 不要只看“生成了多少”

更重要的是看：

- 是否降低了无必要原创
- 是否提高了模式复用率
- 是否减少了 review 中的模式偏差

## 7.2 要把“命中样板”纳入质量指标

如果后续阶段仍然经常从零写，说明 Stage-0 的 `patterns/` 做得不够好。

因此建议将以下指标显式纳入：

- 参考实现命中率
- review 中“风格/模式不符”问题占比
- 后续 plan 中内联样板代码长度

---

## 8. 最终判断

这篇文章对 `spec-graph-bootstrap v2` 最大的价值，不是提供了一个局部技巧，而是再次证明了：

> 提升研发质量，关键不只是“让 AI 更会理解”，而是“让 AI 拿到更好的团队资产”。

落到 v2 设计上，就是：

1. `rules/` 必须独立
2. `patterns/` 必须独立
3. 文档必须任务导向
4. 成功指标必须包含“降低原创比例”
5. Stage-0 必须为后续节点准备可直接消费的执行资产

这篇文章本质上是在外部实践层面，为 `spec-graph-bootstrap v2` 的设计方向提供了一次很强的验证。
