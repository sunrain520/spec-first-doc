# 组织级 Coding Agent 架构与 spec-first 体系对照分析

> 基于四篇文章综合：微信推文《Coding Agent 下半场》+ LangChain Open SWE 博文 + AgentScope Harness 深度搜索结果 + Open SWE 生态搜索结果

## 一、组织级 Coding Agent 全景

### 1.1 三家标杆对比

Stripe、Ramp、Coinbase 在 2025 年底到 2026 年初几乎同时公开了各自的内部 Coding Agent。三家独立开发、无互相参考，最终却不约而同收敛到几乎相同的架构。LangChain 将此总结为："Isolate first, then give full permissions inside the boundary."

| 维度 | Stripe Minions | Ramp Inspect | Coinbase Cloudbot |
|------|---------------|-------------|-------------------|
| **Harness** | Forked (Goose) | Composed (OpenCode) | Built from scratch |
| **沙箱** | AWS EC2 devboxes (pre-warmed) | Modal containers (pre-warmed) | In-house 自建基础设施 |
| **工具数** | ~500, curated per-agent | OpenCode SDK + extensions | MCPs + custom Skills |
| **上下文** | Rule files + pre-hydration | OpenCode built-in | Linear 上下文 + MCPs |
| **编排** | Blueprints (确定性 + Agent 驱动) | Sessions + child sessions | Three modes（三种运行模式） |
| **入口** | Slack + embedded buttons | Slack + web + Chrome extension | Slack-native |
| **验证** | 3-layer (local + CI + 1 retry) | Visual DOM verification | Agent councils + auto-merge |

**核心洞察**：三家在六个维度上的趋同绝非巧合——沙箱隔离、工具精选、Slack 入口、子 Agent 编排、启动时满注上下文、Draft PR 为输出契约，这些已成为组织级 Coding Agent 的行业共识。

### 1.2 Open SWE — LangChain 提炼的开源框架

**定位**：把 Stripe/Ramp/Coinbase 的共同模式提炼为 MIT 许可的开源框架，不作为开箱即用的产品，而是可组合的起点。

**底层**：Deep Agents（Agent Harness）+ LangGraph（编排层）。

**核心设计理念**："Isolate first, then give full permissions inside the boundary"。

**六大组件**：

| 组件 | 能力 |
|------|------|
| Agent Harness | 基于 Deep Agents 组合而非 Fork，保留升级路径 |
| Sandbox | 可插拔（Modal / Daytona / Runloop / LangSmith），每 session 独立容器 |
| Curated Tools | ~15 个核心工具（execute / read_file / write_file / commit_and_open_pr 等），强调"tool curation > tool quantity" |
| Context Engineering | 双层：AGENTS.md（仓库级约定）+ Linear issue / Slack thread（任务级上下文） |
| Subagents + Middleware | task tool 派生子 Agent；middleware 实现消息注入、自动提 PR 等确定性逻辑 |
| 入口 | Slack（@机器人 + `repo:owner/name`）、Linear（评论 `@openswe`）、GitHub（PR 评论） |

**Draft PR 作为输出契约**——Agent 不直接改生产代码，永远需要人类 review 后才能 merge。这是组织级 Coding Agent 的基本安全假设。

### 1.3 AgentScope Harness — Java 生态实现

**定位**：AgentScope Java 2.0 的组织级 Coding Agent 模块，业界首个面向 Java 的智能体工程化框架。

**四个演进阶段**：

| 阶段 | 能力 | 适用场景 |
|------|------|---------|
| Stage 1: CLI | 本机文件系统 + 本地状态存储 + REPL 交互 | 开发验证 |
| Stage 2: Docker 沙箱 | `.filesystem(new DockerFilesystemSpec()...)` 一行切换 | 单机安全隔离 |
| Stage 3: 多副本分布式 | Redis 状态存储 + OSS 快照 + executionGuard 并发控制 | 团队服务 |
| Stage 4: 可观测+限流 | Prometheus 指标 + ThreadBudgetHook + ModelCallLimitHook | 生产就绪 |

**核心工程问题及解法**：

- **沙箱隔离**：FilesystemSpec 统一接口，Docker 容器 / 远端 KV / 本机文件系统可插拔，IsolationScope.SESSION 粒度隔离
- **会话恢复**：快照机制（LocalSnapshotSpec / OssSnapshotSpec / RedisSnapshotSpec），沙箱状态跨进程恢复
- **上下文窗口管理**：四套独立可组合机制——对话摘要压缩（triggerMessages=50, keepMessages=20）、大工具结果卸载（>80K 字符落盘 + 首尾 2K 占位符）、参数截断（maxArgLength=2000）、溢出兜底（context_length_exceeded 紧急压缩重试）
- **长期记忆**：MEMORY.md 周期性合并，Agent 自动学习团队规范（如测试命令、分支命名约定、CI 配置位置）
- **状态持久化**：RedisAgentStateStore，节点崩了会话漂移无缝恢复，零停机滚动发布
- **多通道适配**：GitHub Issue / 飞书 / 钉钉 → SHA-256 → UUID → Agent thread，确定性映射保证同一 issue 路由到同一 session
- **多租户隔离**：IsolationScope（SESSION / USER），会话状态、记忆、子 Agent 任务同粒度隔离
- **并发控制**：RunDispatcher + MessageQueueHook（单 thread 单推理）+ ThreadBudgetHook + ModelCallLimitHook（防预算烧穿）
- **Context Engineering**：workspace 目录体系（AGENTS.md / MEMORY.md / skills/ / subagents/ / knowledge/ / plans/），全部以 Markdown/JSON 表达、可 Git 版本化管理
- **Plan Mode**：只读→写计划→人确认→再动手，与 Coinbase Agent Councils 理念一致
- **子 Agent**：Markdown 声明式定义，agent_spawn 派发，支持同步阻塞与后台委派

**R.E.S.T 模型**——Harness 工程的核心价值维度：
- **R**eliability（可靠性）：检查点恢复、操作幂等、行为一致性
- **E**fficiency（效率）：Token 预算管控、上下文智能规约、资源配额
- **S**ecurity（安全性）：最小权限、沙箱隔离、全链路过滤
- **T**raceability（可观测性）：全链路追踪、决策归因、状态审计

## 二、组织级 vs 个人级：核心差异矩阵

Claude Code 是私家车（信任驾驶员），组织级 Coding Agent 是出租车运营车队（需要行车记录仪、GPS、里程限制、紧急制动）。

| 维度 | 个人级（Claude Code / Cursor） | 组织级（Minions / Open SWE / Harness） |
|------|-------------------------------|----------------------------------------|
| **触发者** | 自己 | 团队任意成员（Slack / Issue / IM） |
| **执行环境** | 本机 | 远端沙箱容器 |
| **信任边界** | 信任本机 | 隔离优先，再放权 |
| **运行模式** | 交互式（盯着改） | 异步（扔出去等结果） |
| **多租户** | 不需要 | 核心需求 |
| **并发** | 单会话 | 几十个 Issue/PR 并行 |
| **上下文管理** | 单会话够用 | 必须防窗口爆炸 |
| **状态持久化** | 本地文件 | 分布式 Redis / OSS |
| **入口** | CLI | Slack / GitHub / 飞书 / 钉钉 |
| **产出形式** | 直接改文件 | Draft PR（人类 review 后 merge） |
| **预算控制** | 自控 | ThreadBudget + 全局额度 |
| **长期记忆** | 不跨会话 | 跨会话累积学习 |

**本质区别**：个人级解决"我一个人写代码更快"；组织级解决"扔给 Agent 跑完开 PR 我 review 就行"——触发者未必是执行者，执行发生在远端，从不盯着看到收到 Draft PR 是异步闭环。

## 四、关键差距深度分析

### 4.1 "单人工具" vs "组织基础设施"

spec-first 当前本质上是一个个人提效工具——一个开发者本地运行，走七阶段生命周期（Specify → Clarify → Plan → Work → Review → Archive → 完成）。这套流程在单人场景下运转良好，但组织级 Coding Agent 追问的是完全不同的工程问题：

- 多用户同时触发怎么隔离？—— spec-first 无多租户概念
- 长任务中断后怎么恢复？—— spec-first 依赖本地文件，无分布式状态管理
- 上下文窗口爆了怎么办？—— spec-first 有上下文文件但无主动防护机制
- 模型预算怎么控制？—— spec-first 无预算维度设计

这些 spec-first 当前都未涉及，因为它的设计出发点就是"一个人在终端里用"。

### 4.2 "同步交互" vs "异步委托"

spec-first 的交互模型是"你说我做，你盯着我做完"——用户在 CLI 里看着 Agent 逐步执行、随时打断纠正。组织级场景是"扔一个 Issue，过半小时收到 draft PR，中间不需要盯着"。

这要求 Agent 具备：
- **自主规划能力**：不仅知道做什么，还要能在无监督下推进到底
- **自动验证能力**：跑测试、过 lint、CI 检查，出错自行修复
- **兜底 middleware**：关键步骤（如开 PR）用确定性逻辑保证，不依赖模型"记得做"

### 4.3 Context Engineering 差距

spec-first 有上下文文件（spec.md / plan.md / decisions.md / tasks.md），但缺乏组织级的 AGENTS.md + MEMORY.md + skills/ + knowledge/ 体系。

组织级 Coding Agent 的 workspace 是一个**可版本化管理、跨会话累积、团队共享**的知识库：

```
workspace/
├── AGENTS.md        ← 人格 + 行为约定（团队规范以文件生效，不硬编码在 prompt）
├── MEMORY.md        ← 长期记忆（自动从对话流水账合并事实）
├── skills/          ← 可复用 SOP（提交规范、测试规范等）
├── subagents/       ← 子 Agent 声明（Markdown 即配置）
├── knowledge/       ← 领域知识（API 文档、代码规范）
└── plans/           ← Plan Mode 计划文件
```

Claude Code 有 CLAUDE.md，GitHub Copilot 有 `.github/copilot-instructions.md`，Open SWE 有 AGENTS.md——repo 级的规约不应该硬编码在 system prompt 里，而应该是文件，能版本化、能 CR、能独立更新。三个工程价值：
1. 团队规范以文件形式生效，不用重启、不用改代码
2. Agent 在用的过程中越来越懂团队（所有同 workspace 对话受益）
3. workspace 当 Git 管理——频繁变化的应该是文件内容，不是 Java/Python 代码

### 4.4 Plan Mode 对齐

AgentScope Harness 的 Plan Mode（只读→计划→确认→执行）与 spec-first 的 Plan→Work 阶段高度相似，但有一个关键差异：**spec-first 的 Work 阶段缺乏"人确认"这一关键节点**。

Coinbase 的"Agent Councils"理念——在高风险操作前加入人类审批节点——值得 spec-first 借鉴。Plan Mode 的实质是用流程约束代替"祈祷模型别出错"，而 spec-first 当前在 Work 阶段是 Agent 全自主执行，没有内置的审批闸门。

### 4.5 交付与运维缺失

spec-first 止于 Review 阶段（代码审查），缺少：
- **Ship**：git commit → tests → push → PR（Gstack 的 /ship）
- **Deploy**：合并 → CI → 生产验证（Gstack 的 /land-and-deploy）
- **Canary**：SRE 金丝雀监控（Gstack 的 /canary）
- **可观测性**：Prometheus 指标、全链路追踪、预算控制

组织级 Coding Agent 的 Draft PR 输出契约——Agent 产出永远是 draft PR，永远需要人类 review 后才能 merge——这是 spec-first 需要补的关键一块。当前 spec-first 的 Review 阶段是"检查代码质量"，但缺少"产出 PR → 人类 review → merge"这个闭环。

## 六、分级改进建议（聚焦"扩展到组织级"）

### P0 — 应立即借鉴

**1. 引入 AGENTS.md + MEMORY.md 机制**

让 spec-first 的 Skill 能从项目仓库读取团队规范，跨会话积累长期记忆。

- AGENTS.md：仓库根目录的团队约定文件，含测试命令、分支命名、CI 配置、架构决策等。替代硬编码在 system prompt 中的规约，支持 Git 版本化和 CR。
- MEMORY.md：Agent 从每次对话中周期性合并长期事实。"这个项目的测试命令是 `npm test -- --coverage`"、"main 分支受保护必须 PR 合并"——Agent 自动学会，不再重复问。

**2. 引入 Draft PR 输出契约**

Work 阶段产出不应是"改了代码"，而应是"开了 Draft PR 等你 review"。这是组织级 Coding Agent 的基本安全假设——Agent 不直接改生产代码，永远需要人类 merge gate。

### P1 — 应在下个迭代引入

**3. Work 阶段加入人类审批节点**

对齐 Plan Mode（先计划→人确认→再动手），不对齐 Coinbase 的 Agent Councils。在 Work 阶段的高风险操作（重构、修改核心模块、跨文件联动修改）前插入人类确认节点，spec-first 现有的 Plan→Work 结构天然适合嵌入这个闸门。

**4. 新增 Ship phase**

覆盖 git → tests → push → PR → 部署管道。对齐 Gstack 的三层交付链（/ship → /land-and-deploy → /canary），让 spec-first 从"代码生成"延伸到"交付上线"。这是从个人工具升级为团队基础设施的关键一步。

**5. 上下文窗口防护**

借鉴 AgentScope Harness 的大工具结果卸载 + 参数截断机制：
- 工具结果 > 80K 字符自动落盘，上下文只留首尾占位符 + 文件路径提示
- write_file 大入参截断（内容已写盘，无需上下文重复持有）
- 消息超阈值时自动触发对话摘要压缩

防止长任务上下文溢出导致任务失败。

### P2 — 中长期探索

**6. 多通道入口适配**

让 spec-first 能接住 Slack / 飞书 / GitHub Issue 的触发。参考 Harness 的通道适配器模式：GitHub Issue / 飞书 / 钉钉 → SHA-256 → UUID → Agent session，确定性映射保证同一对话路由到同一上下文。

**7. 沙箱隔离执行**

Work phase 在容器内执行，隔离 blast radius。"Isolate first, then give full permissions inside the boundary." 本机文件系统足够单人开发，但面向团队时必须让每次执行发生在可丢弃的隔离环境内。

**8. 多租户/并发架构**

让 spec-first 从单人工具升级为团队服务。核心需求：
- IsolationScope：每 session 独立沙箱 + 独立状态 + 独立记忆
- RunDispatcher：同一 thread 同一时间只跑一个推理
- ThreadBudgetHook + ModelCallLimitHook：防单用户失控烧穿全公司额度

**9. 可观测性建设**

Prometheus 指标 + 预算控制 hooks + 全链路追踪。组织级服务必须能回答"当前有多少任务在跑？预算消耗了多少？哪些任务失败率最高？"

## 七、结论

**组织级 Coding Agent 架构对 spec-first 扩展的核心启发**：spec-first 已经解决了"从想法到代码"的研发内循环——七阶段生命周期覆盖了需求澄清、方案规划、编码实现、代码审查的完整链路。但从 Stripe/Ramp/Coinbase 三家独立趋同的实践来看，真正的组织级体系还需要补齐三块拼图：**异步委托的运行模式**（从"盯着改"到"扔 Issue 等 PR"）、**隔离安全的执行底座**（沙箱+预算+多租户）、**交付运维的全链路闭环**（从 Draft PR 到上线监控）。spec-first 的七阶段生命周期不是要被替换，而是需要向外延伸——从研发内循环扩展为研发到交付的全循环。
