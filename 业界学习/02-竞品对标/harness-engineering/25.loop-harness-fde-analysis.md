# 报告 #25：Agentic Engineering 三层体系深度分析 —— Loop工程、Harness工程、FDE

> **源材料**：智能体AI 公众号《一文搞懂Loop工程、Harness工程、FDE——Agentic Engineering的三层体系》（2026-06-13）
> **分析方法**：spec-first 方法论对照框架
> **生成日期**：2026-06-15

---

## 1. 项目概述

### 1.1 文章核心主张

文章从"你还手动喂Prompt吗"这一尖锐提问切入，系统梳理了 AI 工程从 2023 到 2026 年的五层演化路径：**Prompt工程 → Context工程 → Harness工程 → Loop工程 → FDE（Forward Deployed Engineering）**。其核心主张可概括为：

> **"把 Prompt 之外的工程问题工程化"——当模型的单次表现已被 Prompt 和 Context 调至高位后，真正的瓶颈不再是"怎么问得更好"，而是"怎么让 Agent 在真实环境里持续、可靠、自主地跑起来"。**

### 1.2 五层体系概述

| 层级 | 解决的问题 | 核心产出 |
|------|-----------|----------|
| **Prompt工程** | "怎么问"——优化单次指令质量 | 高质量 prompt 模板、few-shot 示例 |
| **Context工程** | "让模型看到什么"——信息环境管理 | RAG 管道、记忆机制、工具描述 |
| **Harness工程** | "在什么条件下干活"——运行环境搭建 | 工具/约束/反馈回路/持久化状态 |
| **Loop工程** | "怎么自己找活干"——自驱动系统设计 | 触发机制 + 可验证目标 + 创作者-检验者分离 |
| **FDE** | "怎么搬进客户地盘"——企业落地摩擦消除 | 数据权限打通、遗留系统集成、组织认知建立 |

### 1.3 Harness / Loop / FDE 三层定义区分

- **Harness工程**：Agent 的运行外壳。用确定性约束（Linter、测试套件、权限边界、审批门控）取代概率性合规，让 Agent "不可能做错"而不只是"希望它做对"。公式：**Agent = Model + Harness**。
- **Loop工程**：在 Harness 之上加装自驱动能力。工程师从"逐轮提示者"转变为"设计循环系统的人"。核心构件：自动化触发、创作者-检验者分离、连接器、隔离工作区、持久化记忆。
- **FDE**：前向部署工程。企业 AI 项目从 PoC 走向生产的最后一公里——数据权限审批、遗留系统对接、合规要求满足、业务方认知建立。95% 的企业 AI 项目卡在此层。

## 2. 架构分析

### 2.1 五层演化全景图

```
Prompt工程 ──→ Context工程 ──→ Harness工程 ──→ Loop工程 ──→ FDE
  "怎么问"       "看到什么"      "干活的环境"     "自己找活干"    "搬进客户地盘"

  无状态         有状态          确定性强约束     自驱动系统      组织级落地
  单次优化       信息环境         工具+验证+      触发+目标+      权限+遗留系统+
                管理             持久化          并行化          合规+认知
```

**关键依赖关系**：每一层依赖内层作为基础，但内层做到极致也无法替代外层。Harness 再完善，没有 Loop 系统就是被动的；Loop 设计再精妙，数据权限没打通就跑不起来；FDE 推进再快，底层 Harness 不可靠则问题越多。

### 2.2 Harness工程：把"概率性合规"变成"确定性约束"

Harness 这个词源于"挽具"——套在马身上传递力量和控制方向的那套装备。由 Mitchell Hashimoto（HashiCorp/Terraform 联合创始人）于 2026 年初正式命名，随后被 OpenAI 的 Ryan Lopopolo 推向大众。

#### 三层核心结构

| 结构层 | 功能 | 关键实践 |
|--------|------|---------|
| **工具与执行层** | 读写文件、调用 Bash、访问 API、操作浏览器 | 工具定义清晰度决定行为可预期性 |
| **控制与验证层** | 测试套件、类型检查、权限边界、审批门控 | 把"希望它这样做"变成"它不可能那样做" |
| **持久化与状态层** | 文件系统 + Git 持久化状态 | 解决长周期任务 AI 失忆问题 |

**设计哲学**：告诉 Agent "请遵守代码规范"是概率性合规（它可能遵守也可能不遵守）；接入一个违反规范时直接阻断 PR 的 Linter 是确定性结构约束。Harness 工程的核心动作是：**把错误类型在结构上直接关掉，而非降低犯错概率**。

#### 案例：Stripe Minions

在 Ruby 代码库上围绕 LLM 编码 Agent 构建完整 Harness。关键流程：

1. Agent 在隔离 devbox 中运行
2. pre-push 阶段自动跑 Linter
3. 从 300 万个测试中精选 CI 执行
4. 任何失败直接反馈给 Agent 自动纠正
5. 通过后才提 PR，人工仅负责审查

**成果**：每周在无人值守下合并超 1000 个 PR。这是 Harness 将"概率性合规"换成"确定性约束"后规模化可达的样子。

### 2.3 Loop工程：自驱动系统设计

Loop工程的核心转变——**把工程师从"逐轮提示 Agent 的人"替换成"设计让系统自动提示 Agent 的系统的人"**。

#### Loop 的两个核心元素

| 元素 | 要求 | 反例 |
|------|------|------|
| **触发器** | 定时（cron）、事件驱动（新 PR → 代码审查）、目标驱动 | 无触发器的 Loop 退化为手动脚本 |
| **可验证目标** | 有明确通过/失败信号（如"所有 CI 通过"） | "帮我优化一下功能"——Agent 无法判断何时算完成 |

#### Loop 的五个核心构件

| 构件 | 功能 | 未实现的后果 |
|------|------|-------------|
| **自动化触发** | 发动机，让 Loop 周期性或事件性启动 | Loop 退化为手动脚本 |
| **创作者-检验者分离** | 独立验证 Agent 评估结果，防止"模型同意自己"幻觉 | 错误得不到纠正 |
| **连接器** | 打通 GitHub/Jira/Slack，感知外部世界 | Loop 成为信息孤岛 |
| **隔离工作区（Worktrees）** | 多 Agent 并行操作同一代码库不污染 | 无法规模化并行 |
| **持久化记忆** | 历史轨迹、失败记录写进可检索存储 | 每次从零开始，成本极高 |

#### 案例：Steve Yegge 的 Gas Town 项目（2026年1月）

- **架构**：一个"Mayor" Agent 协调 20-30 个 Claude Code 实例
- **巡逻 Agent**：持续循环扫描代码库健康状态
- **工作 Agent**：接收巡逻 Agent 发现的问题去修复
- **状态管理**：状态存在 Git 中，崩溃重启后从断点续跑
- **人的角色**：审查 PR + 系统卡住时踢一脚

#### 现实代价警示

| 代价类型 | 具体表现 |
|---------|---------|
| **成本** | 单 Agent Loop 消耗 token 为普通对话 4 倍，多 Agent 约 15 倍；大规模团队月度 token 账单可达七位数 |
| **适用范围** | 仅适用于"高频重复 + 结果可验证 + 工具链已接通"场景；目标模糊的 Loop 可在几小时内烧掉几百美元 |
| **认知负债（Cognitive Surrender）** | Loop 跑得越好，工程师越不知道系统在干什么；关键路径人工审查的价值在出问题前难以感知 |

### 2.4 FDE：企业落地摩擦层

前三层解决系统内部问题，FDE 解决系统与真实世界的接触面问题。95% 的企业 AI 项目卡在落地阶段——不是模型不够好，而是系统性摩擦无人解决。

FDE 角色由 Palantir 于 2003 年发明（当时叫 Delta），将精英工程师派驻军事基地和客户现场。2026 年，OpenAI、Anthropic、Cohere、Databricks 的 FDE 团队已成标配。

#### FDE 四种能力

| 能力 | 说明 |
|------|------|
| **生产级工程能力** | 写真实代码，建真实集成，向生产环境交付——不是 Demo，不是 PPT |
| **领域翻译能力** | 在 CFO 的业务问题与 DBA 的数据约束之间架桥 |
| **Agent 落地专项** | RAG 管道、Eval 框架、生产可观测性——2026 年 FDE 必考项 |
| **产品反哺能力** | 客户现场的集成难题和部署模式反向影响产品路线图 |

## 3. 逐维度对照矩阵（spec-first 方法论基准）

以 **spec-first 方法论**（需求规格驱动、结构化文档、人工闸门、分阶段交付）为基准，对照本文五层体系：

| 维度 | spec-first 方法论 | 本文五层体系 | 对照分析 |
|------|------------------|-------------|---------|
| **Prompt工程** | spec-first 无直接对应层，起点即结构化文档 | 本体系的起点层，"怎么问" | spec-first 跳过单次 prompt 优化，直接从规格化需求起步 |
| **Context工程** | spec-first 的 `docs/` 目录体系 + 前置分析报告链 | Context Pack + RAG + 工具描述 | 功能同构：都解决"AI 看到什么信息"的问题；spec-first 用结构化文档目录，本体系用 RAG 管道 |
| **Harness工程** | spec-first 的工具链定义（`tools/`）+ 验证规则 | 工具/约束/反馈回路工程化 + 确定性约束哲学 | **核心对齐点**：spec-first 的工具链定义可视为 Harness 层雏形，但缺少"把概率性合规变确定性约束"的设计自觉 |
| **Loop工程** | spec-first 的迭代循环（"分析→设计→开发→审查"）+ 人工闸门 | 自驱动系统设计 + 人退出逐轮循环 | **最大分歧点**：spec-first 每一步设人工闸门，Loop 工程追求人退出循环；本质是两种不同的风险模型 |
| **FDE** | spec-first 的组织落地（团队规范、交付流程） | 企业级摩擦处理（权限/遗留系统/合规/认知） | spec-first 侧重方法论层面的落地规范，FDE 侧重物理层面的集成与部署——**FDE 覆盖了 spec-first 近乎空白的维度** |
| **整体** | 方法论骨架：规格→设计→实现→交付 | 工程化分层：Prompt→Context→Harness→Loop→FDE | spec-first 是**纵向流程**，五层体系是**横向能力栈**；两者可互补而非互斥 |

## 4. 关键差异分析

### 4.1 "确定性约束"设计哲学对 spec-first 的启发

Harness 工程最核心的设计理念：**不要告诉 Agent "请遵守规范"，而是在结构上让它不可能违反规范**。

对 spec-first 的启发：

| spec-first 当前做法 | Harness 哲学下的改进方向 |
|---------------------|-------------------------|
| 在规格文档中描述"代码应符合 X 规范" | 将 Linter/类型检查接入 CI，不合格则阻断 Pipeline |
| 审查节点由人工触发 | 在 spec-first 流程中嵌入自动化验证 gate（格式校验、引用完整性检查、API 契约一致性验证） |
| 对 Agent 输出做 post-hoc 审查 | 将验证规则前移为 Agent 执行环境的约束条件 |

**核心洞察**：spec-first 对"质量"的保证依赖人工闸门和审查文化，Harness 哲学则主张**把质量要求编码为 Agent 无法绕过的结构约束**。两者结合起来，spec-first 的方法论骨架会获得更强的执行可靠性。

### 4.2 Loop工程的"人退出逐轮循环" vs spec-first 人工闸门

这是两个框架之间最深层的张力：

| | spec-first 方法论 | Loop 工程 |
|------|------------------|-----------|
| **人的角色** | 每个阶段节点设人工闸门（审查/确认/签字） | 人设计循环系统，系统自主运行 |
| **风险模型** | 假设人必须在关键决策点把关 | 假设可验证目标 + 创作者-检验者分离能替代人工把关 |
| **适用场景** | 创造性工作、高不确定性任务、首次探索 | 高频重复、结果可验证、工具链已接通的任务 |
| **优势** | 质量可控、方向可调 | 规模化、低成本持续运行 |

**并非对立，而是分层适用**：spec-first 适合**定义"做什么"的阶段**（需求分析、架构设计），Loop 适合**执行"怎么做"的阶段**（持续集成、代码审查、测试修复）。一个可行的融合模型是：spec-first 定义 Loop 的边界和目标，Loop 在边界内自主运行，spec-first 的节点审查作为 Loop 的外部闸门。

### 4.3 FDE 层揭示的企业落地摩擦——spec-first 未充分覆盖的维度

FDE 揭示的四类摩擦在 spec-first 方法论中基本处于空白：

| 摩擦类型 | 具体表现 | spec-first 当前覆盖状态 |
|---------|---------|----------------------|
| **数据权限** | 部门审批、数据分类、访问控制 | 未覆盖（假设数据已就绪） |
| **遗留系统** | ERP/OA/自研系统对接 | 未覆盖（假设工具链已存在） |
| **监管合规** | GDPR、行业监管、审计要求 | 仅在文档层面提及，无落地机制 |
| **组织认知** | 业务方对 AI 输出的信任与理解 | 未覆盖（假设组织已准备好） |

**改进方向**：spec-first 可新增一个"部署环境规格"维度，在项目启动前显式评估以上四类摩擦，将 FDE 思维前移到方法论框架内。

### 4.4 Stripe Minions 案例对 #19 组织级 Coding Agent 的补充

#19 报告关注的是组织级 Coding Agent 的架构设计；本文 Stripe Minions 案例提供了**运行时的规模化验证**：

- **#19 关注**：Agent 如何被组织设计、分配、管理
- **Minions 补充**：当 Harness 足够完善（隔离 devbox → Linter → 精选 CI → 自动纠正 → 人工审查），Agent 可做到**周合并千+ PR** 的吞吐量
- **关键启示**：组织级 Agent 的成功不取决于模型能力，而取决于 Harness 的完整程度——测试覆盖率、CI 精准度、自动纠正闭环这三个变量的质量直接决定了 Agent 能从"实验"走到"生产"

### 4.5 成本与认知负债的警示

文章对 Loop 工程的成本警示和"Cognitive Surrender"概念值得纳入 spec-first 实践准则：

| 警示项 | 对 spec-first 的实践含义 |
|--------|-------------------------|
| 成本激增（单 Agent 4x，多 Agent 15x token） | 引入 Loop 前必须先建立成本预算和 token 监控 |
| 目标模糊导致无限烧钱 | Loop 的触发条件必须包含"最大运行时间/预算上限" |
| 认知负债——不知道系统在干什么 | 强制保留关键路径人工审查节点，即使 Loop 运行完美也不删除 |

## 5. 知识链收束：与前序报告的关联

| | #15 pm-skills 报告 | 本文 Harness 工程 |

| **关联** | pm-skills 的工具链整合是 Harness 第一层（工具与执行层）的具体实践 | Harness 在工具层之上叠加了控制验证层和持久化层，是 pm-skills 的**上阶抽象** |

**收束**：pm-skills 解决了"Agent 能调用什么工具"的问题，Harness 进一步回答了"在什么约束下调用、失败了怎么处理、长周期状态怎么保持"。

**收束**：Gstack 的架构分层偏向**系统设计视角**，本文三层体系偏向**工程实践演化视角**。两者共同揭示了一点：AI 工程的复杂度不在模型层，而在模型之上的工程层。

### 5.4 #20 Karpathy 12 条：Rule 6 Token Budgets / Rule 10 Checkpoint vs Loop工程

| Karpathy 规则 | Loop 工程对应构件 | 关系 |
|--------------|-----------------|------|
| **Rule 6: Token Budgets** | 成本控制 + 最大运行时间上限 | Loop 工程将该规则**系统化**：从"建议"升级为 Loop 触发条件的内置约束 |
| **Rule 10: Checkpoint** | 持久化记忆 + Git 状态存储 | Karpathy 的 checkpoint 是手动实践，Loop 工程将其**自动化**为系统级能力 |

**收束**：Karpathy 12 条是个人级别的最佳实践，Loop 工程将这些实践**组织化、自动化**，使其能在多 Agent 并行环境中稳定运行。

### 5.5 #23 业务需求专家 Agent：闭环设计 vs Loop工程的自驱动系统
