# Gstack + Superpowers + GSD 三层组合 vs spec-first 阶段化设计 —— 对照分析报告

> 生成日期：2026-06-12
> 基于六篇文章：微信推文、掘金深度分析、yudesk 对照表、AgentUpdate 三层架构、Flowhat 四层控制、今日头条

---

## 一、项目概述

### 1.1 Gstack（Garry Tan / YC）

**定位**：「一人工程团队」——将 AI 拆分为 23 个虚拟角色。

**创建者**：Garry Tan（Y Combinator CEO），2026 年 3 月开源（MIT），约 100K Stars。

**核心哲学**："瓶颈不是模型智力，是纪律。" —— Garry Tan

Gstack 不提升模型推理能力，而是通过**强制角色分离**，让模型在特定上下文中专注特定任务，避免"万能助理"导致的注意力分散。其交互协议不是对话，而是 slash commands——每个命令召唤一个特定角色接管会话，而非简单地注入 system prompt。

**23 角色按开发阶段分类**：

| 阶段 | 命令 | 角色 | 解决的问题 |
|------|------|------|-----------|
| **规划** | `/office-hours` | YC 创业导师 | 6 个结构化问题澄清需求，避免方向错误 |
| | `/plan-ceo-review` | CEO/创始人 | 4 种范围模式审查产品方案可行性 |
| | `/plan-eng-review` | 工程经理 | 锁定架构设计、数据流、边缘案例 |
| | `/plan-design-review` | 资深设计师 | 设计维度评分（0-10），确保交互体验 |
| | `/autoplan` | 自动化管道 | 一键串联 CEO→设计→工程三轮审查 |
| **开发验证** | `/qa` | QA 负责人 | 真实浏览器端到端测试、截图验证 |
| | `/qa-only` | QA 纯报告 | 仅输出 Bug 报告，不修改代码 |
| | `/browse` | 浏览器操控 | Chrome DevTools Protocol 控制无头浏览器 |
| | `/review` | Staff Engineer | 审查 CI 通过但可能在生产环境出问题的代码 |
| | `/codex` | 第二意见 | 调用 OpenAI Codex CLI 独立审查 |
| | `/cso` | 安全官 | OWASP Top 10 + STRIDE 威胁模型扫描 |
| **发布运维** | `/ship` | 发布工程师 | 同步 main→跑测试→推代码→创建 PR |
| | `/land-and-deploy` | 部署负责人 | 合并 PR→等待 CI→验证生产环境 |
| | `/canary` | SRE | 部署后金丝雀监控循环 |
| **辅助** | `/benchmark` | 性能测试 | Core Web Vitals、页面加载时间 |
| | `/design-shotgun` | 设计探索 | 生成 4-6 种 AI 设计变体 |
| | `/design-html` | 设计交付 | 将设计稿转为可交付 HTML |
| | `/retro` | 工程回顾 | 每周工程回顾 |
| | `/learn` | 记忆管理 | 跨会话记忆管理 |

完整流程循环：**Think → Plan → Build → Review → Test → Ship → Reflect**。

---

### 1.2 Superpowers（Jesse Vincent）

已在 #16 详细分析，此处仅做对比定位：

**定位**：严格的工程导师，流程纪律 + TDD。

**核心价值观**：「宁可慢，不能烂。」

**三个代表性 Skill**：
- **brainstorming**：苏格拉底式提问，逼你动手前把真正要解决的问题想清楚。不是帮你发散，是帮你收敛。
- **verification-before-completion**：你必须先跑测试、确认行为、证明实现真的解决了问题，它才让你标记完成。
- **systematic-debugging**：不许你瞎改。先定位根因，再给修复方案，再验证。

Superpowers 假设你已经知道要做什么——它负责的是"高质量执行"。

---

### 1.3 GSD（中间层）

**定位**：Spec 驱动开发 + 上下文工程。

**核心能力**：
- **Phase-based Roadmap**：Spec→Plan→Tasks 的结构化翻译
- **上下文防腐层**：防止 context rot，主动检查"当前代码是否偏离原始 Spec"
- 不只是存上下文，而是**主动防漂移**

**在三层架构中的位置**：Gstack（决策/角色层）→ **GSD（上下文/规范层）** → Superpowers（执行层）

说人话：Gstack 说"做一扇门"，GSD 把"门"翻译成"高 2 米、宽 0.8 米、材质橡木、铰链在左侧、开门方向朝外"——然后 Superpowers 按这个规格把门做出来。

## 二、三层架构与阶段匹配

### 2.1 三层架构定义

| 层级 | 框架 | 角色 | 核心问题 |
|------|------|------|----------|
| **L1 决策/角色层** | Gstack | CEO、PM、架构师、设计师 | "做什么？值不值得做？方向对不对？" |
| **L2 上下文/规范层** | GSD | Spec 翻译器、防腐层 | "需求怎么变成可执行规格？上下文会不会丢？" |
| **L3 执行层** | Superpowers | Tech Lead、QA | "怎么写对？怎么验证？怎么不出错？" |

这并非竞品关系，而是管道的三个段落——每一层解决不同维度的问题。工具好不好用，80% 取决于你怎么用，20% 取决于工具本身。

### 2.2 阶段匹配：何时用哪层

| 阶段 | 主框架 | 关键命令/能力 | 产出物 |
|------|--------|-------------|--------|
| **方向定义** | Gstack | `/office-hours` → `/plan-ceo-review` → `/design-review` | 产品 Spec |
| **上下文翻译** | GSD | Spec→Plan→Tasks 翻译 + 防腐层 | 可执行技术规格 + 防腐上下文 |
| **工程执行** | Superpowers | brainstorming → coding → verification-before-completion → systematic-debugging | 经验证的高质量代码 |
| **验收交付** | Gstack | `/qa` → `/review` → `/ship` → `/land-and-deploy` | 上线交付包 |

典型串联流程：**Gstack（定方向）→ GSD（译规格）→ Superpowers（高效执行）→ Gstack（验收交付）**。

## 四、关键差异深度分析

### 4.1 "角色驱动" vs "阶段驱动"——架构哲学的根本分歧

Gstack 的核心创新是**显式角色分离**。当你输入 `/plan-ceo-review`，你不是在给 AI 加 system prompt，而是"召唤一个 CEO 接管会话"。每个 slash command 对应一个明确的角色和思维模式——CEO 有 CEO 的"专业直觉"，QA 有 QA 的审查本能。

spec-first 的阶段驱动通过 phase Skill 加载来实现，但阶段内 Agent 的角色是**隐式的**——它在阶段 Skill 内被调度，但不显式暴露"我现在是 CEO"还是"我现在是工程师"。

**关键判断**：角色显式化的价值不在于技术实现，而在于**认知锚定**。当用户明确知道"现在是谁在思考"，他们对该阶段产物的预期和评判标准会更准确。Gstack 的 `slash command` 作为交互协议的巧妙之处在于——它是显式的角色调用，而非隐式的 prompt 注入。

### 4.2 "三段式管道" vs "七阶段生命周期"——同一管道的不同分段方式

Gstack+GSD+Superpowers 不是竞品关系，而是管道的三个段落。spec-first 的七阶段（Bootstrap→Ideate→Brainstorm→Plan→Work→Review→Compound）本质上是同一个管道的另一种分段方式：

| spec-first 阶段 | 三层框架覆盖 | 说明 |
|----------------|-------------|------|
| Bootstrap | Gstack | 方向定义、需求澄清 |
| Ideate | Gstack | 产品构思、方案探索 |
| Brainstorm | Gstack | 方案评审、CEO/设计审查 |
| Plan | GSD | 规范翻译、上下文防腐 |
| Work | Superpowers | 工程执行、TDD 验证 |
| Review | Gstack | QA/CSO/代码审查 |
| Compound | Gstack | retro + learn 复盘记忆 |

关键观察：spec-first 在 Work 阶段之前用四个阶段（Bootstrap+Ideate+Brainstorm+Plan）铺垫，而三层框架用 Gstack 的三个命令（`/office-hours` → `/plan-ceo-review` → `/design-review`）完成同样的工作——但多了一个 GSD 防腐层。

### 4.3 "防腐层" vs "上下文文件"——主动防漂移 vs 被动记录

这是 GSD 最被低估的能力。GSD 不只是存上下文，而是**主动检查"当前代码是否偏离了原始 Spec"**。spec-first 用上下文文件（spec.md / plan.md / decisions.md）做类似的事情，但缺乏这种主动防漂移机制。

| 维度 | GSD 防腐层 | spec-first 上下文文件 |
|------|-----------|---------------------|
| **记录方式** | 结构化 Phase-based Roadmap | Markdown 文件（spec.md / plan.md 等） |
| **防漂移** | 主动检查偏离 | 被动记录，依赖开发者自觉 |
| **上下文传递** | 自动向执行层注入 | 需 Agent 自行读取 |
| **长链路保护** | 持续校验 Spec 一致性 | 缺乏自动化校验机制 |

### 4.4 "安全审查"的缺失——spec-first 最显著的能力空白

`/cso`（OWASP Top 10 + STRIDE 威胁扫描）是 Gstack 的独特能力。spec-first 当前没有等价的安全审查 Agent。

在 AI 编码时代，AI 生成的代码可能存在比人工代码更隐蔽的安全隐患——因为模型可能"学会"了不安全的编码模式。安全审查不应是"上线前找人看一眼"，而应是流程内置的**自动步骤**。

### 4.5 "交付上线"的缺失——spec-first 的后半段空白

Gstack 拥有完整的交付管道：`/ship`（git→tests→push→PR）、`/land-and-deploy`（合并→CI→生产验证）、`/canary`（SRE 金丝雀监控）。spec-first 的七阶段到 Compound 为止，缺乏自动化的交付上线能力。

### 4.6 "设计评审"的缺失——独立开发者失去的视角

`/design-review` 的 0-10 评分维度和 `/design-shotgun` 的 4-6 种设计变体，对独立开发者来说等于请了一个兼职设计顾问。spec-first 缺少这一层，意味着规格→界面的翻译完全依赖开发者的设计判断。

## 五、知识链收束

将本分析与前序四轮调研衔接：

- **#15 pm-skills**：Gstack 的 `/office-hours` 与 pm-skills 的 `/discover` 功能重叠但视角不同——Gstack 从"投资人/CEO"视角（"这个方向值不值得投"），pm-skills 从"产品经理方法论"视角（"需求是否完整"）。Gstack 的 `/ship` 与 pm-skills 的 `/ship-check` 互补——前者关注流程完整性（测试→PR→部署），后者关注 intended-vs-implemented 审计（做出来的和想要的是否一致）。

- **#16 OpenSpec+Superpowers**：GSD 的位置类似于 OpenSpec 的 propose 流程——都做"需求→规格"的翻译。但 GSD 更强调长链路的上下文防腐（防止 context rot），而 OpenSpec 更强调多方案对比和渐进式明确。

- **#17 AI PRD 方法论**：Gstack 的 `/office-hours` + `/plan-ceo-review` 天然覆盖 AI PRD 八大章节中的风险判断和方向验证，但缺失置信度阈值、Eval 集、成本预算等量化章节。Gstack 偏向"定性判断"，AI PRD 偏向"量化评估"。

- **spec-first 自身**：Gstack→GSD→Superpowers 的三层架构与 spec-first 的 Skills→Agents 两层架构形成互补——spec-first **缺少显式的"决策层"**。当前 spec-first 的决策逻辑隐含在阶段 Skill 内，没有独立的、以"角色视角"运作的决策 Agent。

## 六、分级改进建议

### P0（应立即借鉴）

**1. 在 Brainstorm 阶段引入"角色显式化"机制**

让 Agent 声明"我现在以 CEO 视角审视方案"，而非隐式切换思维。具体做法：
- 在 Brainstorm Skill 内嵌入 `role-declaration` 子阶段：Agent 在输出前先声明当前角色
- 借鉴 Gstack 的 slash command 交互协议——用显式命令切换角色视角，而非靠 prompt 暗示
- 目标：让开发者清楚知道"谁在思考"，建立正确的预期

**2. 在 Review 阶段新增安全审查 Agent**

对齐 `/cso` 的 OWASP Top 10 + STRIDE 威胁扫描：
- 新增 `security-review` Agent，在 Review phase 作为必选步骤
- 覆盖 OWASP Top 10（注入、认证失效、敏感数据暴露、XXE、访问控制失效、安全配置错误、XSS、不安全的反序列化、使用已知漏洞组件、日志监控不足）
- 对 AI 生成的代码额外标注"AI 生成代码安全审查"标签

### P1（应在下个迭代引入）

**3. 引入 GSD 式防腐层**

在 Plan→Work 交接处强制检查"产出物是否偏离原始 Spec"：
- 在 Work phase 执行前，运行 `spec-alignment-check`：对比 plan.md 与当前代码状态，标记偏差
- 偏差超过阈值时阻断 Work phase 启动，强制开发者确认或修正
- 这层"中间件"的价值被严重低估——它能防止 Superpowers 式的"又快又好地做出没人要的东西"

**4. 交付上线阶段补充**

新增 **Ship phase**，包含 git→tests→push→PR→部署的自动化管道：
- 借鉴 `/ship`：同步 main、跑测试、推送、创建 PR
- 借鉴 `/land-and-deploy`：合并 PR、等待 CI、验证生产
- 借鉴 `/canary`：部署后金丝雀监控循环（错误率、页面性能、健康检查）

**5. `/autoplan` 式一键串联**

将 Bootstrap+Ideate+Brainstorm 串联为一键入口：
- 类似 Gstack 的 `/autoplan`（CEO→设计→工程三连审）
- 开发者只需一个命令，自动走完方向定义三阶段，产出完整产品 Spec

### P2（中长期探索）

**6. 23 角色体系借鉴**

将 spec-first 的 51 Agents 显式化、角色化：
- 每个 Agent 有明确的"专业直觉"——不是泛化的"工程师"，而是有特定关注点的角色（如"性能架构师"、"安全审计员"、"UX 审查官"）
- 参考 Gstack 的 slash command 设计——用显式命令而非隐式调度来调用角色
- 关键原则：角色分离的价值不在于"让 AI 更聪明"，而在于"让 AI 更专注"

**7. `/retro` + `/learn` 机制**

增强 Compound phase 的复盘和跨会话记忆能力：
- `/retro`：每周工程回顾——"这周做了什么、哪些顺利、哪些卡住、下周改进什么"
- `/learn`：跨会话记忆管理——关键决策、经验教训不随会话结束而丢失
- 这两个能力让 AI 编程助手从"无状态工具"进化为"有记忆力的团队成员"

**8. 浏览器自动化测试**

借鉴 `/qa` 的真实浏览器 E2E 测试能力：
- 在测试 Skill 中集成 Chrome DevTools Protocol 或无头浏览器能力
- 支持截图验证、点击交互、回归测试
- 补齐 spec-first 在真实 UI 验证上的空白

## 七、结论

**核心启发**：spec-first 的阶段化设计强在"规范翻译"和"工程执行"两层，但缺少独立的"决策层"和"安全/交付"环节。Gstack+GSD+Superpowers 的三层架构本质上是把"想清楚→译明白→做正确→安全交付"这四个不可压缩的环节显式化——spec-first 最值得借鉴的不是 23 个角色本身，而是 **"让每个环节有独立的质量守门人"这一设计原则**。

> Gstack 说"先做对的事"，GSD 说"把事翻译对"，Superpowers 说"把事做对"——三层各司其职，不可互相替代。
