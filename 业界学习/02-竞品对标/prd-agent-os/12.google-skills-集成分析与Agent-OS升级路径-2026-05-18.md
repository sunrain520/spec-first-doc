## 0. 结论先行(TL;DR)

> **数据可信度提示**:本节引用的所有具体数值(+26% / 25+ agent / ~1700 tokens / 36.82% / +242.7% / +31.3%)均为 **advisory**,来自二手综述,本会话未核对一手报告;详见 §3 数据可信度声明与 §10.2 未执行清单。

**一句话结论**:

> google/skills 对 spec-first 的真正价值不是 14 个 Google Cloud 产品 skill,
> 而是它**对 spec-first 版本规划 §15 M7 advisory 同源证据做深化解读**(注:§15 M7 *external-capability-source.v1* advisory 草案与本报告均锚定同一 sibling checkout `caeec93`,共同方向判断不构成独立 cross-validation;升级为多源佐证需要至少一条独立证据,例如 Anthropic 官方 17 skills 实物拉取或第三家 vendor 同源调研):
> **AI Coding 的核心竞争力正在从 "Model" 与 "Prompt",迁移到 "Skill Layer + Harness Layer + Governance Layer"**(advisory 判断)[^layer-mapping]。

[^layer-mapping]: 三层(Skill / Harness / Governance)是本报告为传播叙事使用的简化抽象,与版本规划 §4.2 七层 + Skill Governance 横切的对应关系:Skill Layer ≈ Workflow Harness + Artifact Protocol;Harness Layer ≈ Evidence Governance + Context+Tool Routing + Provider Adapter;Governance Layer ≈ Review Closure + Knowledge Compound + Skill Governance 横切。**spec-first 实际能力以 §4.2 七层为准**。
>
> spec-first 当前定位 *"repo-local AI Coding Harness"* 是正确的,但**不够大**;
> 在 2026-05 业界共识窗口内,**M1-M5 文档层使用候选 B 中性升级**(详见 §5.3):
>
> **spec-first is the open repo-local Operating System for evidence-governed, review-closed AI software engineering.**
>
> 候选 C(*The Open Agent Engineering Operating System*)作为 M9 GA 后期外部叙事的升级目标,**当前阶段不直接采用**,以避免 §9.2 反模式 "M9 之前对外宣称 Production-ready Agent OS"。

**三档集成决议(与版本规划 §15 已锚定的 accepted/rejected patterns 对齐并深化)**:

| 档位 | 范围 | Milestone 落点 | 时效 |

**与用户战略愿景的对齐结论**:

用户在本会话提出的 *"The Open Agent Engineering Operating System"* 定位与 P0-P2 优先级清单(SKILL.md 兼容层 / Skill Registry / Metadata Schema / Semantic Skill Router / Recipe / Sandbox / Quality Gate / Benchmark / Evolution / Capability Graph)与版本规划 §1.3 七层能力 + 横切治理结构高度同构,但需要在以下两点上**轻量化收口**而非全盘接受:

1. **不为追求"OS 化"扩张引入重 workflow engine 与 marketplace**。版本规划 §3.4 已声明 "不做 marketplace / 中心化 SaaS / 重 workflow engine";Agent OS 形态应当通过 **轻 contract + 标准协议 + provider adapter** 实现,而不是中心化平台。

**核心定调(回应用户)**:

> **LLM 越强,spec-first 越应该变轻在提示词上,变重在 Harness 上**。
>
> 这与 LangChain Terminal Bench 2.0 控制实验结论一致:同模型下,仅升级 harness 即可让编码 agent 准确率从 52.8% 提升到 66.5%(+26%)(**advisory 单实验**:LangChain Terminal Bench 测的是终端命令任务,与 spec-first 多阶段 repo-local 工程链路不同域;spec-first 自身需在 M3-M4 给出独立 baseline,详见 §7.2 验收)。
> 这意味着:
> - spec-first 的 **核心壁垒不在 prompt 工程**,而在 **artifact protocol / evidence governance / context routing / review closure** 四个"机器可消费"层。
> - skill / agent prose 应该收缩为 **stable instruction prefix + dynamic volatile suffix**(progressive disclosure),让模型决定按需加载,而不是会话启动时一次性灌入。
> - 模型越强,越能消化 light contract;越强的模型 + 越轻的 prompt + 越重的 Harness = 越高的真实研发增益。

### 2.2 设计模式提取(LLM-judgment,advisory)

从 14 个 SKILL.md 中归纳出**六个稳定可移植设计模式**:

#### 模式 1:Progressive Disclosure 三层结构

```text
skills/<name>/
  SKILL.md                # 第一层:frontmatter(name+description,30-80 tokens)+ stable instruction prefix
  references/             # 第二层:深入主题分章(cli-usage / iac-usage / mcp-usage / iam-security 等),按需加载
  assets/                 # 第三层:可复用配置样例(YAML/JSON/Terraform),不进 prompt,由 agent 引用读取
```

启动期只载 frontmatter,触发后载主体,深入时再载 references。**这与 Anthropic 2025-12-18 开放的 Agent Skills 标准三层加载模型完全同构**。

#### 模式 2:四段式 SKILL.md(WAF 风格)

```text
1. Overview            一句话定位 + 适用场景
2. Core Principles     带 grounding document 链接的原则集合
3. Workload Assessment Questions
                       LLM 主动问用户的结构化问题集(10 项左右)
4. Validation Checklist
                       可勾选的客观判定项(8-12 项)
```

适合 **review / audit / lens** 类 workflow。spec-first 已有的 `spec-code-review`、`spec-app-consistency-audit`、`spec-doc-review` 与这套四段式高度同构,但缺第 3、4 段的稳定模板。

#### 模式 3:Clarifying Questions 前置块(Recipe 风格)

```text
## Clarifying Questions
Before providing a specific solution, clarify the following with the user:
1. 维度 A
2. 维度 B
3. 维度 C
4. 维度 D
```

特征:

- 在 `Prerequisites` 与 `Steps` **之前**显式列出
- 问题维度刻意控制在 4-5 个,覆盖人物 / 环境 / 目标 / 偏好
- 每个问题独立成判定分支,避免歧义场景被 LLM 默默选择

spec-first `spec-brainstorm` 已有"先收敛再下钻"的语义,但**没有把 clarifying questions 作为强制章节固化在 SKILL.md**。

#### 模式 4:Core Directive 终止纪律(Diagnostic 风格)

`google-cloud-networking-observability` 顶部:

```markdown
## 🛑 Core Directive: Results First
1. Identify the Primary Source
2. Execute & Present
3. Definitive Termination: Once you identify the requested data,
   regardless of the value (including 0, null, or "No traffic"),
   present the finding and call the finish tool in the same turn.
   Do NOT attempt to find "active" or "busier" resources to provide
   a "better" answer unless specifically instructed.
```

**关键创新**:把 LLM "无意义循环 / over-search / 自动美化答案"的反模式,通过显式 *termination rule* 收口。

这正是 spec-first `spec-debug`、`spec-app-consistency-audit`、reviewer dispatch 在长任务中最容易踩的坑。

#### 模式 5:Validation Logic / Checklist 强收口

每个 Cloud 产品 skill 末尾都有:

```markdown
## Validation Logic
Use this logic to determine if the user has successfully completed X:
- [ ] Created a Project ID?
- [ ] Billing Linked?
- [ ] CLI Authenticated?
- [ ] Resource Verified?
```

特征:

- 可勾选的客观判定项(非主观打分)
- 每项绑定一条可执行验证命令或可观察事实
- 失败项可直接转为 next-action

这是 spec-first **reviewer JSON schema**(`skills/spec-code-review/references/findings-schema.json`)的天然 prose 形态;两者可双向映射。

#### 模式 6:Reference 按主题拆分(子领域分层)

以 `gke-basics/references/` 为例:

```text
gke-app-onboarding.md   gke-batch-hpc.md    gke-cluster-creation.md
gke-compute-classes.md  gke-cost.md         gke-golden-path.md
gke-inference.md        gke-multitenancy.md gke-networking.md
gke-observability.md    gke-reliability.md  gke-scaling.md
gke-security.md         gke-storage.md      gke-upgrades.md
gke-backup-dr.md        cli-reference.md    client-library-usage.md
iac-usage.md            mcp-usage.md        core-concepts.md
```

特征:

- 子领域分层(架构 / 网络 / 存储 / 安全 / 可靠性 / 成本 / 可观测性)
- 跨产品复用模板:`cli-usage.md`、`iac-usage.md`、`mcp-usage.md`、`iam-security.md`、`core-concepts.md` 在多个 skill 重复出现
- **可机器消费的标题命名规范**(短横线 + 主题词,容易被 frontmatter 索引)

这正是 spec-first `skills/<workflow>/references/` 目前**部分已有但不统一**的形态。

### 2.3 google/skills 真正在做什么(战略级判断)

撕开 14 个 Google Cloud 产品 skill 的表层内容,Google 在做的事情有三层:

| 层 | 实际目标 | 给整个生态的信号 |
| --- | --- | --- |
| 表层 | "让 agent 能正确用 GCP 产品" | 域内文档质量提升 |
| 中层 | 统一 SKILL.md 形态,与 Anthropic Agent Skills 开放标准对齐 | **接入 cross-vendor skill ecosystem** |
| 深层 | 证明"企业级 domain skill pack"可作为 **可复用工程资产** 治理(license / commit pin / reference structure) | **AI Coding 的核心竞争力正在从 Model 与 Prompt 迁移到 Skill Layer + Workflow Layer + Governance Layer** |

**对 spec-first 的核心启示**:

> google/skills **不是竞争对手**,也**不是依赖项**;
> 它是同一坐标系下的 *领域能力包提供方*,
> 而 spec-first 应该占据 *领域能力包之上的 Harness / Governance / Routing* 位置。
>
> Google 在做 *能力包内容*,
> spec-first 应在 *能力包接入协议 + 证据治理 + review 闭环* 上做无可替代。

## 3. 业界趋势锚点(2026-05 校准窗口)

本节是 **§5 战略升级判断** 的事实底座。所有引述均来自 §1.1 表格中 4 次 WebSearch 命中的一手综述。

> **数据可信度声明**(2026-05-18 最后校准):本节所有具体数值(LangChain Terminal Bench 52.8%→66.5%、Snyk 36.82% 漏洞率、SkillsMP 400K+ skills、Anthropic 17 skills 1700 tokens、Faros AI +242.7%、Lightrun 43%、DORA ~90%、MIT Tech Review 2.30% 幻觉率、Anthropic Agent Skills 时间线等)**均来自 WebSearch 命中的二手综述,未直接核对原始报告/审计/postmortem 一手来源**;数值精度与算法定义可能与一手报告存在偏差(例如 26% 增幅严格算应为 25.95 个百分点)。本节定位为 *advisory*,不构成 source-of-truth;落地 plan / task-pack 时应重新核对原始数字与定义。本节趋势方向(harness 共识、SKILL.md 标准吸收、SDD 三层、工具痛点收敛、合规临界)的证据强度高于具体数值。

### 3.1 趋势 A:Harness Engineering 已成事实共识

**核心证据**(LangChain 2026 公开实验):

> 在 **不更换模型** 的前提下,仅升级 harness,可让编码 agent 在 Terminal Bench 2.0 上的准确率从 **52.8% → 66.5%**(+26%)。

**业界术语收敛**(Ryan Lopopolo 2026-02-11 OpenAI 帖 + LangChain 转述):

```text
Agent = Model + Harness

Prompt Engineering    : 单轮指令优化
Context Engineering   : 单上下文窗口 token curation
Harness Engineering   : 跨会话、跨阶段、跨 agent 的 context reset / handoff artifact / phase gate
```

**关键判断**:

> 模型能力天花板移交给 vendor(Anthropic / OpenAI / Google);
> 真实差异化在 Harness 层。

这与 spec-first 版本规划 §1.1 一句话定位"repo-local, evidence-governed, review-closed AI coding Harness"**核心方向高度重叠**(spec-first 强调 repo-local + evidence + review-closed,Lopopolo / LangChain 强调 cross-session context reset + handoff artifact + phase gate;两者交集是 *跨会话可治理的 AI 协作过程*,但侧重不同,不严格字面同义)。

### 3.2 趋势 B:Anthropic Agent Skills 开放标准 12 周内被主流 agent runtime 全量吸收

**时间线**:

| 日期 | 事件 |
| --- | --- |
| 2025-10-16 | Anthropic 首发 Agent Skills 概念 |
| 2025-12-18 | 开放为公开标准(SKILL.md + YAML frontmatter + 三层 progressive disclosure) |
| 2026-03 第一周 | OpenAI / Microsoft / JetBrains / Cursor / Gemini CLI / Block Goose 等 25+ agent 产品接入兼容实现 |
| 2026-05 | Anthropic 官方 17 skills 启动期总成本 ~1700 tokens(median ~80 tokens/skill);各 host 默认目录 `~/.claude/skills`、`~/.codex/skills` 收敛 |

**配套生态规模**:

- SkillsMP 等 marketplace 已索引 400K+ skills
- **但 Snyk 2026-02 审计**:36.82% 公开 skill 有至少一处安全漏洞,13.4% 有 critical issue

**关键判断**:

> SKILL.md 已经在 12 周内成为 *跨 vendor 的事实标准*,这是历史上罕见的 prompt-layer 协议收敛速度。
> spec-first 若不接入,会在 18-24 个月内变成生态孤岛。
>
> 但 spec-first 不应"无脑接入第三方 skill",36.82% 安全漏洞率说明:
> **接入协议** 与 **接入治理(quality gate + sandbox + trust score)** 必须同时设计。

### 3.3 趋势 C:SDD 三层分类已成评估框架(spec-first 在第一层但需向第二层演进)

**Martin Fowler 团队 + arxiv 论文**已经把 spec-driven development 分成三个递进层级:

| 层 | 含义 | 典型工具 |
| --- | --- | --- |
| 1. **spec-first** | 写好 spec 后用于本次 AI-assisted 任务 | GitHub Spec Kit(93K stars, v0.8.7, 30+ agents) |
| 2. **spec-anchored** | spec 完成任务后**继续保留**,用于该 feature 的演化与维护 | AWS Kiro(GA + CLI, 默认 Auto router 跨 Claude/Qwen/DeepSeek/GLM/MiniMax) |
| 3. **spec-as-source** | spec 是主源,代码是 *generated*,人只编辑 spec | Tessl(Framework + Spec Registry 10K+ specs) |

**spec-first 项目命名本身就锁在第一层**,但**当前能力实际上已经在向第二层挪**(`evidence-packet`、`spec-anchor`、`provenance` 链已在 M2 落地)。

**业界横向对比**(2026-05):

- Thoughtworks Tech Radar Vol.33 把 SDD 列入 *Assess* ring(2025-11)
- 早期采用者数据:SDD 让非琐碎任务的 **首通过率提升 3-10×**(GitHub / AWS 实测)
- **SDD 三大代价**(各家都没解决):
  1. spec review 开销随 spec 啰嗦度线性增长
  2. 非确定性仍存在(spec 准了 agent 也可能 ignore / over-interpret)
  3. 探索性原型、<50 行 bug fix、hackathon、需求每日变化的项目**不适用**

**关键判断**:

> spec-first 应当**保留项目名**(第一层定位有品牌价值),
> 但在 M2(spec-anchor / provenance 链)与 M6(compound + standards)上,把能力**显式声明覆盖到 spec-anchored 层**。
> *spec-as-source* 是 Tessl 当前在探的远期形态,spec-first 不必抢这一层。

### 3.7 关键反命题与本报告判断(advisory)

战略报告应正面回答会动摇主线判断的反命题。本节列 4 个最尖锐的反命题与 spec-first 立场:

#### 反命题 1:MCP 已成事实标准,SKILL.md 是否冗余?

**反命题陈述**:Anthropic 自己的 MCP(Model Context Protocol)已在 2024-2025 成为工具调用事实标准,GitHub / Cloudflare / 数十家厂商接入;若 MCP 已经覆盖"agent 调用工具"的协议,SKILL.md 是否只是 prompt 包装?

**本报告判断**:**MCP 与 SKILL.md 是互补而非竞争**。

- MCP 解决 *"agent 用什么工具"* 的连接性问题(transport + tool schema + 权限)
- SKILL.md 解决 *"agent 怎么用这套工具完成一个任务"* 的过程性知识问题(workflow + clarifying questions + validation checklist)
- Anthropic Murag 原话:*"MCP provides secure connectivity to external software and data, while skills provide the procedural knowledge for using those tools effectively"*(§3.2 引用)
- spec-first 不重写 MCP;`/spec:mcp-setup` 与 `runtime-capabilities.json` 已经把 MCP 作为 *provider adapter* 消费

**对 spec-first 的含义**:M1 SKILL.md compat 不影响 MCP 接入策略;M3 `tool-budget.v1` 与 MCP provider 是同一坐标系下的不同维度(budget 是 spec-first 增量,不是 MCP 替代品)。

#### 反命题 2:Gemini / Claude 1M context 已落地,progressive disclosure 是否过度优化?

**反命题陈述**:Anthropic Claude Opus 4.7 1M context beta、Gemini 1.5/2.5 标配 1M context、GPT-5 系列同向扩张;若上下文窗口足够装下整个 SKILL.md + references/ + assets/,progressive disclosure 是优化还是冗余?

**本报告判断**:**长 context 不消除 progressive disclosure 价值,反而提高其杠杆**。

- **成本**:1M context 标价远高于短 context;每会话灌满 context 的边际成本仍显著(Anthropic 2026-04 postmortem 涉及的 reasoning regression 部分与默认 effort 上调相关)
- **缓存命中**:prompt cache 5min TTL + stable instruction prefix 对长 context 仍是核心优化(`spec-code-review/SKILL.md` 已显式声明 *Cache-Friendly Context Layout*)
- **注意力衰减**:即使窗口 1M,模型对中段内容的注意力 / 召回率仍弱于头尾;Anthropic / Gemini 的 needle-in-haystack 数据普遍在长上下文下有性能 degradation
- **进入决策**:Anthropic Agent Skills 三层加载(discovery 80 tokens → activation 全文 → references 按需)即在 1M context 时代仍是默认推荐设计

**对 spec-first 的含义**:F-05 修订的分层 token 预算(discovery ≤ 200 / activation ≤ 2000 / references 不限)在 1M context 时代仍合理;**长 context 是 progressive disclosure 的赋能而非替代**。

#### 反命题 3:AGENTS.md / CLAUDE.md / SKILL.md 是否层级混乱?

**反命题陈述**:三个 markdown 入口同时存在,概念边界不清;企业用户会问"我应该写哪个"。

**本报告判断**:**三者是层级关系,不是替代关系**。

| 文件 | 作用域 | 加载时机 | 内容性质 |
| --- | --- | --- | --- |
| `AGENTS.md` | 仓库根,**跨 host 共享** | session 启动注入 | 仓库级 + 团队级约定(coding style / commit policy / 测试命令) |
| `CLAUDE.md` | 仓库根,**Claude 专属增量**(host-specific) | Claude session 启动注入 | Claude host 特有的入口治理 / 工具偏好 / 行为约束 |
| `SKILL.md` | `skills/<name>/`,**单一 skill 边界** | skill 触发时按 progressive disclosure 加载 | 单个 workflow / capability 的 contract + 规则 + references |

- spec-first 已实践:`CLAUDE.md` 与 `AGENTS.md` 双入口共存,**managed bootstrap block** 自动同步两者(详见 §1 表格;CLAUDE.md L4-L42 的 Workflow 入口治理 block)
- 写哪个的判断:跨 host 通用 → AGENTS.md;Claude 特异 → CLAUDE.md;workflow 单元 → 对应 skill 的 SKILL.md

**对 spec-first 的含义**:M1 落地双宿主入口治理时,**严格保持三层关系**,不让 SKILL.md 内容回流到 AGENTS.md / CLAUDE.md(否则会破坏 progressive disclosure 设计)。

#### 反命题 4:多 agent / sub-agent context isolation 是否要求 SKILL.md 重新设计?

**反命题陈述**:Claude Code subagent / OpenAI Codex parallel threads / Gemini CLI multi-agent 已成趋势;sub-agent 默认无主会话历史,只接收 task contract;此时 SKILL.md 顶部的"上下文导向"规则是否失效?

**本报告判断**:**sub-agent context isolation 反而强化 SKILL.md 设计的必要性**。

- **stable instruction prefix 即 sub-agent 入口契约**:sub-agent 启动时,SKILL.md 顶部的 Workflow Contract Summary 就是它的 *task contract*
- **bounded scope**:SKILL.md 中显式声明的 *When To Use / When Not To Use / Inputs / Outputs / Failure Modes* 是 sub-agent 不越权的核心保障
- **artifact-summary.v1** 已是 spec-first 跨 sub-agent 的 handoff 协议(version 规划 §11);SKILL.md 与 artifact-summary 双层契合
- 反例校验:google/skills `Definitive Termination Rule`(§2.2 模式 4)对 sub-agent 比对主 agent 更关键,因为 sub-agent 缺乏全局视野更易陷入 over-search

**对 spec-first 的含义**:M1 SKILL.md compat 必须显式声明 *sub-agent 加载契约*(子集);M3 context-router 必须支持 *主 agent 与 sub-agent 不同的 context budget*。

#### 反命题 5:Anthropic 24mo 内 deprecate Agent Skills 标准?

**反命题陈述**:Anthropic Agent Skills 标准自 2025-12-18 开放至今约 5 个月;无开源治理委员会、无第三方维护者承诺、无 RFC 流程。Anthropic 在 2026-2028 内可能因战略转向(纳入 MCP / 合并到新协议 / 直接弃用)而 deprecate 该标准;若发生,§7.1 M1 SKILL.md compat 工作量将变沉没成本。

**本报告判断**:**主线不动摇,但需独立 fallback 文档**。

- 双协议 compat 层本质是适配器(`compat/skill-loader/`),Anthropic 字段是子集而非依赖
- M1-T04 落地时需明确 backend 抽象边界:即使 Anthropic 标准 deprecate,spec-first 原生 contract.yaml 仍可独立运行
- §9.1 R-06 已识别"业界标准向另一方向收敛"风险,但缓解措施不够具体;应在 M1 落地后加 deprecation playbook(独立文档,不在 SKILL.md compat 内)
- 历史参考:OpenAI Function Calling → Tool Use 演进,Anthropic 自身也在 Agent Skills 之前迭代过若干内部草案

**对 spec-first 的含义**:M1-T04 必须把 Anthropic 字段 mapping 收敛在 `compat/skill-loader/MAPPING.md` 单点;不让 Anthropic 字段名渗透到 spec-first 原生 schema。

#### 反命题 6:信任瓶颈是人不是工具,review/evidence 改善边际增益小?

**反命题陈述**:§3.4 数据中"0% leader 完全信任 AI 代码"+"AI 改动后 incidents/PR +242.7%"+"未审 PR merge +31.3%"同时存在;这些数字可能并非说明"工具 review 不够",而是"人没有时间/意愿审"。若瓶颈是人,那么 spec-first 加重 evidence/review/governance 层的 ROI 会显著低于预期。

**本报告判断**:**⚠️ 部分动摇——这是本报告战略主轴最薄弱的一环**。

- harness/evidence 改善是 *必要不充分* 条件:更好的证据可降低审查者认知成本,但不等于审查者会真的审
- §8.1 P-01..P-10 已被本次 doc-review 重新定性为 *roadmap gap* 而非 *用户实证痛点*(详见 §8.1 重定性);真实 user pain 验证仍 deferred
- 若 M3 启动前 5+ 用户访谈或 issue tracker scan 显示瓶颈在"人意愿"而非"工具能力",M4 review-closure 投资量级应下修,转向 **降低审查门槛**(更短的 review surface、更多 auto-resolve、更智能的 default)而非 **增加证据深度**
- 不动摇 spec-first 整体定位(repo-local Harness),但动摇 §7.2 M4 投资规模

**对 spec-first 的含义**:§7.2 M4 子任务在 plan 阶段需先做 user evidence gate,不能直接全量进入 review-closure 重投资。

#### 反命题 7:中文 host 不接入 Anthropic Skills,协议分叉?

**反命题陈述**:§3.6 已 advisory 标注通义灵码 / 文心快码 / Trae / CodeGeeX 等国内主流 agent 走自家路线,且与厂商云生态 + 信创 + 私有化深度绑定。若 12-24mo 内中文 host 不接入 Anthropic Agent Skills 标准(可能性中等以上),spec-first 要么放弃中文侧、要么承担双协议 + 双 runtime 设计。

**本报告判断**:**⚠️ 部分动摇——§3.6 advisory 需升级为 plan-level decision**。

- 中文 host 若不接入,spec-first 单一 SKILL.md compat 层无法触达;需在 M1-T04 设计时预留"国产 agent 适配子层"接口

## 5. 战略升级路径:从 *AI Coding Workflow Tool* 到 *The Open Agent Engineering Operating System*

> **架构定义边界**:**主框架以版本规划 §4.2 七层 + Skill Governance 横切为准**;本节 OS 类比仅作为非工程受众的辅助说明,§5.2 表格不作为 spec-first 的实际架构定义,plan / task-pack 引用时必须回到 §4.2 七层。

### 5.1 升级语句的工程化解读

用户提出的战略愿景:

> **The Open Agent Engineering Operating System**

不应理解为"做一个 OS"(那会违反版本规划 §3.4 "不做平台/不做 marketplace"边界),
而应理解为**四个维度的语义升级**:

| 旧定位 | 新定位 | 关键增量 |
| --- | --- | --- |
| AI Coding Workflow Tool | **Agent Engineering OS** | 把 workflow harness **协议化**,允许任意 host / 任意 skill 在协议上运行 |
| 服务个体开发者 | **服务 individual + team + ecosystem** | M8 governance + M7 ecosystem 同时推进 |
| 单一 Claude Code 入口 | **host-agnostic 双宿主标准** | Claude / Codex 已就位,Cursor / Gemini CLI / Goose 通过 SKILL.md compat 接入 |
| Spec-driven workflow | **Spec + Evidence + Review 三轴闭环** | 强化第二轴(evidence)和第三轴(review),不仅强化第一轴(spec) |

### 5.2 与 "OS" 类比的边界声明(避免叙事过度)

**spec-first 是 OS** 仅在以下意义上:

> **类比边界声明**:下表是为传播叙事服务的工程类比,**至少 3 项映射 imperfect**:Process scheduling 是 OS 核态强制调度而 Workflow Harness 是用户主动进入(强制性差一档);IPC 是同步异步双向通讯而 artifact handoff 是异步串行文档传递(语义不重合);Sandbox 核心是隔离边界而 tool budget 是 quota/cost 限制(完全不同维度)。**spec-first 实际能力以版本规划 §4.2 七层 + Skill Governance 横切为准**,本表仅供叙事使用。

| OS 概念 | spec-first 对应 | 落地形态 |
| --- | --- | --- |
| Process scheduling | Workflow Harness 阶段调度 | `/spec:*` 入口 + stage-contract |
| File system | Artifact Protocol + Repo-local Memory | `.spec-first/` + `docs/` 双轨 |
| IPC | Artifact handoff between stages | `artifact-header.v1` |
| Driver layer | Provider Adapter | `runtime-capabilities` / `graph-facts` |
| System call interface | Public SKILL invocations + CLI | `/spec:*` + `spec-first` CLI |
| Package manager | M7 Capability Pack intake | `external-capability-source.v1` |
| Sandbox / permissions | Tool budget + provider risk | M3 `tool-budget.v1` + `provider-risk.v1` |
| Logging / monitoring | OTel GenAI facts | M8(advisory facts,不内化 dashboard) |

**spec-first 不是 OS** 在以下意义上(必须明确):

- ✗ **不内化 agent runtime**(Claude Code / Codex 已经是 runtime)
- ✗ **不内化 model inference**
- ✗ **不内化 IDE / GUI**
- ✗ **不内化 marketplace / billing**
- ✗ **不内化 dashboard / viz**
- ✗ **不做中心化注册表**(advisory facts 不等于 registry)

**判断标准**:
> spec-first 是 *Engineering OS*,不是 *Compute OS*。
> 它治理的是"AI 协助下的软件工程过程",
> 不治理"AI 模型的算力调度"。

### 5.3 升级后的一句话定位(建议词条)

候选定位语(三选一,advisory):

| 候选 | 文本 | 适用语境 |
| --- | --- | --- |
| A(保守) | spec-first is a repo-local Harness for evidence-governed AI-assisted software engineering. | 现行版本规划 §1.1 已写,不变 |
| B(中性升级) | spec-first is **the open repo-local Operating System** for evidence-governed, review-closed AI software engineering. | 接入"OS"叙事但保留"repo-local" |
| C(战略激进) | spec-first is **The Open Agent Engineering OS** —— protocol, evidence, and review-closure on top of any AI coding agent. | 用户原话,更具传播性;**启用条件 = M9 GA 已达成**(而非时间窗口到达);若启用,文中所有 OS 概念必须显式注明类比关系,不能作为架构契约 |

**推荐**:在 M1 文档层先用 B(降低承诺风险),M9 GA 之后在外部叙事中升级到 C。

| 10 | Recipe 与核心 workflow 重叠,出现"两条路做同一件事" | Recipe **只调用核心 workflow**,不实现 workflow 逻辑;Recipe 文件本身有 `recipe.v1` schema(强约束) |
| 12 | capability-registry advisory facts 被外部误读为 *authoritative registry* | 在 `capability-registry.json` 顶部增加 `authority_level: advisory` + `consumer_warning` 字段 |
| 13 | trust-score 被滥用为"自动屏蔽"机制 | trust-score **只 advisory**;实际拦截由 user 显式策略 + capability-manifest permission 决定 |
| 14 | spec-first 不内化 sandbox 导致企业用户认为安全能力缺失 | 文档明确"sandbox 由 host runtime 提供";signature 验证由 spec-first 提供;两者分层 |
| 16 | Semantic Router 误匹配 skill 导致工作流走偏 | facts(metadata)优先;router 必须可审计(输出 reason_code);user 可手动覆盖 |
| 17 | "不直接集成自进化"被误读为 spec-first 不支持 skill 演化 | 角色契约 §11 已声明"AI 改动需有审查闭环";compound-delta 即等效演化机制 |

### 8.2 当下机遇判断(2026-05 窗口期)
