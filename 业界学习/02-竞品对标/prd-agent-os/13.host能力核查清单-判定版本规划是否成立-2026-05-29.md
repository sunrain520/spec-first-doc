# Host 能力核查清单(判定版本规划是否仍然成立)

## 用途

`docs/00-版本路线/版本规划.md` 的战略正确性是**条件性**的:它默认 host runtime 演进对 spec-first 是利好(§2 正文),但 §2.1 Anti-thesis 指出 host 可能从下往上吃掉 M4/M5/M6,且 spec-driven 范式本身可能降温。

这张清单把"规划对不对"从主观判断推进到**可证伪核查**:每项给出能力点、要回答的精确问题、一手来源、以及 `发现 → 对路线图的含义` 的判定规则。

**这不是"去搜一下 host 动态",而是"回答这个具体问题,按判定规则得出对路线图的含义"。**

## 时效边界(必读)

- 本清单创建于 2026-05-29,创建者所在会话**无联网工具**,因此所有"当前 host 能力状态"列**留空待填**,不预填猜测。
- 版本规划基线为 v1.8.2 / 2026-05-27;作者知识截止 2026-01。2026-02..05 的 host 原生能力进展是规划最大盲区,正是本清单要消除的。
- 每项的"判定规则"是中立 if/then,不夹带预判结论。

## 执行约束

- **最小有效集 = A1 + A2 + A3**。只做这三项即可定住威胁 A(规划最大盲区)约 70% 的判断。
- 每项产出 = **一句结论 + 一手链接 + 核实日期**,不写长报告。判定规则已把"发现 → 含义"写死,填空即可。
- 复核节奏绑定版本规划已有的 90 天 reality-check 周期(见 frontmatter `review_cycle_days`),不另起炉灶。
- 谁来填:需联网/一手源的会话或人工。填写后更新本文件 `status: open → in-progress → resolved`,并把结论回写版本规划对应 Open Question 与 §20 决策 7 的 complement/compete 标注。

## 🔴 威胁 A:host 是否在吃掉 M4 / M5 / M6(最高杠杆,优先做)

判定尺子:host 几乎必然提供**单宿主原生版**;关键是它是否侵入 spec-first 的护城河——**跨宿主中立 + source/runtime 边界 + 结构化可消费 artifact**。只有当 host 提供"跨 host 中立 + 结构化"的版本时,才算 compete。

### A1 ⭐ 原生 code review 的结构化程度

- **精确问题**:Claude Code / Codex 的原生 review 是否输出**结构化、可被外部 workflow 消费的 finding**(severity / category / evidence schema),还是只给人读的散文?
- **一手来源**:Anthropic Claude Code docs + changelog;OpenAI `developers.openai.com/codex` + blog
- **判定规则**:
  - 仅散文 / 单宿主 → **M4 仍是 complement**,规划对。
  - 已输出结构化、可跨用 finding → **M4 转 compete**,须按 §20 决策 7 声明差异化或收缩为消费 host review facts。
- **当前状态**:_(待填:结论 / 链接 / 日期)_

### A2 ⭐ 原生 session 恢复 / 后台长任务

- **精确问题**:Claude Code 的 resume/checkpoint、Codex 的 cloud background tasks / threads / automations,是否已覆盖"中断恢复 + 决策日志 + 下一步",且**状态可跨 host 迁移**?
- **一手来源**:Claude Code release notes(resume、checkpoints);Codex automations / cloud tasks 文档
- **判定规则**:
  - 状态 host-locked → **M5 的"跨 host 中立 run-state"仍有价值**。
  - 已能跨 host 导出 / 恢复 → **M5 大幅贬值**,应收缩为消费 host session facts。
- **当前状态**:_(待填)_

### A3 ⭐ 原生 memory 的证据化程度

- **精确问题**:Claude Code 的 memory 工具 / `CLAUDE.md`、Codex 的 `AGENTS.md`,是否已支持"**带证据的、confirmed-only 的项目知识沉淀**",还是仅 freeform 记忆?
- **一手来源**:Claude Code memory 文档;Codex AGENTS.md 规范
- **判定规则**:
  - 仅 freeform → **M6 的 evidence-governed compound(§6.8 只沉淀 confirmed)仍是差异化**。
  - 已带 provenance / confirmed 机制 → **M6 需重新定位**。
- **当前状态**:_(待填)_

### A4 原生 subagent / 并行调度

- **精确问题**:host 的 subagent 是否已能做"selective reviewer dispatch"(按改动类型分派专家)?
- **一手来源**:Claude Code subagent 文档;Codex parallel threads
- **判定规则**:若是 → **M4 的 reviewer dispatch(§12)从能力降为"在 host 原语上的编排薄层"**,定位要降级。
- **当前状态**:_(待填)_

### A5 原生 diff / 影响面 / 代码图谱

- **精确问题**:host 是否原生提供 repo 级代码图谱 / 影响分析?
- **一手来源**:各 host docs;GitHub Copilot codebase indexing
- **判定规则**:若 host 内置 graph → **M2 的 GitNexus provider 层价值收窄到"跨 host 中立 + 可替换"**,护城河更窄。
