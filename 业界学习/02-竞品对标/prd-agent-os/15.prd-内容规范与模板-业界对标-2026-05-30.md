# PRD 内容规范与模板:业界是怎么定义的(2026-05-30 对标)

## 结论先行

业界对 PRD「内容规范 / 模板」的定义,**没有单一统一标准**,但收敛出一个跨流派稳定的**核心骨架**,再叠加各流派的差异化取向。关键趋势:

1. **传统 PM 流派(ProductPlan / Aha / Wiegers)**:模板偏完整、面向 release,核心是 `Objective + Features(每个带 use case)+ System/Env Requirements + Assumptions/Constraints/Dependencies`。明确「**PRD 描述 WHAT 不规定 HOW**」(may hint at implementation, may not dictate it)。
2. **敏捷流派(Atlassian Confluence)**:模板偏轻量,强调 `Objective + Success metrics + Background/strategic fit + Assumptions + User stories + Not doing(显式 out-of-scope)+ Open questions`,反对重仪式。
3. **AI-native / spec-driven 流派(GitHub Spec Kit / testany / CodeStable)**:模板**结构化、可机器消费、可追溯**,核心是 `Prioritized User Stories(独立可测)+ Acceptance Scenarios(Given/When/Then)+ Functional Requirements(稳定 ID + MUST)+ Measurable Success Criteria + Assumptions`,并显式标注未决点。

跨三流派的**最小公共核心(business-grade 通用)**:
> Objective/Problem · Scope(含显式 Non-Goals)· Requirements/User Stories · Acceptance/Use Cases · Success Metrics · Assumptions/Constraints/Dependencies · Open Questions

## 一手证据 1:GitHub Spec Kit `spec-template.md`(AI-native 标杆,curl 完整抓取)

Spec Kit 是「spec-driven development」最具代表性的开源标杆,其 spec 模板就是 AI-native 的 PRD。结构(原文 section,逐字):

| Section | 标记 | 关键约束 |
| --- | --- | --- |
| User Scenarios & Testing | **mandatory** | User stories 必须**按优先级排序(P1/P2/P3)**、**每条独立可测**(实现单条即一个可用 MVP) |
| → Acceptance Scenarios | 每条 story 内 | 强制 **Given / When / Then** |
| → Edge Cases | | 边界条件 + 错误场景 |
| Requirements | **mandatory** | Functional Requirements 用 **稳定 ID(FR-001…)+ MUST** 句式 |
| → 未决点标记 | | `[NEEDS CLARIFICATION: ...]` 内联标注,不许猜 |
| Key Entities | include if data | 只描述含义/属性/关系,**不含实现** |
| Success Criteria | **mandatory** | **technology-agnostic 且 measurable**(如「2 分钟内完成」「1000 并发不降级」)|
| Assumptions | | 记录"描述未指定时所选的合理默认",含对现有系统/服务的依赖 |

**对 spec-first 的直接启示**:
- 三个 `*(mandatory)*` 标记 = 业界标杆已经在用 **core(强制)vs optional(条件)** 的分层,这正印证 `003` 的 R16/R16b core+conditional 决策。
- `[NEEDS CLARIFICATION]` 内联标记 = 与 `003` R22「无证据内容必须进 Assumptions/Outstanding Questions」同向。
- 稳定 ID(FR-/SC-)= 与 spec-first 的 R/A/F/AE traceability 同向。
- Success Criteria 强制 **measurable + technology-agnostic** = 与 `003` Success Signals「不编造业务指标、衡量 workflow 质量」部分同向,但 Spec Kit 要求每个 spec 内必须有可度量验收,比 `003` 更硬。

## 一手证据 2:ProductPlan PRD glossary(传统 PM 权威,curl 整页抓取)

传统/waterfall-leaning 定义,逐条摘录其「What Should a PRD Contain」:

- **Objective/Goal**:为什么建、想达成什么。
- **Features**:每个 feature 至少含 description + goal + use case;复杂的拆 sub-items,各自带 use case;可含 out-of-scope。
- **UX Flow & Design Notes**:描述整体用户工作流,**不放 pixel-perfect mockup/wireframe**(那是 PRD 之后的事)。
- **System & Environment Requirements**:支持的浏览器/OS/内存等;可用性需求(如移动端单手导航)。
- **Assumptions, Constraints & Dependencies**:期望前置(未保证)/ 实现不能依赖的限制 / 外部依赖。

权威立场两条(逐字):
- 「PRDs **may hint at** a potential implementation to illustrate a use case, but they **may not dictate** a specific implementation.」→ 与 `003` R23「PRD 不得含 implementation units/schema/接口字段/任务拆解」**完全一致**。
- 「The PRD itself **does not touch on market opportunity or revenue**, but is firmly rooted in **use cases and desired functionality**.」→ 市场/营收归 MRD,PRD 只管 use case 与功能。与 `003` 不默认做竞品/市场研究(NG5)同向。

## 一手证据 3(prior-verified,来自 001 的 curl)

## 综合:业界 PRD 模板的「定义方式」共性

1. **核心骨架统一,展开动态**——所有流派都有一个稳定核心(Objective + Requirements + Use Cases/Acceptance + Scope/Non-Goals + Assumptions),再按方法论轻重/产品阶段动态裁剪。**没有谁主张"一个硬模板全场景全填"**;Spec Kit 直接用 `*(mandatory)*` vs optional 分层。
2. **WHAT not HOW 是铁律**——传统(ProductPlan「may not dictate implementation」)与 AI-native(Spec Kit「without implementation」「technology-agnostic」)一致。
3. **显式 Non-Goals / Out-of-scope / Not doing 是标配**——三流派都单列。
4. **Assumptions/Constraints/Dependencies 是标配**——传统流派尤其强调,AI-native 用内联 `[NEEDS CLARIFICATION]` 强化。
5. **AI-native 增量**:稳定 ID、Given/When/Then 验收、独立可测的优先级 user story、measurable success criteria、机器可消费结构。
6. **业界普遍弱的点(spec-first 差异化空间)**:① 按 surface(App/H5/Admin/Backend/CLI)差异化 lens——未见成熟公开实践;② current-state 证据分级 + 「现状不反向扩 scope」治理——Sourcegraph/CodeScene 有 graph 但未明文化此治理。

## 对 `003` 的校准提示(advisory)

| `003` 决策 | 业界对标结论 |
| --- | --- |
| R16/R16b core+conditional 分层 | ✅ 与 Spec Kit `*(mandatory)*` vs optional **完全同构**,有一手先例支撑 |
| R23 禁 HOW | ✅ 传统 + AI-native 双流派铁律,证据最强 |
| Non-Goals 单列 | ✅ 三流派标配,`003` 已有 |
| Evidence/Assumptions 治理 | ✅ 同向;Spec Kit 的内联 `[NEEDS CLARIFICATION]` 可作 PRD 内未决点标注的轻量参考 |
| Success Signals 不编业务指标 | ⚠️ 与业界部分张力:Spec Kit/Aha/Lenny 都要求 PRD 内含 **measurable success criteria**。`003` 的 S1-S5 是"评估 spec-prd workflow 质量"的信号,**不是 PRD 文档内的成功度量**——两者不是一回事。建议 plan 阶段确认:`prd-output-template.md` 是否要求**每份产出 PRD** 含 measurable success criteria(business-grade PRD 业界标配),与 spec-first 既有"不编造指标"如何调和(答案大概率是:有证据/用户给出时填,无则进 Assumptions)。 |
| 按 surface 差异化 lens(R24-R30) | ⚠️ 业界未见成熟公开等价物,是真差异化但样本少,维持 advisory |

**唯一值得 plan 注意的张力**:business-grade PRD 业界标配「文档内 measurable success criteria」,而 `003` 当前把 Success 放在 workflow 评估层、且 NG 倾向不编指标。这两者不矛盾(一个是文档内容、一个是 skill 质量信号),但 `prd-output-template.md` 设计时要明确:**产出 PRD 是否设 Success Metrics section(conditional)**,有证据则填、无则入 Assumptions——既符业界标配又守 spec-first 不编造原则。
