## 一、AI PRD 思维框架

### 1.1 核心差异：确定性 vs 概率性

AI PRD 与传统 PRD 的本质差异不是格式变化，而是底层思维范式的转换。传统 PRD 建立在确定性逻辑之上——定义输入、预期输出、边界条件，工程实现后 QA 验证，功能"通过"或"不通过"。AI PRD 必须面对一个不可规避的事实：同一个输入可能每次产生不同输出，"正确"不是二值判断而是一个概率分布。

| 维度 | 传统 PRD | AI PRD |
|------|---------|--------|
| **假设前提** | 系统是确定的（Deterministic） | 系统是不确定的（Probabilistic） |
| **核心问题** | "系统要做什么" | "系统能做多好" |
| **输出特点** | 给定输入 X，产生输出 Y | 给定输入 X，产生输出 Y（以 Z% 准确率），置信度低于阈值触发兜底行为 W |
| **评估方式** | 功能是否完成（Binary Pass/Fail） | 模型效果好不好、能否上线、能否持续学习（Continuous Spectrum） |
| **版本迭代** | 功能迭代 | 数据迭代 + 模型迭代 + 策略迭代 |
| **风险点** | 业务需求变更 | 数据质量、偏见、安全、模型漂移、性能不稳定 |
| **验收标准** | "功能正常工作" | "输出在定义的质量阈值内，且翻车时有兜底" |
| **文档生命周期** | 一次性产物，上线即结束 | 活文档，上线才开始真正的迭代 |

**从三个维度理解这一范式转换**（源自微信推文框架 + ainna.ai + ideaplan.io 三方共识）：

- **输入侧**：AI 的输入不确定——自然语言、多模态、非结构化数据，需要 Confidence 阈值和 Eval 集来管理质量。传统软件输入是结构化的，不需这些。
- **输出侧**：AI 的输出不确定——同一输入可能产生不同输出、可能幻觉、可能出错，需要兜底链、Token 成本预算、灰度回滚。传统软件输出确定，不需要。
- **运营侧**：AI 需要持续优化——数据回流、模型漂移监控（MAD）、偏见审计、提示词变更回归测试。传统软件上线后基本稳定。

### 1.2 AI PRD 八大新增章节详析

以下八章是 AI 产品 PRD 必须新增的核心章节，综合了微信推文（大厂审批实践）、51CTO（六大支柱框架）、LinkedIn（Alena Zakharova 的 AI Product PRD 定义）、ideaplan.io（2026 模板）和 sigos.io（B2B 实践指南）的交叉共识。

---

#### 章节一：风险等级与 Confidence 阈值矩阵

**放置位置**：项目背景后，核心需求前。

**写什么**：
- **项目风险等级**：高（医疗诊断、金融风控）、中（智能客服、内容审核）、低（内容推荐、文案生成）。风险等级直接决定后续所有章节的严格程度。
- **Confidence 阈值矩阵**：高风险场景模型置信度 ≥0.90 才输出；中风险 ≥0.80；低风险 ≥0.70。低于阈值触发兜底链。
- **预估触发率**：预估多少比例的请求会触发兜底链——这个数字直接决定成本和用户体验。L1 理想覆盖 80%、L2 覆盖 15%、L3 控制在 5% 以内。

**为什么不写不行**：评审一定会问"翻车了怎么办"。没有明确的风险边界，项目过不了评审。风险等级是后续 Eval 集设计、兜底链、灰度方案的"输入参数"——没定义风险等级，这些章节就不知道以什么标准写。

**与传统 PRD 章节的映射**：传统 PRD 的「风险分析」章节仅列出业务风险，不量化到模型级别。AI PRD 的风险等级是模型行为约束的起点。

---

#### 章节二：Eval 集设计与上线 NoGo 标准

**放置位置**：核心功能后，数据章节前。

**写什么**：
- **Eval 集规模**：最少 200 条。包含正常 case、bad case、对抗样本、边界样本。
- **Eval 集三层金字塔**：
  - 单元级（Unit）：单一模型能力的独立测试，如"分类是否正确""抽取是否完整"
  - 链路级（Integration）：多组件协作场景，如"意图识别 → 槽位填充 → API 调用 → 回复生成"
  - 端到端级（End-to-End）：完整用户场景，如"用户说'帮我查物流' → 正确返回物流信息"
- **评估维度分离**（源自 sigos.io / ideaplan.io）：模型指标（precision、recall、F1、幻觉率）和业务指标（用户满意度、采纳率、干预率）分别定义。不把两者混为一个数字。
- **上线 NoGo 标准**：准确率低于多少不能上、翻车率超过多少不能上、P95 延迟超过多少不能上。量化到具体数字。按用户群体、风险等级、使用场景分档设定——"95% 准确率"在最重要分类上可能是 60%。
- **A/B 测试方案**：每次 Prompt 或模型变更需跑完整的 Eval Suite，设定最大退化阈值（如"任一维度退化不超过 2%"）。

**为什么不写不行**：没有 Eval 集的产品上线后覆盖不足导致投诉率激增。工程团队不知道什么时候算"做完了"。"看起来很对"不是上线标准。原型能告诉你系统做什么，不能告诉你"多好才算好"。

**与传统 PRD 的映射**：传统 PRD 用测试用例（Test Case），覆盖 happy path 和异常路径。AI PRD 的 Eval 集覆盖的是概率分布中的典型、边界和对抗样本——测试的不是"对不对"而是"多对"。

---

#### 章节三：兜底链分层 SOP

**放置位置**：错误处理 / 异常流程章节。

**写什么**：
- **三层兜底结构**：
  - L1 自动降级（目标覆盖 80%）：换更可靠的模型或规则、降低温度参数、缩减上下文窗口
  - L2 规则兜底（目标覆盖 15%）：硬编码安全回复、返回"我无法确定，建议您…"、展示置信度指标让用户判断
  - L3 人工兜底（目标控制 5% 以内）：转人工处理、异步审核后通知用户
- **每层触发条件**：L1 触发条件（置信度低于阈值、P95 延迟超标）、L2 触发条件（L1 降级后仍不可接受）、L3 触发条件（L2 无法覆盖的场景）
- **触发率告警阈值**：L3 从 5% 跳到 20% → 模型可能出问题需要紧急排查
- **用户体验设计**：用户看到什么？旋转图标？错误提示？置信度指示器？还是静默降级？

**为什么不写不行**：工程团队需要按这个 SOP 写代码。兜底逻辑不清晰，月成本可能高到被叫停。关键不在防止 AI 犯错，在 AI 犯错之后系统如何优雅降级——这是产品逻辑，必须写入 PRD。

**与传统 PRD 的映射**：传统 PRD 的「异常处理」章节处理网络超时、数据为空等确定性异常。AI PRD 的兜底链处理的是概率性失败——模型没崩但输出不够好。

---

#### 章节四：Token 成本预算

**放置位置**：成本估算 / 商业可行性章节。

**写什么**：
- **单次请求成本分位数**：P50（典型情况）和 P95（极端情况）同时提供。P50 代表日常开销，P95 代表风险敞口。
- **日 / 月预算上限**：基于 DAU 预估和 P50 成本计算基线，P95 预留 1.5-2x buffer。
- **超预算降级策略**：自动切换到更便宜模型（如 GPT-4 → GPT-4o-mini）、限速（QPS 上限）、关闭非核心功能（如禁用辅助推荐只保留主回复）。
- **12 个月成本建模**（源自 ideaplan.io）：原型阶段 $0.03/query 看似便宜，乘以 50,000 DAU 后完全不同。必须在 PRD 阶段建模 12 个月增长后的成本。
- **异常消耗告警**：超预算 20% 自动告警。

**为什么不写不行**：CFO 在 Review 时一定会问成本。很多 AI 项目上线后月消耗远超预期，被老板直接叫停。Token 成本不是技术细节，是商业可行性判断的核心输入。

**与传统 PRD 的映射**：传统 PRD 的成本估算在服务器和人力。AI PRD 的成本按 token 计费——输入越长、模型越强、并发越高，成本呈非线性增长。

---

#### 章节五：数据脱敏与 Bad Case 回流机制

**放置位置**：运营 / 数据章节。

**写什么**：
- **用户反馈闭环**（源自 ideaplan.io / sigos.io）：
  - 拇指向上/向下：简单二值反馈
  - 纠错界面：用户修正具体错误 → 生成标注数据
  - 报告流程：用户标记有害/不准确/不当输出
- **Bad Case 标注 SOP**：谁收集（PM 还是运营）、谁标注（内部团队还是外包）、怎么标（标注规范文档）、多久审一次（周度还是月度）。
- **模型迭代周期**：周迭代（快速修复 Bad Case）还是月迭代（系统优化）。
- **数据脱敏字段清单**：用户 PII（姓名、手机、身份证）、财务数据、医疗信息等必须脱敏到什么程度才可进入训练/标注流程。
- **改进工作流**（源自 ideaplan.io）：
  - 周度：审查标记输出，识别模式
  - 月度：分析聚合质量指标，对比阈值，决定 Prompt 调整或模型变更
  - 季度：评估 Eval 标准本身是否需要根据真实使用情况更新

**为什么不写不行**：没有数据回流机制，模型无法持续优化。半年后竞品迭代三轮，你的模型还在用上线时的版本。数据质量属于 PRD 的职责，不应埋在技术文档里。

**与传统 PRD 的映射**：传统 PRD 的运营章节关注 Bug 收集和版本规划。AI PRD 需额外定义"模型层面的反馈采集→标注→回流→再训练"完整闭环。

---

#### 章节六：合规审核

**放置位置**：合规 / 风控章节。

**写什么**：
- **数据脱敏 SOP**：用户数据进入模型前必须经过的处理（脱敏字段、脱敏方式、脱敏程度）
- **内容安全双层过滤**：输入过滤（用户输入是否包含越狱/注入/有害内容）和输出过滤（模型输出是否包含敏感/有害/违规内容）各需一层。任一环节缺失都是合规缺口。
- **偏见审计**：覆盖性别、地域、年龄、收入等维度。保留审计痕迹（谁审的、什么时间、发现什么、如何处置）。
- **可回溯性**：所有模型决策必须可解释可回溯。用户问"为什么给我这个结果"，要能给出解释和溯源路径。
- **数据确权与权限**：哪些数据源是权威的（如 Zendesk vs. 销售电话记录，哪个更能反映真实痛点）。明确数据排除清单。
- **第三方模型数据使用政策**：审查模型供应商的数据使用政策。GDPR 擦除权、CCPA 删除请求、HIPAA 医疗数据、SOC 2 企业合规——AI 特性往往触发传统功能不会激活的合规要求。

**为什么不写不行**：GDPR、个保法、网信办都有明确要求。不合规的项目法务直接驳回，没有商量余地。AI 产品的合规面比传统软件多一个数量级——数据采集、模型训练、内容输出各有合规关卡。

**与传统 PRD 的映射**：传统 PRD 的合规章节侧重数据存储和传输安全。AI PRD 增加了模型偏见、可解释性、内容安全三个新维度。

---

#### 章节七：灰度方案与回滚 SOP

**放置位置**：上线计划章节。

**写什么**：
- **灰度四档分桶**：5%（内部员工/种子用户，1-2 天）→ 20%（活跃用户样本，3-5 天）→ 50%（半量，1 周）→ 100%（全量）。每档跑多久、看什么指标、什么条件进入下一档。
- **Kill Switch（一键关停）**：紧急情况下的一键关停机制——谁有权触发、触发后多久生效、关停后用户看到什么。
- **自动回滚条件**：翻车率超过阈值 X%、触发率超过阈值 Y%、P95 延迟超过阈值 Z 毫秒 → 自动回滚到上一稳定版本或规则兜底。
- **回滚 SOP**：谁有权回滚（值班工程师/PM/技术负责人）、多久内完成（5 分钟/15 分钟）、回滚后如何通知用户和干系人。
- **Prompt 变更回归测试**：每次 Prompt 修改跑完整 Benchmark Suite。模型升级（如 GPT-4→GPT-4.5、Claude 3.5→Claude 4）必须跑完整 Eval——不要假设新模型对你的场景一定更好。

**为什么不写不行**：AI 产品翻车风险远高于传统软件。全量上线出问题，DAU 可能一天掉一半。灰度不是"有了就行"——必须预设退出条件和自动回滚触发线。

**与传统 PRD 的映射**：传统 PRD 的灰度看功能有没有 Bug。AI PRD 的灰度看模型表现稳不稳定——Bug 是确定性故障，模型退化是渐进的、不易察觉的。

---

#### 章节八：持续运营指标

**放置位置**：运营 / 监控章节。

**写什么**：
- **模型漂移监控（MAD）**：模型表现随时间变差的监测——真实输入分布与测试集分布偏差的持续追踪。
- **翻车率监控与告警阈值**：按类型分层的翻车率（幻觉翻车、格式翻车、安全翻车），各设独立告警阈值。
- **触发率分层报表**：多少请求走 L1（自动降级）、多少走 L2（规则兜底）、多少走 L3（人工兜底）。L3 突然飙升是最危险的信号。
- **周复盘 SOP**：每周看一次数据，调整阈值和策略。审查真实验证样本（100 条/天起步，稳定后改为周度）。
- **月预算偏差告警**：超预算 20% 自动告警。
- **用户干预率追踪**：用户修改/覆盖 AI 输出的频率——如果干预率持续上升，模型可能在漂移。
- **Prompt/模型变更后的自动回归对比**。

**为什么不写不行**：没有监控指标，问题可能积累几个月才被发现。到时候用户已经流失，赔偿已经产生。AI 产品上线才是真正的开始——监控不是运维的事，是 PM 要在 PRD 中定义"什么叫正常、什么叫异常"。

**与传统 PRD 的映射**：传统 PRD 上线后监控 QPS、延迟、错误率。AI PRD 还需监控模型本身——模型漂移、翻车率、触发率、成本偏差。这些是 AI 产品独有的健康度指标。

## 二、现有 PRD 相关 Skill 能力盘点

### 2.1 pm-skills 的 PRD 能力

pm-skills（phuryn/pm-skills，13.3k Star，MIT License）是将经过验证的产品管理方法论编码为 AI 可调用工作流的开源项目，其 PRD 相关能力集中在 `pm-execution` Plugin 中。

**已覆盖能力**：

| Command | 功能 | 核心价值 | 对应 AI PRD 章节 |
|---------|------|---------|-----------------|
| `/write-prd` | 结构化 PRD 生成，基于 Dan Olsen 方法论 | 从灵感 → 可执行方案的端到端输出，含 pre-mortem 自检 | 传统 PRD 全部章节（背景/目标/范围/用户故事/验收标准） |
| `/red-team-prd` | 攻击性假设审查——从批评者视角攻击 PRD 中最危险的 5 个假设 | "大多数 PRD 只能经受住礼貌反馈"——在评审前主动暴露盲点 | △ 兜底链（挑战翻车场景但不量化阈值） |
| `/plan-launch` | 上市发布计划，包含上线步骤、沟通、风险预案 | 部分覆盖灰度/回滚逻辑 | △ 灰度与回滚（部分覆盖） |

**缺失能力**：

pm-skills 的 PRD 相关技能全部基于传统确定性软件的方法论（Dan Olsen、Marty Cagan 等），**没有针对 AI 概率性产品的专项能力**。八大 AI PRD 章节中：

- **完全未覆盖**：Confidence 阈值矩阵、Eval 集与 NoGo 标准、Token 成本预算、数据脱敏与 Bad Case 回流、持续运营指标
- **部分覆盖**：兜底链（`/red-team-prd` 挑战翻车场景但不定义分层兜底 SOP）、灰度与回滚（`/plan-launch` 有部分逻辑但不区分模型表现 vs. 功能 Bug）、合规审核（传统 PRD 有合规章节但无 AI 专项偏见审计和内容安全双层过滤）

### 2.2 OpenSpec+Superpowers 的 PRD 相关能力

OpenSpec（Fission-AI，37K+ Star）+ Superpowers（Jesse Vincent / obra）组合提供了"规格驱动开发"的完整框架。

**已覆盖能力**：

| 组件 | 产出物 | 核心价值 | 对应 AI PRD 章节 |
|------|--------|---------|-----------------|
| OpenSpec proposal.md | 为什么做：背景、目标、范围、边界 | 在编码前对齐预期 | 传统 PRD 背景/目标/范围 ✅ |
| OpenSpec design.md | 怎么做：技术方案、模块设计 | 固化技术决策，可复用 | 传统 PRD 技术方案 ✅ |
| OpenSpec tasks.md | 分几步：原子任务拆解（输入/输出/验收标准三元组） | 每个 step 的验收标准是 Eval 集的前置条件 | △ Eval 集与 NoGo（tasks 有验收标准但非 AI Eval 视角） |
| OpenSpec spec.md | 做成什么样：功能规格定义（Delta Specs） | 可验证的验收标准 | 传统 PRD 功能规格 ✅ |
| Superpowers Socratic Questioning | 方案对比 + 推荐 + 确认 + 决策记录 | Brainstorming 阶段的主动追问 | 传统 PRD 需求澄清 ✅ |

**缺失能力**：

OpenSpec+Superpowers 的核心设计面向传统确定性软件开发，"规格驱动"中的"规格"定义的是确定性行为——"给定输入 X，应产生输出 Y"。**整个四件套体系中没有任何处理概率性输出的机制**：

- proposal.md 中没有 AI 风险评估章节
- design.md 中没有模型选择、置信度阈值、兜底链设计
- tasks.md 的验收标准是"是否正确"而非"有多对"
- spec.md 定义的是精确输出而非可接受范围

### 2.3 spec-first 的 PRD 相关能力

spec-first 是 AI 编码研发体系（37 Skills + 51 Agents），覆盖 Bootstrap → Ideate → Brainstorm → Plan → Work → Review → Compound 七阶段。

**已覆盖能力**：

|------|--------|-----------------|

| Brainstorm | 需求澄清 + decisions.md | 传统 PRD 需求澄清 ✅ |
| Plan | plan.md（实施规划） | 传统 PRD 技术方案 ✅ |
| Review | 代码审查 | 无 AI PRD 专项章节 ❌ |

**缺失能力**：与 OpenSpec 类似，spec-first 完全缺失 AI 产品的专项能力——无置信度管理、无兜底链设计、无成本预算、无 Eval 集标准、无模型漂移监控。

4. **传统 PRD 章节（背景/目标/范围/用户故事）三大体系都已成熟覆盖**——不需要从零建设，仅需在现有基础上追加 AI 专项章节。

## 四、PRD Skill 优化方法论

基于差距矩阵的分析结论，提出分层优化方案。方案的核心设计原则是：**不另起炉灶，而是在现有能力上增量构建**——pm-skills 的传统 PRD 生成能力、OpenSpec 的四件套文档体系、spec-first 的阶段化工作流都是已成熟的基础设施，优化方案是让它们"长出 AI 原生能力"。

### P0：AI PRD 核心思维注入（立即行动）

**目标**：让现有的 PRD 生成流程具备 AI 产品的最基本思维——识别不确定性、管理不确定性、量化不确定性。

#### 4.1 创建 `ai-prd-framework` Skill

- **定位**：作为 `/write-prd` 的前置或增强 skill，当用户描述的需求涉及 AI/ML/LLM 时自动激活（关键词触发：模型、AI、LLM、大模型、智能、推荐、生成、分类、预测、对话、Agent）。
- **核心机制**：强制输出八大章节中**不可跳过的四章**——这四个是"不写就过不了评审"的硬门槛：
  1. **风险等级与 Confidence 阈值矩阵**：自动追问"这个场景的风险等级是什么？"，引导用户定义高/中/低风险场景及其对应置信度阈值
  2. **兜底链分层 SOP**：自动追问"模型翻车了怎么办？"，引导用户定义 L1/L2/L3 三层兜底策略及触发条件
  3. **Eval 集设计与 NoGo 标准**：自动追问"上线的最低标准是什么？"，引导用户定义 Eval 集规模、结构和量化 NoGo 指标
  4. **Token 成本预算**：自动追问"预算上限是多少？"，引导用户提供 P50/P95 成本分位数和超预算降级策略
- **Socratic 追问模式**（借鉴 Superpowers Brainstorming）：当用户描述需求时，Skill 内置主动追问机制——每次只问一个关键决策点，给出 2-3 个可选方案 + 优劣分析 + 推荐方案 + 理由，用户确认后记录并进入下一个决策点。
- **输出格式**：在原 `/write-prd` 产出的传统 PRD 基础上，追加四个 AI 专项章节作为 PRD 的 Part II，形成"传统骨架 + AI 专项"的两段式 PRD。

#### 4.2 改造 `/red-team-prd` 为 AI 感知版本

- **现有能力复用**：`/red-team-prd` 的核心机制（找出最危险的 5 个假设 + 按"最需优先验证"排序 + 推荐最便宜的验证实验）无需改动，直接复用。
- **新增 AI 专项攻击面**：
  - 攻击置信度阈值：0.90 是否合理？低风险场景 0.70 会不会太松？有没有场景应该用 0.95 但只设了 0.85？
  - 攻击兜底链：L1/L2/L3 的覆盖比例是否合理？有没有翻车场景三层都兜不住？人工兜底的响应时间能保证吗？
  - 攻击 Eval 集：200 条够不够？有没有覆盖对抗样本？正常 case 占比是否过高导致虚假信心？
  - 攻击成本模型：P95 成本是基于什么数据的？MAU 增长 10 倍后的成本还算过吗？超预算降级后的用户体验退化多少？
  - 攻击合规盲区：偏见审计覆盖了所有必须的维度吗？内容安全过滤是单层还是双层？模型决策可回溯吗？
  - 攻击模型漂移：上线三个月后如果输入分布变了怎么办？监控指标足够灵敏吗？

### P1：审查与验证闭环（下个迭代）

**目标**：建立 AI PRD 的审查和验证机制，让 PM 写完 PRD 后能自动获得一份"攻击报告"和一套"验证框架"。

#### 4.3 创建 `/red-team-ai-prd` Command

- **定位**：在传统 `/red-team-prd` 基础上，专门针对 AI PRD 的六类攻击面（置信度、兜底链、Eval 集、成本、合规、运营）进行系统化攻击。
- **工作原理**：读取已生成的 AI PRD（含四大核心章节），对每个章节按预设攻击模板逐一审查，输出结构化攻击报告——每条攻击包含：被攻击的假设、攻击理由（为什么这个假设可能不成立）、风险等级（高/中/低）、推荐验证方式（最便宜的验证实验）。
- **输出**：一份可以附在 PRD 末尾的"AI PRD Red Team 审查报告"，供评审会议使用。

#### 4.4 tasks.md 验收标准增强为 AI 版本

OpenSpec tasks.md 的原子 step 三元组（输入/输出/验收标准）是最接近 Eval 集的现有能力。将其升级为 AI 版本：

```
传统版本（当前）：
## Step 3.2：实现意图识别
- 输入：用户原始消息文本
- 输出：意图类别（查物流/改地址/退款/其他）
- 验收标准：返回正确的意图类别

AI 增强版本（改进后）：
## Step 3.2：实现意图识别
- 输入：用户原始消息文本
- 输出：意图类别 + 置信度分数（0-1）
- 验收标准：
  - 正确率达到 92% 以上（基于 200 条标注 Eval 集）
  - 置信度低于 0.75 时触发"意图确认"追问而非直接执行
  - 对抗样本（故意混淆意图的输入）不导致高风险操作误触发
  - 未知意图正确归类为"其他"而非高置信度误分类
- 边界 case：
  - 空输入 → 追问
  - 超长输入 → 截断 + 降级兜底
  - 混合意图 → 识别主导意图 + 追问是否同时处理次要意图
```

**核心变化**：验收标准从"对不对"升级为"多对"+"不对时怎么办"+"边界有多远"。

#### 4.5 Eval 集自动生成骨架

基于 tasks.md 中每个 AI 相关 step 的验收标准，自动生成初始 Eval 集骨架：

- 每个验收标准自动生成 3-5 条对应的 Eval case
- 每个边界 case 自动生成 2-3 条 Eval case
- 自动按三层金字塔（单元/链路/端到端）分类
- 自动标注每条 Eval case 的类型（正常/bad case/对抗/边界）
- 输出为结构化 JSON/CSV，可直接导入评估平台

**注意**：自动生成的仅是最小可行骨架（预计 50-80 条），需要人工审核和补充到最少 200 条。

### P2：全生命周期运营集成（中长期）

**目标**：将 AI PRD 的思维框架延伸到产品上线后的持续运营阶段，形成从"写 PRD"到"看 PRD 是否还在生效"的完整闭环。

#### 4.6 创建 `ai-prd-ops` Skill

- **定位**：上线后的 AI 产品监控与管理 skill，基于 AI PRD 中定义的质量阈值和告警条件自动配置监控。
- **核心能力**：
  - 从 AI PRD 中自动提取监控指标（置信度阈值、触发率告警线、成本预算上限）
  - 生成监控 Dashboard 配置（翻车率、触发率分层、Token 消耗趋势、模型漂移 MAD）
  - 周/月复盘模板自动生成（基于 AI PRD 中定义的复盘 SOP）
  - 模型漂移告警：真实输入分布 vs. Eval 集分布的偏差检测
  - 与 CodeGraph 集成：上线后追踪模型调用链路，自动标记高 Token 消耗路径，与 PRD 中定义的成本预算对比

#### 4.7 PRD 模板市场——按 AI 产品类型分层

借鉴 pm-skills 的 Plugin 模式（9 个 Plugin 按 PM 领域打包），建立按 AI 产品类型的 PRD 模板体系：

| 模板类型 | 适用场景 | 重点关注章节 | 典型产品 |
|---------|---------|-------------|---------|
| **对话式 AI PRD** | 聊天机器人、客服助手、AI 助手 | 兜底链（防幻觉/越狱）、合规（内容安全双层过滤）、持续运营（翻车率监控） | ChatGPT-like、企业客服 Bot |
| **RAG PRD** | 知识检索增强生成 | 置信度阈值（召回质量 vs. 生成质量分别设）、Eval 集（检索命中率 + 答案正确率双维度）、成本预算（Embedding + LLM 双链路） | 企业知识库问答、文档助手 |
| **推荐/排序 AI PRD** | 内容推荐、搜索排序、Feed 流 | 偏见审计（推荐多样性）、持续运营（CTR/转化率 vs. 模型准确率的双指标）、灰度（A/B 实验设计） | 电商推荐、短视频 Feed |
| **分类/预测 AI PRD** | 风控、标签、意图识别 | 置信度阈值（高风险场景 0.95+ 才自动处理）、兜底链（高风险 L3 人工兜底）、合规（决策可回溯） | 金融风控、内容审核 |
| **生成式 AI PRD** | 文案生成、代码生成、图片生成 | Eval 集（质量维度需更细分——事实性/创意性/风格一致性）、兜底链（有害内容过滤）、成本预算（生成类 Token 消耗高） | 营销文案生成、AI 代码助手 |

## 五、知识链收束

此分析与前序两轮调研的完整衔接：

### 5.1 pm-skills 分析（第一轮）→ 本报告的继承与深化

- **`/write-prd`**：第一轮分析确认其传统 PRD 生成能力成熟（基于 Dan Olsen 方法论），本报告明确了其"成熟但不完整"的定位——缺的不是格式而是 AI 不确定性管理的思维方式。解决方案不是替换，而是通过 `ai-prd-framework` Skill 增强。
- **`/red-team-prd`**：第一轮分析高度评价其"先攻击自己"的思维模式——"在周二杀死一个坏下注，而非一个季度后在领导层面前为它辩护"。本报告确认了这一攻击性思维可直接复用到 AI 专项攻击面（置信度、兜底、成本），仅需扩展攻击维度。
- **`/plan-launch`**：第一轮分析指出其部分覆盖灰度/回滚逻辑。本报告明确了 AI 产品的灰度与"传统灰度"的本质差异——传统灰度看 Bug，AI 灰度看模型漂移。需要从"看功能对不对"升级为"看模型稳不稳"。

### 5.2 OpenSpec+Superpowers 分析（第二轮）→ 本报告的继承与深化

- **proposal → design → tasks → spec 四件套**：第二轮分析指出这是"结构化共识的起点"，本报告进一步明确其局限——四件套定义的是确定性行为，proposal 中缺乏 AI 风险评估章节。解决方案：在 proposal.md 中增加"AI 风险与不确定性管理"子章节。
- **tasks.md 的原子 step 验收标准**：第二轮分析将其定性为"最被低估的产出"——同时是 Plan 产出、Work 执行脚本、Review 检查清单。本报告发现它也是最接近 AI PRD Eval 集前置条件的现有能力——只需从"对不对"升级为"有多对"，即可成为 AI PRD Eval 集骨架的自动生成源。
- **Superpowers Socratic Questioning**：第二轮分析将其定性为"Brainstorm 的增强版"。本报告将其完整注入到 `ai-prd-framework` Skill 的设计中——AI PRD 的核心四章每章都应是"Socratic 追问"的结果，而非 PM 独自填写的模板。
- **OpenSpec Archive 的知识复利**：第二轮分析指出 Archive 是"被严重低估的环节"——让每次开发经验变成可复用资产。本报告进一步点明：AI PRD 归档后的复用价值更高——后续 AI 需求可以参考前序 AI PRD 的 Eval 集设计、兜底链方案、成本模型，形成 AI 产品开发的"知识复利"效应。

### 5.3 跨轮次共识——三个体系共享的核心理念

前序两轮调研和本轮分析共同指向一个核心理念：**"在行动之前穷尽所有假设"**。

| 体系 | 机制 | 本报告中的对应 |
|------|------|-------------|
| pm-skills `/red-team-prd` | 攻击自己的 PRD 假设 | → AI PRD 六类攻击面（P1 优化） |
| Superpowers Socratic Questioning | 追问每一个设计决策点 | → `ai-prd-framework` 内置追问模式（P0 优化） |
| OpenSpec Archive | 不让知识随会话消散 | → AI PRD 归档后可复用（P2 运营集成） |

AI PRD 方法论将这一理念推向了更高维度：不仅要在编码前穷尽假设，还要**穷尽"不确定性"本身**——系统可能在哪些场景犯错、犯错概率多大、犯错后如何兜底、兜底的成本多少、模型随时间退化后怎么办。这不是传统 PRD 的"加几个章节"，而是思维框架的升维。

---

## 六、结论

**AI PRD 方法论对当前 PRD skill 优化的核心启示**：当前所有 PRD 相关 skill 体系（pm-skills、OpenSpec、spec-first）均建立在确定性软件的思维框架之上——它们能很好地回答"系统要做什么"，但完全无法回答"系统能做多好"。PRD skill 的优化不是另起炉灶，而是让现有的成熟体系"长出 AI 原生能力"：让 `/write-prd` 在写完传统 PRD 骨架后自动追问四个硬门槛（置信度阈值 × 兜底链 × Eval 集 NoGo × 成本预算）；让 `/red-team-prd` 的攻击性思维从挑战功能假设升级为挑战概率假设；让 tasks.md 的验收标准从"对不对"升级为"多对 + 不对时怎么办"。一句话——**让 PRD 从"确定性蓝图"进化为"概率性行为契约"，这是 AI 时代产品管理最关键的范式升级。**
