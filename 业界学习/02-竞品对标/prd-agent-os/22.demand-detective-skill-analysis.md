# Demand Detective（需求侦探）Skill 深度分析

> 分析日期：2026-06-14 | 来源：微信公众号文章 / GitHub SKILL.md / README

---

## 一、项目概述

**demand-detective（需求侦探）** 是一个以 Claude Code / Codex 等 AI Coding 工具为载体的 Skill，定位为 **需求前置澄清工具**，而非 PRD 生成器。作者 wdy121 从多年 ERP/业务系统实战中提炼出一个核心洞察：**很多项目真正难的地方不是开发，而是开发前面那一步——需求根本没被说清楚。**

| 维度 | 说明 |
|---|---|
| **定位** | 需求前置澄清工具（非 PRD 生成器） |
| **核心设计立场** | 不急着给方案，先把问题摆到桌面上 |
| **核心理念** | "需求不是写出来的，是问出来的" |
| **适用场景** | ERP / CRM / PLM / 进销存 / 审批流 / 报表系统等业务逻辑重的场景 |
| **不接的活** | 纯 UI 体验优化、不需要业务规则决策的需求 |
| **载体** | Claude Code / Codex / 小龙虾 / Workbuddy 等 AI Coding 工具的 Skill |
| **开源地址** | https://github.com/wdy121/demand-detective-skill |

作者指出了一个行业通病：业务说"我要一个功能"，产品说"好的我画页面"，开发说"好的我做出来"，上线后所有人发现理解的根本不是同一件事。系统一旦做出来，错的流程被固化了，模糊的规则被固化了，临时拍脑袋的字段被固化了，后续的返工和补丁成本指数级上升。

demand-detective 的解法不是帮你"写需求"，而是帮你"把需求问清楚"——在写 PRD 之前，把模糊的、一句话的、靠感觉讲出来的需求，先拆成一组可以确认的业务决策。

## 二、架构分析

### 2.1 三阶段工作流

Skill 将需求澄清拆解为三个克制但递进的阶段，每个阶段有明确的输入、动作和输出：

```
场景共建（Stage 1）  →  决策驱动（Stage 2）  →  驾驭输出（Stage 3）
   强制反讲                A/B 选择题             分级任务单 + 决策日志
   "故事确认"              "把讨论变决策"           "开发-业务-测试三方对齐"
```

**阶段一：场景共建** —— 不急着写方案，先用自己的话把业务故事复述一遍。使用"强制反讲"模板确认角色、时间、操作、结果、风险。输出"场景清单"。这一步的核心价值是**建立共享语境**：让 AI 和业务方先在同一个"故事"里对话，避免各说各话。

**阶段二：决策驱动** —— 把含糊讨论变成可选择、可承担后果的业务决策。核心原则是**按决策影响半径由大到小推进**（L1→L2→L3→L4），配合 A/B 选择题方法论、默认方案+责任声明、业务生命周期 12 维扫描。少问"你想怎么做"，多给"选 A 还是选 B，各有什么代价"。

**阶段三：驾驭输出** —— 输出分级开发任务单（核心确定区 vs 假设验证区）和需求决策日志。前者让开发明确哪些必须严格实现、哪些做成可配置；后者保留决策痕迹，防止日后扯皮。

### 2.2 L1-L4 决策层级设计

这是本 Skill 最核心的结构设计。四个层级按影响半径从大到小排列，**高层决策未定，低层细节不展开**——这与传统需求讨论中"一上来就陷入按钮位置"的问题形成鲜明反差。

| 层级 | 名称 | 影响范围 | 典型决策问题 | 输出决定 |
|---|---|---|---|---|
| **L1** | 战略/边界级 | 项目目标、业务边界、系统建设范围 | 服务谁？全公司共享还是区域隔离？目标优先级？ | 模块边界、组织架构、权限模型、项目一期范围 |
| **L2** | 业务模式/流程级 | 核心流程走向、状态机、库存口径、跨部门协作 | 借出是否审批？是否影响库存？是否跨公司调拨？ | 流程图、状态流转、库存计算、核心业务规则 |
| **L3** | 节点规则/操作级 | 具体页面、按钮、字段、校验、权限控制 | 谁可以新建？借出必填哪些字段？是否支持批量？ | 页面交互、字段规则、权限点、校验逻辑 |
| **L4** | 体验/报表/运营级 | 使用体验、提醒机制、统计报表 | 逾期如何提醒？报表看什么指标？搜索支持哪些筛选？ | 用户体验、运营看板、提醒策略、效率优化 |

L1-L4 的层级递进设计体现了**决策拓扑排序**的思维：L1 的决策会直接影响 L2 的展开范围（如确定"仅服务总部统一管理"后，L2 就不会再展开跨子公司调拨的决策题），L2 的决策又决定 L3 的节点范围。这种设计天然防止了需求膨胀和过度设计。

### 2.3 A/B 选择题方法论

这是 Skill 最核心的工作方式转变：**少问开放问题，多给决策题**。

传统的"你想怎么做？"让业务方难以回答——他们脑子里没有产品模型。A/B 选择题则把问题转化为业务选择：

```
传统方式："你想怎么借？"          → 业务方茫然
A/B 方式："A. 必须审批后出库（代价：流程慢；好处：责任清晰）
          B. 直接扫码借出（代价：易丢失；好处：快速灵活）"
                                   → 业务方："我们公司就应该选 A"
```

每个选项附带**业务代价和系统影响的完整说明**，让决策变成对后果的确认而非对方案的凭空想象。当用户犹豫时，提供"默认选择 + 责任声明"机制——先按成本最低方案上线验证，同时明确后续数据/业务兜底责任。

### 2.4 业务生命周期扫描规则（12 维度）

在 L2/L3 阶段，Skill 内置了一套针对 ERP/业务系统/流程系统的 12 维扫描规则，确保不遗漏关键业务决策点：

| # | 维度 | 核心检查点 |
|---|---|---|
| 1 | **对象定义** | 管理对象是什么？款式/SKU/样衣/色号/尺码/图片/BOM？ |
| 2 | **生命周期** | 从创建→启用→流转→使用→冻结→归档→报废的完整过程 |
| 3 | **操作动作** | 新增/编辑/审核/借出/归还/调拨/盘点/锁定/删除/导出/分享 |
| 4 | **状态流转** | 每个动作前后状态如何变化，是否需要状态机 |
| 5 | **审批控制** | 哪些动作需审批、谁审批、是否允许越级或自动通过 |
| 6 | **库存影响** | 动作是否影响库存、占用库存、可借库存、在途库存、异常库存 |
| 7 | **权限边界** | 总部/子公司/业务员/设计/样衣间/管理员分别能看什么、改什么、借什么 |
| 8 | **共享策略** | 哪些资料公司内共享，哪些按区域/客户/业务员隔离 |
| 9 | **留痕审计** | 哪些操作必须记录人、时间、原因、前后值、附件 |
| 10 | **异常处理** | 逾期未还/丢失/破损/重复建档/无库存借出/跨公司调拨怎么处理 |
| 11 | **数据质量** | 编码规则、必填字段、重复校验、图片/二维码/标签规范 |
| 12 | **报表指标** | 最终看哪些经营指标：样衣利用率、借出周转、款式推广次数、成交转化 |

这 12 维度构成了一个**业务系统的决策清单**——不只是问"怎么做"，而是从对象本体到生命周期到异常兜底全覆盖。它与 L1-L4 决策层级构成二维矩阵：每个维度在不同层级（L1/L2/L3/L4）有不同的决策粒度。

### 2.5 分级输出：核心确定区 vs 假设验证区

这是 Skill 对现实世界中"需求不可能一开始就完全确定"这一事实的诚实承认和工程化应对。

| 区域 | 含义 | 开发要求 | 典型示例 |
|---|---|---|---|
| **核心确定区** | 已确认的业务规则 | 必须严格实现 | "审批后出库"、"归还必须质检" |
| **假设验证区** | 当前不完全确定，先按默认方案做 | 做成可配置、易修改 | "逾期提醒默认提前1天+当天+3天提醒主管" |

假设验证区的设计避免了两大陷阱：一是"假装已经确定了"导致后续改动成本高；二是"因为不确定就不做"导致项目停滞。**承认它是假设，然后设计成可配置**——这一工程思维对实际项目极具价值。

## 三、逐维度对照矩阵

以 spec-first 方法论为基准，对照 demand-detective 体系：

| 维度 | spec-first 方法论 | demand-detective 体系 | 对照分析 |
|---|---|---|---|
| **需求澄清机制** | PRD 阶段通过 spec 文件声明用户故事、验收条件、边界约束 | 场景反讲 + A/B 决策树 + 12 维生命周期扫描 | spec-first 更偏"声明式输出"，demand-detective 更偏"引导式对话"。前者适合需求已相对清晰的场景，后者专攻模糊需求的破冰 |
| **需求颗粒度控制** | 按阶段拆分：PRD → 技术方案 → 实施计划，逐层细化 | L1→L2→L3→L4 层级递进，按决策影响半径由大到小 | 两者都遵循"先粗后细"原则。区别在于颗粒度的划分依据：spec-first 以"工程阶段"为轴，demand-detective 以"决策影响半径"为轴 |
| **模糊需求处理策略** | 通过 specs 文件承载，允许标注 TODO/待确认 | A/B 选择题 + 默认方案 + 责任声明 + 假设验证区 | demand-detective 对模糊需求的容忍度和处理机制更成熟：不假装确定，不阻塞推进，给默认方案并标记为假设。spec-first 可以借鉴这一机制 |
| **可验证性** | spec 中的验收条件作为测试依据 | 分级任务单（核心区 vs 假设区）+ 决策日志，开发-业务-测试三方对齐 | 两者都强调可验证性。demand-detective 的分级输出让测试团队更清楚"哪些必须严格验证，哪些允许弹性" |
| **知识沉淀** | docs 体系：PRD、技术方案、实施计划分层归档 | 决策日志（日期/关键词/不确定点/暂定决策/决策人/潜在风险） | spec-first 沉淀的是"最终方案"，demand-detective 沉淀的是"决策过程和上下文"。后者对于后续需求变更和新人 Onboarding 更有价值 |
| **业务系统适配** | 通用软件工程方法论，适用于各类项目 | 专为 ERP/CRM/PLM/进销存/审批流/报表系统设计，内置 12 维业务生命周期扫描 | demand-detective 在业务系统领域的领域知识注入更深入，spec-first 的通用性更强但缺乏业务域的针对性扫描规则 |
| **协作对齐机制** | 通过文件交付实现团队成员间的信息同步 | 开发-业务-测试三方对齐的任务单，决策日志可追溯 | demand-detective 更强调"谁在什么时间做了什么决策"的可追溯性，这对防止扯皮有直接价值 |

## 四、关键差异分析

### 4.1 互补关系：demand-detective 与 spec-first PRD 阶段

demand-detective 与 spec-first 的 PRD 阶段不是竞争关系，而是**时序上的互补**。demand-detective 解决的是"PRD 写之前"的问题——把模糊的一句话需求澄清为一组可确认的业务决策；spec-first 的 PRD 阶段则在决策相对明确后，将其规范化为结构化的 PRD 文档。

```
demand-detective          spec-first PRD 阶段
    │                          │
    ├─ 场景反讲                  │
    ├─ L1-L4 决策驱动            │
    ├─ A/B 选择题                │
    ├─ 12 维生命周期扫描          │
    │                          │
    ▼                          ▼
【决策清单 + 决策日志】  →  【结构化 PRD 文档】
```

**建议融合方式**：将 demand-detective 作为 spec-first PRD 阶段的**前置步骤**。当用户输入是"一句话需求"时，先跑 demand-detective 的三阶段工作流，产出决策清单和决策日志后，再将其作为 spec-first PRD 阶段的输入。

### 4.2 A/B 选择题 vs Specs 声明式方法

这是两种不同的需求表达范式：

| 方法 | 范式 | 优势 | 局限 |
|---|---|---|---|
| **A/B 选择题** | 引导式对话 | 降低业务方认知负担，每个选项附带代价说明 | 受限于 AI 的领域知识，可能遗漏第三种方案 |
| **Specs 声明式** | 规范性文档 | 精确、可验证、可自动测试 | 要求需求方已具备较清晰的认知，对模糊需求不友好 |

两者的关系是**A/B 选择题产生决策，Specs 记录决策**。demand-detective 的 A/B 选择题方法对 spec-first 的直接启示是：在 spec 文件形成之前，可以用 A/B 选择题作为需求破冰的对话工具，将最终决策回填入 spec 的验收条件和约束字段。

### 4.3 业务生命周期 12 维扫描对 spec-first 的可注入点

demand-detective 的 12 维生命周期扫描规则可以作为 spec-first 的 **PRD 模板增强插件**。具体注入方式：

1. **PRD 的"业务规则"章节**：用 12 维扫描规则逐项检查是否遗漏决策点，特别是"异常处理""审批控制""留痕审计""数据质量"这四个在传统 PRD 中容易被忽略的维度
2. **验收条件模板**：为每个维度预设验收条件骨架（如"状态流转：从[状态A]到[状态B]的触发条件已明确"）
3. **PRD Review Checklist**：将 12 维度转化为 PRD 评审清单，逐项打勾确认

这种注入不改变 spec-first 的整体架构，而是增强其在业务系统场景下的覆盖度。

### 4.4 "假设验证区"与 "Spec as Source of Truth" 的张力与融合

spec-first 的核心原则之一是 **spec as source of truth**——spec 文件是项目唯一的事实来源。demand-detective 的"假设验证区"则明确承认某些决策在当前是不确定的，先按默认方案推进。

两者表面存在张力：

| 原则 | 立场 |
|---|---|
| Spec as Source of Truth | 所有决策都应被精确记录和遵守 |
| 假设验证区 | 有些决策当前无法精确，先按假设推进 |

**融合方案**：在 spec 文件中引入 **"假设标记"机制**。具体做法：在 spec 的每个条目上标注确定性等级（Confirmed / Assumed），Assumed 条目附带默认方案、回滚触发条件和验证周期。这样既保持了 spec 的单一事实源地位，又诚实地标注了不确定性。

```
示例：
- [Assumed] 逾期提醒频率：默认提前1天+当天+逾期3天提醒主管
  验证周期：上线后2周内根据业务反馈调整
  回滚触发条件：主管投诉提醒过于频繁
```

### 4.5 适用范围差异

| 维度 | demand-detective | spec-first |
|---|---|---|
| **核心战场** | ERP/CRM/PLM/进销存/审批流/报表系统 | 通用软件工程 |
| **需求特征** | 业务规则复杂、角色多、权限细、状态机关键 | 各类项目 |
| **优势** | 业务域深度扫描、领域知识注入 | 方法论通用性、结构化交付 |
| **局限** | 纯 UI/体验优化不适用；强依赖 AI 的领域知识 | 对 ERP 等重业务系统缺乏领域专项能力 |

两者的适用范围存在交集但不是完全重叠。对于业务系统项目，建议以 demand-detective 为澄清工具、spec-first 为交付框架；对于纯技术项目或体验优化项目，spec-first 单独使用即可。

| **#21 AI 软件工程通关攻略** | 第4关（AI 需求分析） | #21 提出的"AI 需求分析关卡"中的痛点（模糊需求、范围蔓延、验收标准缺失）正是 demand-detective 要解决的问题。demand-detective 可以视为第4关的"攻略装备"——用对话式决策树替代一次性写死需求文档 |

**知识体系演进脉络**：从 #15（需求分析职业方法论）到 #17（AI PRD 八大章节）到 #18（三层架构）到 #20（编码纪律）到 #21（工程通关），再到本报告 #22（需求澄清 Skill），形成了一条完整的"需求分析→PRD 生成→架构映射→工程纪律→工程落地→工具实例"的知识链。demand-detective 作为首个被深度分析的外部 Skill，为这条知识链注入了"对话式需求澄清"的新范式。

## 六、分级改进建议

### P0：可立即注入 spec-first 的方法论要点

1. **PRD 前置澄清步骤**：在 spec-first 的 PRD 阶段开头，增加一个"需求澄清检查"环节。当输入需求为模糊的一句话或场景描述时，先使用 demand-detective 的场景反讲模板确认理解，再进入结构化 PRD 撰写。操作方式：在 PRD 模板中增加 `## 0. 需求澄清确认` 章节。

2. **A/B 选择题话术集成**：将 demand-detective 的 A/B 选择题模板作为 spec-first 的需求讨论工具。当与业务方沟通不明确的需求点时，不要直接写进 spec 然后问"这样对吗"，而是给两个具体选项让选择。在 spec-first 的工作流文档中加入 A/B 选项生成模板。

3. **假设标记规范**：在 spec 文件的条目中引入 `[Confirmed]` / `[Assumed]` 标注，Assumed 条目需附带默认方案、验证周期和回滚条件。这解决了"spec as source of truth"与"需求不确定"之间的根本矛盾。

### P1：近期可融合的能力

4. **12 维生命周期扫描作为 PRD Review 插件**：将 demand-detective 的 12 维扫描规则转化为 spec-first 的 PRD Review Checklist。在完成 PRD 初稿后，用 12 维逐项检查是否遗漏关键决策点，特别是异常处理、审批控制、留痕审计、数据质量四个维度。建议以独立 Markdown 文件形式存放于 `docs/01-需求分析/review-checklist.md`。

5. **L1-L4 决策层级作为 PRD 章节结构**：借鉴 L1-L4 的层级思维，调整 spec-first PRD 的撰写顺序：先定战略边界（L1），再画业务流程（L2），然后细化节点规则（L3），最后补充体验报表（L4）。这比按八大章节线性推进更能防止"过早陷入细节"。可在 PRD 模板中增加章节间的依赖声明。

6. **决策日志机制**：在 spec-first 的 docs 体系中引入决策日志。为每个 PRD 配套一个 `decisions.md` 文件，记录每个关键决策的时间、上下文、选项对比、最终选择、决策人和潜在风险。这对后续需求变更和新成员理解项目历史有直接价值。

### P2：中长期关注方向

7. **demand-detective Skill 的形式化集成**：将 demand-detective 的完整 Skill 定义（SKILL.md）分析后，考虑作为 spec-first 工具链中的独立 Skill 模块。触发条件：当用户输入为模糊的一句话需求且涉及业务系统时，自动路由到 demand-detective 工作流。这需要评估与 spec-first 现有工作流的集成成本和冲突点。

8. **领域知识库构建**：demand-detective 的 L1-L4 决策题和 12 维扫描规则的输出质量高度依赖 AI 的领域知识。中长期可考虑为常见业务领域（服装 PLM、生产 ERP、物流 WMS 等）构建领域知识库，让决策题的选项更精准、代价说明更具行业针对性。

9. **与 spec-first 验证体系的闭环**：探索将 demand-detective 的"假设验证区"与 spec-first 的验证体系打通——Assumed 条目自动生成验证任务，上线后根据实际数据反馈自动更新 spec 中的确定性标注。这需要与 CI/CD 和监控体系配合，属于中长期工程化方向。

---

> **总结**：demand-detective 的核心贡献不在于技术复杂度，而在于**对"需求澄清"这一工程环节的深刻拆解和工程化设计**。它用三阶段工作流（场景共建→决策驱动→驾驭输出）、四层决策递进（L1-L4）、A/B 选择题方法论、12 维生命周期扫描和分级输出机制，把"问需求"这个看似简单实则最难的环节变成了可执行、可追踪、可复现的工程流程。对 spec-first 方法论而言，demand-detective 提供了一个在 PRD 之前、在架构之前、在编码之前的那一步——**把问题先摆到桌面上**。
