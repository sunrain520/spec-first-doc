## 一、项目概述

### 1.1 系统定位

本系统不是"更聪明的代码生成器"，而是一个**端到端业务需求专家 Agent**。其目标是将业务需求从进入到结项的完整过程组织成 Agent 能自主推进、人只在关键节点确认的研发闭环。核心主张一句话：**代码生成只是最标准化的一段，真正贵的是代码之外的串联成本。**

### 1.2 要解决的三个工程化问题

| 问题 | 表象 | 本质 |
|------|------|------|
| **上下文组织** | 需求文档、业务知识、代码事实、配置平台、日志分散在不同地方 | Agent 拿不到完整上下文，推理再好也无从发挥 |
| **工具体系串联** | Aone、CR、代码仓、SLS、HSF、监控等需人工手动编排 | 平台切换造成上下文丢失和人工判断累积 |
| **反馈产生复利** | 澄清问错、方案漏看、CR 反复指出的问题只留在聊天记录里 | 下一个需求大概率再犯同样的错 |

### 1.3 落地平台与设计原则

系统落地于 **Multica**，通过 skill 编排实现。选择 Multica 的三个关键基础设施优势：独立 issue 工作区（上下文隔离）、Agent 与 skill 绑定（减少手动拼装）、多运行时和沙箱（需求可持续推进，不依赖本地开发机在线）。

方案本身不绑定特定平台，核心仍是 skill 的组合编排和提示词设计。团队选择锚定 Aone（钉钉群绑定 + 状态推送），Agent 做评论和回刷，团队在钉钉群里直接收到通知。

### 1.4 跑通状态

当前已在真实业务需求上跑通完整链路：需求进入 → 澄清 → 方案 → 实现（TDD + 质量门） → CR/Issue 协同 → 验收验证 → 发布观察 → 结项沉淀。三个工程化问题——上下文组织、工具串联、反馈复利——在链路中被系统化解决。

## 二、架构分析

### 2.1 四层横向架构

```
┌─────────────────────────────────────────────────────────┐
│                  业务需求专家 Agent                       │
├─────────────────────────────────────────────────────────┤
│  第四层：反馈学习层                                       │
│  结项蒸馏 → 长期 wiki/codemap 更新 → skill/提示词迭代      │
├─────────────────────────────────────────────────────────┤
│  第三层：工具执行层                                       │
│  Aone / CR / 代码仓 / 配置平台 / SLS / HSF / 监控 / 文档   │
├─────────────────────────────────────────────────────────┤
│  第二层：业务专家编排层                                   │
│  阶段判断（澄清/方案/实现/验收/发布/结项）                  │
│  人工确认门定位 · 自主推进决策 · 证据不足回退策略           │
├─────────────────────────────────────────────────────────┤
│  第一层：上下文输入层                                     │
│  长期业务知识 + 当前需求材料 + 代码事实 + 运行证据           │
│  按阶段拉取和组织，非一次性塞入 prompt                      │
└─────────────────────────────────────────────────────────┘
```

四层不是并列模块，而是从事实输入 → 阶段判断 → 工具落地 → 反馈回收的闭环。没有第四层，系统就是无状态的，每个需求从零开始。

### 2.2 八阶段纵向流程

四层架构是静态能力地图，八阶段纵向流程回答"一个需求实际上怎么走完"：

| 阶段 | 编号 | 核心动作 | 产出物 | 人工确认 |
|------|------|---------|--------|---------|
| **需求进入** | 3.1 | 拉取需求材料 + 长期业务知识，构建初始上下文快照 | 上下文快照 | 仅事实缺失时 |
| **需求澄清** | 3.2 | `superai-clarify` 结构化澄清，生成 requirements | requirements 文档 | **必须确认** |
| **技术方案** | 3.3 | `superai-plan` 编写方案，配置取证 + schema 校验 | plan 文档 | **必须确认** |
| **实现与质量门** | 3.4 | TDD 驱动编码，pre-push PMD/lint/内部 review | 代码 + 测试 | 自动化门禁 |
| **CR/Issue 协同** | 3.5 | 读取 unresolved 评论 → 修改 → resolve，milestone 回刷 | CR 评论 + commit | 持续人机协作 |
| **验收验证** | 3.6 | 独立环境部署 + HSF 调用 + SLS 查询 + 监控核对 | verification 证据 | **环境准入确认** |
| **发布与观察** | 3.7 | 变更材料准备 + 发布后日志/监控回读 | 观察记录 | **发布动作确认** |
| **结项沉淀** | 3.8 | `superai-finish` 选择性蒸馏到长期 wiki | closeout log + wiki 候选 | **每类写入确认** |

### 2.3 三仓库上下文管理

```
代码仓库              项目记忆仓库              长期 wiki 仓库
(master)              (feature 分支)            (master)

源码 + 测试          requirements/plan         业务规则
                    progress/CR 评论           系统边界
                    验收证据                    验收规范
                    临时调试记录                排障经验
                                                 codemap

                    需求推进中持续积累          结项时选择性蒸馏 →
                    ─────────────────────────────────────────
                    不直接升级，先留在记忆层      经审视+人工确认后进入
```

**设计关键**：项目记忆在 feature 分支上独立演进，结项时才从过程材料中筛选稳定知识蒸馏回长期 wiki。这样避免噪声污染长期知识库。

### 2.4 Skill 体系清单（12+ 个）

| Skill | 职责 | 对应阶段 |
|-------|------|---------|
| `superai-clarify` | 结构化需求澄清，生成 requirements | 3.2 |
| `superai-plan` | 技术方案编写，验收条件前置 | 3.3 |
| `superai-execute` | TDD 驱动编码实现 | 3.4 |
| `superai-code-review` | 内部结构化代码 review | 3.4 |
| `superai-aone` | Aone CR 评论读写、milestone 回刷、分支管理 | 3.5/3.7 |
| `superai-sls` | SLS 日志查询与验证 | 3.6/3.7 |
| `superai-ops` | HSF 泛化调用、监控指标核对 | 3.6/3.7 |
| `superai-tjx` | 天机星配置取证（基于 tjx-cli） | 3.3 |
| `superai-mt` | 工作台配置取证 | 3.3 |
| `superai-memories` | 项目记忆读写、长期 wiki 候选生成 | 3.1/3.8 |
| `superai-finish` | 结项编排入口 | 3.8 |
| `superai-workflow` | 阶段流转编排，控制横向各层调度 | 贯穿 |

### 2.5 关键设计决策

| 设计决策 | 内容 | 设计哲学 |
|---------|------|---------|
| **TDD 驱动实现** | 先根据 plan 里的验收标准写测试，让测试定义"什么是对的"，再让实现去满足测试 | 测试是质量约束，不是事后覆盖率装饰 |
| **pre-push 质量门禁** | git pre-push hook 硬绑定 PMD/lint/覆盖率检查，不通过拒绝 push | "Agent 应该做" → "不做就推不上去" |
| **CR 评论留痕** | 人机反馈全部落在 CR 平台，不留在聊天框 | 反馈可追溯，结项时可系统回看 |
| **真实环境验收取证** | 独立项目环境部署 + HSF 调用 + SLS + 监控，非共享预发 | 用可回读的证据证明结果 |
| **结项选择性蒸馏** | 稳定知识 → wiki/codemap，流程问题 → skill 改进候选，一次性的 → 归档 | 不直接升级，防止噪声污染 |

## 四、关键差异分析

### 4.1 方法论骨架 vs 工程化肌肉

spec-first 提供了一套完整的方法论骨架：PRD → Plan → Work → Review 的阶段化设计，docs/ 目录结构，验收标准前置，知识库沉淀。这套方法论回答了"应该怎么做"的问题。

本系统在此基础上注入了**工程化肌肉**——不是方法论创新，而是将方法论落地为真实运行的工具链、平台集成和自动化门禁：

| 层面 | spec-first（方法论骨架） | 本系统（工程化肌肉） |
|------|------------------------|---------------------|
| 上下文管理 | docs/ 目录按阶段存放 | 三仓库架构 + feature 分支隔离 + 结项蒸馏 + Multica issue 工作区 |
| 工具串联 | 依赖人工调用 | 12+ skill 编排，Aone/SLS/HSF/配置平台按阶段自动调用 |
| 质量门禁 | Review 阶段人工检查 | pre-push git hook 硬绑定，PMD/lint/覆盖率/内部 review 自动化 |
| 验收取证 | 人工测试 | 独立环境自动部署 + HSF 调用 + SLS 查询 + 监控核对 |
| 反馈循环 | 无系统化机制 | CR 评论 → 结项分析 → skill/提示词迭代 |
| 协作模式 | 文件交付 | CR 评论体系 + milestone 回刷 + 钉钉群通知 |

### 4.2 "先写验收标准再写代码"的深层一致性

本系统 3.3 节明确提出"方案阶段就应该把怎么验收定清楚"，验收方式、测试策略和验证入口写入 plan。3.4 节进一步通过 TDD 驱动实现——先根据 plan 里的验收标准写测试，再让实现去满足测试。

这与 spec-first "PRD 定义验收条件，Plan 细化验证策略"的理念**完全一致**，但本系统做了两处关键强化：

1. **验收维度扩展**：不只是"功能是否正确"，而是正例/反例/回归三分类 + 真实环境取证（HSF/SLS/监控），覆盖了 spec-first 中依赖人工判断的边界验证
2. **验收链闭环**：验收条件从 plan → TDD 测试 → 独立环境实际调用 → verification 文件，形成了一条可追溯的验收证据链

### 4.3 pre-push Quality Gate 的设计哲学

这是本系统最具工程化价值的设计决策之一。核心洞察：**质量门禁不能只靠提示词约束，必须变成 Agent 绕不过去的硬规则。**

```
传统模式：提示词说"push 前必须跑 PMD" → Agent 可能跳过
本系统：  git pre-push hook 硬绑定 PMD + 覆盖率 → 不通过就拒绝 push
```

这道门包含三件事：diff-to-test 映射、PMD/lint 检查、Agent 内部 code review。落地时通过 git hook 硬绑定，不经过 skill 调度。设计哲学：**"不做就推不上去"是真正的工程化卡口，"Agent 应该做"靠自觉。**

对比 spec-first：spec-first 通过阶段 Review 实现质量门，依赖人工审查。本系统的 pre-push hook 可以被视为 spec-first Review 阶段的**自动化工程化版本**——将人审查的部分死规则提前到 push 前由机器强制执行。

### 4.4 三仓库架构对 spec-first docs/ 的启发

spec-first 的 docs/ 目录结构（PRD.md, plan.md, tasks.md, spec.md）是按阶段切分的单层结构。本系统的三仓库架构引入了一个关键概念：**知识的分层生命周期**。

| spec-first docs/ | 本系统三仓库 | 启发 |
|-----------------|-------------|------|
| 所有文档同级存放 | 代码仓（源码）+ 项目记忆（过程产物）+ 长期 wiki（稳定知识） | 区分"当前需求的过程产物"和"跨需求复用的稳定知识" |
| 需求完成后统一归档 | 过程产物先在项目记忆 feature 分支积累，结项时选择性蒸馏 | 不直接升级——防止一次性的、有噪声的产物污染长期知识库 |
| 归档是一次性的 | 蒸馏是持续累积的——每做一个需求，长期 wiki 就充实一点 | 知识复利效应——Agent 理解业务的基线逐步提高 |

**对 spec-first 的启发**：可考虑引入"项目记忆层"概念——在 docs/ 中增加显式的"临时产物区"和"长期知识区"的区分，结项时审视筛选。同时，feature 分支隔离的实践对 spec-first 的多需求并行场景有直接参考价值。

### 4.5 结项蒸馏机制对知识沉淀的增量价值

spec-first 的知识沉淀依赖手动归档——需求完成后将 docs/ 整理入项目知识库。本系统的结项蒸馏机制（3.8）提供了三个增量价值：

| 增量 | 说明 |
|------|------|
| **三分类决策** | 稳定知识 → wiki/codemap，流程问题 → skill 迭代候选，一次性材料 → 归档。每个产物有明确的命运归属 |
| **人工确认门** | 每类蒸馏写入都需人工确认，防止"自动蒸馏"引入噪声 |
| **闭环合上** | 蒸馏出的稳定知识进入长期 wiki → 成为下一个需求进入时的"长期业务知识"拉取来源 → 每做一个需求，基线就高一点 |

这三个增量构成了 spec-first 知识管理演进的可参考路线——从"归档"到"蒸馏"，从"一次性写入"到"持续累积+选择性升级"。

### 4.6 从单 Agent 到 Agent Team 的演进蓝图

本系统 4.3 节勾勒了一个重要的演进方向——Agent Team：

```
当前：单 Agent 后端闭环
      ↓
未来：Agent Team
      ├── PD Agent（需求产出，生成 PRD + 验收标准）
      ├── 前端 Agent（页面交互 + 组件实现）
      ├── 后端 Agent（接口 + 业务逻辑）
      ├── 测试 Agent（端到端集成验证）
      ├── 数据专家 Agent（离线数据 + 报表 + 分析）
      └── 队长 Agent（需求拆解 + 任务派发 + 进度协调 + 契约对齐）
```

关键设计约束：**共享长期 wiki，各自维护视角**。所有 Agent 共享同一份长期业务知识，但每个 Agent 只拆解与自己职责相关的需求部分。

这与 #19 组织级 Coding Agent 中 Stripe/Ramp/Coinbase 的"沙箱隔离 + 子 Agent 编排"趋同，也与 #18 Gstack 的 23 虚拟角色体系形成对照——Gstack 是"按流程步骤机械切分"，Agent Team 是按"独立判断/产物/执行/责任"维度拆分。本系统的约束原则（"拆分应看是否真的存在独立判断、独立产物、并行执行和责任隔离的需要"）是对 Gstack 模式的重要修正。

## 五、知识链收束

### 5.1 #15 pm-skills：职业级 Agent 的工具链整合

**关联点**：pm-skills 将产品管理方法论编码为 68 个 Skills + 42 个 Commands + 9 个 Plugins，本质是对"产品决策工具链"的整合。本系统通过 12+ skill 体系完成了"需求研发工具链"的整合。

**收束**：pm-skills 证明了"方法论编码为技能"的可行性，本系统则将其推到了"端到端研发流程"维度。两者共享一个核心假设——工具体系必须被系统化编码和编排，人工串联是研发瓶颈。

### 5.2 #17 AI PRD 方法论：PRD 八大章节 vs requirements 文档

**关联点**：#17 报告的 AI PRD 八大章节（不确定性假设/验收标准兜底/Confidence 阈值/Eval 集/数据飞轮等）。本系统的 requirements 文档结构（目标行为/非目标边界/验收标准/风险/待确认问题）与八大章节在"验收标准前置"和"边界明确"两个维度上高度一致。

**收束**：两者都强调"结构化"优于"聊天追问"。差别在于 #17 面向 AI 产品特有的概率性需求管理，本系统面向传统业务需求的确定性场景。如果将 #17 的"Confidence 阈值/兜底行为定义"注入本系统的 requirements 模板，可覆盖更多 AI 产品类需求。

### 5.3 #18 Gstack 三层架构：决策层/规范层/执行层 vs 四层横向架构

**关联点**：Gstack 的三层架构（决策层 `plan-*`、规范层 `/review`、执行层 `/autoplan`）与本系统的四层架构（上下文输入层/编排层/工具执行层/反馈学习层）在结构上形成对照。

| Gstack | 本系统 | 差异 |
|--------|-------|------|
| 决策层（plan-* 角色） | 编排层 | Gstack 按角色切分，本系统按阶段切分 |
| 规范层（review 角色） | 工具执行层中的质量门 | Gstack 依赖角色审查，本系统通过 git hook 硬绑定 |
| 执行层（autoplan 管道） | 工具执行层 | Gstack 是一键串联，本系统是 workflow 编排 |
| **缺失** | 反馈学习层 | **关键差异**——Gstack 缺少反馈闭环层 |

**收束**：本系统最大的增量是将 Gstack 缺失的"反馈学习层"显式化，并将质量门禁从"角色审查"升级为"硬 hook 卡口"。

### 5.4 #19 组织级 Coding Agent：沙箱隔离+Draft PR vs 质量门+CR 协同

**关联点**：#19 报告中 Stripe/Ramp/Coinbase 的六大共识：沙箱隔离、工具精选、Slack 入口、子 Agent 编排、满注上下文、Draft PR 为输出契约。本系统在三个维度上与之趋同：

| #19 共识 | 本系统对应 | 趋同度 |
|---------|-----------|--------|
| 沙箱隔离 | Multica 独立 issue 工作区 + 独立项目验收环境 | 高 |
| 满注上下文 | 三仓库上下文管理 + 按阶段拉取 | 高 |
| Draft PR | CR 评论体系 + milestone 回刷 | 中高（本系统更强调评论留痕而非 PR 本身） |
| 子 Agent 编排 | 12+ skill 体系 + workflow 编排 | 高 |

**收束**：本系统的 pre-push 质量门禁（git hook 硬绑定）是 #19 三家尚未强调的能力——Stripe 的三层验证（local + CI + 1 retry）更偏事后，本系统的设计将质量前移到了推代码之前。

### 5.5 #20 Karpathy 12 条规则：行为契约的本系统映射

**关联点**：#20 报告的 12 条 AI 编码行为规则与本系统各环节存在精确映射：

| Karpathy 规则 | 本系统对应环节 | 映射逻辑 |
|--------------|--------------|---------|
| Rule 4: Goal-Driven Execution | 3.3 方案阶段"验收条件写入 plan" | 目标不是"完成编码"而是"满足验收标准" |
| Rule 5: Use Model Only for Judgment | 3.4 内部 review（`superai-code-review`） | 判断力留给模型，确定性逻辑给 PMD/lint |
| Rule 6: Token Budgets | 3.1 上下文按阶段拉取（非全量塞入） | 上下文预算的工程化管理 |
| Rule 9: Tests Verify Intent | 3.6 验收验证（正例/反例/回归） | 测试验证业务意图，非表面覆盖率 |
| Rule 10: Checkpoint | 3.5 milestone 回刷 + 3.6 verification 文件 | 关键节点状态留痕 |
| Rule 12: Fail Loud | 3.4 pre-push hook 不通过拒绝 push | 失败必须可见可阻断 |

**收束**：本系统在 Karpathy 规则的基础上做了**工程化升维**——12 条规则是"不要做什么"的行为契约，本系统是将这些契约固化为"不做就通不过"的硬门禁。

### 5.6 #21 AI 软件工程通关攻略：全流程闭环 vs 工程化实现

**关联点**：#21 报告的游戏化通关攻略覆盖了从个人玩具到团队战力的全流程（13 关、36 项流水线工具），本系统可视为其"第 4-8 关（主线任务）"的工程化深度实现。

**收束**：#21 提供了广度（覆盖需求→开发→测试→发布→运维→治理的 1080P 全景），本系统提供了深度（在需求→结项这一完整区间内的 4K 工程化细节）。两者互补——#21 的测试/发布/运维/治理关卡内容可作为本系统 3.6/3.7/3.8 阶段的扩展素材。

### 5.7 #22 demand-detective：需求澄清前置 vs structured clarification

**关联点**：#22 demand-detective 的三阶段工作流（场景共建→决策驱动→驾驭输出）和本系统的 `superai-clarify`（结构化澄清→requirements 生成→人工确认）在需求澄清前置这一点上完全一致。

| dimension | demand-detective | 本系统 |
|-----------|-----------------|--------|
| 澄清方式 | A/B 选择题 + L1-L4 决策层级 | requirements 结构化文档 |
| 产出物 | 分级任务单 + 决策日志 | requirements.md |
| 后续衔接 | 直接进入开发 | requirements → plan → 实现 |
| 决策驱动 | 核心方法（按 L1→L4 逐级决策） | 未显式定义决策层级 |

**收束**：两者互补——本系统从 requirements 到 plan 到结项的完整链路是 demand-detective 所缺失的下游工程化能力；demand-detective 的 L1-L4 决策层级 + A/B 选择题方法论可视为本系统 `superai-clarify` 的内部方法论升级。

## 七、总结

本系统的核心贡献不在于方法论创新，而在于将一套已有共识的研发流程（需求澄清→方案设计→编码实现→验收验证→知识沉淀）**工程化地跑通了**。三个关键工程化突破：

1. **硬门禁替代软约束**：pre-push git hook 硬绑定 PMD/lint/覆盖率，将质量保障从"Agent 应该做"升级为"不做就推不上去"
2. **反馈闭环系统化**：CR 评论不留在聊天框，结项时系统性回看 → 蒸馏稳定知识到长期 wiki → 识别流程问题推动 skill 迭代
3. **知识复利效应**：三仓库架构 + 结项蒸馏 → 每做一个需求，Agent 理解业务的基线就高一点

对于 spec-first 而言，本系统最有价值的启示是：**方法论的骨架需要工程化的肌肉才能持续运转**。PRD → Plan → Work → Review 的四阶段设计是正确的，但如果缺少 pre-push 硬门禁、缺少验收证据链、缺少结项蒸馏机制、缺少反馈闭环，这些阶段很大概率会退化为"文档写完就忘"的一次性仪式。
