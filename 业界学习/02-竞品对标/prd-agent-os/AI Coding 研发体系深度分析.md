# 第 1 章：六层架构全景提炼

> 来源：AI Coding 研发体系（一）：企业级 AI 编码的六层架构

## 1.1 核心判断：从工具采购到研发体系设计

该系列的开篇即抛出一个根本性判断：**AI Coding 不是工具采购问题，而是研发体系设计问题。** 这个判断的论证路径如下：

1. **工具视角的局限**：很多团队推进 AI Coding 的第一步是买许可证——Cursor、Copilot、Codex、Claude Code 被装上开发机，知识库接入，编码规范喂入。短期个人效率提升，但进入企业级代码库后，问题迅速切换为"谁给 AI 权限、谁验证改动、Token 成本怎么算、测试失败谁负责、AI 生成代码能否直接进主干"。
2. **架构视角的必要性**：从架构角度出发，真正要回答的是：AI 在什么环境里工作、承担什么角色、进入哪段流程、由谁监督、用什么指标评价、被哪些治理规则约束。
3. **关键隐喻**：AI Coding 应被视为一套"研发操作系统"，而不是一组工具清单。

> **买工具只能让 AI 进门，不能保证它进入研发生产线。**

## 1.2 六层架构全景图

```
┌─────────────────────────────────────────────────────────────┐
│  L6 治理体系层 │ 版本追踪 / 版权归属 / 安全拦截 /           │
│                │ Token 成本 / Agent 治理                    │
├─────────────────────────────────────────────────────────────┤
│  L5 评价指标层 │ 交付周期 / Bug 率 / Review 打回率 /        │
│                │ 一次成功率 / AI 返工率 / Intent Efficiency │
├─────────────────────────────────────────────────────────────┤
│  L4 组织能力层 │ 监督式工程师 / AI Tech Lead /              │
│                │ AI Engineering Manager                     │
├─────────────────────────────────────────────────────────────┤
│  L3 研发流程层 │ Intent Engineering → Context Engineering   │
│                │ → Verification Engineering → 反馈沉淀      │
├─────────────────────────────────────────────────────────────┤
│  L2 Agent 执行层│ PM Agent / Architect Agent / Coding Agent  │
│                │ / Review Agent / QA Agent / Ops Agent       │
├─────────────────────────────────────────────────────────────┤
│  L1 基础设施层 │ 模型 / 工具 / 知识库 / 外部集成            │
│                │ (Git / Jira / 数据库 / CI/CD)              │
└─────────────────────────────────────────────────────────────┘
```

**这张图不是一张采购清单，而是一张责任分布图。** 越往下越接近模型、工具、代码库和系统集成；越往上越接近组织、指标、权限、质量和责任。

## 1.3 逐层定义与核心职责

### L1：基础设施层

| 组件 | 内容 | 解决的问题 |
|------|------|-----------|
| 模型 | GPT 系列、Claude、Gemini、通义千问 | AI 能不能"理解"代码 |
| 工具 | Codex、Cursor、Claude Code、Roo Code | AI 能不能"操作"代码 |
| 知识层 | 代码库、设计文档、API 文档、历史 Issue | AI 能不能"认识"项目 |
| 外部集成 | Git、Jira、Confluence、数据库、测试平台、CI/CD | AI 能不能"接入"研发链路 |

**关键判断**：基础设施层只回答"AI 能不能接进来"——买工具只能让 AI 进门，不能保证它进入研发生产线。企业最常见的误判是把"接进来"当成"跑起来"。

### L2：Agent 执行层

个人使用时一个 Coding Agent 就够（写代码、修 Bug、补测试、解释报错），但企业研发需要角色分工：

| Agent | 职责 | 对接的研发环节 |
|-------|------|--------------|
| PM Agent | 拆需求 | 产品拆解 |
| Architect Agent | 做方案比较 | 架构设计 |
| Coding Agent | 改代码 | 代码实现 |
| Review Agent | 查规范和安全 | Review |
| QA Agent | 设计测试 | 测试 |
| Ops Agent | 看发布和故障处理 | 发布和回滚预案 |

**关键判断**：分工本身还不够，更关键的是流程层——这些 Agent 什么时候介入、读什么上下文、产出什么结果、谁来验收、失败后回到哪一步。

### L3：研发流程层

传统流程：需求 → 设计 → 开发 → 测试 → 上线。

AI 进入后新增：AI 规划 → AI 编码 → AI 测试 → 人工验证 → 反馈修复。

**核心变更**：不是在现有流程中插入几个步骤，而是重排软件研发的责任链。流程层包含 Intent Engineering（意图工程）、Context Engineering（上下文工程）、Verification Engineering（验证工程）和反馈沉淀四组机制，将在第 2 章深度拆解。

> **单个 Agent 再强，如果流程仍然靠人手动搬运上下文，效率会被很快吃掉。**

### L4：组织能力层

AI Coding 带来的最深变化不在工具层面，而在"人"的位置：

- **监督式工程师**：工作重心从编码迁移到表达需求、组织上下文、设计验收和纠偏
- **AI Tech Lead**：管理多个 Agent 的任务编排、架构边界和技术决策
- **AI Engineering Manager**：设计指标、治理和 ROI 口径

> **当 AI 能写更多代码，人类工程师的价值不会消失，而是转移到意图、上下文、验证和责任。**

第 3 章将详解组织能力层的五级分级模型。

### L5：评价指标层

文章区分了两类指标：

| 类别 | 指标示例 | 问题 |
|------|---------|------|
| 采纳指标（易被带偏） | 工具覆盖率、DAU、WAU、AI 代码占比、Prompt 次数 | 说明采纳情况，不能证明研发体系变好 |
| 交付指标（真正关心的） | 交付周期、PR 周期、Bug 率、Review 打回率、人均交付 | 问的是交付、质量和责任有没有一起变好 |
| Agent 时代新增指标 | 一次成功率、平均反馈轮次、AI 返工率、验证成本、Intent Efficiency | 问的是人机协作有没有真正变顺 |

> **AI 写了多少代码不是终点，企业要问的是交付、质量和责任有没有一起变好。**

### L6：治理体系层

治理层决定了企业级 AI Coding 的上限，包含六个维度：

| 维度 | 内容 | 核心问题 |
|------|------|---------|
| 版本追踪 | 模型版本、Prompt 版本、Agent 版本可追踪 | 谁在什么时候用了什么配置 |
| 版权归属 | AI 代码来源、版权和归属可说明 | 生成的代码是谁的 |
| 安全拦截 | 敏感代码、数据泄露、供应链风险可拦截 | 有没有东西不该被 AI 看到或生成 |
| Token 成本 | Token 消耗、调用成本和预算归因可算 | 钱花在哪里 |
| 质量可见 | 代码质量、测试覆盖率、发布质量可看 | 生成的东西到底好不好 |
| Agent 治理 | Agent 权限边界、操作审批、发布门禁 | 能不能防止 AI 改错文件直接上线 |

> **个人可以容忍一次 AI 改错文件，企业不能容忍一套没人负责的自动化研发链路。**

## 1.4 层间依赖关系

六层架构的关键不在某一层是否完美，而在层与层之间是否形成了生产链路：

```
L1 基础设施 ← 供给
    ↓
L2 Agent 执行 ← 分工
    ↓ (需流程设计)
L3 研发流程 ← 编排 (什么时候介入/读什么/产出什么/谁验收/失败回哪步)
    ↓ (需组织保障)
L4 组织能力 ← 管理 (谁监督/谁编排/谁治理)
    ↓ (需指标牵引)
L5 评价指标 ← 衡量 (交付/质量/责任有没有一起变好)
    ↓ (需规则约束)
L6 治理体系 ← 兜底 (版本/版权/安全/成本/Agent 治理)
```

**企业做 AI Coding，难点通常不是某一层完全没有，而是层与层之间没有被设计成一条生产链路。**

这一判断在"订单列表 CSV 导出"的贯穿案例中得到了具体印证：如果只解决 L1（装工具）和 L2（让 AI 写个导出按钮），而不解决 L3（定义导出范围、准备上下文、设计验证），那么 AI 产出会变成"看起来合理但业务上已经错了"的代码。

# 第 2 章：流程层三工程深度拆解

> 来源：AI Coding 研发体系（二）：AI 如何进入研发流程

## 2.1 贯穿案例：订单列表 CSV 导出

第二篇文章用一个常见需求贯穿全文——"给订单列表增加 CSV 导出"。这个看似简单的功能，却足以暴露 AI Coding 进入企业流程后的核心问题：

| 传统流程（靠人脑补） | AI 流程（需工程约束） |
|-------------------|---------------------|
| 一句需求 → 工程师脑补 → 写代码 → 测试发现问题 → 返工 | 意图说明 → 上下文包 → 生成修改 → 自动验证 → 人工判断 → 反馈沉淀 |

**关键洞察**：问题通常不在 AI 不会写代码，而在旧流程没有给 AI 准备可执行的输入和可验证的出口。

## 2.2 Intent Engineering：任务约束化

### 2.2.1 与 Prompt 工程的根本区别

Intent Engineering 不是 Prompt 技巧，也不是把一句话包装得更像命令。它要解决的是：**任务目标、边界、验收标准和禁止事项能不能在 AI 开始行动前说清楚。**

- **Prompt 工程**：如何让 AI 更好地"理解"我的意思
- **Intent Engineering**：如何让我的意思本身变得"可执行"

传统需求写给人看，人看到不完整的地方会开会、追问、翻代码、找同事确认。AI 也可以追问，但如果每次都靠对话临时补，它就会把研发流程拖回"边做边猜"。

### 2.2.2 任务约束化的要素

以订单导出为例，好的意图不是"做导出"，而是一组明确约束：

| 约束类型 | 示例 | 作用 |
|---------|------|------|
| 目标 | 只导出当前筛选结果 | 明确范围 |
| 边界 | 复用现有权限判断 | 不重新发明轮子 |
| 验收标准 | 字段顺序按财务模板 | 可验证 |
| 性能边界 | 超过 5000 行给提示 | 防性能问题 |
| 禁止事项 | 不改订单查询接口 | 设红线 |
| 测试要求 | 补一组导出字段测试 | 质量保障 |

**关键判断**：需求表达越含糊，AI 生成越勤快，返工就越稳定。

> **AI 不怕任务复杂，怕的是任务边界一直在对话里漂。**

## 2.3 Context Engineering：小上下文包

### 2.3.1 为什么不是"越多越好"

AI Coding 进入真实代码库后，最耗的往往不是写代码，而是理解环境：订单列表入口在哪里、已有导出组件在哪、权限判断怎么写、测试怎么跑、哪些模块不能碰、历史上为什么这么设计。

但上下文越多未必越好。长日志、过期文档、无关代码和泛泛规范一起丢进去，只会让 AI 花更多成本判断什么重要。

### 2.3.2 合格上下文包的组成

对订单导出这类需求，上下文包至少应包含：

| 类别 | 内容 | 作用 |
|------|------|------|
| 入口 | 订单列表页面和数据请求入口 | 知道从哪里开始 |
| 相似实现 | 已有 Excel 或 CSV 导出的相似实现 | 复用模式 |
| 映射规则 | 字段映射、脱敏规则和权限判断 | 知道怎么转 |
| 验证工具 | 相关测试文件和本地验证命令 | 知道怎么测 |
| 禁止区域 | 禁止修改的订单查询接口和公共组件 | 知道哪里不能碰 |

**关键判断**：上下文不是资料堆得越多越好，而是要让 AI 少走弯路。

## 2.4 Verification Engineering：验证前置

### 2.4.1 传统测试滞后的风险

传统流程里，测试常常在开发之后才出现：代码写完了再跑测试，PR 提了再做 Review，出了问题再补修复。

AI 流程不能这样。原因很简单：AI 让生成和试错成本变低，未经验证的改动也更容易堆起来。如果验证仍然滞后，团队会得到大量"看起来合理"的代码，然后把时间花在筛选、解释和回滚上。

### 2.4.2 验证前置的原则

Verification Engineering 要做的，是在任务开始前就说明验证方式。不是等 AI 写完才问"能不能跑一下测试"，而是一开始就告诉它：

| 验证项 | 示例 | 作用 |
|--------|------|------|
| 功能正确性 | 导出文件字段顺序必须正确 | 防业务错误 |
| 逻辑生效 | 筛选条件必须生效 | 防逻辑错误 |
| 性能边界 | 超过 5000 行必须提示 | 防性能问题 |
| 权限控制 | 未授权用户不能导出 | 防安全漏洞 |
| 回归保障 | 旧的订单查询测试不能挂 | 防破坏性修改 |

**关键判断**：AI 可以加速写代码，但不能替一个没有验证体系的项目承担质量责任。

## 2.5 反馈沉淀

### 2.5.1 个人与团队的分水岭

很多团队的反馈停在聊天窗口里："字段顺序不对"、"这里要复用权限判断"、"这个模块不能碰"、"测试挂了再修一下"。当次订单导出需求可能被救回来了，但这些纠偏没有进入团队资产。

下次换一个人、换一个 Agent、换一个任务，错误还会重来。

### 2.5.2 四类沉淀资产

更好的反馈循环，应该把一次次纠偏沉淀成四类东西：

| 资产类型 | 订单导出示例 | 作用 |
|---------|------------|------|
| 需求模板 | 导出类需求模板 | 下次同类需求直接套用 |
| 上下文说明 | 订单模块上下文说明 | 下次不用重新组织 |
| 测试用例 | 字段顺序、权限和大数据量测试用例 | 下次自动验证 |
| 禁止事项 | Agent 修改订单模块时的禁止事项 | 下次自动拦截 |

**关键判断**：个人可以靠经验临场补救，团队必须让经验进入流程。

> **如果每次纠偏都只停在聊天窗口里，团队不会真的变强，只是当次任务被救回来了。**

## 2.6 三工程的关系与价值

Intent Engineering、Context Engineering、Verification Engineering 共同构成 AI Coding 的"输入-执行-出口"三约束：

```
Intent Engineering (输入)
    ↓ 定义"做什么、不做什么、怎么才算好"
Context Engineering (执行)
    ↓ 提供"最小必要信息、让 AI 少走弯路"
Verification Engineering (出口)
    ↓ 确保"产出可验证、质量有保障"
    ↓
反馈沉淀 (循环)
    ↓ 将纠偏变成团队资产
```

**流程层的核心不是把"开发"换成"AI 开发"，而是把研发流程改成 AI 能理解、能执行、能验证、能复盘的形态。**

这也是为什么文章不建议企业只从工具培训开始。会用工具当然重要，但如果需求仍然含糊、上下文仍然分散、验证仍然滞后、反馈仍然不沉淀，AI 只会把原来的流程问题放大。

# 第 3 章：五级能力模型详解

> 来源：AI Coding 研发体系（三）：AI 研发团队的五级能力模型

## 3.1 模型的定位："团队能力覆盖图"而非"个人晋升标准"

五级能力模型（L1-L5）的定位非常明确：

- **不是公司职级**：L1-L5 指的是能力成熟度，不是给人贴标签
- **不是晋升阶梯**：个人提效阶段 L2 已够用；企业要把 AI Coding 接入流程，L3 才变分水岭
- **是团队能力覆盖图**：说明一个团队从"个人提效"走向"Agent 研发组织"需要补哪些能力

**核心判断**：当 AI 承担更多执行动作，人类工程师就要承担更清晰的监督动作。

## 3.2 L1-L5 逐级详解

| 级别 | 角色 | 核心能力 | 关键转变 | 典型工作方式 |
|------|------|---------|---------|------------|
| **L1** | 独立开发 | 手动完成需求全部环节（理解、设计、编码、调试、测试） | — | 自己写每一行代码 |
| **L2** | AI 辅助开发 | 用 AI 写代码、修 Bug、补测试、解释报错 | **个人提效**：AI 成为编码加速器 | 让 AI 帮忙写导出按钮、补测试、解释报错 |
| **L3** | 监督式工程 | 定义目标 + 准备上下文 + 设计验证 + 纠偏结果 | **分水岭**：人机协作的工程分工 | 先定义导出范围和权限边界，再准备模块上下文，最后设计测试和验收 |
| **L4** | AI Tech Lead | 多 Agent 编排 + 任务边界 + 状态同步 + 结果汇总 | **工程编排者**：从管人到管 Agent | 决定哪些任务适合交给 Agent，哪些模块禁止自动改，哪些测试必须跑 |
| **L5** | AI Engineering Manager | 指标设计 + 治理 + ROI + 风险追踪 | **研发管理视角**：从看个人效率到看系统产出 | 关心一次成功率、反馈轮次、AI 返工率、验证成本是否吞掉编码收益 |

## 3.3 L3 分水岭论证

### 3.3.1 L2 vs L3：本质差异

L1 和 L2 仍然更接近个人能力：一个是自己实现，一个是用 AI 提速。到了 L3，工程师开始**管理 AI 完成需求**。这不是简单会写 Prompt，而是能把任务目标、上下文、验证方式和纠偏动作组织起来。

以"订单导出"为例：

| 维度 | L2 工程师 | L3 监督式工程师 |
|------|---------|---------------|
| 工作方式 | 让 AI 帮忙写导出按钮、补一个测试、解释一段报错 | 先定义导出范围和权限边界，再准备订单模块上下文，最后设计测试和验收方式 |
| 时间分配 | 编码时间减少，但解释和回滚时间增加 | 前置准备时间增加，但返工和验证时间大幅减少 |
| 责任边界 | AI 产出代码，工程师最终兜底 | 工程师定义约束，AI 在约束内执行，工程师验证 |
| 技能要求 | 会写 Prompt、会用工具 | 能表达任务、组织上下文、设计验收、纠偏结果 |

### 3.3.2 L3 为什么是分水岭

L3 的出现标志着三件事同时发生：

1. **工作模式质变**：从"人写代码、AI 帮忙"变为"人定约束、AI 执行、人验证"
2. **技能重心迁移**：核心能力从"亲自编码"转向"意图表达 + 上下文组织 + 验证设计 + 责任承担"
3. **组织含义显性化**：AI Coding 开始从个人工具使用，进入人机协作的工程分工

> **L3 的出现，说明 AI Coding 开始从个人工具使用，进入人机协作的工程分工。**

## 3.4 L4：Tech Lead 要管 Agent

到了 L4，问题再上一个台阶。一个 AI Tech Lead 不能只关心某个工程师会不会用工具，而要关心多个 Agent 如何协作：

| 关注维度 | 具体问题 |
|---------|---------|
| 任务分配 | PM Agent 拆需求、Architect Agent 比较方案、Coding Agent 生成修改、Review Agent 检查规范、QA Agent 设计测试 |
| 任务边界 | 哪些任务适合交给 Agent，哪些模块禁止自动改 |
| 上下文共享 | Agent 之间需要共享什么信息 |
| 状态同步 | Agent 的进度如何同步和汇总 |
| 结果汇总 | 多个 Agent 的产出如何合并和验证 |
| 失败处理 | 哪些失败要立刻停止，哪些可以重试 |

**关键判断**：此时 Tech Lead 的工作更像"工程编排者"——决定哪些任务适合交给 Agent、哪些模块禁止自动改、哪些测试必须跑、哪些结果必须人工确认、哪些失败要立刻停止。

> **单个 Agent 的能力决定一次任务的上限，多 Agent 的编排能力决定团队能不能稳定交付。**

## 3.5 L5：管理者要换指标

到了 L5，AI Coding 成为研发管理问题。AI Engineering Manager 要关心的不再是工具覆盖率、使用人数和 Prompt 次数，而是这套系统有没有真正改变交付结果：

| 传统管理指标（不够） | 新管理指标（更关键） |
|-------------------|-------------------|
| 工具覆盖率 | 一次成功率有没有提高 |
| DAU / WAU | 平均反馈轮次有没有下降 |
| AI 代码占比 | AI 返工率是否可控 |
| Prompt 次数 | 验证成本有没有吞掉编码收益 |
| — | 风险事件能不能追踪 |

**关键判断**：这也是组织能力层和评价层的连接点。没有组织能力，指标只会变成报表；没有指标，组织能力也很难复盘。

## 3.6 五级能力与六层架构的对应

```
L5 AI Engineering Manager  → 治理体系层 + 评价指标层
L4 AI Tech Lead            → Agent 执行层编排 + 研发流程层设计
L3 监督式工程师              → 研发流程层（Intent/Context/Verification）
L2 AI 辅助开发              → Agent 执行层（使用者）
L1 独立开发                 → 传统研发模式
```

组织能力层不是软话题——它决定流程层能不能真正运行，也决定后面的评价层和治理层有没有对象可管。

## 4.2 五大结构性差距总结

```
spec-first 当前最突出的五个结构性差距：

1. 【流程层三工程未分离】—— 核心缺失
   Intent/Context/Verification Engineering 以零散形式存在，
   无独立的、系统化的三层分离，未形成"输入-执行-出口"链式约束。
   这与前序 Harness 分析中"无 dispatcher 状态机"的本质相同：
   流程驱动仍依赖 LLM 自觉而非结构化约束。

2. 【无 L1-L5 能力分级模型】—— 核心缺失
   用户默认处于 L1-L2 区间，缺乏监督式工程（L3）、编排（L4）、管理（L5）的角色定义。
   这与前序 Harness 分析中"主会话承载全部推理"的问题一脉相承——
   没有 L3 角色去承担约束定义和验证设计的职责。

3. 【无反馈沉淀机制】—— 核心缺失
   纠偏经验停留在聊天窗口和 docs/solutions/ 中，
   没有自动化沉淀为需求模板/上下文说明/测试用例/禁止事项四类资产。

4. 【无评价指标和门禁体系】—— 与前序一致
   无法回答"改规范到底变好还是变坏"，无法阻断不合规产物进入下一环节。
   前序 Harness 分析已将此列为 P0。

5. 【Agent 体系偏技术审查】—— 结构偏斜
   缺少 PM/QA/Ops 三类上游和下游 Agent。
   这与 spec-first 的历史路径有关——它从代码审查起家，
   Agent 体系自然偏技术审查（安全/性能/数据/前端/测试），
   但企业级 AI Coding 需要覆盖从需求到上线的全链路。
```

## 4.3 spec-first 的意外优势

尽管存在上述差距，spec-first 在以下维度具有独特优势，可成为升级基础：

| 优势维度 | 具体表现 | 升级潜力 |
|---------|---------|---------|
| **审查体系丰富度** | spec-code-review 12+ 角色化审查 + merge/dedup pipeline | 可直接映射为 Review Agent + QA Agent 的能力底座 |
| **合同驱动治理** | dual-host-governance 定义明确的治理边界 | 可直接扩展为 Agent 治理体系的基础设施 |
| **Skill 体系厚度** | 37 个 Skill 覆盖编码、测试、部署全流程 | 为 Intent/Context/Verification Engineering 提供了可组合的原子能力 |
| **知识库分层编排** | 前序分析已构建的金字塔五层 + 7 种有向边 | 为 Context Engineering 提供了分层加载方案 |
| **brooks-lint 集成** | 六维衰减框架已深度集成到 spec-code-review | 为 Verification Engineering 提供了细粒度诊断工具 |

| 本文六层架构 | Harness 五层架构 | 映射关系 | 关键差异 |

### 5.2.2 同构判断

两者在三个层面高度同构：

1. **分层思想**：都将 AI Coding 拆分为多个独立关注层（工具→Agent→流程→治理），而非视为单一工具
2. **约束外置**：都主张将规则和门禁从 LLM 推理中剥离——本文提"流程层三工程"（前置约束），Harness 提"G1-G8 门禁"（后置阻断）
3. **上下文预算**：都反对无节制堆上下文——本文提"小上下文包"，Harness 提"上下文 ≤8K + 按需加载"

**关键分歧**：本文的"组织能力层"（L4）是 Harness 完全没有覆盖的维度。Harness 把关注点全放在技术架构上，而本文提出了"谁监督、谁编排、谁治理"的组织命题——这恰好解释了为什么 Harness 分析中反复发现 spec-first "主会话承载全部推理"的问题：**缺少 L3 监督式工程师这个角色定义，导致约束定义和验证设计的职责无处安放。**

## 5.3 流程层三工程 × brooks-lint × 测试决策单：互补矩阵

| 本文流程层 | brooks-lint 维度 | 测试决策单 | 互补关系 |
|-----------|-----------------|-----------|---------|
| Intent Engineering | R6 领域模型失真 | — | 意图定义"做什么"，brooks-lint 诊断"领域模型是否被 AI 扭曲" |
| Context Engineering | R1 认知过载 / R3 知识重复 | — | 上下文组织"给什么信息"，brooks-lint 诊断"信息是否造成过载或重复" |
| Verification Engineering | R2 变更传播 / R4 意外复杂度 / R5 依赖混乱 | T1-T6 全维度 | 验证设计"怎么验证"，brooks-lint 诊断"哪些维度会衰减"，测试决策单指导"测什么" |
| 反馈沉淀 | R1-R6 全维度 | T1-T6 全维度 | 反馈沉淀"把教训变成资产"，brooks-lint 和测试决策单提供"资产化的诊断维度" |

**核心洞察**：本文提供了"流程应该怎么设计"，brooks-lint 提供了"代码会在哪些维度衰减"，测试决策单提供了"测试应该覆盖什么"——三者分别解决 **Design（设计）、Diagnose（诊断）、Verify（验证）** 三个不同层面的问题。

## 5.4 流程层三工程 × 知识库金字塔：上下文工程的落地方案

前序分析中构建的知识库金字塔五层（L0 Core → L1 Foundation → L2 Domain → L3 Experience → L4 Meta）与 7 种有向边，恰好为本文的 Context Engineering 提供了落地方案：

| Context Engineering 需求 | 知识库金字塔对应层 | 实现方式 |
|------------------------|------------------|---------|
| 入口（从哪开始） | L2 Domain：模块入口索引 | 按模块组织，7 种有向边中的"入口关系" |
| 相似实现（复用模式） | L3 Experience：历史实现库 | "相似度关系"边 + 向量检索 |
| 映射规则（字段/权限/脱敏） | L1 Foundation：规则与约定 | L0 Core → L1 Foundation 的"约束关系"边 |
| 验证工具（测试文件） | L2 Domain：测试配置 | 关联到测试决策单的 T1-T6 映射 |
| 禁止区域（不能碰的模块） | L0 Core：核心约束 | "禁止关系"边 + 门禁阻断 |

**这意味着，如果 spec-first 实现了知识库金字塔分层，Context Engineering 的"小上下文包"生成就可以不再靠人工整理，而是从知识库中按阶段自动提取最小必要上下文。**

**五维的定义**：

| 维度 | 来源 | 核心问题 |
|------|------|---------|
| 约束层 | Harness 五层 + G1-G8 | 如何强制 AI 遵守规则（外部约束） |
| 流程层 | 本文（三工程 + 反馈） | 如何设计 AI 能理解的研发流程（输入-执行-出口） |
| 验证层 | brooks-lint + 测试决策单 | 如何诊断 AI 产出在哪些维度衰减（质量诊断） |
| 知识层 | 知识库金字塔分层 | 如何组织和管理 AI 需要的知识（上下文供给） |
| 能力层 | 本文（L1-L5 分级） | 谁负责定义约束、组织上下文、验证结果（角色分工） |

这五个维度互相依存：约束层给流程层提供执行保障（门禁）、流程层给验证层提供验证点（Verification Engineering）、验证层的诊断结果沉淀到知识层（反馈沉淀）、知识层的高效供给降低能力层（L3）的负担、能力层的角色定义驱动约束层的设计（谁定义、谁执行）。

## 5.6 知识链收束的意义

本次分析将三篇文章的"六层架构 + 三工程 + 五级能力"与前序 Harness/brooks-lint/测试决策单/知识库金字塔分析串联后，形成了一条从"体系顶层设计 → 流程工程化 → 门禁实现 → 质量诊断 → 知识管理 → 角色分工"的完整链条。这不是简单的堆砌，而是揭示了 AI Coding 研发体系的六个必要维度——缺少任何一个，体系都会有结构性漏洞。

# 第 6 章：分级改进建议

> 基于第 4 章全景对照中识别的五大结构性差距，结合第 5 章知识链收束中形成的五维一体图谱，按 P0/P1/P2 三级给出改进建议。

## 6.1 P0：立即行动（不解决则体系有结构性漏洞）

### P0-1：建立流程层三工程骨架

**对应差距**：流程层三工程未分离（第 4 章差距 #1）

**当前状态**：Intent/Context/Verification Engineering 以零散形式存在于 spec-prd、spec-work 和 spec-code-review 中，没有形成独立的系统化分离。

**建议方案**：

1. **Intent Engineering 独立化**：将 spec-prd 的"需求模板"升级为结构化约束模板，强制包含以下字段：
   - 任务目标（做什么）
   - 边界约束（范围限定）
   - 验收标准（可量化的完成条件）
   - 禁止事项（明确红线）
   - 测试要求（必须覆盖的测试场景）

2. **Context Engineering 结构化**：基于知识库金字塔五层（L0-L4），设计按阶段自动提取的"小上下文包"生成方案：
   - L0 Core → 禁止事项（自动注入）
   - L1 Foundation → 映射规则与字段约定
   - L2 Domain → 入口文件与相似实现
   - L3 Experience → 历史纠偏案例
   - 控制 Token 预算，避免上下文过载

3. **Verification Engineering 前置化**：将 spec-code-review 从"事后审查"扩展为"前置验证 + 事后审查"双模式：
   - 前置验证：任务开始前自动注入验证清单（brooks-lint R1-R6 + 测试决策单 T1-T6）
   - 事后审查：保持现有审查管道，新增 Intent 一致性验证

**与知识链的衔接**：方案直接复用前序 Harness G1-G8 门禁（约束层）、brooks-lint 六维衰减（验证层）、测试决策单（验证层）、知识库金字塔分层（知识层），四者在此处交汇。

### P0-2：建立反馈沉淀闭环

**对应差距**：无反馈沉淀机制（第 4 章差距 #3）

**当前状态**：纠偏经验停留在聊天窗口和 docs/solutions/ 中，未形成自动化沉淀。

**建议方案**：

设计四通道自动化沉淀管道：

| 纠正类型 | 沉淀目标 | 自动触发条件 | 沉淀为 |
|---------|---------|------------|--------|
| 字段/约束错误 | 需求模板库 | Intent 验证失败 | 结构化约束模板 |
| 上下文遗漏 | 模块上下文库 | Context 不足导致返工 | 模块上下文说明 |
| 验证盲区 | 测试用例库 | 测试遗漏导致线上问题 | 测试用例 |
| 误修改模块 | 禁止事项库 | 审查发现误修改 | Agent 禁止事项 |

**实现路径**：在每个任务完成后，由 Review Agent 分析纠偏记录，自动提取三类资产并写入知识库的 L3 Experience 层。

### P0-3：定义 L3 监督式工程师角色

**对应差距**：无 L1-L5 能力分级模型（第 4 章差距 #2）

**当前状态**：用户默认处于 L1-L2 区间，缺少监督式工程的角色定义。

**建议方案**：

在前序 Harness 分析的 dispatcher 状态机基础上，新增一个 **"监督者"（Supervisor）** 角色，职责边界如下：

| 职责 | 对应文章概念 | 在 spec-first 中的落地方式 |
|------|------------|------------------------|
| 定义任务约束 | Intent Engineering | 调用 Intent Engineering 模板，生成结构化约束 |
| 准备上下文包 | Context Engineering | 从知识库金字塔按阶段提取最小必要上下文 |
| 设计验证清单 | Verification Engineering | 调用前置版 spec-code-review，生成验证检查单 |
| 纠偏与决策 | 反馈沉淀 | 判断 Agent 产出是否合格，决定继续/回退/重做 |

**监督者不是取代 dispatcher，而是在 dispatcher 之前增加一个"任务预处理层"**——将模糊的用户需求转化为可执行的约束包，降低 dispatcher 的意图漂移风险。

## 6.2 P1：尽快推进（补齐能力盲区）

### P1-1：建立门禁与评测体系

**对应差距**：无评价指标和门禁体系（第 4 章差距 #4）

**当前状态**：与前序 Harness 分析的发现一致——spec-first 没有门禁体系。

**建议方案**：

1. **引入 G1-G8 门禁框架**（从前序 Harness 分析复用）：
   - G1-G4：代码级（格式/安全/性能/测试）
   - G5-G8：任务级（范围/流程/质量/交付）

2. **设计 AI Coding 专属指标**（从本文评价层萃取）：
   - 一次成功率（任务一次通过率）
   - 平均反馈轮次（人均纠偏次数）
   - AI 返工率（AI 产出被回滚的比例）
   - Intent Efficiency（意图表达的清晰度，可用歧义追问次数衡量）

### P1-2：扩展 Agent 角色覆盖

**对应差距**：Agent 体系偏技术审查（第 4 章差距 #5）

**当前状态**：缺乏 PM Agent（需求拆解）、QA Agent（测试设计）、Ops Agent（发布和故障处理）。

**建议方案**：

| Agent | 现有能力可复用 | 新增能力 |
|-------|-------------|---------|
| PM Agent | spec-prd 需求模板 | 需求拆解规则 + 任务优先级排序 |
| QA Agent | spec-code-review + 测试决策单 T1-T6 | 测试用例生成 + 测试覆盖率分析 |
| Ops Agent | — | 发布检查清单 + 回滚预案验证 + 故障分类 |

**现有能力复用优势**：spec-code-review 的 12+ 角色化审查可直接为 QA Agent 提供验证维度支撑；测试决策单的 T1-T6 可直接映射为 QA Agent 的检查维度。

### P1-3：设计 L4 AI Tech Lead 编排框架

**对应差距**：无编排角色定义（第 4 章差距 #2 中的 L4 维度）

**当前状态**：dispatcher 只有单任务编排能力，无法处理多 Agent 协作。

**建议方案**：

在现有 dispatcher 基础上新增多 Agent 编排 Primitive：

| 编排 Primitive | 功能 | 实现方式 |
|---------------|------|---------|
| Agent 路由 | 根据任务类型分派到不同 Agent | 任务分类器 + Agent 能力注册表 |
| 上下文共享 | Agent 之间共享中间产物 | 共享 Session Context |
| 状态同步 | 追踪每个 Agent 的状态 | 状态机扩展为多 Agent 并行态 |
| 结果汇总 | 合并多个 Agent 产出 | 结果合并引擎 + 冲突检测 |
| 失败处理 | 单 Agent 失败的级联处理 | 定义"继续"、"重试"、"停止"策略 |

## 6.3 P2：长期建设（体系成熟度提升）

### P2-1：设计 Agent 治理体系

**对应差距**：版本追踪/Token 成本/版权归属缺失（第 4 章六层对照中 L6 治理层的差距）

**建议方案**：

| 维度 | 短期（P1） | 长期（P2） |
|------|----------|----------|
| 版本追踪 | Prompt 版本记录 | Agent 版本快照 + 回滚能力 |
| Token 成本 | 调用次数统计 | 部门级归因 + 预算预警 |
| 版权归属 | AI 生成代码标记 | 来源追溯 + 归属审计 |
| Agent 权限 | 读写权限分级 | 细粒度操作授权 + 审批链 |

**现有资产复用**：dual-host-governance 合同体系可直接扩展为 Agent 治理的基础设施。

### P2-2：构建 AI Engineering Manager 仪表盘

**对应差距**：无管理视角指标（第 4 章差距 #2 中的 L5 维度）

**建议方案**：

设计三级仪表盘：

| 层级 | 受众 | 指标 |
|------|------|------|
| L1 工程师仪表盘 | 个人开发者 | AI 辅助时间节省、Prompt 有效性、返工次数 |
| L2 Tech Lead 仪表盘 | Tech Lead | Agent 使用率、编排成功率、任务完成率 |
| L3 管理仪表盘 | Engineering Manager | 一次成功率趋势、AI 返工率趋势、验证成本/编码收益比、团队能力分级分布 |

### P2-3：知识库金字塔分层落地

**对应差距**：Context Engineering 自动化（第 4 章差距 #1 的深化）

**当前状态**：前序分析已设计完整的金字塔五层 + 7 种有向边，但未落地。

**建议方案**：

分两步走：
1. **P1**：将金字塔五层设计写入 spec-first 的 using-spec-first 注入机制，确保每次会话自动加载 L0 Core + L1 Foundation
2. **P2**：实现 L2 Domain → L3 Experience 的按需加载，基于任务类型自动匹配对应知识层

## 6.4 改进优先级总览

```
P0（不解决则体系结构性漏洞）：
  ├─ P0-1: 流程层三工程骨架（Intent/Context/Verification 独立化）
  ├─ P0-2: 反馈沉淀闭环（四通道自动化沉淀）
  └─ P0-3: L3 监督式工程师角色（Supervisor → dispatcher）

P1（补齐能力盲区）：
  ├─ P1-1: 门禁与评测体系（G1-G8 + AI Coding 专属指标）
  ├─ P1-2: Agent 角色扩展（PM/QA/Ops Agent）
  └─ P1-3: L4 AI Tech Lead 编排框架（多 Agent 编排 Primitive）

P2（长期体系成熟度提升）：
  ├─ P2-1: Agent 治理体系（版本/成本/版权/权限）
  ├─ P2-2: AI Engineering Manager 仪表盘（三级视图）
  └─ P2-3: 知识库金字塔分层落地（按需加载）
```
