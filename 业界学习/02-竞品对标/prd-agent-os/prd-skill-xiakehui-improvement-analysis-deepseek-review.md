# spec-prd 改进分析——深度二轮评审（DeepSeek 视角）

> 对 `prd-skill-xiakehui-improvement-analysis.md` 的深度二次分析
> 方法：质疑每项建议的隐含假设、发现被忽略的系统性风险、揭示架构层面的深层张力
> 分析日期：2026-06-08

---

## 前置声明

第一轮分析建立了一条清晰的映射——侠客汇做了什么、spec-prd 缺了什么、怎么补。这是必要的第一步。但映射式分析有一个固有盲区：它倾向于把侠客汇的做法当作"正确答案"，而忽略 spec-prd 中那些**刻意不做的设计决策**。

本轮分析的任务是：
1. 找到第一轮分析中被简化的复杂性问题
2. 暴露每条改进建议在落地时会遇到的真实阻力
3. 识别侠客汇方案中"在特定上下文下有效，但在 spec-prd 的上下文下可能失效"的部分
4. 提出比第一轮更根本的架构问题

---

## 一、对差距总览的再审视

第一轮分析将差距定性为"spec-prd 是工具化的，侠客汇是自进化的"。这个二元对立本身有误导性——它暗示"工具化"是落后的、应该被替换的，而"自进化"是更高级的形态。

**被忽略的事实**：spec-prd 的"工具化"是经过深思熟虑的设计选择，不是能力缺失。证据：

1. **Readiness Reviewer 刻意不实体化**：SKILL.md 明确写道"readiness reviewer 是一个 eval-gated 内部 helper——只有当 fresh-source eval 证明自审稳定失效时，才会实体化为独立 agent"。这不是缺位，是刻意的渐进演化策略。侠客汇式的"先建机制再跑通"，和 spec-prd 的"先跑通再建机制"，是两种不同的工程哲学，不该预设孰优孰劣。

2. **Feature Slices 已经是一个轻量级的"引用系统"**：每个 slice 有自己的 identity trace 和 source excerpt preservation。虽然不如侠客汇的引用账本那样全面，但在"不增加概念负担"的前提下已经解决了最紧迫的问题——确保需求到实现的追溯性。

3. **Domain-Glossary Promotion 的保守设计是故意的**：≥2 个 PRD 才能磨锐、只晋升术语不晋升规则、preview 式审批——这套设计刻意限制了自动化的边界，因为"自动晋升业务规则"一旦出错，后果远比"术语晋升出错"严重。

**结论修正**：差距不是"有没有"，而是"自动化到什么程度、在什么边界下自动化"。第一轮分析的改进建议可以在方向上正确，但在边界设定上过于激进。

## 二、逐项深度质疑

### 2.1 输入策略：会议纪要的隐含脆弱性

第一轮建议："force 弃 BRD 保会议纪要"。

**深度问题**：

1. **会议纪要本身的质量方差极大**。侠客汇之所以能走通，是因为他们同时做了另一件没说透的事——"统一需求会议标准，反向推动业务一起从源头提升需求沟通质量"。如果没有这个前提，"会议纪要优先"可能变成"垃圾进，垃圾出"的更隐蔽形式。spec-prd 目前的 `input_posture` 不做来源区分，恰好避免了在没有标准化工序的情况下过度依赖单一来源。

2. **BRD 的"噪音"在某些场景下是必要信息**。侠客汇说 BRD 有"方案越界"的问题——运营写了"我要这个功能"而不是"要解决什么问题"。但对棕地系统而言，运营提出的"方案越界"本身就是一个隐性的约束信号：它说明运营已经基于他们对系统的理解做了一层翻译，这个翻译无论对错都反映了业务方的 mental model。丢弃它就是丢弃了一条关于"业务方怎么理解系统"的信息。

3. **`input_fidelity` 的三级分法有错误的精确性**。"consensus-confirmed" 和 "single-source-stated" 之间的边界在实际中非常模糊——会议纪要里面也会有争议点没被记录、也会有强势一方主导结论的真实共识缺失。给一个来源贴上 "consensus-confirmed" 标签，可能会创造一个虚假的安全感。

**修正建议**：

- 不做"会议纪要优先"的范式迁移，而是做"来源标记 + 不确定性量化"：每个输入来源不是被判定为高/中/低可信度，而是携带一个 `uncertainty_profile`（来源种类、参与者数量、确认轮次、是否有反向验证），由 AI 在 Phase 1 中基于这些 profile 做动态的可信度权衡。
- `noise_warning` 不是建议用户"改用会议纪要"，而是建议"在当前的 BRD 上追加一次需求对齐会议进行确认"——这是可操作的、不要求用户改变工作习惯的。

### 2.2 分步生成：确认节点的隐藏成本

第一轮建议：在 Phase 3 中插入 3 个 `section_anchor`，每个锚点等待 owner 确认方向。

**深度问题**：

1. **确认节点本身就是新的瓶颈**。侠客汇的产品线提效很大程度上来自减少"等人的时间"。如果 spec-prd 每次起草 PRD 都要等 owner 在 3 个锚点确认方向，那么从"一次输出 + 一次审核"变成了"分段输出 + 分段审核"，总 wall-clock time 可能不降反升。特别是当 owner 不是实时在线时（异步协作场景），每多一个确认节点就多一次上下文切换成本。

2. **Section Anchor 的"确认"粒度是一个极难调对的参数**。如果确认太粗（如只确认"Core Sections 方向"），owner 看不到足够细节来做出有效判断，可能点头通过但实际上方向已经偏了。如果确认太细（如逐段确认），就退化成"人裹挟着 AI 写"，失去了 AI 提效的核心价值。

3. **owner 在中间锚点的决策质量可能比在终点的决策质量更低**。因为缺少完整的上下文（后面的 Conditional Sections 还没起草），owner 在 Section Anchor 1 确认"方向正确"时，可能不知道这个方向会导致后面的某个 Conditional Section（如 Negative Acceptance）出现矛盾。这在第一轮分析中被完全忽略。

4. **spec-prd 现有的 Phase 2 Grill 已经在做这件事**——只是它问的是"最关键的 1-3 个问题"而不是"确认每个 section"。Grill 的设计理念是：如果关键决策点被准确回答了，中间生成的偏差是可接受的，最终在 Readiness Lens 中收敛。这是刻意用"前向干预"替代"中程打断"。

**修正建议**：

- 不引入 `section_anchor` 作为常规流程，而是作为 `section_anchor_on_demand`：仅在 `quality_diagnosis` 标记为高风险（如 `input_fidelity` = `hybrid-inferred`、或涉及 `topology-heavy-prd` 中的 `replace`/`migrate` 高风险拓扑）时才启用。常规场景保留现有的一次起草模式。
- 替代方案：用"起草后回溯确认"代替"起草中分段确认"——Phase 3 一次性输出后，不直接进入 Phase 4，而是先输出一个 `draft-summary`（每个 section 一句话 + 关键决策点），等 owner 快速扫一眼确认方向后再进入正式 Readiness Lens。这保持了 AI 起草的连贯性，同时把人的确认提前了但没有碎片化。

### 2.3 错题本：自进化机制的真实边界

第一轮建议：人工纠正自动沉淀为规则门禁，下次主动规避。

**深度问题**：

1. **错题本的"匹配精度"是致命变量**。侠客汇用一个"领域关键词匹配"来检索相关纠正条目。但在 spec-prd 的场景中，"领域"的粒度可能远细于侠客汇的会员系统——如果关键词是"积分"，而错题本中有一条"会员升级后积分清零是错的"，当新的 PRD 是关于"积分兑换"时，这条纠正是相关的。但如果新的 PRD 关于"积分规则"中某个更细的条款呢？匹配阈值设太宽会导致大量误报（把不相关的纠正强行注入上下文），设太窄会导致漏报（真正相关的纠正没被匹配到）。

2. **纠正条目的冲突管理是第一轮分析完全没提的问题**。假设：
   - C-001（2026-03）：积分在会员降级时清零
   - C-005（2026-06）：积分在会员降级时保留，但会员等级积分清零
   
   两条纠正表面矛盾，但实际上是补充关系。当 AI 在 Phase 1 加载到两条时，它应该遵循什么规则？如果 C-005 更近，是否覆盖 C-001？还是合并？这个冲突解决机制不存在，错题本就可能在关键时刻给出矛盾信号。

3. **错题本越厚，系统的"可预测性"越低**。这是自进化系统的根本悖论——每条新纠正都在增加系统的特异性（更懂这个项目），但也在降低系统的通用性（更难被新人理解、更难迁移到其他项目）。当纠正条目超过 50 条时，AI 的行为将变得对 owner 不透明：你不知道它为什么在某处表现得特别保守，因为它可能在匹配一条你自己都忘了的三个月前的纠正。

4. **纠正的"反向适应性"问题**：如果被纠正的不是 AI 的编造，而是 owner 个人的偏好（如"用这个术语而不用那个"），这条纠正将把 owner 的个人偏好固化为系统规则。换一个 owner、换一个项目阶段，这条规则可能就过时了——但错题本不知道。

**修正建议**：

- 错题本必须加上 `granularity` 和 `scope` 字段：区分是"全局规则纠正"（如"这个系统里积分和会员独立"）还是"局部偏好纠正"（如"这个 PRD 中请用术语 A 替代术语 B"）。局部纠正不自动注入到其他 PRD，除非 owner 手动晋升为全局。
- 错题本需要 `conflict-resolution` 策略：按时间（新覆盖旧）+ 来源（owner 确认覆盖 AI 自行总结）+ 显式标记（owner 可以显式将两条矛盾的纠正标记为"合并"或"替代"）。
- 错题本必须有一个 `attenuation` 机制：纠正条目被创建 6 个月后，如果没有被再次验证（即后续 PRD 没有触发这条纠正的匹配），自动降级为 `dormant`，不再主动注入上下文，仅在被显式触发时加载。

### 2.4 知识漏斗：收敛算法的隐蔽脆弱性

第一轮建议：用"目录页→卡片→整条经验"三级漏斗从几百条知识中精准挑选 3-5 条。

**深度问题**：

1. **"关键词匹配"作为收敛算法在领域知识上极度脆弱**。侠客汇用一个隐式的"锚点→取知识"映射来做检索——这在他们的单一业务线（会员系统）中有效，因为领域范围有限、关键词覆盖率高。但 spec-prd 是一个通用 PRD skill，适用任何棕地系统。从电商订单到证券交易到内部管理后台，不同领域的"关键词"几乎没有交集。如果 knowledge-directory.md 中的关键词是由 owner 手动录入的，那维护成本很快就超过提效收益。

2. **"top-5 知识卡片"这个数字是拍脑袋的**。对于一个小型项目，5 条可能已经包含了所有相关知识；对于一个大型棕地系统（像侠客汇的会员系统），5 条可能只涵盖了冰山一角。第一轮分析没有给出"top-N"的 N 如何根据项目规模动态计算。

3. **"被漏斗过滤掉的知识"可能恰恰是最关键的**。如果一个 PRD 涉及一个全新领域（比如从会员系统扩展到积分商城），knowledge-directory 中可能根本就没有相关卡片。漏斗收敛后输出 0 条卡片，AI 在"没有历史经验"的情况下起草 PRD——这本身是合理的，但系统应该显式地标记这一点（"本次无历史知识可参考"），而不是静默地跳过。

4. **漏斗的"第二层筛选"（卡片→整条经验）需要 AI 做判断——而 AI 做判断本身需要上下文**。这形成了一个循环：为了选出哪些知识卡片值得展开，AI 需要理解卡片内容；但为了理解卡片内容，AI 需要展开它们。第一轮分析的"目录页→卡片→经验"链条把过滤逻辑假设为机械的关键词匹配，但实际上机械匹配在语义检索上的准确率不足以支撑决策。

**修正建议**：

- 漏斗的第一层不做关键词匹配，而是做**向量化语义检索**。knowledge-directory.md 中的每条卡片都有一个 `embedding` 或至少一个 `description` 段落（非关键词，而是自然语言描述），通过语义相似度排序取 top-N。
- N 的动态计算：基于当前项目的 knowledge-base 总条目数来设定（total_entries < 20 → N=3; 20-100 → N=5; 100+ → N=8），且当语义相似度最高的条目与 PRD 主题的 cosine similarity 低于 0.3 时，标记为 `no-relevant-knowledge`。
- 漏斗的最后一道是"显式声明"：AI 在 Phase 3 起草前输出它实际加载了哪些知识卡片、每张卡片的匹配理由。这不仅用于质量闸门（owner 可以快速判断是否正确匹配），也为引用账本提供数据源。

### 2.5 知识进化：成熟度体系的失效模式

第一轮建议：引用次数驱动 draft→verified→proven 三级成熟度。

**深度问题**：

1. **"被引用 N 次"不等于"被正确使用 N 次"**。一条关于"支付回调幂等"的知识可能被 10 个 PRD 引用，但如果其中 7 次的引用者只是机械地把它贴在 Phase 1 evidence 中、从未真正驱动决策，那这条知识实际上没有价值——但成熟度系统会把它误判为 proven。

2. **成熟度的"锁定效应"**：一条知识升到 proven 后强制加载、不受 context_budget 限制。如果这条知识后来被证明是错误的（或部分过时），它的 proven 标签反而会把它变成一个"持续犯错、强制加载"的污染源。成熟度越高的错误知识，危害越大。

3. **"不同 PRD"作为计数单位的模糊性**。同一个需求被拆成 3 个 child PRD，它们都引用同一条知识，这算 3 次引用还是 1 次？如果是 3 次，大型需求会不成比例地推高知识的成熟度。如果是 1 次（按 parent PRD 级），则需要额外的 parent-child 追溯机制。

4. **draft 和 verified 之间的"冷启动困境"**：一条新的高质量知识需要被 2 次引用才能从 draft 升到 verified，但在升到 verified 之前它的加载优先级很低（可被 context_budget 裁剪）。如果它总是被裁剪，就永远得不到引用次数，从而陷入"好知识无法晋升"的死循环。

**修正建议**：

- 成熟度升级不仅要看引用次数，还要看**引用质量**。每次引用时记录 `引用深度`（advisory-only / decision-driving / blocking-constraint），只有 `decision-driving` 及以上级别的引用才计入升级计数。
- 引入 **`challenged` 状态**：如果一条 proven 知识在后续的 PRD 中被 owner 纠正过（进入错题本），它不直接降级，而是进入 `challenged` 状态——仍然强制加载，但加载时附带"此条曾被纠正"的提示，直到 3 个后续 PRD 的引用中无新的纠正出现，才恢复 proven。
- parent PRD 去重：同一 feature 下的多个 child PRD 对同一条知识的引用合并计为 1 次。
- draft→verified 的跃迁增加**"紧急晋升"通道**：owner 可以手动将一条 draft 知识标记为 `priority-load`，绕过 context_budget 限制，给它一个公平的验证机会。

### 2.6 跨角色衰减：一个被过度简化的模型

第一轮建议：在 quality_diagnosis 中新增 `role_attenuation_risk` 维度，检测信息在角色之间的传递衰减。

**深度问题**：

1. **角色边界在 spec-prd 的上下文中是模糊的**。spec-prd 不是侠客汇——侠客汇有一个明确的组织架构（运营/产品/UI/前端/后端/测试），但 spec-prd 的使用者可能是一个人在做 PRD、也可能是 3 个人、也可能是有产品经理+架构师但没有专门 UI。第一轮分析中预设的"运营→产品→UI→开发→测试"角色传递链在通用场景下不存在。

2. **真正的衰减不在角色之间，而在"上下文"之间**。侠客汇之所以要收敛成 1+1，是因为他们发现了"信息每过一道角色就走样一次"。但 spec-prd 的用户面对的衰减更可能是时间维度的——需求在两周
的排期等待中被遗忘的细节、跨 sprint 交接时的信息丢失。这个维度的衰减第一轮分析完全没有涉及。

3. **`role_attenuation_summary` 可能制造不必要的焦虑**。如果 AI 在 PRD 末尾输出"本次 PRD 跨 4 个角色边界，存在中等转述失真风险"，但使用者根本控制不了角色结构（你能让运营和开发直接沟通吗？），这个提示除了增加一份"我知道但我无能为力"的焦虑之外没有任何价值。

**修正建议**：

- 不预设角色，而是检测 **"信息传递的断层"**：PRD 中哪些段落是基于二手转述、哪些是基于原文引用、哪些缺少直接来源。这个检测不绑定到角色，而是绑定到证据链条的完整性。
- 针对 spec-prd 的真实场景新增 **`temporal_attenuation`** 检测：如果 PRD 的创建时间与它引用的会纪纪要/原始需求之间存在显著的时间间隔（> 1 周），自动建议对关键决策点做一次 refresh 确认。
- `attenuation_summary` 的输出形式不是"告诉你有多危险"，而是"给出具体的可操作动作"——哪些段落需要找谁确认、确认什么。

### 2.7 产品-技术闭环：知识冲突的实现困境

第一轮建议：PRD ↔ Plan 建立回流通道，两类知识共享同一份 knowledge-base。

**深度问题**：

1. **产品侧的知识和技术侧的知识有根本性的时间尺度差异**。产品侧的业务规则（如"积分独立于会员"）是长期稳定的，技术侧的实现约束（如"Facade 层禁止直接调用 DAO"）可能因框架升级而迅速过时。把它们放在同一个 knowledge-base 中用同一套成熟度体系管理，产品的长期规则会被技术的短期约束"稀释"——knowledge-base 的成熟度统计会偏向大量高频变化的技术条目，让真正重要的产品规则淹没其中。

2. **回流通道的方向性非对称**。Plan → PRD 的回流理论上合理，但实际上 plan 阶段识别出的 PRD 层面的问题往往是"这个需求在当前技术栈下不可行/成本过高"，这不是 PRD 写错了，而是"需求和技术不匹配"。回流这个信息到 PRD 只能触发"降低需求范围"，不能触发"让 PRD 写得更准"——这对 PRD skill 自身的改进没有价值。真正的价值在于：把这个信息回流到**下一个 PRD 的 feasibility pre-check** 中——让 AI 在起草时就避开已知的技术禁区。

3. **共享 knowledge-base 意味着共享写入权限**。谁有权限把 Plan 中发现的问题写入 knowledge-base？如果 Plan agent 能写，那它写的内容会不会污染 PRD agent 的知识空间？如果只有人才能写，那"自动回流"的自动化程度在哪里？

**修正建议**：

- knowledge-base 强行分层：`business-knowledge/`（产品侧，长期稳定）和 `technical-knowledge/`（技术侧，可能快速变化），两层的成熟度体系和淘汰周期分别独立管理（产品侧淘汰周期 6 个月，技术侧淘汰周期 3 个月）。
- 回流不通到 PRD 的自身改进，而是通到 "feasibility-constraint 注入"：Plan 中的技术约束不是告诉 PRD"你写错了"，而是作为下一个 PRD 的 pre-check 约束（"这个需求在规划阶段就被告知不可行/高成本"）。
- 自动回流只生成"建议草稿"（和错题本一样），不自动入库。人和 PRD owner 的共同确认是入库的必要条件。

## 三、被第一轮分析系统性忽略的四个深层张力

### 张力 1：自进化 vs 可预测性

spec-prd 的 6 条核心原则明确要求"Brownfield first"和"Evidence-tag current-state claims"——这些原则本质上在追求**可预测、可审计、可追溯**的 PRD 产出。但"自进化"机制（错题本、成熟度升级、知识漏斗收敛）每一次运行都可能在后台改变 AI 的行为。当 owner 发现 PRD 输出"变好了"或"变差了"，他无法知道是模型升级了、知识库变了、还是纠正条目被匹配了。

**建议**：每次 PRD 生成完成后，不仅要输出 Decision Card，还要输出一个 `evolution-trace`——本次运行加载了哪些纠正条目、匹配了哪些知识卡片、它们的成熟度和来源。只有 owner 能回溯"为什么这次输出和上次不同"，自进化才不是黑箱。

### 张力 2：特异性 vs 通用性

侠客汇的文章都在讲同一个项目（会员系统），他们的所有机制都在"单一业务线深度积累"的假设下成立。但 spec-prd 的定位是通用 PRD skill——它今天可能在用在一个电商项目，明天可能用在一个金融项目。knowledge-base 中的知识如果不能跨项目迁移，积累的价值就只在单个项目中；如果能跨项目迁移，就需要在各项目之间做知识隔离（金融项目的业务规则污染电商项目的知识库？）。

**建议**：knowledge-base 必须支持 `namespace` 隔离（project-level），同时支持 `cross-project` 晋升（被 3+ 个项目引用的知识 → 预览式晋升到 shared-global-knowledge）。跨项目晋升需要 owner 显式审批。

### 张力 3：人机协同的"注意力的经济学"

侠客汇的设计处处体现一个原则："人只做决策和审核"。但决策和审核本身是消耗注意力的稀缺资源。如果每次 PRD 起草都需要人在 3 个 section_anchor 确认方向、在 Readiness Lens 中审核、在纠正条目入库时审批，人的总注意力消耗可能比"人自己写 PRD"还高——分散的小决策不是一个"小"成本，它们是上下文切换的累积成本。

第一轮分析没有问一个根本问题：**所有这些机制加起来，人的总注意力净增还是净减？**

**建议**：引入 `attention-budget` 概念——在 Decision Card 中追踪本次 PRD 运行中人对系统的干预次数和干预类型。当干预次数/类型的分布偏离健康基线（如 section_anchor 确认被 100% 跳过但 Readiness Lens 的 revision 次数上升），触发"流程健康度"审计。

### 张力 4：知识的"熵增"问题

知识库只增不减（或只按零引用淘汰）会导致一个更隐蔽的问题：知识库中的知识总量增长，但知识的"平均新鲜度"在持续下降。新知识被创建、旧知识没有被淘汰（只是没被引用，但没到 3 个月），知识库的信息熵在持续增加。最终 AI 在 Phase 0 的漏斗收敛中，面对几十条"两月前创建、没被引用但也没被淘汰"的 draft 知识，检索的精度会持续下降。

**建议**：引入 `知识新鲜度评分`——创建时间 + 最后引用时间 + 引用频率的加权评分。新鲜度低于阈值的 draft 知识在漏斗收敛中自动排到末尾。同时引入定期"知识库瘦身"，不仅仅是淘汰零引用知识，也提醒 owner 重新审阅创建超过 6 个月的已验证知识。

## 五、总结：三句话

1. 第一轮分析的方向是对的——spec-prd 需要从"工具"走向"会学习的工具"。但"会学习"不是"把侠客汇的机制搬过来"——侠客汇在单一业务线、明确角色结构、高频率需求迭代的上下文中有效，spec-prd 在通用 PRD skill、模糊角色边界、多样化项目类型的上下文中需要不同的边界设计。

2. "可观测性优先于自动化"应该成为改进的核心原则。在错题本能证明自己不会制造更多问题之前，在知识漏斗能证明自己不会漏掉关键信息之前，在成熟度体系能证明自己不会锁定错误知识之前——不要自动化写入、不要自动化升级、不要强制加载。

3. spec-prd 当前设计中那些"看起来是缺失"的部分（无专用 agent、无自进化、无知识漏斗），有一部分是刻意不做——因为 spec-prd 的设计哲学是"先跑通质量闸门，再考虑自动化"。这个哲学不应该被侠客汇的范式简单否定——但可以被侠客汇的"由可观测数据驱动迭代"的工程方法丰富。
