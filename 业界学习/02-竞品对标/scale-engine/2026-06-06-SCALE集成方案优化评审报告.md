## 3. 两处前瞻性承重裂缝(v1.16 / spike 启动前必须想清楚)

无法用当前磁盘证实(provider 未接入、spike 未启动),属设计判断;但对抗审查论证扎实,建议保留为待你拍板的决策。

### 裂缝 A · provider-readiness 轴 A 被定位为「deterministic 机械字段」,代码里实为 caller 自报的 enum 归一化

`setup-facts.js` 的 `normalizeProviderReadiness` 只做 enum 归一,没有任何确定性手段判断 index 是否 `stale`;而方案又钉死「刷新归工具、spec-first 不代刷」。到 v1.16 必然二选一破裂:

- 要么 spec-first 去探 index mtime(开始拥有 lifecycle,违反「消费不耦合」);
- 要么承认 `fresh` 是工具自报(轴 A 不是 deterministic)。

**两者必居其一,方案当前两头都占。** 建议现在就把填值责任钉死:若不探新鲜度,provider 自报的 readiness 必须降为轴 B advisory,不得进入声称 deterministic 的轴 A 字段;或在 v1.16 引入最小 deterministic 新鲜度探针(artifact mtime vs git HEAD 时间)作为唯一可写 `fresh/stale` 的来源。

### 裂缝 B · honest-closeout 的「牙齿」建立在与它要防的谎报同类的信任假设上

§4.4 自承 capture 的 exit_code 可信度 =「workflow 如实转录」,弱于进程级监督。失败场景:一个会在散文里谎称「测试通过」的 workflow,同样能向 capture 提交伪造 `exit_code=0`,此时 honest-closeout 判 `consistent/verified` 并盖结构化背书章 —— 使虚假声明**比纯散文谎报更可信**,而非更不可信。这是 P0 整条「先让事实可见」链路的承重假设。

- **建议**:把 `verified` 明确限定为 honest-but-transcribed(非 process-verified),最终回复披露该限制等级;或给 capture 加 caller-independent 校验(log_path 存在且非空、命令时刻与 capture 时刻一致性)。

## 4. 三个高风险架构论点的一手证据验证(2025–2026)

| 论点 | 裁决 | 关键证据 |
| --- | --- | --- |
| 强 deterministic harness(hooks/gates/checkpoint/evidence/recovery)+ 轻语义合同,而非强状态机或纯 prompt | **supported** | Claude Code hooks 官方定义「ensuring certain actions always happen rather than relying on the LLM to choose to run them」;best-practices 把 hooks(deterministic)与 CLAUDE.md(advisory)显式对立;Building Effective Agents 的 restraint 原则 + Bitter Lesson 共同支持「语义判断要轻、blocking 延后」 |
| 外部能力工具走 setup 帮装 + 原生 MCP 消费 + 能力类别抽象(不写死工具名)+ advisory 回源 | **supported** | MCP 2025-06-18 capability negotiation + `tools/list` 动态发现;CodexGraph(2024)/RepoGraph(ICLR 2025)证明 code/project graph 对编码任务有可度量增益且支持「可插拔」范式;Thoughtworks Radar 提示「abstraction tax」—— 抽象要轻量按需,不为低频能力叠协议层 |
| 知识沉淀优先 checked-in docs/solutions ledger + context budgeting + summary-first handoff,而非默认集成外部 memory 工具 | **supported** | Anthropic context engineering(context rot / attention budget / structured note-taking 即文件型 ledger);Chroma context rot 实证 ~300-token 聚焦优于 113k 全量;Mem0/MemGPT 的 refutes 证据属「跨会话对话记忆/个性化」域,与 coding 知识沉淀域不同,不构成能力缺口 |

**对方案的含义**:三个方向均无需改向。建议在对应 contract 各补一句边界声明:(1) 确定性投资集中在 harness 轴,语义判断走 LLM-as-judge / review subagent 而非正则硬规则;(2) capability-aware 显式写明「按 MCP capability 动态发现 + 能力类别名 + degraded/advisory 回源」三条边界 + 一条反过度设计护栏(低频能力不叠协议层);(3) Knowledge Harness 补一句 scope 声明(本机制服务 engineering knowledge sedimentation,跨会话对话/个性化记忆不在范围内),堵住未来用 refutes 证据扩张范围的口子。

## 5. v2 / spec-first-on-scale 研究线:围栏有效,但有一条承重前提被自己证伪

对抗审查给出 **blocking** 级发现:两份报告共同的立身前提「SCALE owns 去 FSM 的 deterministic runtime,spec-first owns content」**在当前 SCALE 上不存在** —— FSM(11个)+ G0-G22 GateStage + 5 hook 不可分离,`gateInlineCommands.ts:24` 的 gate 就是 `process.exit(2)` 物理阻断(源码取证)。可行性报告的增-4 取证与轻合同报告的推荐架构在这点上直接矛盾。配套问题:

- v2 的 8 条成功标准**全是 capability-parity,零 Evaluation Harness 指标** —— 即便全绿也回答不了「对用户值不值得」,违反角色契约 §7。
- Phase 0 控制 artifact `spec-first-on-scale-v2-spike.md` **被两份报告引用但磁盘上不存在**(ls 确认)—— 隔离边界活在 3 份文档的散文里,没有单一权威 kill-criteria 文档。
- 防 `.scale`/`.spec-first` 双 runtime truth 目前只靠纪律,无 tripwire。
- 行业证据被框定到它够不到的结论:所有被引系统都是 one-system-internal 的强弱分层,无一是「外部独立厂商 runtime + 内容包联邦」;最贴近的 Eino「Graph-as-Tool」证据其实支持 OPT-B(spec-first 主导、SCALE 当 provider),而非报告主推的 OPT-C 换底座。

**建议:启动 spike 前补三道门(若启动)**

1. 硬性 Evaluation 指标 gate —— parity 通过但指标(closeout 诚实度 / review 漏判率 / debug 命中率)不改善 = 关闭该线而非推进;
2. `src/` 与 `docs/contracts/` 零 SCALE 耦合 schema 的提交门禁 —— 防止 pack manifest / evidence mapping / agent shadow 经 source 静默渗入(这是真正的不可逆向量,目前几乎裸奔);
3. 先向上游确认「runtime-without-FSM」能力,拿不到就把 spike 重定义为 OPT-B(把 SCALE 当 provider 工具),而非 OPT-C 换底座。

另:把「FSM/gate 不可分离且 `process.exit(2)` blocking」从增补提升为 Phase-0 决定性 kill-criterion(若确认不可选择性禁用即 abort,不要等 Phase 3 实跑撞墙);把 spike 第一步明确为 spec-first 侧、零 scale 依赖的只读 manifest emitter(纠正「对不存在命令做 dry-run」的标签错位)。
