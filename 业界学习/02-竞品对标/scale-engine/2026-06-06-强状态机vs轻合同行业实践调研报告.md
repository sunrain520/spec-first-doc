# 强状态机 vs 轻合同行业实践调研报告

日期：2026-06-06
作者：leokuang
对象：判断 `spec-first` 与 `scale-engine` 集成时，应采用强状态机、轻合同，还是二者分层组合

## 结论

业界最佳实践不是“强状态机”和“轻合同”二选一，而是：

```text
强运行底座 + 轻语义合同

强运行底座：
  约束工具调用、权限、hook、checkpoint、retry、evidence、artifact、human approval、恢复与观测。

轻语义合同：
  约束 workflow 输入输出、source/runtime 边界、artifact 证据、角色职责和 handoff，
  但不把需求理解、架构取舍、review 判断、root cause、上下文展开等语义决策硬编码成状态机。
```

对 `spec-first` 的直接判断：

```text
不要把 spec-first 做成强状态机。
可以把 scale-engine 作为强 runtime 候选。
但 spec-first 应保留 light contract / skill / agent / workflow semantics。
```

如果后续采用 `scale-engine` 当底座，正确姿态不是“用 SCALE 状态机替代 spec-first 语义流程”，而是：

```text
SCALE owns deterministic runtime:
  hooks / gates / state checkpoints / evidence ledger / tool permission / recovery

spec-first owns semantic content:
  Codebase -> Spec -> Plan -> Tasks -> Code -> Review -> Knowledge
  skills / agents / artifact contracts / source-runtime boundary / review semantics
```

这与 `spec-first` 角色契约中的 `Light contract + Explicit boundaries + Scripts prepare, LLM decides` 不冲突。真正冲突的是把整个研发链路变成中心化审批状态机，让脚本或 runtime 替代 LLM 做语义判断。

## 研究边界

本报告回答三个问题：

1. 业界做 AI Agent / AI workflow 时，更偏强状态机还是轻合同？
2. 阿里、腾讯、字节公开产品与框架文档体现了什么实践？
3. 这些实践对 `spec-first` 是否应该让 `scale-engine` 做 runtime 底座有什么启发？

限制：

- “阿里、腾讯、字节实践”只基于公开官方文档与开源框架资料，不声称代表内部研发治理全貌。
- 本报告不是 v2 迁移计划，不定义具体 pack manifest 或 SCALE adapter schema。
- 本报告不修改 generated runtime mirrors：`.claude/`、`.codex/`、`.agents/skills/`。

## 判定框架

### 什么应该强

这些领域适合强 runtime、强 checkpoint、强 gate：

| 领域 | 为什么要强 | 对 spec-first / SCALE 的含义 |
| --- | --- | --- |
| 工具权限与危险操作 | 不能靠提示词要求模型“自律” | hooks / permission gate / mutation boundary 应由 runtime owning |
| 文件写入与命令执行 | 有副作用，失败后需要恢复与审计 | pre-tool / post-tool evidence、exit code、raw log ref 要确定性记录 |
| 长任务恢复 | 会中断、compact、resume、并发 | run id、checkpoint、artifact freshness 应由 runtime 负责 |
| 验证与 closeout | 最容易发生 fake completion | tests/logs/verification summary 必须有 confirmed evidence |
| human approval | 对不可逆动作、发布、删除、权限升级必须可停可续 | HITL checkpoint 是 runtime 问题，不是 prompt 问题 |
| retry / timeout / idempotency | 模型循环无法可靠处理副作用重复 | runtime 应保存 activity / tool call evidence，避免重复副作用 |
| knowledge promotion | 历史经验会污染后续判断 | verified promotion gate 应强，recall 命中仍是 advisory |

### 什么应该轻

这些领域不能被状态机替代：

| 领域 | 为什么要轻 | 正确合同形态 |
| --- | --- | --- |
| 需求理解 | 用户意图与产品边界需要语义判断 | PRD/brainstorm output contract，而非流程硬转移 |
| 架构取舍 | tradeoff 不可枚举 | 要求 evidence / assumptions / non-goals，不要求固定答案 |
| 计划拆解 | 任务结构依赖代码形态和风险 | plan/task artifact contract，不做中心化调度器 |
| review finding | 严重性与可修复性依赖语义 | finding schema + source refs，LLM 判定 |
| root cause | 需要假设、验证、反证 | debug ledger + deterministic facts，LLM 综合 |
| 上下文展开 | 是否读全文取决于当前问题 | summary-first + expand-on-trigger，而非固定全读 |
| agent / skill 选择 | 不同任务的最佳角色会变化 | light trigger contract + orchestrator judgment |

## 外部行业实践

### Anthropic：简单组合优先，workflow 与 agent 分工明确

Anthropic 的 agent 工程文章把 agentic systems 区分成两类：

- workflow：LLM 与工具沿预定义代码路径编排，适合任务可分解、流程可预测的场景。
- agent：LLM 动态决定流程和工具使用，适合步骤数量不可预判、需要灵活决策的开放任务。

它同时强调最成功的实现通常使用简单、可组合的模式，而不是复杂框架；复杂度应在需要时才增加。

对 `spec-first` 的启发：

- `spec-first` 的公开 workflow 应保持轻合同和可组合 skill，而不是上来构造全局状态机。
- `scale-engine` 的强 runtime 适合承接权限、hook、evidence 和恢复，不应强行接管所有语义路径。
- 对编码 agent，复杂文件修改数量和改动性质常常无法预判，应该让 LLM 在证据约束下决策，而不是写死路径。

### OpenAI Agents SDK：用 guardrails / handoffs / tracing 包住 agent loop

OpenAI Agents SDK 的公开文档把 agentic 应用描述为模型使用上下文和工具、在需要时 handoff 给专门 agent，并保留完整 trace。Guardrails 被设计在输入、输出和工具调用边界，Tracing 记录模型生成、工具调用、handoff、guardrails 等事件。

关键点不是把每个语义步骤做成硬状态，而是在 agent loop 的关键边界上加可观测和可拦截机制：

- handoff 仍然作为模型可选择的动作，而不是纯状态表跳转；
- tool guardrails 围绕每次 function-tool 调用；
- tracing 让流程可调试、可监控、可评估。

对 `spec-first` 的启发：

- 多 agent 审查、handoff 和 skill 调用可以保留 LLM 决策，但必须留下 trace/evidence。
- 该强的地方是工具边界、handoff 输入过滤、输出验证，而不是把 review/plan/debug 的语义判断脚本化。

### LangGraph / Temporal：强 durable execution 服务长任务与恢复

LangGraph 文档把 durable execution 定位为在关键点保存执行进度，支持人机协作、长运行任务、错误中断后的恢复；Human-in-the-loop middleware 会在敏感工具调用前暂停并等待人类决策。Temporal 的核心价值则是 durable workflow、activity retry、signals、timers 和状态恢复。

这类系统证明：强状态或强工作流 runtime 在以下场景有明显价值：

- 长任务；
- 需要中断后恢复；
- 有外部副作用；
- 需要审批；
- 需要 retry / timeout / idempotency；
- 需要可观测运行状态。

但这不等于所有 AI 推理都应变成 FSM。Temporal/LangGraph 强的是执行可靠性、恢复、HITL 和 side-effect 管理；LLM 节点内部仍然可以是语义判断。

对 `spec-first` 的启发：

- `spec-work` closeout、verification run、review gate、knowledge promotion 适合强 evidence gate。
- `spec-brainstorm`、`spec-plan`、`spec-debug` 内部的推理过程不应被硬状态机约束到失去弹性。

### Claude Code hooks：在生命周期和工具边界加确定性拦截

Claude Code hooks 文档展示了 lifecycle hooks、tool hooks、matcher、permission decision 等机制。`PreToolUse` 可以在工具调用前阻断危险命令，`PostToolUse` 可以在调用后记录或验证，hook 可以按工具名或参数匹配。

这类机制的意义是把关键治理点从提示词下沉到 runtime：

- 危险命令阻断；
- 文件写入后 lint/test；
- MCP 写操作审计；
- session start/end 或 compact 前后处理。

对 `spec-first` 的启发：

- SCALE 的 hook/gate 层与业界强 runtime 方向一致。
- 但 hook 应服务明确边界，不应成为所有语义流程的中心化控制器。

## 国内大厂公开实践

### 阿里云百炼：Agent / Workflow / 高代码三模式并存

阿里云百炼公开文档把应用分为智能体、工作流和高代码三种核心模式：

- 智能体应用由提示词驱动，能够理解用户意图、自主决策，并调用知识库、MCP 服务等工具。
- 工作流应用通过可视化节点编排，把多步骤复杂任务串成稳定、可控、可复现的执行链路。
- 高代码应用由 Python 代码控制逻辑和执行路径。

这是一种非常明确的混合架构：

```text
开放式对话 / 规划 / 动态工具选择 -> Agent
固定流程 / 审批 / 可复现链路 -> Workflow
复杂后端 / 专业系统集成 -> Code
```

对 `spec-first` 的启发：

- `spec-first` 不能只选强 workflow，也不能只靠 agent prompt。
- 对需求、计划、debug、review 这类语义密集工作，应保留 Agent/LLM 自主判断。
- 对验证、closeout、runtime projection、evidence、clean/init 这类确定性工作，应交给 runtime/script 强约束。

### 腾讯云智能体开发平台：自由转交、工作流编排、Plan-and-Execute 三类协同

腾讯云 ADP 的 Multi-Agent 文档把协同方式分成：

- 自由转交：模型驱动转交，配置简单、自由度高，但稳定性依赖 Agent 名称、转交描述和提示词质量。
- 工作流编排：通过固定流程编排 Agent 节点，任务执行稳定可控。
- Plan & Execute：由 Planner 拆解任务，Executor 执行，适合深度分析等耗时复杂任务。

腾讯还允许把 Agent 节点嵌入工作流。Agent 节点内部具备自主任务规划、工具选择调用和按预设逻辑执行能力；工作流层可以配置超时、重试和异常处理。

这说明腾讯的公开平台实践也是混合式：

```text
模型自主转交解决灵活性；
工作流编排解决稳定性；
Plan & Execute 解决深度复杂任务；
Agent 节点嵌入 Workflow，允许强流程中保留局部自治。
```

对 `spec-first` 的启发：

- `$spec-doc-review` 多 persona 审查类似 Plan & Execute / multi-agent review，但 finding 仍由最终 orchestrator 去重、判定和收敛。
- `scale-engine` 可以承接超时、重试、异常处理、evidence ledger。
- 但不能把 reviewer finding 的成立性、计划拆解质量、根因判断交给状态机。

### 字节 / CloudWeGo Eino：Agent 与 Graph 是互补范式

CloudWeGo Eino 官方文档把 Agent 和 Graph 明确区分：

- Agent 以 LLM 为决策中心，适合开放、连续、多轮任务。
- Graph 由开发者预设拓扑，强调过程确定性和任务闭合。
- 推荐的集成点是把 Graph 封装成 Agent 可调用的 Tool。

Eino ADK 也同时提供：

- `ChatModelAgent` / ReAct loop：模型作为 decision maker，工具作为 action space。
- Sequential / Parallel / Loop / Supervisor / Plan-Execute 等协作原语。
- Runner、checkpoint/resume、interrupt/resume、callbacks 和 tracing。
- Graph/Chain/Workflow orchestration，用于类型检查、流处理、并发管理、状态读写和字段映射。

这套体系非常贴近本报告结论：

```text
Graph/Workflow = deterministic capability
Agent = semantic/autonomous decision center
Graph can become Agent's Tool
```

对 `spec-first` 的启发：

- 如果 `scale-engine` 是 Graph/Workflow/runtime，那么 `spec-first` skill 可以是 Agent 可调用的语义能力包。
- 正确组合不是把 Agent 关进 Graph，而是让 Graph 提供可靠工具与流程边界，让 Agent 决定何时使用。

## 对 scale-engine 作为 spec-first 底座的判断

本地 `scale-engine` 定位文档把 SCALE 定义为 Agent Governance Runtime / AI Engineering OS，核心包括 workflow state machines、hard gates、verification evidence、hook interception、artifact persistence、context budgets、memory provider routing、skill/tool orchestration。其战略表述是用系统约束、证据和 runtime gates 替代 agent self-discipline。

这与外部行业实践中的“强运行底座”一致，尤其适合：

- 防 fake completion；
- 防 skipped tests；
- 防危险 tool overreach；
- 保存 evidence；
- 支持 resume / handoff；
- 管理 context budget；
- 记录 memory / lessons；
- 在多宿主间复用治理语义。

但 `scale-engine` 不能直接无差别替代 `spec-first`，原因仍然是 owner 边界：

| 面 | SCALE 适合 owning | spec-first 应继续 owning |
| --- | --- | --- |
| host hooks/settings | 是 | 否，若换底座应让出 runtime projection |
| evidence ledger | 是 | 定义 evidence 对 workflow 的语义消费方式 |
| verification gate | 是 | 定义哪些 claim 需要什么 evidence |
| state checkpoint | 是 | 不把 checkpoint 当 workflow 语义真相 |
| skill/agent content | 可安装/投影 | 是，作为 content pack source |
| review finding | 不应判定 | 是，LLM/persona/orchestrator 语义判断 |
| plan/debug/root cause | 不应脚本化 | 是，轻合同约束输出与证据 |
| knowledge recall | 可做 provider | `docs/solutions` 与 promotion boundary 仍要明确 source truth |

当前 `scale-engine` 的 Codex adapter 会写 `.scale/`、`.codex/hooks.json`、`.codex/config.toml` 和 `AGENTS.md`。这说明它一旦作为底座，就必须成为 runtime owner；不能同时让 `spec-first init` 和 `scale init` 双写同一 host 入口与 runtime 资产。

## 推荐架构：强 runtime + 轻 workflow content pack

如果推进 `spec-first-on-scale` spike，建议用以下边界：

```text
spec-first package:
  skills/
  agents/
  workflow entrypoint manifest
  artifact contracts
  source-runtime boundary docs
  knowledge promotion rules

scale runtime:
  host projection
  hooks / gates
  checkpoint / resume
  evidence ledger
  permission / mutation boundary
  runtime doctor / setup
  optional memory / context provider
```

### 强 gate 只放五类地方

1. mutation gate：写文件、删文件、改 host runtime、运行危险命令。
2. verification gate：声明完成、声明测试通过、声明修复时必须有 evidence。
3. source/runtime gate：禁止把 generated runtime mirror 当 source 修。
4. handoff gate：跨 workflow 的 artifact 必须有 summary、source refs、freshness 和 limitations。
5. knowledge promotion gate：只有 verified、可复用、有 invalidation condition 的经验进入 durable knowledge。

### 轻 contract 保留七类判断

1. 需求是否明确。
2. 计划是否合理。
3. 任务拆分是否过细或过粗。
4. review finding 是否成立。
5. root cause 是否被证据支持。
6. 是否需要展开更多上下文。
7. 是否需要 human owner 介入。

### 不建议的反模式

| 反模式 | 后果 |
| --- | --- |
| 把 `Codebase -> Spec -> Plan -> Tasks -> Code -> Review -> Knowledge` 变成硬 FSM | 会把语义工作伪装成流程推进，降低真实判断质量 |
| 每个 workflow 都必须经过全局状态注册 | 增加状态噪声，轻量任务成本过高 |
| 让 runtime 判定 review/plan/debug 结论 | 脚本替代 LLM，违反 `Scripts prepare, LLM decides` |
| 同时保留 `.spec-first` 和 `.scale` 两套 runtime truth | source/runtime 冲突，doctor/clean/init 都会失真 |
| 把 provider 输出当 confirmed context | 重演 code graph/provider 过耦合问题 |
| 为低频边界做完整状态机 | 过度设计，增加维护和测试矩阵 |

## 最小落地顺序

### Step 1：先做只读映射，不动 runtime

输出 `spec-first-on-scale` spike 文档或 manifest 草案：

- skill manifest：`skills/*/SKILL.md` 如何成为 SCALE skill。
- agent manifest：`agents/*.agent.md` 如何映射到 SCALE agent loader。
- entrypoint manifest：`$spec-*` / `/spec:*` 如何保持用户心智。
- evidence mapping：`verification-run-summary`、`honest-closeout`、`review-finding` 如何映射到 SCALE evidence ledger。

验收：

- 不运行 `scale init`。
- 不写 `.scale/`、`.codex/`、`.claude/`。
- 只生成 docs / manifest draft。

### Step 2：做 dry-run projection

让 SCALE 或桥接脚本只输出“将会写什么”：

- host settings/hooks diff；
- `AGENTS.md` / `CLAUDE.md` managed block diff；
- skill/agent install diff；
- evidence directory diff；
- clean/uninstall plan。

验收：

- preview-first；
- 无 silent write；
- 能发现与现有 `spec-first init` 的冲突点。

### Step 3：选择单宿主试点

先选 Codex 或 Claude 之一，验证：

- 一个入口能被投影；
- 一个 skill 能运行；
- 一个 reviewer agent 能被发现；
- 一条 verification evidence 能回写；
- `clean` 能完全回滚。

验收：

- 不要求全量替换；
- 不删除 `src/cli/**`；
- 不改变 v1.x 主线。

### Step 4：再评估是否退役 spec-first CLI runtime 逻辑

只有当以下条件满足，才考虑删除或迁移 `spec-first` 原 CLI 逻辑：

- SCALE runtime build/install 稳定；
- SCALE 能等价承载双宿主投影；
- SCALE 能清晰处理 `source-of-truth` 与 generated runtime；
- `spec-first` skills/agents/content pack 格式兼容；
- evidence 与 knowledge promotion 语义能保真；
- 用户入口心智不破坏。

## 最终判断

行业实践支持的不是“强状态机替代轻合同”，而是“强运行边界保护轻语义判断”。

因此：

```text
spec-first 当前哲学是正确的：
  Light contract + Explicit boundaries + LLM decides。

scale-engine 的 runtime 方向也是有价值的：
  hooks + gates + evidence + checkpoint + memory/context orchestration。

二者正确组合方式：
  SCALE 做底座，spec-first 做 content/semantics pack。

二者错误组合方式：
  SCALE 的状态机吞掉 spec-first 的语义 workflow，
  或 spec-first 同时保留 runtime owner 与 SCALE 双写 host。
```

换句话说，`scale-engine` 可以成为 `spec-first` 的运行底座，但前提是它只强在 deterministic runtime boundary 上；`spec-first` 不能放弃 light contract，也不能把 AI coding harness 演化成中心化流程引擎。
