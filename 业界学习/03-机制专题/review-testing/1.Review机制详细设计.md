# Review 机制详细设计

> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、`CRG Stage-0`、`ECC`、`compound-engineering-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 背景

这个项目里的 Review 不是单一动作，而是两套相关但不同的机制：

1. `spec-doc-review` 用来审 **需求文档 / 计划文档**
2. `ce:review` 用来审 **代码变更 / PR / 分支 diff**

两者都属于“质量闸门”，但关注对象不同，处理方式也不同。

## 核心结论

项目的 Review 机制可以概括为：

**多 persona 并行审查 + 置信度门禁 + 冲突合并 + 自动修复 / 人工确认分流 + 必要时回退到 work。**

它不是“看一眼给建议”，而是一个结构化流水线：

- 先确定审查对象
- 再选择审查维度
- 再并行产出 findings
- 再过滤低置信度结果
- 再去重与合并冲突
- 再按可自动化程度路由
- 最后把未完成的问题外化给后续阶段处理

## 一、`spec-doc-review`：审需求和计划文档

参考对象：

- [spec-doc-review](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/spec-doc-review/SKILL.md)
- [ce:plan](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-plan/SKILL.md)
- [ce:brainstorm](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-brainstorm/SKILL.md)

### 1. 它审什么

- `docs/brainstorms/` 下的需求文档
- `docs/plans/` 下的计划文档

### 2. 它怎么审

#### 2.1 先分类文档类型

- requirements
- plan

#### 2.2 再按内容选择 reviewer persona

**always-on：**

- `coherence-reviewer`
- `feasibility-reviewer`

**conditional：**

- `product-lens-reviewer`
- `design-lens-reviewer`
- `security-lens-reviewer`
- `scope-guardian-reviewer`
- `adversarial-document-reviewer`

#### 2.3 并行 dispatch

所有 persona 并行运行，每个 persona 返回结构化 findings。

#### 2.4 合并与过滤

审查结果会经过一条固定流水线：

1. schema 校验
2. confidence gate
3. dedup
4. residual concern promotion
5. contradiction resolution
6. route by autofix class

### 3. 它的硬规则

- 低于置信度阈值的 findings 会被压掉
- 文档里的冲突不会简单删掉，而是合并成 tradeoff 或 combined finding
- `docs/brainstorms/`、`docs/plans/`、`docs/solutions/` 是受保护产物，不能被 review 建议删除
- 最终会输出 `Review complete` 作为终止信号

### 4. 它和 `ce:plan` 的关系

`ce:plan` 自带一个 confidence check，和 `spec-doc-review` 不是同一个层次：

- `spec-doc-review` 更偏向清晰度、完整性、范围控制
- `ce:plan` 的 confidence check 更偏向理由是否足够强、顺序是否合理、风险是否补足、系统性思考是否到位

`ce:plan` 在后续阶段会推荐或调用 `spec-doc-review`，尤其是在这些情况下：

- Deep plan
- 高风险计划
- confidence check 深化了多个 section
- 或者 plan 的 grounding 比较薄

## 二、`ce:review`：审代码变更 / PR

参考对象：

- [ce:review](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-review/SKILL.md)

### 1. 它审什么

- PR number
- GitHub URL
- branch
- diff
- untracked changes

### 2. 它的入口模式

`ce:review` 有三种模式：

- **Interactive**
  - 默认模式
  - 允许问用户问题
- **Autofix**
  - 自动修 `safe_auto`
  - 不提交、不 push、不创建 PR
- **Report-only**
  - 纯只读
  - 不改文件、不建 todo、不提交

### 3. 它的整体流程

#### 3.1 先确定审查范围

- 如果给了 `base:`，直接用这个 base
- 如果是 PR，就 checkout PR branch，再算 diff base
- 如果是 branch，就算 merge-base
- 如果是 standalone，就拿当前 branch 对比 base

#### 3.2 可选读取 plan

`ce:review` 会尝试找到 plan 文档，用来做 requirements completeness verification。

它读取的核心内容是：

- `Requirements Trace`
- `Implementation Units`

然后在最后的 review 里检查：

- 计划里的要求有没有被实现
- 实现单元有没有对应完成

这里有一个重要区别：

- **explicit plan** 没满足的要求，会被视为更严重的问题
- **inferred plan** 只是提示，不是强契约

#### 3.3 选择 reviewer

**always-on：**

- `correctness-reviewer`
- `testing-reviewer`
- `maintainability-reviewer`
- `project-standards-reviewer`
- `agent-native-reviewer`
- `learnings-researcher`

**按 diff 追加：**

- `security-reviewer`
- `performance-reviewer`
- `api-contract-reviewer`
- `reliability-reviewer`
- `adversarial-reviewer`
- 以及语言/领域特定 reviewer

**migration 相关：**

- `schema-drift-detector`
- `deployment-verification-agent`

#### 3.4 并行审查

所有 reviewer 并行运行，每个 reviewer 返回结构化 findings。

#### 3.5 合并结果

结果合并时也有明确规则：

- 先做 schema validation
- 再做 confidence gate
  - 低于 **0.60** 的 findings 会被压掉
- 再做 dedup
  - 通过 `file + line bucket + title` 做 fingerprint
- 再归类路由
  - `safe_auto`
  - `gated_auto`
  - `manual`
  - `advisory`

#### 3.6 处理 `requires_verification`

如果某个 finding 标了 `requires_verification: true`，那么修复后还不能算结束，还要补：

- targeted tests
- focused re-review
- 或 operational validation

### 4. 它的输出

`ce:review` 的最终输出包含：

- Findings
- Requirements completeness
- Applied fixes
- Residual actionable work
- Pre-existing
- Learnings & past solutions
- Agent-native gaps
- Schema drift check
- Deployment notes
- Coverage
- Verdict

### 5. 它的后续动作

按模式不同，后续动作不同：

- **Autofix**
  - 只自动修 `safe_auto`
  - 写 `.context/compound-engineering/ce-review/<run-id>/` artifact
  - 必要时生成 todo
- **Report-only**
  - 只出报告
- **Interactive**
  - 让用户决定修、只看报告，还是继续

## 三、这套 Review 机制的本质

如果把它抽象成工程机制，可以总结成 6 步：

### 1. 先定义审查对象

- 文档 or 代码

### 2. 再挑审查维度

- coherence
- feasibility
- security
- correctness
- testing
- maintainability
- performance

### 3. 并行出 findings

避免单一 reviewer 视角过窄。

### 4. 再做置信度门禁

低置信度结果直接压掉，减少噪音。

### 5. 再做去重和冲突处理

避免重复意见，也避免把真正的 tradeoff 简单吞掉。

### 6. 再按可自动化程度路由

- 自动修
- 批量确认
- 只展示给用户

最后把没完成的部分外化到：

- todo
- artifact
- 回退到 work

## 四、和整条工作流的关系

在 `brainstorm -> plan -> work -> review -> compound` 这条链路里，Review 是质量闸门：

- `brainstorm` 产需求文档
- `plan` 产实现计划
- `work` 产代码成果
- `review` 检查成果是否真的可以交付
- `compound` 把已验证解决方案沉淀成知识

所以 Review 的作用不是终点，而是：

- 把 work 的成果过滤成“可合格交付”
- 把问题明确回传给 work
- 把已解决的问题沉淀给 compound

---

## 五、结论

这个项目的 Review 机制可以概括为：

> 多 persona 并行审查 + 置信度门禁 + 冲突合并 + 自动修复 / 人工确认分流 + 必要时回退到 work。

它不是一个单一动作，而是一套控制质量、过滤噪音、推动回修、并把已解决问题沉淀下来的结构化机制。
