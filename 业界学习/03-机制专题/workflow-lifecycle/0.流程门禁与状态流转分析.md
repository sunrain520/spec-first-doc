# 流程门禁与状态流转分析

## 背景

在 `brainstorm -> plan -> work -> review -> compound` 这条主流程里，项目并不是用一套统一的“状态机”去硬编码所有阶段，而是采用了更贴近实际协作的方式：**每个阶段自己有门禁、自己有交付物、下一阶段读取前一阶段的产物**。

这意味着，流程控制不是靠一个中心状态表去统一调度，而是靠各阶段 skill 的输入输出契约来串联。

## 核心结论

项目里的执行过程，更像下面这三层结构的组合：

1. **阶段门禁**：当前阶段必须满足最低条件，才能进入下一阶段；
2. **阶段产物**：每一阶段都要产出明确文档或结果，作为下一阶段输入；
3. **阶段继承**：后续阶段不会重新发明前一阶段已经确认的内容，而是直接读取并延续。

所以，真正管控流程的方式，不是“一个全局状态机”，而是“每一段自己负责守门”。

## 各阶段规则

### 1. Brainstorm -> Plan

Brainstorm 的职责是把问题、目标、边界和可选方向说清楚。它的门禁重点是：

- 如果还有真正阻塞 planning 的问题，不能进入 Plan；
- 如果还需要去“发明”产品行为、范围边界或成功标准，Brainstorm 还没完成；
- 只有当问题已经足够清楚，Plan 才能接手把 WHAT 变成 HOW。

也就是说，这里不是硬性要求“候选方案必须 >= 2 个”，而是要求**问题已经可规划**。

### 2. Plan -> Work

Plan 的职责是把需求收敛成可执行方案。它必须产出：

- Problem Frame
- Requirements Trace
- Scope Boundaries
- Implementation Units
- Verification / Test Scenarios
- 风险、依赖、技术决策

如果这些内容不够明确，Work 阶段就会缺少执行依据，所以 Plan 的本质门禁是：**方案必须可执行**，而不是只有方向。

### 3. Work -> Review

Work 的职责是把 Plan 落成真实交付物，通常包括：

- 代码变更
- 测试结果
- 任务完成状态
- 必要时的增量提交

如果 Work 没有形成明确交付物，Review 就没有可审查对象。因此这里的实际规则是：**必须先有可验证的成果，Review 才有意义**。

### 4. Review -> Work

Review 的作用不是终点，而是质量反馈回路。

如果 Review 发现阻断问题，尤其是 P0 / P1，或者出现 `requires_verification: true` 的 finding，说明当前成果还不能算完成，必须回到 Work 修正。

所以这里的流转逻辑是：

- Review 通过 -> 流程可以继续；
- Review 不通过 -> 回到 Work 修复；
- 需要验证的问题 -> 必须补验证后再收敛。

### 5. Review / Work -> Compound

Compound 的职责是把“已经解决的问题”沉淀成可复用知识。

它更看重的是：

- 这是一个 recent solved problem；
- 解决方案已经被验证；
- 最终交付已经稳定下来；
- 如果已有高重叠文档，优先更新旧文档，而不是重复创建。

所以 Compound 在语义上更接近“已验证后的知识沉淀”，而不是“尚未收敛时的随手记录”。

## 实际流程控制方式

从项目文档看，流程控制主要靠下面几种机制：

### 1. 阶段输入门槛

每个阶段都有自己能接受的输入类型：

- Brainstorm 接收问题和目标；
- Plan 接收需求文档或足够清晰的想法；
- Work 接收计划文档或可执行任务；
- Review 接收实际变更；
- Compound 接收已验证的解决结果。

### 2. 阶段输出契约

每个阶段都必须输出下一阶段能用的产物：

- Brainstorm 输出需求文档；
- Plan 输出实现计划；
- Work 输出可审查成果；
- Review 输出审查结论和修正意见；
- Compound 输出沉淀文档。

### 3. 阶段内自检

项目不是等到最后才检查，而是在每段内部就做约束：

- Brainstorm 检查是否还有阻塞问题；
- Plan 检查 Requirements Trace、Implementation Units、Verification 是否完整；
- Work 检查每个 unit 的完成标准；
- Review 检查是否存在阻断问题和需要补验证的 finding；
- Compound 检查是否与已有知识重复、是否应该更新旧文档。

## 对“状态流转规则”的判断

如果把这个项目抽象成状态流转规则，可以写成下面这个更准确的版本：

- Brainstorm 先确保问题可规划；
- Plan 先确保方案可执行；
- Work 先确保产物可审查；
- Review 先确保问题已收敛；
- Compound 先确保知识已验证。

也就是说，**项目强调的是阶段门禁和产物传递，而不是统一状态机的硬切换**。

## 结论

这条链路的本质是：

> 每个阶段自己守门，每个阶段自己产物，下一阶段只消费上一阶段已经确认过的结果。

因此，最贴近项目实际的表达不是“一个状态机控制所有阶段”，而是：

**brainstorm -> plan -> work -> review -> compound**

这是一条由门禁、产物和反馈回路共同构成的协作链路。
