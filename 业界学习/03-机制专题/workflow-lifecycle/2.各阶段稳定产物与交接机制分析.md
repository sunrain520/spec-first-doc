# 各阶段稳定产物与交接机制分析

> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、`CRG Stage-0`、`ECC`、`compound-engineering-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 背景

在这个项目的工作流里，`brainstorm -> plan -> work -> review -> compound` 不是简单的顺序动作，而是由一组阶段性产物串联起来的协作链路。

如果从“每阶段是否有稳定产物文件、下一阶段如何消费”这个角度看，当前项目**大体成立，但不是完全对称**：

- 有些阶段产物是明确落盘的文档
- 有些阶段产物是运行时 artifact 或 todo
- 有些阶段的产物就是代码 diff / PR 本身，而不是单独的 handoff 文件

## 核心结论

当前项目的真实形态可以概括为：

**Brainstorm / Plan / Compound 的文件化最明确，Review 有稳定外化路径，Work 主要以代码变更本身作为产物。**

也就是说，这条链路不是“每阶段都产出一类完全对称的稳定文件”，而是：

1. 前两段把问题和方案写成文档
2. Work 直接改变代码仓库状态
3. Review 把问题和修复建议外化成运行产物和 todo
4. Compound 再把已验证的解决方案沉淀成长期知识

## 一、Brainstorm 的稳定产物

Brainstorm 的 durable output 是 **requirements document**，也就是需求文档。

参考：

- [ce:brainstorm](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-brainstorm/SKILL.md#L13-L23)

### 产物落点

- `docs/brainstorms/YYYY-MM-DD-<topic>-requirements.md`

### 它怎么被下一阶段消费

`ce:plan` 会直接读取这个 requirements doc，把它作为 planning 的输入。

Brainstorm 的职责不是写实现细节，而是把这些内容稳定下来：

- 问题是什么
- 为什么要做
- 目标和边界是什么
- 哪些问题必须先解决

### 这一阶段的特点

这是最典型的“稳定产物文件”：

- 文件路径明确
- 内容职责明确
- 下一阶段明确依赖它

## 二、Plan 的稳定产物

Plan 的产物是实现计划文档。

参考：

- [ce:plan](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-plan/SKILL.md#L473-L503)
- [ce:plan](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-plan/SKILL.md#L609-L617)

### 产物落点

- `docs/plans/YYYY-MM-DD-NNN-<type>-<descriptive-name>-plan.md`

### 它怎么被下一阶段消费

`ce:work` 直接把 plan 作为主输入，尤其会读：

- `Requirements Trace`
- `Implementation Units`
- `Verification`

这意味着 plan 文件不是写给人看的“会议纪要”，而是后续执行的可操作说明书。

### 这一阶段的特点

Plan 也是稳定文件，但它更像：

- 需求到实现的中间契约
- 实现阶段的决策依据
- review 阶段的要求对照表

## 三、Work 的产物不是单独文档，而是代码变更本身

Work 阶段和前两段不同。

参考：

- [ce:work](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-work/SKILL.md)

### 它的主要产物

Work 的核心产物通常是：

- 代码 diff
- 测试结果
- 必要时的增量 commit
- 任务完成状态

### 它怎么被下一阶段消费

`ce:review` 不是读一个固定的 work 文件，而是直接审：

- PR
- branch
- diff
- untracked changes

也就是说，Work 阶段的“交付物”就是仓库状态的变化本身。

### 这一阶段的特点

Work 是整个链路里**最不像“稳定文件交接”**的一段：

- 没有统一的 work handoff 文件
- 代码仓库本身就是产物
- 下一阶段通过 diff / PR 来消费结果

这也是为什么它更像“执行阶段”，而不是“文档阶段”。

## 四、Review 的产物是运行时 artifact + todo 外化

Review 阶段有稳定外化路径，但它不是一个单一 canonical 文档文件。

参考：

- [ce:review](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-review/SKILL.md#L37-L52)
- [ce:review](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-review/SKILL.md#L494-L504)
- [todo-create](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/todo-create/SKILL.md#L11-L20)

### 产物落点

#### 1. Review 运行产物

- `.context/compound-engineering/ce-review/<run-id>/`

这里会记录：

- synthesized findings
- applied fixes
- residual actionable work
- advisory-only outputs

#### 2. Residual todo

- `.context/compound-engineering/todos/`

这是 review 发现的、还需要后续处理的持久化任务。

### 它怎么被下一阶段消费

Review 的输出不是只“看一下就完了”，而是会进入后续动作：

- residual work 回到 work 修复
- todo 进入后续 triage / resolve 流程
- report-only 内容保留在报告里

### 这一阶段的特点

Review 有稳定落盘，但它的稳定性更多体现在：

- 运行 artifact 可追踪
- todo 可跨会话持续存在
- 结果可回流到 work

而不是像 brainstorm / plan / compound 那样直接对应一个长期主文档。

## 五、Compound 的稳定产物

Compound 的目标非常明确：把解决方案写成长期可复用知识。

参考：

- [ce:compound](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-compound/SKILL.md#L13-L20)
- [ce:compound](/Users/kuang/xiaobu/compound-engineering-plugin/plugins/compound-engineering/skills/ce-compound/SKILL.md#L155-L160)

### 产物落点

- `docs/solutions/[category]/[filename].md`

### 它怎么被下一阶段消费

Compound 本身就是把“已验证的解决方案”沉淀下来，供之后的：

- brainstorm
- plan
- review

复用。

### 这一阶段的特点

它和 brainstorm / plan 一样，属于非常明确的长期文档产物。

而且它强调：

- 要从 verified fix 出发
- 要有 YAML frontmatter
- 如果已有高重叠文档，优先更新旧文档而不是重复创建

## 六、按阶段对照

| 阶段 | 稳定产物 | 典型落点 | 下一阶段消费方式 |
|---|---|---|---|
| Brainstorm | 是 | `docs/brainstorms/*-requirements.md` | `ce:plan` 读取 requirements doc |
| Plan | 是 | `docs/plans/*-plan.md` | `ce:work` 读取 plan 的 trace / units / verification |
| Work | 否，主要是代码变更本身 | diff / PR / branch | `ce:review` 直接审代码变更 |
| Review | 部分是 | `.context/compound-engineering/ce-review/<run-id>/`、`.context/compound-engineering/todos/` | 后续回修 work，或进入 todo 流程 |
| Compound | 是 | `docs/solutions/*/*.md` | 作为后续 brainstorm / plan 的知识输入 |

## 七、结论

如果你的判断标准是：

> 每阶段都有稳定产物文件，供下一阶段消费

那么这个项目的答案是：

**基本成立，但 Work 阶段是例外。**

更准确地说，当前项目是一个“文档驱动 + 代码变更驱动 + 运行产物驱动”的混合结构：

- 前半段靠 requirements / plan 文档交接
- 中间段靠代码 diff 交接
- Review 通过 run artifact 和 todo 保持连续性
- Compound 再把最终结果沉淀回文档库

所以，这条链路的稳定产物并不是完全对称的五件套，而是：

**requirements doc -> plan doc -> diff/PR -> review artifact/todo -> solution doc**
