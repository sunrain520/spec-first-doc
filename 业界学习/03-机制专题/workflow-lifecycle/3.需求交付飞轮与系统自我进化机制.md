# 需求交付飞轮与系统自我进化机制

## 核心判断

这套系统的终局，不是“有多少个 skill”，而是能否形成一个持续增强的交付飞轮：

- 需求交付越来越快
- 质量越来越好
- 经验会沉淀
- 系统能自我进化
- 每个需求都在给下一个需求铺路

## 飞轮图

```text
                 +--------------+---------------+

                 +--------------+---------------+
                 |                              |
                 |            Plan              |
                 |   把 HOW 说清楚、定方案      |
                 |                              |
                 +--------------+---------------+
                                |
                                v
                 +--------------+---------------+
                 |                              |
                 |            Work              |
                 |     把东西做出来、可验证     |
                 |                              |
                 +--------------+---------------+
                                |
                                v
                 +--------------+---------------+
                 |                              |
                 |           Review             |
                 |   把风险、缺口、回退拦住     |
                 |                              |
                 +--------------+---------------+

                                v
                 +--------------+---------------+
                 |                              |

                 +--------------+---------------+
                 |                              |
                 |   知识库 / 决策库 / 模式库   |
                 |   下一个需求直接复用         |
                 |                              |
                 +--------------+---------------+
                                |
                                v
                         回到新需求 / 新问题
```

## 每一环的作用

### 1. Brainstorm

- 先把“要解决什么”说清楚
- 明确用户问题、范围边界、成功标准
- 避免后面的 Plan 自己发明需求

### 2. Plan

- 把 WHAT 翻译成 HOW
- 明确实现路径、依赖、验证方式和风险
- 让 Work 能直接开工，而不是继续猜

### 3. Work

- 真正完成实现
- 产出代码、配置、测试、PR 等可审查结果
- 把计划落成现实

### 4. Review

- 检查 Work 是否真的能交付
- 找出阻断问题、遗漏项、风险点
- 决定是通过、回修，还是继续深化

### 5. Compound

- 把已经解决的问题写成可复用知识
- 记录解决方案、为什么这么做、哪些方案被否决
- 让以后遇到同类问题时，不必从头开始

## 为什么这个飞轮重要

如果只做“单次交付”，系统每次都要重新理解、重新规划、重新试错。  
如果做成飞轮，每一次需求都会留下：

- 更清晰的边界
- 更可靠的方案
- 更强的验证
- 更有价值的沉淀

最后，系统会从“帮你做完这次”变成“帮你越做越快、越做越稳”。

## 长期目标

这个飞轮的目标不是把流程做得更复杂，而是把系统做得更会学习：

- 交付速度越来越快
- 返工越来越少
- 决策越来越少重复
- 经验越来越可复用
- 系统越来越接近自我进化
