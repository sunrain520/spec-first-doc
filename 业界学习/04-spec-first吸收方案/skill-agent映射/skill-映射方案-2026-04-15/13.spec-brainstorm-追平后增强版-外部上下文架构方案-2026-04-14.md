## Part A：上游基线与严格追平事实

### A.1 目标定义

从上游事实视角看，`spec-brainstorm` 要“严格追平” `ce-brainstorm`，含义是：

- 上游 `ce-brainstorm` 已具备的能力，在 `spec-brainstorm` 中都必须可用
- 上游 `ce-brainstorm` 的缺失文件、规则、分支、引用关系，必须补齐
- 允许存在的差异，只能是 `ce-* -> spec-*` 的命名空间替换，以及 `ai:compound -> ai:spec-first` 的产品标识替换
- 任何会导致能力降级、交互退化、输出质量下降、上下文路由缺失的差异，都不应被视为“可接受分叉”

换言之，严格追平的意思不是“差不多”，而是：

**把 `spec-brainstorm` 恢复成以 `ce-brainstorm` 为 source of truth 的等价实现。**

### A.3 当前结论

基于上游事实，当前 `spec-brainstorm` **不满足严格追平**。

原因不是命名差异，而是存在真实能力缺口：

1. 缺少**非软件任务分流能力**

## Part B：当前产品的正式实施方案

### B.1 当前产品决策

在当前产品方案中，边界已经明确：

- `Slack context` **不纳入当前方案**
- `Slack context` 仅保留在 Part A 中作为上游事实参考

这意味着当前产品方案不是“继续对齐 Slack”，而是：

- 用更贴近团队真实工作方式的多源补充上下文系统替代上游的单一组织上下文设计

### B.2 方案目标

当前产品方案要回答的问题是：

**在不引入 `Slack context` 的前提下，如何把 `spec-brainstorm` 扩展成支持 GitHub、网页、在线文档、本地文档等多源补充上下文的更强版本。**

### B.3 当前方案的核心结论

更合理的架构是：

1. 在 `spec-brainstorm` 中新增一个更高层抽象：`Supplemental Context`
2. 让 `Local Docs`、`GitHub URL`、`Web URL`、`Docs URL`、`Issue/PR` 成为 `Supplemental Context` 下的不同 source adapter
3. `find-skills` 只作为 **fallback discovery**，不作为主执行路径

当前方案的本质不是：

- “沿用 Slack 再额外挂几个入口”

而是：

- **“按当前产品边界，构建一个不依赖 Slack、且把本地文档纳入一等公民的多源补充上下文系统。”**

### B.3.1 当前开发范围冻结

基于最新决策，当前方案不再分“先做一部分、其余后做”。  
当前开发范围直接冻结为：

1. `local-doc-reader`
2. `github-context-reader`
3. `web-context-reader`
4. `docs-context-reader`
5. `find-skills fallback`
6. `spec-brainstorm` 的 `1.1 Existing and Supplemental Context Scan` 路由改造

也就是说：

- `web-context-reader`
- `find-skills fallback`

这三项不再后置，直接纳入本轮开发范围。

### B.3.2 当前开发前置约束

虽然当前范围已经冻结为全量开发，但要避免返工，仍需要先把下面三个 contract 写清楚，再进入代码实现：

1. `research digest` schema
2. `Repo Context` 与 `Local Docs Context` 的边界
3. `find-skills fallback` 的触发条件与失败降级规则

这三项不是新阶段，而是当前开发前必须先钉死的设计 contract。

### B.4 为什么当前方案可以删除 Slack context

#### 原因 1：基线事实与当前方案已经分层

Part A 已完整记录上游 `ce-brainstorm` 的 `Slack context` 事实与依赖。  
因此 Part B 可以聚焦“当前产品要做什么”，而不需要继续把 Slack 带入实现方案。

#### 原因 2：团队真实场景不以 Slack 为主

如果团队主要使用：


那么继续保留 Slack 只会带来：

- 路由冗余
- 能力叙事模糊
- 实现资源分散

#### 原因 3：当前产品需要的是“真实可用的补充上下文”

当前真正高价值的补充上下文是：

- GitHub 工程上下文
- Web / Docs 在线内容上下文
- 本地文档上下文

### B.5 增强版目标状态

增强后的 `spec-brainstorm` 应从当前的“repo context only”升级为：

| 层级 | 能力目标 |
|---|---|
| `Brainstorm Core` | 保持上游 brainstorm 主流程、探索护栏、requirements capture 与 handoff 不变 |
| `Context Router` | 在 brainstorm 过程中判断是否需要补充 repo 外或 repo 内的附加上下文 |
| `Source Adapters` | 支持 Local Docs、GitHub、Web URL、在线文档、Issue/PR 等来源 |
| `Synthesis Layer` | 把多源内容统一转成 brainstorm 可消费的约束、历史决策、风险、已知方案 |
| `Fallback Discovery` | 当没有显式 adapter 时，可尝试用 `find-skills` 发现能力，但不作为主执行路径 |

### B.5.1 当前实现必须吸收的上游非 Slack 差异

| 至少一个非显然角度 | 是 | 回补到 `Phase 2 Explore Approaches` |

### B.6 架构原则

#### 原则 1：增强不能破坏当前产品边界

- 不再把 Slack 纳入当前方案
- 文档上下文优先收敛到本地文档与在线文档
- 增强层服务当前团队真实工作方式，而不是服务上游兼容叙事

#### 原则 2：source adapter 必须显式，而不是隐式动态猜测

增强版应显式支持若干已知 source 类型，而不是把所有补充上下文都交给“临时找个 skill 试试”。

#### 原则 3：本地文档是一等公民，不是 fallback

很多对 brainstorm 最有价值的输入，并不来自公网，而来自：

- `docs/` 下已有分析文档
- 本地 markdown / txt / pdf
- 用户显式给出的需求文档、纪要、方案说明

#### 原则 4：`find-skills` 只做 fallback，不做主执行器

`find-skills` 的定位是：

- 发现能力
- 提示安装
- 兜底 fallback

不是主 contract 的稳定执行路径。

#### 原则 5：统一 digest 输出

不管 source 是本地文档、GitHub 还是网页，最终都应被收敛成统一的 `research digest`，而不是把原始内容直接倒给 brainstorm。

#### 原则 6：`Repo Context` 与 `Local Docs Context` 必须边界清晰

为了避免重复扫描、重复摘要与冲突覆盖，必须明确：

- `Repo Context` 负责代码与仓库级治理约束
- `Local Docs Context` 负责用户显式指定文档，以及 topic-driven 命中的本地文档材料
- 同一份文档不能在两个阶段被重复读取并重复摘要，除非显式要求强制重读

### B.7 能力模型

#### 1. Repo Context

这是当前已存在的基础能力：

- `AGENTS.md` / `CLAUDE.md` 约束扫描
- repo 代码与文档扫描
- 现有 brainstorm / plan / spec / skill artifact 扫描

#### 2. Local Docs Context

- Source: repo 内或工作目录中的本地文档
- 用途：读取已有 brainstorm、plan、需求文档、会议纪要、分析文档、本地 markdown/txt/pdf
- 执行器：`local-doc-reader`
- 输出：本地文档摘要与约束提炼

#### 3. Online Context

- `GitHub URL`
- `Web URL`
- `Docs URL`
- `Issue / PR / Discussion URL`

#### 4. Discovery Fallback

当用户要求补充上下文，但当前环境没有已知 adapter 时：

- 可尝试 `find-skills`
- 或给出缺失提示与安装建议

### B.8 Source Adapter 设计

| source type | 触发信号 | 主要内容 | 推荐执行器 | 统一输出 |
|---|---|---|---|---|
| `local-doc` | 提供本地文件路径，或明确说本地文档 / 会议纪要 / markdown / pdf / `docs/` 下文档 | 本地文档上下文 | `local-doc-reader` | research digest |
| `github-url` | `github.com/...` URL，或明确说 issue / PR / repo discussion | issue、PR、discussion、README、代码说明 | GitHub adapter | research digest |
| `web-url` | 通用 http/https URL | 网页正文、公告、博客、帮助页 | web/doc fetch adapter | research digest |
| `docs-url` | 明确是文档、规范、说明页 | 产品文档、技术文档、接口说明 | docs adapter | research digest |
| `unknown-context` | 只说“查一下上下文”或“找一个工具读一下” | 不确定 | `find-skills` fallback 或请求明确 source | 能力发现结果 / 提示 |

### B.8.1 `research digest` 统一 contract

为避免多个 adapter 各自输出不同结构，所有 adapter 必须统一输出为同一份 digest contract。

最低 schema 定义如下：

```yaml
source_type: local-doc | feishu-chat | feishu-doc | github-url | web-url | docs-url
source_locator: string
fetched_at: ISO-8601 string
relevance:
  level: high | medium | low
  rationale: string
key_findings:
  - string
constraints:
  - string
decisions_or_existing_direction:
  - string
open_questions:
  - string
evidence:
  - label: string
    locator: string
    note: string
```

补充要求：

- `source_type`、`source_locator`、`fetched_at` 必填
- `key_findings` 必须至少有 1 条，除非明确返回“无高价值结果”
- `evidence` 不能为空数组；若 source 无法稳定定位证据，必须显式写明原因
- `decisions_or_existing_direction` 与 `constraints` 应尽量区分，不要混写

### B.8.2 “无结果”输出 contract

如果某个 adapter 没有找到有价值内容，也不能返回空白，而要返回统一的空结果结构：

```yaml
source_type: ...
source_locator: ...
fetched_at: ...
relevance:
  level: low
  rationale: "No useful context found for the requested topic."
key_findings: []
constraints: []
decisions_or_existing_direction: []
open_questions: []
evidence: []
```

同时必须在调用层显式区分：

- “没找到内容”
- “工具不可用”
- “权限不足”
- “source 无法解析”

### B.9 路由策略

增强版建议采用**显式优先、推断次之、fallback 最后**的三层路由。

#### 层 1：显式 source 优先

当用户已经明确给出 source 时，直接路由：

- 本地文件路径 / `docs/` 文档 -> `local-doc`
- GitHub URL -> `github-url`
- 通用网页 URL -> `web-url`
- 在线文档链接 -> `docs-url`

#### 层 2：弱推断路由

如果用户没有显式说 source，但输入中有明确线索，可以做弱推断：

- 路径形如 `/Users/.../*.md`、`docs/**/*.md`、本地 pdf/txt -> `local-doc`
- URL 域名明显属于 GitHub -> `github-url`
- URL 明显是文档站 -> `docs-url`

要求是：

- 线索足够强
- 一旦不强，就不应自作主张

#### 层 3：fallback discovery

如果用户只表达了“想要补充上下文”，但 source 不明确、执行器也不明确，则：

1. 先提示当前支持的补充上下文类型
2. 如果本地没有对应 adapter，再尝试 `find-skills`
3. 如果仍然没有，就明确告知能力缺失，而不是假装已完成上下文研究

### B.9.1 `find-skills fallback` 触发条件

`find-skills` 不是默认路径，只有同时满足以下条件时才触发：

1. 用户明确要求补充上下文
2. 输入没有足够强的 source 线索，无法稳定落到已有 adapter
3. 当前环境里没有显式的本地 adapter 可直接处理

若不满足以上条件，则：

- 优先请求用户明确 source
- 或直接走已知 adapter

> **环境约束**：`find-skills` 是 Claude Code IDE 内置能力，**不在 spec-first 的 `skills/` 目录中**，不随 `spec-first init` 部署到用户项目。在 **Codex 环境**下不可用，遇到此场景直接跳过，执行 B.9.2 降级第 1 步。

### B.9.2 `find-skills fallback` 失败降级

`find-skills` fallback 失败后（或 Codex 环境下直接跳到此步），必须按以下顺序降级：

1. 明确告知当前未发现可用能力
2. 列出当前已知支持的 source 类型
3. 引导用户改为：
   - 提供本地文档路径
   - 提供 GitHub URL
   - 提供网页 / 在线文档链接

不得在 fallback 失败后伪造外部研究结果。

### B.10 为什么 `find-skills` 不应作为主执行机制

#### 问题 1：不稳定

相同问题在不同环境里，`find-skills` 可能找到不同 skill，导致行为不可预测。

#### 问题 2：不可测试

如果核心路径依赖运行时能力发现，就很难写稳定的 unit / smoke contract。

#### 问题 3：不可治理

你无法清晰回答：

- 哪些 source 被支持
- 哪个 source 的输出格式是什么
- 失败时怎么降级

因此增强版的正确位置是：

- `find-skills` = fallback / capability discovery
- `source adapter` = 主执行路径

### B.11 对 workflow 的改造方式

当前最合理的改造点是 `Phase 1.1 Existing Context Scan`。

建议升级成：

#### `1.1 Existing and Supplemental Context Scan`

推荐结构：

1. `Repo context` 扫描
2. `Local docs context` 判断
3. `Supplemental context` 判断
4. 进入具体 adapter

### B.11.1 `Repo Context` 与 `Local Docs Context` 的边界 contract

为避免重复处理，当前方案明确：

- `Repo Context`
  - 负责：`AGENTS.md` / `CLAUDE.md` / 代码模式 / 目录结构 / 仓库内治理约束
  - 不负责：对具体本地文档做逐篇 digest 摘要
- `Local Docs Context`
  - 负责：用户显式提供的本地路径，以及 topic-driven 命中的本地文档
  - 负责输出统一 digest
  - 不负责：代码结构与代码实现扫描

优先级规则：

1. 用户显式给路径时，`Local Docs Context` 优先
2. 未给路径时，先跑 `Repo Context`，再按 topic-driven 命中决定是否补 `Local Docs Context`
3. 同一份本地文档只应由 `Local Docs Context` 摘要一次

其中 `Supplemental context` 下再细分：

- `Local docs context`
- `GitHub / URL context`
- `Fallback discovery`

推荐在 `spec-brainstorm/SKILL.md` 中把补充上下文段落写成类似结构：

- **Local docs context**
- **Online context**
- **Fallback discovery**

### B.12 建议新增的执行器与落点

5. 不同 source 的输出都能收敛成统一 digest

### B.15 风险与注意事项

#### 风险 1：把增强版当成追平版

增强版必须建立在“上游基线事实已被记录清楚”之上。  
否则你很容易得到一个“当前方案与上游事实混在一起”的版本。

#### 风险 2：source router 写得太智能

如果 router 过度依赖推断，很容易：

- 误判 source
- 跳错 adapter
- 难以测试

所以增强版应坚持：

- 显式优先
- 推断保守
- fallback 明确

#### 风险 3：把本地文档降级成附属来源

这会直接错过当前项目中最容易获取、且通常最有价值的上下文材料。

#### 风险 4：过度依赖 `find-skills`

这会让主流程不可预测，最终伤害 `spec-brainstorm` 的 contract 稳定性。

## 最终结论

作为唯一完整方案文档，本文的最终结论是：

### 从上游基线看

- `ce-brainstorm` 的完整事实已经被保留
- `Slack context` 仍然是上游能力的一部分
- 这些事实不能被删改，否则审计口径会失真

### 从当前产品方案看

- `Slack context` 不进入当前产品实现
- 当前实现应转向 `Supplemental Context`
- `Local Docs` 必须成为一等上下文源
- `GitHub`、`Web/Docs URL` 与 `find-skills fallback` 共同构成新的多源补充上下文系统

因此，当前唯一正确的对外表述应是：

**`spec-brainstorm` 的完整方案已经从“上游基线记录”与“当前产品实施方案”两个层面统一收口；实际实施以 Part B 为准，事实基线以 Part A 为准。**
