本文同时承载两层内容：

1. **Part A：上游 `ce-ideate` 的代码基线事实**

### A.3 代码事实：`ce-ideate` 与 `spec-ideate` 的逐类差异

下面的分类全部基于当前文件 diff，而不是根据审计结论倒推。

#### A.3.1 命名空间与宿主适配差异

这部分不是能力差距，而是 `spec-first` 当前运行时架构的必然结果。

| 差异项 | `ce-ideate` | `spec-ideate` | 结论 |
|---|---|---|---|
| source `name` | `name: ce:ideate` | `name: ideate-workflow` | 本地 runtime contract，不能简单回退 |
| handoff 名称 | `ce:ideate` → `ce:brainstorm` → `ce:plan` | `spec:ideate` → `spec:brainstorm` → `spec:plan` | 正常命名迁移 |
| canonical Task agent 名称 | `compound-engineering:research:*` | `spec-first:research:*` | 正常命名迁移 |
| path 示例 | `plugins/compound-engineering/skills/` | `plugins/spec-first/skills/` | 文案例子迁移 |

证据：

- `ce-ideate/SKILL.md` 第 1-15 行定义 `name: ce:ideate` 与 `ce:*` handoff。
- `spec-ideate/SKILL.md` 第 1-15 行定义 `name: ideate-workflow` 与 `spec:*` handoff。
- `spec-first/.claude-plugin/plugin.json` 第 14-18 行把 `/spec:ideate` 命令映射到 `spec-ideate`。
- `spec-first/src/cli/plugin.js` 第 181-219 行把 command-backing skills 安装到 runtime workflows 路径。
- `spec-first/src/cli/adapters/codex.js` 第 80-88 行与 182-188 行会把 source `name` 重写成 runtime 的 `spec-ideate`。

这部分说明：

1. `spec-first` source skill 使用内部名称，是当前 CLI 安装架构的一部分。
2. 如果直接把 `spec-ideate` source 文件改回 `name: spec:ideate`，会违反现有 smoke test 对“内部 workflow name”的保护，不是简单追平。

#### A.3.2 `spec-first` 已经强于上游的 ideate 方法论增强

`spec-ideate` 不是单纯删改上游文案，而是在 ideation orchestration 上显式增强了 contract。

##### 核心原则层

相比 `ce-ideate/SKILL.md` 第 38-42 行的 3 条原则，`spec-ideate/SKILL.md` 第 38-46 行扩展为 7 条：

- 增加 `Diverge before judging`
- 增加 `Use adversarial filtering`
- 增加 `Preserve the original prompt mechanism`
- 增加 `Use agent diversity to improve the candidate pool`
- 增加 `Preserve the artifact before handoff`

这不是文案膨胀，而是把原来隐含在上游流程中的执行约束显式化，降低后续 drift 风险。

同时还有两条需要单独冻结、不能只笼统归入“more frames”的行为差异：

- `ce-ideate/SKILL.md` 第 100 行示例是 `model: "haiku"`；`spec-ideate/SKILL.md` 第 104 行改成 `the platform's cheapest capable model`
- `ce-ideate/SKILL.md` 第 145 行在 issue-tracker 模式下 `Cap at 4 total`；`spec-ideate/SKILL.md` 第 157 行是 `Cap at 6 total frames`

前者代表 `spec-first` 已经把 quick-scan 子代理从 Claude 特定示例收敛为跨宿主表达；后者代表 issue intelligence 激活时，本地允许更宽的 frame 并行度与候选池覆盖面。

##### Phase 2 ideation orchestration 层

`ce-ideate/SKILL.md` 第 134-156 行是较紧凑的 Phase 2 说明；`spec-ideate/SKILL.md` 第 132-200 行把以下能力显式写进 contract：

- 把“先发散再评判”拆成固定的 13 步执行机制
- 明确 4-6 个 ideation frame，而不是固定 3-4 个 sub-agents
- 新增默认 frame：
  - `unmet need or missing capability`
  - `extreme cases, edge cases, or power-user pressure`
- 明确 `cross-cutting synthesis` 是 orchestrator 责任，不只是 dedupe 后的可选补充
- 明确 sub-agent 返回结构需要包含：
  - `title`
  - `summary`
  - `why_it_matters`
  - `evidence or grounding hooks`
  - `optional local signals`

这部分在 `ce-ideate` 中是较松散的说明，在 `spec-ideate` 中已经被写成更稳定的编排 contract。

##### Post-ideation critique 层

`ce-ideate/references/post-ideation-workflow.md` 第 5-27 行要求 orchestrator 直接过滤；`spec-ideate/references/post-ideation-workflow.md` 第 5-40 行进一步引入：

- 两层 critique：
  - skeptical critique sub-agents
  - orchestrator final scoring
- 显式区分：
  - critique agents 可以给局部判断
  - orchestrator 保留最终排序权
- survivor rubric 被拆成逐项列表，便于后续测试与审计

这同样不是“偏离上游”，而是把方法论增强为更易维护的 contract。

#### A.3.3 上游独有、当前未承接的真实能力

这一类差异必须被认真看待，因为它们不是表述差异，而是**真实执行分支缺失**。

##### 差异 1：Slack context 路由被删除

`ce-ideate/SKILL.md` 第 126-130 行存在完整的 `Slack context` opt-in 分支：

- tools available + user asked → dispatch `compound-engineering:research:slack-researcher`
- tools available + user didn’t ask → 提示可用
- no tools + user asked → 返回安装说明

当前 `spec-ideate/SKILL.md` 第 124-130 行之后直接结束 grounding summary，没有任何对应分支。

进一步核对仓库代码可知，这不是“文案忘了写”，而是 `spec-first` 当前确实没有运行依赖：

- 上游存在：
  - `plugins/compound-engineering/skills/ce-slack-research/SKILL.md`
  - `plugins/compound-engineering/agents/research/slack-researcher.md`
- 当前仓库不存在对应 skill 与 agent。

因此：

- 从**严格上游等价**口径看，`spec-ideate` 当前不等价；
- 从**当前产品边界**口径看，这个能力是否补回，需要单独决策，不能混入一般文本同步。

##### 差异 2：缺少显式的 sub-agent dispatch 执行约束

`ce-ideate/SKILL.md` 第 138 行明确写了：

- ideation sub-agents 运行在 inherited model
- 不要 tier down
- omit `mode` parameter，尊重用户权限设置

而 `spec-ideate/SKILL.md` 的 Phase 2 虽然保留了“使用 sub-agents”的思想，但**没有把这三条执行约束写成明确 contract**。

这会带来两个问题：

1. 未来再次同步时，容易误判为“没有这个要求”。
2. 当 skill 被不同宿主或不同执行器消费时，creative ideation 的 model / permission 约束没有统一的 source-of-truth。

注意：这不只是“contract 不够完整”，而是一个 **workflow correctness / portability contract 缺口**：

1. creative ideation 是否继承当前 run 的 model / reasoning level，没有明确 source-of-truth
2. 执行器是否可以擅自 tier down，到当前文本层面没有被禁止
3. `mode` 参数是否应省略以继承宿主权限设置，也没有统一约束

当前代码未必每次都会出错，但如果继续缺少这三条，未来不同宿主或 orchestrator 消费 `spec-ideate` 时，就可能在不违反文本 contract 的情况下发生质量下滑或权限行为漂移。

#### A.3.4 `spec-first` 当前新增但仍需收口的局部问题

`spec-ideate` 目前还有一个应该在本地修正的细节问题。

##### 问题：数量 contract 在上游与本地都不够严格自洽

`spec-ideate/SKILL.md` 第 136-137 行写：

- 每个 sub-agent 约 7-8 ideas
- 4-6 agents 可得到 30-40 raw ideas

但按字面计算：

- 4 × 7 = 28
- 6 × 8 = 48

因此“4-6 agents + 7-8 ideas → 30-40 raw ideas”不是严格自洽的数量 contract。

同时，上游 `ce-ideate/SKILL.md` 第 138 行的表述也不是严格数学自洽：

- 3 × 8 = 24
- 4 × 10 = 40

但上游写的是 `~30 raw ideas`。

这类问题虽然不影响主流程设计，但会造成三个负面结果：

1. 执行者难以判断当前 run 是否偏离预期。
2. 后续 contract test 无法写得严格。
3. 与上游同步时，容易继续出现“吸收语义但数字漂移”的情况。

因此，这里的修正目标不是“回退到上游字面数字”，而是把 `spec-first` 本地 contract 收紧为一个严格自洽、可以测试的版本。

#### A.3.5 邻接依赖的本地分叉

`spec-ideate` 依赖 `learnings-researcher`，而该 agent 在 `spec-first` 中已经继续演化。

对比：

- 上游 `learnings-researcher.md` 第 64-72 行：总是读取 `critical-patterns.md`
- 当前 `learnings-researcher.md` 第 64-68 行：仅当文件存在时读取，缺失不报错

同时，`spec-first` 版本新增了：

- `applies_when` 字段支持
- “No critical patterns file in this repo” 的显式输出
- `spec-compound` 的 schema reference 路径更新

这说明：`spec-ideate` 的 grounding 阶段并不是简单调用上游同一 agent，而是运行在本地更宽容的 institutional-knowledge contract 上。

结论是：

- 这属于**本地增强**，不是 ideate 落后；
- 但如果未来要做严格同步，必须把“ideate skill 差异”和“research agent 差异”分开处理。

### A.4 现状判断

基于代码事实，当前结论应拆成两层：

#### 结论 1：按上游严格等价口径，`spec-ideate` 仍未完全追平

原因只有两类：

1. 缺少 `Slack context` 路由及其运行依赖
2. 缺少上游明确写出的 ideation sub-agent 执行约束

#### 结论 2：按当前 `spec-first` 产品边界与核心 ideate 能力口径，`spec-ideate` 已经达到“能力同构且局部更强”

原因是：

1. ideation 主链路仍然完整保留：
   - grounding
   - divergent ideation
   - adversarial filtering
   - survivor presentation
   - durable artifact
   - brainstorm handoff
2. `Phase 2` 与 `post-ideation` reference 已经显式增强为更强的 orchestration contract
3. runtime 安装、命令入口、host transform、打包与 smoke tests 都已接入 `spec-ideate`

证据：

- `spec-first/templates/claude/commands/spec/ideate.md` 第 8-16 行定义了 `/spec:ideate` 的 command wrapper。
- `spec-first/.claude-plugin/plugin.json` 第 14-18 行将 `ideate` 命令绑定到 `spec-ideate`。
- `spec-first/tests/smoke/cli.sh` 第 98-113 行、第 157-215 行、第 700-730 行验证了命令生成、内部命名、runtime 安装与打包产物。
- `spec-first/src/cli/adapters/claude.js` 第 131-143 行与 `src/cli/adapters/codex.js` 第 150-188 行负责运行时 agent/path/name 改写。

## Part B：当前产品边界下的正式实施方案

### B.1 目标定义

当前方案的目标不是“把 `spec-ideate` 文本回退成 `ce-ideate`”，而是：

1. 让 `spec-ideate` 在**非 Slack 核心链路**上与上游保持能力一致
2. 保留并固化 `spec-first` 当前更强的 orchestration 增强
3. 将现有增强收敛成**可测试、可验证、可继续同步**的 contract
4. 为未来外部上下文能力演进预留扩展点，但不把外围平台能力强行拉回当前范围

### B.2 非目标

本方案明确不做以下事情：

1. 不把 source `name` 从 `ideate-workflow` 改回 `spec:ideate`
2. 不为了文字追平而移除 `spec-first` 当前已经更强的 Phase 2 / critique contract
3. 不在本阶段引入 `Slack context`、`spec-slack-research` 或 `slack-researcher`
4. 不把 ideate 的差异与 research agents、CLI adapters、插件安装架构混成一个改动

### B.3 当前实施冻结口径

为避免开发时再次出现“到底按 strict parity 还是按当前产品边界”的争论，先冻结以下口径：

1. **Part A 只保存上游事实，不直接进入当前验收范围。**
2. **Part B 是当前唯一实现范围。**
3. **Part A 中与 ideate 方法论质量直接相关、且不依赖 Slack 的能力，必须在 Part B 中吸收。**
4. **`Slack context` 与 `slack-researcher` 明确排除，不得以“严格追平”为理由回流进当前开发范围。**

### B.4 分阶段技术方案

## Phase 0：先固化“什么叫对齐”

#### 目标

先把 `spec-ideate` 的对齐目标从“文本接近”收敛为“能力 contract 对齐”。

#### 动作

1. 新增本方案文档，作为 `spec-ideate` 的唯一完整方案文档。
2. 以后所有关于 `ce-ideate` / `spec-ideate` 的同步，统一按三类判定：
   - 命名空间/宿主适配
   - 核心 ideation contract
   - 外围能力依赖
3. 后续审计时，必须逐文件核对以下真实落点：
   - `skills/spec-ideate/SKILL.md`
   - `skills/spec-ideate/references/post-ideation-workflow.md`
   - `agents/research/issue-intelligence-analyst.md`
   - `agents/research/learnings-researcher.md`
   - `templates/claude/commands/spec/ideate.md`
   - `src/cli/adapters/claude.js`
   - `src/cli/adapters/codex.js`
   - `tests/smoke/cli.sh`
   - `tests/unit/spec-ideate-contracts.test.js`（本方案新增）

#### 验收标准

- 后续任何“是否追平”的讨论，必须能够落到具体文件和具体 contract，而不是凭印象判断。

## Phase 1：补齐非 Slack 的上游核心 contract

#### 目标

让 `spec-ideate` 在不引入 Slack 的前提下，把上游仍然有价值的执行约束补齐。

#### 计划修改文件

- 修改：`skills/spec-ideate/SKILL.md`
- 修改：`skills/spec-ideate/references/post-ideation-workflow.md`

#### 修改内容

##### 1. 回补 ideation sub-agent 的执行约束

在 `skills/spec-ideate/SKILL.md` 的 Phase 2 中显式加入以下 contract：

- creative ideation sub-agents 默认继承当前 run 的 model / reasoning level
- 不为了省 token 盲目把 ideation sub-agents tier down 到更弱模型
- dispatch 时 omit `mode` parameter，尊重用户或宿主权限设置

理由：

- 这是 `ce-ideate` 当前真实存在的执行约束
- 它不依赖 Slack，不超出当前产品边界
- 它属于 workflow correctness，不是外围增强

这是本阶段的第一优先级修改项。

##### 2. 收紧数量 contract

把 `4-6 agents × 7-8 ideas ≈ 30-40 raw ideas` 的表述改成严格自洽的版本，建议二选一：

方案 A：

- 保持 `4-6 agents`
- 把 raw count 改为 `~28-48`

方案 B：

- 保持 `~30-40 raw ideas`
- 收敛 agent 数量或 per-agent target，使数字严格匹配

建议采用 **方案 A**，原因是：

- 对当前 `spec-first` 的增强结构侵入最小
- 不需要回退现有 4-6 frame 的设计
- 只修正文案 contract，不改变 orchestration 思路

同时需要在方案文档中显式说明：

- 上游 `ce-ideate` 的 `3-4 × 8-10 ≈ ~30` 也不是严格数学自洽
- 本轮目标是把 `spec-first` 的数量 contract 收紧成可测试版本，而不是追随上游的松散写法

##### 3. 显式冻结“当前不引入 Slack context”

不建议在 user-facing skill 里大段写产品史，但建议在适合的位置补一句冻结说明，含义如下：

- 当前 ideate grounding 只依赖 repo 内置 research agents
- 如果未来引入组织级外部上下文，应通过新的 repo-owned context source 进入，而不是隐式回补 Slack-only 分支

这句的目标不是告诉最终用户“为什么没有 Slack”，而是防止维护者在未来同步时再次误判。

##### 4. 保留并明确 `spec-first` 的增强 contract

以下增强不应被回退，而应保留并在文案上进一步收紧：

- 更多 ideation frames
- issue-tracker 模式下 `Cap at 6 total frames`
- cross-cutting synthesis
- critique sub-agents + orchestrator scoring 的双层过滤
- post-ideation split 以降低前置 token 成本

同时要补一条说明：

- quick context scan 使用 `the platform's cheapest capable model` 是本地已完成的跨宿主改进，应保留，不回退到 Claude 特定示例

##### 5. 明确 `post-ideation-workflow.md` 的具体修改边界

既然 `skills/spec-ideate/references/post-ideation-workflow.md` 被列为 P0-P1 必改文件，本方案必须把它的修改目标写清。

本轮对该 reference 的修改目标不是回退到上游 `orchestrator direct filter`，而是：

- 保留 `Prefer a two-layer critique`
- 保留 `final scoring authority belongs to the orchestrator`
- 在方案口径中明确：这里的“增强”是指 **quality gate 更显式、contract 更稳定**，不是未经验证就断言“效果一定优于上游”
- 如需补充说明，增加一句约束：两层 critique 的目的是提供不同视角的攻击性过滤，但最终排序与 survivor selection 仍由 orchestrator 统一裁决，避免多代理局部判断导致全局排序漂移

这样，`B.4 Phase 1` 与 `B.6` 的文件清单才是对齐的，执行者不会再遇到“文件列在必改里，但不知道改什么”的歧义。

#### 验收标准

- `spec-ideate` 仍然保留当前增强结构
- 同时补回上游非 Slack 的关键执行约束
- `SKILL.md` 与 post reference 不再存在数量 contract 模糊点

## Phase 3：为后续演进拆出更稳定的 ideation contract
