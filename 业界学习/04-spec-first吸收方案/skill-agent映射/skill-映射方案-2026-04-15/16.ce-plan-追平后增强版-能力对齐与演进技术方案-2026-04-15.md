### A.2 代码事实：差异不是单纯 rename

从目录级 diff 看，`ce-plan -> spec-plan` 的真实差异包含三类：

1. **命名空间与 runtime contract 迁移**
2. **`spec-first` 新增的 Stage-0 上下文预载与 planning 范围收口**
3. **上游存在但本地未承接的非软件 planning / visual communication / Slack context 分支**
4. **上游新增的 Interactive deepening 模式未同步**
5. **上游已修复的 gate bug 未同步**

### A.3 关键代码差异

#### A.3.1 `spec-plan` 增加了 Stage-0 上下文预载

`spec-plan/SKILL.md` 在开头新增了完整的 Stage-0 预载流程：

- 基于 control plane contract 读取 `.spec-first/workflows/bootstrap/<slug>/`
- 优先消费 `context-routing.json`、`artifact-manifest.json`、`minimal-context/plan.json`
- 以 `selected_assets / fallback_reason / level / skipped_rules` 为 evaluator 真源
- 允许 Level 1 / Level 2 / Level 3 降级

这部分在 `ce-plan/SKILL.md` 中完全不存在。

需要特别说明：

- `injection-index.yaml` 仍保留在 `docs/contexts/<slug>/` 下，但当前代码已明确它只是**人类视图**，不是运行时唯一判定逻辑
- 因此 `spec-plan` 的 Stage-0 不应再被表述为“按 yaml 路由”，而应表述为“按 evaluator/control-plane contract 取上下文”

这说明 `spec-plan` 已与 `spec-graph-bootstrap` 的 evaluator 输出形成真实联动，不是简单文案增强。

#### A.3.2 `spec-plan` 明确删除了非软件 planning 分流

`ce-plan/SKILL.md` 有 `0.1b Classify Task Domain`，用于把非软件任务路由到 `references/universal-planning.md`。

`spec-plan/SKILL.md` 当前没有这一段，且目录中也不存在 `references/universal-planning.md`。

这意味着：

- 按上游严格口径看，`spec-plan` 当前缺少"多领域 planning"能力。
- 但按当前 `spec-first` 的整体产品链路看，这个收口已经不再稳固：`spec-brainstorm` 现已支持非软件 brainstorming，因此 `spec-plan` 若继续不承接非软件 planning，会把 `WHAT -> HOW` 链路在 plan 阶段截断。

因此，这一项不应继续作为长期产品边界，而应视为**待恢复的上游能力缺口**。

#### A.3.3 `spec-plan` 删除了 Slack context 分支

`ce-plan/SKILL.md` 在 Phase 1.1 有 opt-in Slack context 分支。

`spec-plan/SKILL.md` 当前已删除该段。

结合仓库事实：

- source 有 `ce-slack-research` / `slack-researcher`
- target 没有等价 skill / agent

因此这不是文案遗漏，而是外围能力未引入。

#### A.3.4 `spec-plan` 收紧了 spec-doc-review handoff contract

`spec-plan/references/plan-handoff.md` 与上游相比，关键差异是：

- 明确 **不请求 `mode:headless`**
- 如果调用方无法承接交互 review，返回固定说明：
  - `Interactive spec-doc-review still required before execution handoff.`

这与仓库现有治理是一致的：

- 当前 `spec-doc-review` 明确保留 interactive 路线
- 不回退到上游 headless 契约

这是本地有意分叉，不是缺失。

#### A.3.5 `spec-plan` 增加了 external-delegate posture，但 issue creation 并非本地新增

`spec-plan/SKILL.md` 新增：

- `Execution target: external-delegate`

同时，`Create Issue` 路径与 tracker 探测逻辑也存在于 `spec-plan/references/plan-handoff.md` 中。

但这里需要区分两类事实：

1. **`Execution target: external-delegate` 是本地新增能力**
   - 上游 `ce-plan` 的执行姿态检测只覆盖 test-first / characterization-first
   - `spec-plan` 新增了对 external delegation / token conservation 场景的 execution posture 承接

2. **`Create Issue` 不是本地新增能力**
   - 上游 `ce-plan/references/plan-handoff.md` 中已存在同样的 Issue Creation section
   - 因此它属于沿袭上游已有能力，而不是 `spec-plan` 相对 `ce-plan` 的新增增强

说明 `spec-plan` 相对上游真正新增的是面向 `spec-work-beta` / 外部委派场景的 execution posture，不是 issue creation 本身。

**注意**：`spec-plan/SKILL.md` 末尾（Line 748-770）有一段重复的 Issue Creation 章节，与 `plan-handoff.md` L67-88 内容完全一致。这是维护风险：`ce-plan/SKILL.md` 末尾只有 `NEVER CODE!`，Issue Creation 仅在 `plan-handoff.md` 维护。

因此"visual communication"的缺失是联动的：Section 4.4 + 5.1 检查 + reference 文件三处同时缺失。

#### A.3.7 5.3.2 Gate 逻辑 Bug：confidence check 通过时跳过了 spec-doc-review

#### A.3.8 Interactive Deepening 模式整体缺失，deepening-workflow.md 有 dead code

`ce-plan/SKILL.md` Line 684-690 新增了 Two deepening modes 定义：

```
- Auto mode（默认，计划生成时）
  → 无需用户审核，AI 直接整合 agent 发现
- Interactive mode（用户主动 re-deepen 已有计划时激活）
  → 5.3.6b 逐个向用户展示每个 agent 的发现，用户可 Accept / Reject / Discuss
```

`ce-plan/SKILL.md` Phase 0.1 短路逻辑（Line 71-72）指定了激活路径：

```
short-circuit to Phase 5.3 (Confidence Check and Deepening) in interactive mode.
```

**spec-plan 的现状：**

- SKILL.md Phase 5.3 描述："This phase runs without asking the user for approval." — 永远是 auto mode
- SKILL.md Phase 0.1 写："short-circuit directly to Phase 5.3" — 缺少 "in interactive mode"
- `references/deepening-workflow.md` 第 188 行有完整的 `5.3.6b Interactive Finding Review` 逻辑

**后果**：deepening-workflow.md 的 5.3.6b 是 dead code — SKILL.md 从不激活它。  
用户用 re-deepen 快速路径时会静默走 auto mode，agent 发现被直接整合进计划，用户无法审核。

#### A.3.9 source-of-truth 中误植了 runtime naming 指令，导致 Task naming contract 自相矛盾

这是当前 `spec-plan` 源文件自身的一处 contract 漂移，文档必须单列，不应混入 Stage-0 或 Interactive deepening 差异中。

当前 `skills/spec-plan/SKILL.md` 在 Phase 1.1 写着：

```md
Use bare agent names inside Task calls.
```

但其后紧跟的实际调用却是 canonical source 形式：

```md
- Task spec-first:research:repo-research-analyst(...)
- Task spec-first:research:learnings-researcher(...)
```

同时，`skills/spec-plan/references/deepening-workflow.md` 第 94 行仍明确要求：

```md
Use fully-qualified agent names inside Task calls.
```

且该文件中的 Task 调用也确实全部使用 fully-qualified canonical 名称。

这说明：

1. `spec-plan` 的 source-of-truth 本应继续使用 canonical `spec-first:category:name` 形式
2. Claude runtime 的 bare-agent 形式属于 adapter 产物，不应反向写入 source 文件指令
3. `Use bare agent names inside Task calls.` 这行是错误移植进 `spec-plan/SKILL.md` 的 runtime 口径，不是源文件真实 contract

对照上游也能确认这一点：

- `ce-plan/SKILL.md` 对应位置没有 `Use bare agent names...` 这行
- `ce-plan/references/deepening-workflow.md` 同样要求 `Use fully-qualified agent names inside Task calls.`

**结论**：`spec-plan` 这里的问题不是 agent 名称没迁移，而是 source 文件混入了 runtime naming instruction，造成同文件内自相矛盾。最干净的修复是删除 `skills/spec-plan/SKILL.md` 中这行错误指令，保留 canonical Task 示例不动；runtime bare-name 继续由 adapter 与 smoke test 保障。

**范围边界**：仓库里 `skills/spec-brainstorm/SKILL.md` 也存在同类问题，但那属于 `ce-brainstorm -> spec-brainstorm` 文档范围，不并入本文的 `ce-plan` 实施改动清单，只在此作为仓库级共性现象备注。

### A.4 现状判断

#### 结论 1：按上游严格等价口径，`spec-plan` 未完全追平

原因包括：

1. 缺少非软件 task routing（应恢复，不宜继续排除）
2. 缺少 Slack context 分支（外围能力未引入）
3. 缺少 visual communication 支持（Section 4.4 + 5.1 检查 + reference 文件）
4. 缺少 Interactive deepening 模式
5. 5.3.2 gate bug 未修复

#### 结论 2：按当前 `spec-first` 产品边界，`spec-plan` 已经演化为更聚焦的软件技术规划器

增强点包括：

1. Stage-0 预载
2. external-delegate posture
3. interactive `spec-doc-review` handoff 冻结

#### 结论 3：除能力差异外，`spec-plan` 还有一处 source-of-truth contract 漂移需要顺手修正

即 Phase 1.1 中误植的 `Use bare agent names inside Task calls.`。这不是能力边界问题，而是源文件与 runtime 口径混线后的文案性 contract bug；若不修，后续维护者会误以为 source 文件也应该改成 bare-agent 形式。

## Part B：当前产品边界下的正式实施方案

### B.1 目标

当前方案目标不是把 `spec-plan` 回退成 `ce-plan`，而是：

1. 保持 `spec-plan` 作为**软件技术规划**入口的聚焦定位
2. 恢复 `ce-plan` 中已经存在、且已被 `spec-brainstorm` 上游链路证明需要的非软件 planning 路由
3. 修复上游已发现的 workflow bug
4. 补齐上游中仍然有价值、且不超出当前产品边界的 planning quality contract
5. 修复 source-of-truth 与 runtime contract 混线造成的文案性漂移
6. 把本地增强固化为可测试 contract

### B.2 非目标

本方案明确不做：

1. 不补 `Slack context`
2. 不回退 interactive `spec-doc-review` 契约
3. 不让非软件 plan 进入 `spec:work` 的软件执行链路

### B.3 冻结口径

1. `spec-plan` 恢复非软件 planning 路由，但软件 planning 仍是主路径，继续优先服务代码与技术规划场景。
2. 非软件 planning 采用独立 `universal-planning.md` 路由，不进入软件专用的 confidence check、`spec:work` 执行链路与 issue handoff 菜单。
3. `spec-doc-review` 保持 interactive contract。
4. `Slack context` 继续排除。
5. `Stage-0` 预载与 external-delegate posture 作为本地正式能力保留；其中 Stage-0 仅作用于软件 planning 主路径。
