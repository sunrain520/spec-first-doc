## 3. 需要修正的原报告口径

### 3.1 “source-of-truth 开发基本完成”原表述不够严谨

原报告把以下两件事混在了一起：

1. 主 SKILL contract 是否恢复
2. source-of-truth 全套交付是否闭环

更准确的拆分应为：

- `主 SKILL contract`：基本完成
- `配套 reviewers / tests 文件内容`：已写出
- `source-of-truth 是否形成完整可交付变更集`：当前仍不能下“已完成交付”结论，因为关键文件还处于 untracked 状态

因此，不能简单写成“source-of-truth 开发基本完成”，而应写成：

- `主 SKILL contract 已恢复`
- `配套 source-of-truth 资产已落盘，但尚未全部纳入正式 tracked 变更集`
- `因此尚不能视为完整交付`

### 3.2 `plugin.js` 死函数删除不应与主能力恢复并列

`src/cli/plugin.js` 的死函数删除是关联清理项，不是 `spec-code-review` 主能力 contract 的核心组成。

它可以记录，但不应与下面这些主线恢复项并列：

- reviewer catalog 恢复
- reviewer agents 补回
- tests 补回
- interactive / autofix / merge heuristics 恢复

因此更合理的归类应是：

- 主线能力恢复项
- 配套验证与运行时链路项
- 顺手清理项

### 3.3 状态字段必须改成可判定枚举

原报告里的“`源码主线开发完成：是，或接近完成`”不是可执行状态，不能回填映射表。

应改成以下固定状态：

- `方案分析完成`
- `主 SKILL contract 完成`
- `配套 reviewer agents 完成`
- `contract tests 完成`
- `source-of-truth 已纳入 tracked 变更集`
- `runtime 刷新验证完成`
- `完整可交付`

每一项都只允许 `是 / 否`。
