## Part A：上游基线与代码事实

### A.1 差异分层总览

`ce-review -> spec-code-review` 不是简单 rename，而是四类差异叠加：

1. **本地正式增强**
   - Stage-0 上下文预载
   - `.spec-first/workflows/spec-code-review/` artifact 路径
   - Stage-0 telemetry 与 evaluator contract 消费
2. **能力缺口**
   - 缺少 `cli-readiness-reviewer`
   - 缺少 `previous-comments-reviewer`
   - 缺少上游若干 merge/synthesis contract
3. **行为漂移**
   - interactive 模式不再默认自动应用 `safe_auto`
   - untracked 文件处理策略更保守
   - 交互策略与上游不同
4. **显式收口**
   - 删除 `mode:headless`
   - 删除 headless output/template/schema 依赖

这四类必须分开处理，不能混成一个“是否恢复 headless”的问题。

### A.2 已成立的本地正式增强

#### A.2.1 Stage-0 上下文预载是正式能力，不是临时分叉

`spec-code-review/SKILL.md` 开头新增了 review 阶段 Stage-0 预载，统一消费：

- `context-routing.json`
- `artifact-manifest.json`
- `selected_assets / fallback_reason / level / skipped_rules`
- Stage-0 telemetry

这不是文档描述而已，已有测试保护：

- `tests/unit/workflow-stage0-consumption.test.js`
- `tests/unit/workflow-telemetry.test.js`

因此：

- **不能回退**
- **应视为 `spec-first` review 体系的正式增强**

#### A.2.2 artifact 路径迁移是本地 runtime contract

上游使用：

- `.context/compound-engineering/ce-review/<run-id>/`

当前使用：

- `.spec-first/workflows/spec-code-review/<run-id>/`

并已有 smoke guard：

- `tests/smoke/cli.sh` 会校验 source-of-truth 与 runtime 不再包含旧 `.context/spec-first/...` 路径

因此：

- **不能机械回退**
- 所有后续对齐都应建立在 `.spec-first/workflows/spec-code-review/` 之上

### A.3 真实能力缺口

#### A.3.1 reviewer catalog 少了 2 个上游条件 persona

对比 `references/persona-catalog.md`：

- 上游：17 reviewer personas
- 当前：15 reviewer personas

缺失：

- `cli-readiness-reviewer`
- `previous-comments-reviewer`

这不是命名差异，而是实打实的 selector 链路缺口。

当前仓库只有：

- `agents/review/cli-agent-readiness-reviewer.md`

它是 standalone deep-audit agent，不是上游接入 `ce-review` 的 conditional persona，不能替代：

- `agents/review/cli-readiness-reviewer.md`

同时，当前仓库完全缺少：

- `agents/review/previous-comments-reviewer.md`

结论：

- `spec-code-review` 当前对 CLI agent-readiness 的自动覆盖弱于上游
- `spec-code-review` 当前对 PR 历史 review thread 的复核能力弱于上游

#### A.3.2 缺失 `previous-comments` 的同时，也缺失其 PR-only selector contract

上游不仅有 agent 文件，还在 `ce-review/SKILL.md` 和 `persona-catalog.md` 中明确：

- `previous-comments` 只在 PR metadata 存在时激活
- standalone branch review 不应激活

当前 `spec-code-review` 不仅没有 persona，也没有这条选择规则。

这意味着即使后续补 agent 文件，如果不同步 selector 规则，也仍然不算追平。

### A.4 真实行为漂移

#### A.4.1 interactive 模式 contract 发生了行为变化

上游 interactive 语义：

1. 自动应用 `safe_auto -> review-fixer`
2. 仅对 `gated_auto` / `manual` 剩余项提问

当前 `spec-code-review` 语义：

1. 先根据 actionable findings 询问总策略
2. 不再明确承诺 interactive 默认自动应用 `safe_auto`

这不是“表述更清晰”，而是 **产品行为变化**。

如果目标是追平 `ce-review` 的主流程体验，这一项必须明确决策：

1. 恢复上游 interactive safe-auto 默认语义
2. 或明确冻结为 `spec-first` deliberate divergence，并说明原因

不能保持模糊。

#### A.4.2 untracked 文件处理策略比上游更保守

上游：

- `mode:autofix`
- `mode:headless`

在发现 untracked 文件时，不会阻断；会继续基于 tracked diff 审查，并在 Coverage 中记录排除项。

当前 `spec-code-review`：

- 统一要求用户先 `git add` 再继续

这对 interactive 模式未必是错的，但对 `mode:autofix` 来说，是上游 contract 的收缩。

因此需要显式判断：

1. `autofix` 是否恢复为“tracked-only continue”
2. interactive / report-only 是否继续保守

推荐不要一刀切。

#### A.4.3 Stage 5 merge heuristics 被简化

上游 Stage 5 有两条当前缺失的合成规则：

1. **cross-reviewer agreement boost**
   - 多 reviewer 指向同一问题时，提升 merged confidence
2. **disagreement annotation**
   - reviewer 在 severity / route 上冲突时，保留透明度

当前 `spec-code-review` 已删去这两条。

这不是小差异，而是 review synthesis 质量 contract 的缩减。

如果要追求“审查质量对齐”，这一项优先级高于 `headless`。

#### A.4.4 model tiering 约束被删除

上游明确规定：

- persona / CE sub-agents 使用 fast mid-tier model
- orchestrator 留在 default model

当前 `spec-code-review` 已删掉 model tiering 说明，只保留“不传 mode:auto”的 permission 约束。

这会带来两类后果：

1. 成本与延迟行为不再受 contract 约束
2. 不同平台迁移时，review 批量 sub-agent 的运行层级可能失控

这项不一定必须原样恢复，但必须在方案中显式决策。

### A.5 显式收口：`headless` 已被完整删除

当前 `spec-code-review` 对 `headless` 的删除是全链路的，不是只删了一行 mode：

1. `SKILL.md` 的 mode 表删掉 `mode:headless`
2. headless rules 全段删除
3. report-only / checkout switch / no diff scope / completion signal 等 headless 规则全部删除
4. `review-output-template.md` 删除 headless format 说明
5. `findings-schema.json` 删除“headless output depends on detail tier”说明
6. `subagent-template.md` 中 full artifact 的 downstream consumer 不再包含 headless output
7. `spec-plan/references/plan-handoff.md` 已明确要求 pipeline mode 不要请求 `mode:headless`

结论：

- 当前 `spec-first` 体系已经有意识地放弃 `spec-code-review headless`
- 这不是遗漏，而是产品边界选择

### A.6 当前测试保护的真实状态

当前仓库对 `spec-code-review` 已有的保护主要是：

1. Stage-0 contract 测试
2. telemetry 测试
3. `.spec-first/workflows/` 路径 smoke guard
4. runtime asset 安装与 managed-state 相关测试

但当前 **缺少专门的 `tests/unit/spec-review-contracts.test.js`**。

因此下面这些关键 contract 现在都没有独立测试锁住：

1. reviewer catalog 数量与关键 persona
2. `cli-readiness` / `previous-comments` selector 规则
3. interactive / autofix 的行为口径
4. merge heuristics
5. `headless` 删除边界
6. source -> Claude/Codex runtime 转换后的 review skill 内容契约

### A.7 现状判断

#### 结论 1：按上游核心能力口径，`spec-code-review` 还没有追平

主要差异不是一个，而是三组：

1. reviewer 能力缺口
2. orchestration 行为漂移
3. synthesis 质量 contract 收缩

#### 结论 2：按当前 `spec-first` 产品边界，不应回退的本地增强已经很明确

应保留：

1. Stage-0 预载
2. `.spec-first/workflows/spec-code-review/` artifact contract
3. telemetry contract

#### 结论 3：`headless` 可以继续作为非目标，但不能再占据主线优先级

当前最应该优先修的，不是 `headless`，而是：

1. reviewer 缺口
2. interactive / autofix 行为 contract
3. merge heuristics
4. spec-code-review 专项 contract tests

## Part B：正式实施方案

### B.1 目标

本轮目标分三层：

1. **追平上游非 headless 核心 review 能力**
2. **保留 `spec-first` 已成立的本地正式增强**
3. **用自动化测试锁住 review contract，防止继续漂移**

### B.2 非目标

本轮明确不做：

1. 不回退 Stage-0 预载
2. 不回退 `.spec-first/workflows/spec-code-review/` artifact 路径
3. 不为了“人数追平”而盲目恢复所有上游文本
4. 不把 `headless` 作为本轮主线目标

### B.3 冻结口径

以下边界本轮冻结：

1. Stage-0 预载属于正式能力，继续保留
2. artifact 继续写入 `.spec-first/workflows/spec-code-review/`
3. telemetry 继续作为 Stage-0 消费的正式输出
4. `headless` 暂不恢复，但保留后续扩展空间

### B.4 恢复口径

以下属于本轮应优先追平的上游 contract：

1. `cli-readiness-reviewer`
2. `previous-comments-reviewer`
3. `previous-comments` 的 PR-only selector 规则
4. interactive safe-auto 默认语义
5. `mode:autofix` 的 tracked-only continue 语义
6. Stage 5 的 cross-reviewer agreement / disagreement transparency

### B.5 待决策项

以下项要在开发前显式定口径：

1. **model tiering**
   - 选项 A：恢复上游 mid-tier sub-agent contract
   - 选项 B：继续不写死具体 model，但补“cheap capable / mid-tier equivalent”跨平台约束
   - 不建议继续空白
2. **interactive 交互措辞**
   - 可保持 `spec-first` 当前更简洁的话术
   - 但行为上应先恢复“safe_auto 默认自动应用”

推荐：

- **行为追平**
- **文案可本地化**

### B.6 分阶段开发方案

#### Phase 0：先修正文档与实施口径

在进入代码开发前，先把本方案文档本身修正到以下状态：

1. `headless` 降为非目标，不再作为主线差异中心
2. 明确 interactive / autofix / merge heuristics 才是本轮主线
3. 明确 `cli-agent-readiness-reviewer` 不是 `cli-readiness-reviewer` 的替代品

这一步完成后，方案才能进入开发。

#### Phase 1：补齐 reviewer 能力缺口

目标：先把缺失的 persona 与 selector 链路补回。

必须修改：

1. 新增 `agents/review/cli-readiness-reviewer.md`
2. 新增 `agents/review/previous-comments-reviewer.md`
3. 修改 `skills/spec-code-review/references/persona-catalog.md`
4. 修改 `skills/spec-code-review/SKILL.md`

具体要求：

1. `cli-readiness-reviewer` 按上游 conditional persona 形态接入 review pipeline
2. `previous-comments-reviewer` 按上游 PR-only persona 形态接入
3. `spec-code-review/SKILL.md` 的 reviewer 列表、selector 说明、team announce 示例同步更新
4. `previous-comments` 的 PR-only 规则必须落在主 skill 与 persona-catalog 两处

#### Phase 2：恢复非 headless orchestration parity

目标：把主流程行为追平到上游可接受水平，但保留本地 Stage-0 / artifact contract。

建议优先顺序：

1. interactive 模式恢复：
   - 默认自动应用 `safe_auto -> review-fixer`
   - 仅对 `gated_auto` / `manual` 提问
2. `mode:autofix` 恢复：
   - 遇到 untracked 文件时继续 tracked scope
   - Coverage 中记录 excluded untracked files
3. 恢复 Stage 5：
   - cross-reviewer agreement boost
   - disagreement annotation / conservative merge transparency
4. 明确 model tiering：
   - 至少恢复“sub-agent 用较低 tier，orchestrator 用 default”这层 contract

#### Phase 3：收口 template / schema / reference

目标：把 Phase 1-2 的变更同步到 supporting references，防止 skill 与 reference 不一致。

必须核对并按需要修改：

1. `skills/spec-code-review/references/subagent-template.md`
2. `skills/spec-code-review/references/review-output-template.md`
3. `skills/spec-code-review/references/findings-schema.json`

本阶段重点不是恢复 headless，而是：

1. 确保 compact return / artifact full schema 的说明与主 skill 一致
2. 确保 reviewer merge tier / detail tier 说明与当前实现一致
3. 确保 route / confidence / reviewer attribution 说明与 Stage 5 一致

#### Phase 4：补 contract tests

新增：

- `tests/unit/spec-review-contracts.test.js`

至少覆盖以下断言：

1. source `skills/spec-code-review/SKILL.md` 保留 Stage-0 contract 关键字符串
2. source `skills/spec-code-review/SKILL.md` 不包含 `mode:headless`
3. persona catalog 包含：
   - `cli-readiness`
   - `previous-comments`
4. `previous-comments` 明确带 PR-only 说明
5. `spec-code-review/SKILL.md` 含：
   - interactive 自动 safe-auto 语义
   - autofix tracked-only continue 语义
   - cross-reviewer agreement / disagreement handling 关键字符串
6. `review-output-template.md` 不再包含 headless 段
7. `findings-schema.json` 不再包含 headless detail-tier 依赖
8. Claude runtime 转换后：
   - canonical agent names 被改写为 bare names
   - source contract 未被错误破坏
9. Codex runtime 转换后：
   - review skill name / shared path / agent path 改写正确

#### Phase 5：可选增强，不阻断主线

本 phase 只在前四个 phase 完成后再评估：

1. 是否恢复 `mode:headless`
2. 是否为 `spec-work` / 其他 workflow 增加 programmatic review handoff contract
3. 是否为 `spec-code-review` 增加更细粒度 benchmark / benchmark smoke

当前不应前置。

### B.9 开发入口建议

进入开发时，建议按以下顺序执行：

1. 先补 `cli-readiness-reviewer` 与 `previous-comments-reviewer`
2. 再修改 `skills/spec-code-review/SKILL.md` 主流程 contract
3. 再同步 `persona-catalog.md` / `subagent-template.md` / `review-output-template.md`

不建议先写测试再反推主流程，因为当前需要先定 interactive / autofix / merge heuristics 的恢复口径。

### B.10 最终建议

`spec-code-review` 当前最合理的推进方式不是“先讨论 headless”，而是：

1. **先补 reviewer 能力缺口**
2. **再恢复非 headless 核心 orchestration contract**
3. **保留 Stage-0 / artifact / telemetry 这些已成立的本地正式增强**
4. **最后用 spec-code-review 专项 contract tests 锁住**

换句话说：

- 本轮要追求的是 **review 质量与主流程行为对齐**
- 不是把 `ce-review` 的所有文本一字不差搬回来
- 也不是继续让 `spec-code-review` 在关键行为上无测试漂移
