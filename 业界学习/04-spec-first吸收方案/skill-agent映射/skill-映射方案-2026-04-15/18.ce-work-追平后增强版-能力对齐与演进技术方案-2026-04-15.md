# spec-work 完整方案文档：`ce-work` 基线对齐与可演进增强（2026-04-15）

## 文档定位

本文基于 `ce-work` / `spec-work` 目录级 diff 与 shipping reference 写成。

---

## Part A：上游基线与代码事实

### A.1 核对范围

- source:
  - `plugins/compound-engineering/skills/ce-work/`
- target:
  - `skills/spec-work/`
- 邻接：
  - `skills/spec-code-review/`
  - `skills/git-commit-push-pr/`
  - `tests/smoke/cli.sh`

### A.2 关键差异

#### A.2.1 `spec-work` 删除了 bare prompt triage

`ce-work/SKILL.md` 支持：

- plan document
- bare prompt
- Phase 0 input triage

`spec-work/SKILL.md` 当前已收敛为：

- 以 work document 为主
- 删除 Phase 0 bare prompt triage

这说明 `spec-work` 更依赖上游规划/规格输入，而不鼓励从模糊 prompt 直接执行。

#### A.2.2 `spec-work` 新增 Stage-0 预载

`spec-work` 与 `spec-plan` / `spec-code-review` 一样加入了 Stage-0 预载能力。

#### A.2.3 `spec-work` 增加了 swarm mode / quality checklist / reviewer usage guidance

相比上游，`spec-work` 新增：

- Swarm Mode with Agent Teams
- 更完整的 Quality Checklist
- “何时使用 reviewer agents”的限制说明

这些都是本地执行治理增强。

#### A.2.4 branch naming / bare prompt routing 被收紧

`ce-work` 中关于 feature branch meaningful name 与 bare prompt default 的说明，在 `spec-work` 中有所删除或收口。

这代表执行入口从“泛化工作执行器”收敛到“基于 plan/spec/todo 的 work executor”。

#### A.2.5 shipping-workflow 的 review / badge / evidence 口径更新

`references/shipping-workflow.md` 的主要差异：

- `ce:review` → `spec-code-review`
- `Compound Engineered badge` → `Spec First badge`
- 移除 `ce-demo-reel` 直接引用，改成更泛化的 evidence capture 说明

### A.3 现状判断

#### 结论 1：按上游严格等价口径，`spec-work` 未完全追平

因为它删除了 bare prompt triage 与部分更泛化执行分支。

#### 结论 2：按当前 `spec-first` 产品边界，`spec-work` 已经演化成“以 plan/spec/todo 为核心输入的执行器”

增强点包括：

1. Stage-0 预载
2. swarm mode
3. quality checklist
4. reviewer usage guidance

## Part B：当前产品边界下的正式实施方案

### B.1 目标

1. 保持 `spec-work` 作为 plan-driven executor 的定位
2. 继续保留本地执行治理增强
3. 把“bare prompt 是否支持”从隐式删除改成显式边界

### B.2 非目标

1. 不恢复完整 bare prompt triage 流程
2. 不回退 Stage-0
3. 不删 swarm / quality checklist 等本地增强

### B.3 建议方案

#### Phase 1：显式冻结输入边界

建议在 `skills/spec-work/SKILL.md` 中明确写出：

- 主输入是 plan/spec/todo file
- 如果只有模糊 prompt，优先先去 `spec:brainstorm` / `spec:plan`

避免当前“行为已收口，但边界说明不够明确”的状态。

#### Phase 2：补 contract test

建议新增：

- `tests/unit/spec-work-contracts.test.js`

覆盖：

- source skill 内部命名
- 存在 Stage-0 预载
- 不包含 bare prompt triage 段
- 包含 swarm mode / quality checklist / reviewer guidance
- shipping-workflow 引用 `spec-code-review`

#### Phase 3：验证 shipping contract

建议对 `references/shipping-workflow.md` 做独立 contract 保护，防止 future sync 回退：

- review tier 默认值
- evidence capture 口径
- badge 文案

### B.4 影响文件

- `skills/spec-work/SKILL.md`
- `skills/spec-work/references/shipping-workflow.md`
- `tests/unit/spec-work-contracts.test.js`

### B.5 建议验证

```bash
npx jest tests/unit/spec-work-contracts.test.js --runInBand
npm run test:unit
```

### B.6 最终建议

`spec-work` 当前应继续沿“plan-driven execution”方向走，而不是回退成对 bare prompt 更宽松的通用执行器。需要补的是 contract clarity 和测试，不是把本地增强删掉。
