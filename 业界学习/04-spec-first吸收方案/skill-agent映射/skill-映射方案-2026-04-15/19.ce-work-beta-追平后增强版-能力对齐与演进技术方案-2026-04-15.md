## 执行摘要

结论先行：

- `spec-work-beta` 不是“更快的 `spec-work`”，而是“显式 opt-in 的 delegation 执行轨”
- 它不应回退成上游 `ce-work-beta` 的 bare-prompt 入口
- 它也不应比本地稳定版 `spec-work` 少上下文增强和治理能力
- 当前最正确的目标态是：
  - 与 `spec-work` 共享执行主骨架
  - 在共享主骨架之上保留并校准 delegation 参数解析、host-aware routing、scratch/config contract、delegation reference
  - 显式消费 `spec-plan` 已经产出的 `Execution target: external-delegate`

按优先级排序，当前真实需要补齐或澄清的不是“文字 parity”，而是下面四项：

1. `spec-work-beta` 缺少与 `spec-work` 一致的 `/spec:plan` redirect 文案
2. `spec-plan` 已能产出 `Execution target: external-delegate`，但 `spec-work-beta` 还没有确定性的消费与路由 contract
3. Stage-0 当前虽已存在，但 `stage = work` 是否应改为 `work-beta` 尚未完成依赖证明；在证明前不应贸然改写
4. 实施方案需要同步现有测试与治理事实：`spec-work-beta` contract test 已存在，`CHANGELOG.md` 必须纳入代码改动落地流程；delegation reference 仅在逐条 diff 发现真实缺口时才修改

因此，本方案不再把 Stage-0 预载、reviewer usage guidance、delegation reference 主不变式、专项 contract test 当作“待新增项”。真正应推进的是入口边界、signal 闭环、依赖证明和实施治理。

## Part A：代码事实与正确目标边界

### A.1 `ce-work-beta` 的真实定位

上游 `ce-work-beta` 不是独立于 `ce-work` 的另一套工作流，而是：

- 继承 `ce-work` 的执行主流程
- 在其上叠加 Codex delegation 模式
- 在 beta 期保持 handoff 仍指向稳定版 `ce:work`

从真实代码看，上游的主要能力面包括：

1. `plan/spec/todo` 与 bare prompt 双入口
2. `Phase 0: Input Triage`
3. `delegate:codex` / `delegate:local` 参数与 config resolution
4. 单独拆出的 `codex-delegation-workflow.md`
5. 与主执行流程共享的 shipping/review contract

### A.2 当前 `spec-work-beta` 的真实定位

当前 `spec-work-beta` 已经形成了不同于上游、但更符合 `spec-first` 产品边界的本地定义：

1. 只接受 `plan/specification/todo` 文档输入，不再接 bare prompt
2. 保留 `delegate:codex` / `delegate:local` 参数解析
3. 使用 `.spec-first/config.local.yaml` 作为 delegation 配置入口
4. 使用 `.spec-first/workflows/work-beta/codex-delegation/<run-id>/...` 作为 scratch path
5. 使用 host-aware platform gate，而非“仅 Claude Code 支持”
6. shipping 仍默认走 `spec-code-review`

这说明当前 `spec-work-beta` 的问题已经不是“没有 delegation”，而是“beta 轨内部是否自洽、是否与本地稳定基线保持一致”。

### A.3 正确的产品分层

结合 `spec-work`、`spec-work-beta`、`spec-plan` 当前代码，正确的产品分层应该是：

| Workflow | 正确定位 | 默认性 | 输入边界 | 主要价值 |
|---|---|---|---|---|
| `spec-work` | 稳定默认执行轨 | 默认 | plan/spec/todo | 稳定、本地执行、上下文增强、治理完整 |
| `spec-work-beta` | delegation-aware 的实验执行轨 | 显式 opt-in | plan/spec/todo | 外部 delegate、token 节省、批处理实现 |
| `spec-plan` | 执行前计划轨 | 默认前置 | feature/request/doc | 输出可执行 plan 与 execution posture |

因此，`spec-work-beta` 的正确目标不是“比 `spec-work` 更轻”，而是：

- 保持与 `spec-work` 一样完整的执行主骨架
- 在此基础上增加 beta delegation contract

### A.4 差异分类矩阵

当前真实差异可分成五类：

| 类别 | 条目 | 当前状态 | 正确决策 |
|---|---|---|---|
| 已具备 | Stage-0 上下文预载 | `spec-work-beta` 已有完整 block | 不重复插入；只决定是否调整 `stage` / telemetry 口径 |
| 必须集成 | `/spec:plan` redirect | `spec-work` 有，beta 无 | 补进 beta |

| 保持分叉 | 不放开 bare prompt | beta 当前 plan-only | 保持 |
| 保持分叉 | `.spec-first` config / scratch path | 本地已落地 | 保持 |
| 保持分叉 | host-aware platform gate | 本地已落地 | 保持 |
| 保持分叉 | `spec-code-review` / `Spec First badge` | 本地已落地 | 保持 |

## Part B：哪些内容必须集成，哪些必须保持分叉

### B.1 必须集成进 `spec-work-beta` 的内容

#### B.1.1 校准 Stage-0 口径，不重复插入预载块

当前 `spec-work-beta` 已经包含完整的 Stage-0 consumption contract：

- `context-routing.json`
- `artifact-manifest.json`
- `minimal-context/work.json`
- `selected_assets / fallback_reason / level / skipped_rules`

因此，本轮不再把 Stage-0 视为“待移植能力”。

**正确做法：**

1. 保持当前 `spec-work-beta` 的 Stage-0 block 为基线，不重复插入整段内容
2. 若团队希望把 `本 workflow stage 标识` 或 telemetry 中的 `stage` 从 `work` 改成 `work-beta`，必须先证明该字段不会影响 Stage-0 产物命名、选择逻辑或 evaluator contract
3. 在未完成依赖证明之前，保持 `stage = work`，并在方案中明确这是“复用 stable work Stage-0 产物”的有意选择，而不是遗漏

换言之，本轮 Stage-0 的任务不是“补块”，而是“证明是否需要改口径；若不能证明，就不改”。

#### B.1.2 继承 `spec-work` 的 plan-only redirect

`spec-work` 当前明确告诉用户：

- 如果只有 feature idea 或 rough description，应先去 `/spec:plan`

`spec-work-beta` 已经在事实上是 plan-only，但文案上缺少这条入口约束，容易被误解为“更强万能入口”。

**正确做法：**

在 `spec-work-beta` 的 introduction 中补入一条显式 redirect：

- rough description 不直接执行
- 先跑 `/spec:plan`
- `spec-work-beta` 仅负责 plan-driven execution

#### B.1.3 显式消费 `spec-plan` 的 `Execution target: external-delegate`

当前 `spec-plan` 已经会在 implementation unit 中写入：

- `Execution note: Execution target: external-delegate`

这是 execution posture signal，不应该只停留在 plan 文档中。

**正确做法：**

`spec-work-beta` 需要在主流程里明确消费这个信号：

1. 在 `Read Plan and Clarify` 阶段检查 implementation units 的 `Execution note`
2. 标记出 delegation candidates
3. 在 `Choose Execution Strategy` 阶段把这些 candidates 作为 delegation routing 的优先输入

这里不能只写成“读到信号”，必须落成确定性 contract。

**规范形态（v1）：**

- 只把精确的 `Execution note: Execution target: external-delegate` 视为 canonical signal
- `spec-plan` 是该 signal 的 canonical producer；`spec-work-beta` 在 v1 不扩展同义短语、大小写变体或模糊匹配
- 先把生成与消费两端收敛到同一个精确字符串，再考虑兼容层

**决策表：**

| 条件 | 行为 |
|---|---|
| `delegation_active = false` 且无 candidates | 维持标准 / 本地执行，不额外提示 |
| `delegation_active = false` 且存在 candidates | 明确提示一次：是否仅对 candidate units 临时开启 delegation；若用户拒绝，继续本地执行；若用户接受，只对当前 session 生效，不自动写入 config |
| `delegation_active = true` 且存在 candidates | 默认只把 candidate units 送入 delegation batch；non-candidate units 保持标准 / 本地执行；只有用户显式要求 full-plan delegation 时才覆盖 |
| `delegation_active = true` 且无 candidates | 保持现有 `delegate:codex` / config-selected delegation 行为，不因为缺少 signal 而关闭 |

这里的关键原则是：

- 这是 recommendation signal，不是强制自动开 delegation
- 不能偷偷改变当前默认行为或悄悄写入用户 config
- 但也不能让 `spec-plan` 的信号白写，必须能从输出或 routing 行为中复核其效果
- 一旦出现 mixed execution（candidate delegate + non-candidate local），执行总结与 PR 说明必须能看出分工边界

#### B.1.4 delegation reference 只改真实缺口，不做预设性重写

当前本地 `spec-work-beta/references/codex-delegation-workflow.md` 已经包含以下硬约束：

- background launch 是单独调用
- polling 是新的独立调用
- 不允许 improvisation CLI flags
- `run_in_background: true` 必须是 tool parameter，而不是 shell `&`

所以，Phase 2 不能再预设“这部分需要整体补回”。

**正确做法：**

1. 先对上游与本地 reference 做逐条 diff inventory，列出“当前本地仍缺什么”
2. 只有当 inventory 证明某条 contract 真实缺失时，才对本地 reference 做定点修补
3. 若 inventory 结果为“现有 contract 已覆盖”，则把 Phase 2 记为 no-op verified，不再做重写
4. 不允许为了“追上游文字一致”而重写已经正确的本地 contract

### B.2 必须保持为本地分叉的内容

#### B.2.1 不恢复 bare prompt triage

这是当前最重要的产品边界之一。

原因不是做不到，而是不应由 `work-beta` 来承接模糊输入：

- 模糊需求应交给 `spec:brainstorm` / `spec:plan`
- `spec-work` / `spec-work-beta` 都应保持 plan-driven
- validation 文档也已明确“不放开 bare prompt”

因此：

- 不恢复 `Phase 0: Input Triage`
- 不恢复 bare prompt 路由
- 不恢复“description of work. Blank to auto use latest plan doc”这类 argument-hint

#### B.2.2 保持 `.spec-first` config / scratch path

这些不是“暂时 rename”，而是本地 runtime contract：

- `.spec-first/config.local.yaml`
- `.spec-first/workflows/work-beta/codex-delegation/<run-id>/...`

不应回退到：

- `.compound-engineering/config.local.yaml`
- `.context/compound-engineering/codex-delegation/...`

#### B.2.3 保持 host-aware platform gate

上游表述是“只有 Claude Code 支持 delegation”；本地表述已经升级成：

- 只要当前宿主会导致 Codex 递归或违反宿主模型，就禁用 delegation

这更符合 `spec-first` 的跨宿主产品边界，因此必须保持。

#### B.2.4 保持本地 shipping / review 命名与品牌

这些已完成，不应回退：

- `spec-code-review`
- `Spec First badge`
- 去除 `ce-demo-reel` ad hoc 调用

### B.3 可以进一步优化，但不是当前骨架缺口的内容

#### B.3.1 reviewer usage guidance

`spec-work` 当前有：

- `## When to Use Reviewer Agents`

`spec-work-beta` 当前也已经有这段。

因此它不是当前缺口。正确动作是只复核其表述与 `shipping-workflow.md` 的 review tier contract 没有冲突，而不是再补一遍。

#### B.3.2 shipping reference 共享化

当前 `spec-work` 与 `spec-work-beta` 的 `references/shipping-workflow.md` 已经同构。

这不是能力缺口，而是未来的维护成本点。可后续考虑：

- 共享文件
- 生成逻辑
- 组合式 reference

## Part C：目标态设计

### C.1 目标态定义

改造完成后的 `spec-work-beta` 应满足：

1. 与 `spec-work` 共享完整执行主骨架
2. 与 `spec-plan` 共享 execution posture 信号链
3. 在此之上增加 beta delegation contract
4. 保持 plan-only 输入边界
5. 保持 `spec-first` 本地 runtime contract

### C.2 目标态结构图

```text
spec-plan
  └─ emits Execution note / Execution target: external-delegate

spec-work
  └─ stable execution skeleton
     ├─ Stage-0 preload
     ├─ task execution loop
     ├─ testing / review / shipping governance
     └─ shipping-workflow.md

spec-work-beta
  └─ stable execution skeleton + delegation beta layer
     ├─ inherits Stage-0 preload
     ├─ consumes external-delegate signal
     ├─ parses delegate:codex / config.local.yaml
     ├─ routes into codex-delegation-workflow.md
     └─ reuses shipping-workflow.md
```

### C.3 设计原则

1. **共享主骨架，分离 beta 增量**  
   beta 只能增加 delegation 能力，不能丢掉稳定版已有能力。

2. **plan 是唯一主输入**  
   模糊输入留给 `spec:plan`，执行轨不承担产品澄清职责。

3. **signal 要能闭环消费**  
   `spec-plan` 产出的 execution posture 必须能被 `spec-work-beta` 读懂。

4. **reference 的描述必须足够硬**  
   delegation reference 的改动必须建立在真实 diff 上；已经固化的工具调用不变式不重复重写。

5. **contract 先于优化**  
   先修真实缺口，再补测试与 rehearsal；不要把已完成项重新包装成新增工作。

## Part F：验收与状态更新规则

### F.1 完成态判断

`spec-work-beta` 要从“部分完成”推进到“已完成”，建议至少满足：

1. `spec-work-beta/SKILL.md` 已补 `/spec:plan` redirect
2. `Execution target: external-delegate` 的消费 / 路由 contract 已确定且可复核
3. `tests/unit/spec-work-beta-contracts.test.js` 已在现有基础上追加断言并通过
4. Claude / Codex runtime transform 被验证
5. `CHANGELOG.md` 已按仓库治理要求补齐
6. Phase 2 若发现真实 reference 缺口，则已补齐并验证；若未发现，则已有明确 no-op 结论

### F.2 rehearsal 的地位

rehearsal 非主映射表“代码修复完成”的唯一前置，但它是高价值增强项。

建议口径：

- Contract + tests 全部落地后，可先进入“代码修复已完成”
- rehearsal 作为后续增强项单独跟踪

如果团队希望提高 beta 质量门槛，也可以把 rehearsal 提升为完成态前置，但这需要同步调整所有 skill 的完成标准，不能只对 `work-beta` 单独加码。

### F.3 映射表更新建议

当前总映射表仍写：

- `代码修复状态 = 部分完成（Stage-0 / reviewer guidance / delegation reference invariants / contract tests 已落地，待补 /spec:plan redirect 与 external-delegate 确定性消费）`

这个判断更符合当前代码事实。

在本方案执行完成后，可改写为：

- `代码修复状态 = 已完成（plan-only redirect、external-delegate deterministic routing、existing contract tests/runtime 验证、CHANGELOG 治理已落地；delegation reference 经 diff 核对无缺口或已补齐）`

若 rehearsal 继续作为增强项保留，可单独在差异说明里备注，而不必继续卡在“部分完成”。

## Part G：最终建议

`spec-work-beta` 的正确改造方向，不是“继续追上游文字一致”，而是建立一个内部自洽的 beta 执行轨：

1. 先修 `SKILL.md` 里真实未完成的入口与 routing contract
2. 再按逐条 diff 决定 `codex-delegation-workflow.md` 是否需要定点修补
3. 在现有 tests 上追加 guardrails，而不是重复新建同类保护
4. 用可重复的 rehearsal 提高 beta 轨置信度

一句话概括：

`spec-work-beta` 应该是“稳定 `spec-work` 主骨架 + `/spec:plan` 入口约束 + `spec-plan` posture 消费闭环 + delegation beta layer”，而不是“把已完成项重新包装成缺口，再去追上游文字一致”。

这也是当前两端代码与本仓库内部一致性下，最正确、最可演进的目标态。
