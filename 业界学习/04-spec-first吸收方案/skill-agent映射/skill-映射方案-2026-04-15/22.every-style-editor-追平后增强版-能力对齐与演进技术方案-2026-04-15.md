### C.2 输出模板 Markdown 化

#### 问题定位

`SKILL.md` Step 4 的输出模板使用 ASCII art 分隔线风格：

```
DOCUMENT REVIEW SUMMARY
=====================
...
DETAILED CORRECTIONS
===================
```

在 Claude Code 的 Markdown 渲染环境下，这种格式**不会被渲染**——`===` 不是
Markdown 标题，输出结果是一段不可读的等号行。相比之下，使用 `##`/`###` Markdown
heading，输出在 Claude Code、VSCode、GitHub 等任何 Markdown 渲染器中都清晰可读。

#### 增强方案（本地分叉）

将 SKILL.md 的输出模板从 ASCII art 改为标准 Markdown：

```markdown
## Document Review Summary

**Document Type:** [type]
**Word Count:** [approximate]
**Overall Assessment:** [brief overview]
**Errors Found:** [total number]

---

## Detailed Corrections

### Error 1

- **Location:** Paragraph #, Sentence #
- **Issue Type:** Grammar / Punctuation / Mechanics / Style Guide
- **Original:** "[exact text with error]"
- **Correction:** "[corrected text]"
- **Rule Reference:** [specific style guide rule violated]
- **Explanation:** [brief explanation]

---

## Recurring Issues

[List patterns of errors that appear multiple times]

## Style Guide Compliance Checklist

- ✅ [Rule followed correctly]
- ❌ [Rule violated — with count of violations]

## Final Recommendations

[2-3 actionable suggestions for improving the draft]
```

#### 触发分叉的条件

当 spec-first 用户在 Claude Code 中使用 `every-style-editor` 时，如果反馈输出格式
难以阅读，则启动本地分叉。上游若接受 Markdown 化改进，可提 PR 回去。

### C.3 description 补中文触发词

#### 问题定位

当前 `SKILL.md` frontmatter：

```yaml
description: "This skill should be used when reviewing or editing copy to ensure
adherence to Every's style guide..."
```

description 全为英文，在中文工作流中，当用户说"帮我审查这篇文章的英文风格"或
"按照 Every 风格规范润色" 时，skill discovery 无法匹配，用户需要手动找到
`every-style-editor` 这个 skill 名称。

#### 增强方案（本地分叉）

在 description 末尾追加中文触发词：

```yaml
description: "This skill should be used when reviewing or editing copy to ensure
adherence to Every's style guide. It provides a systematic line-by-line review
process for grammar, punctuation, mechanics, and style guide compliance.
适用场景：英文文章审查、按照 Every 风格规范润色、英文语法和标点检查。"
```

#### 触发分叉的条件

当 spec-first 用户在中文工作流中频繁需要手动指定 `every-style-editor` 时启动。

### C.4 Stage-0 / CRG 接入：为何不适用

`every-style-editor` 是**内容审查型 skill**，操作对象是用户输入的文本（文章、博文），
不是仓库代码。它：

- 不需要预载仓库上下文（操作对象不是代码）
- 不依赖 CRG 图谱（style guide 是静态参考文件，不是动态产物）
- `references/EVERY_WRITE_STYLE.md` 已在 skill 目录中随 skill 分发

接入 Stage-0 或 CRG 无任何收益。**结论：不适用。**
