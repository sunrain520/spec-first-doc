### C.2 最高优先级：AI 工具生态探测（通用增强）

#### 问题定位

`inventory.mjs` 已有 AI/LLM 服务检测，但仅靠 env 文件中的字符串匹配：

```js
// 当前实现（scripts/inventory.mjs:774）
if (/OPENAI|ANTHROPIC|CLAUDE/i.test(content)) services.push("AI/LLM API");
```

**盲区**：当项目使用 Claude Code / Codex / Gemini 等 AI 工具驱动开发时（如
spec-first、compound-engineering-plugin），`CLAUDE.md`、`AGENTS.md`、`.claude/`
目录等约定文件是该项目最重要的架构信号，当前 inventory 完全忽略它们。这会导致
生成的 ONBOARDING.md 漏掉"这是一个 AI 工具驱动项目"这一核心上下文。

#### 增强方案

在 `findInfrastructure()` 函数中，在现有 env 文件检测逻辑之后补充 AI 工具约定探测：

```js
// ── AI 工具约定文件探测 ──────────────────────────────────────────────────────

// Claude Code / Claude CLI convention files
const claudeSignals = [];
if (rootFiles.includes("CLAUDE.md")) claudeSignals.push("CLAUDE.md");
if (await exists(join(root, ".claude")))  claudeSignals.push(".claude/");

// Codex / OpenAI Agents convention files
const codexSignals = [];
if (rootFiles.includes("AGENTS.md"))  codexSignals.push("AGENTS.md");
if (rootFiles.includes("AGENTS.json")) codexSignals.push("AGENTS.json");
if (await exists(join(root, ".codex"))) codexSignals.push(".codex/");

if (claudeSignals.length > 0 || codexSignals.length > 0) {
  services.push("Claude Code / AI Agent tooling");
}
```

#### 产出影响

inventory 输出新增 `services: ["Claude Code / AI Agent tooling"]`，LLM 在写
Section 1（What Is This?）和 Section 3 外部依赖表时可以准确标注该项目的 AI
工具属性，而不是仅凭 README 猜测。

#### 上游策略

这是**通用增强**，与 spec-first 无耦合。建议：

1. 在本地 fork 验证效果
2. 提 PR 给 `compound-engineering-plugin` 上游
3. 上游合并后 spec-first 通过正常同步吸收，保持字节级一致

### C.3 Stage-0 接入：为何不适用

其他 workflow skill（`spec-work`、`spec-plan`、`spec-code-review`）接入 Stage-0 的原因是：
它们在执行复杂工程任务前需要预载仓库的历史上下文（pitfalls、code-facts、进度等）。

`onboarding` 的机制完全不同：

- Phase 1 运行 `inventory.mjs` **自行采集**结构信息（没有上下文依赖）
- Phase 2 **按需 Read** 关键文件（主动拉取，不需要推送上下文）
- 产出是 `ONBOARDING.md`，是一次性文档生成，不是迭代执行

接入 Stage-0 只会增加噪音，不会提升产出质量。**结论：不适用。**

### C.4 本地分叉增强（暂缓项的存档）

以下增强有 spec-first 使用价值，但因破坏字节级一致而暂缓，记录备查：

**`EXCLUDED_DIRS` 加 `.spec-first`**

```js
// 增加位置：inventory.mjs:26-33
const EXCLUDED_DIRS = new Set([
  // ... 现有内容 ...
  ".spec-first",   // spec-first graph/workflow 产物目录，非源码
]);
```

效果：防止 `.spec-first/` 出现在 `topLevel` 结构列表中，减少 ONBOARDING.md 噪音。

**触发分叉的条件**：当 spec-first 自身或高价值宿主仓库使用 `onboarding` skill
生成文档，且因 `.spec-first/` 被列为顶层目录而产生明显误导时，再启动本地分叉。
