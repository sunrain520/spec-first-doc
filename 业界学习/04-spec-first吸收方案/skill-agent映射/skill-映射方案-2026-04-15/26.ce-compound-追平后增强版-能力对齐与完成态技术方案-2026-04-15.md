### A.8 当前仍保留的真实差异

当前还存在的差异主要是以下几类：

#### A.8.1 source 命名与 public identity 分离

- 上游 source: `name: ce:compound`
- 当前 source: `name: compound-workflow`

这不是功能缺失，而是 `spec-first` 当前运行时模型的一部分：

- Claude command 通过 workflow 路径接入
- Codex runtime public name 由 adapter 重写为 `spec-compound`

因此这属于**允许保留的宿主适配差异**。

#### A.8.2 文案与命名空间迁移

例如：

- `ce:*` → `spec:*`
- canonical agent name 从 `compound-engineering:*` → `spec-first:*`
- 若干表达被改写成更贴近当前仓库风格的文本

这同样不构成功能缺口。

## Part B：完成态判断

### B.1 判断口径

对 `spec-compound` 是否可以标记为“代码修复完成”，本轮采用以下口径：

1. 关键行为合同已真实落地
2. support files 已恢复到双 track 模型
3. session enrichment 不是纸面设计，而是已经接入 runtime
4. consumer 与验证器已一起切换
5. `docs/solutions/` 的最小合法数据面已清洗完成
6. 剩余差异仅限宿主命名与本地运行时 contract，不影响能力等价

### B.2 结论

按上述口径，`ce-compound -> spec-compound` 可以标记为：

- `方案分析状态 = 已完成（已归档）`
- `代码修复状态 = 已完成`

### B.3 不应混入本结论的事项

以下事项不应被拿来否定 `spec-compound` 的完成态：

1. `spec-compound-refresh` 仍未完全 parity
2. source frontmatter 名称与上游不同
3. 若干说明性文案与上游不完全同字节

这些问题要么属于**相邻 skill**，要么属于**允许保留的宿主适配差异**。
