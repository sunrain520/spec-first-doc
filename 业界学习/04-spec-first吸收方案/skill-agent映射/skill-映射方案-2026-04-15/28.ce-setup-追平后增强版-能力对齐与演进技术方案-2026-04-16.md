## B. 与上游的剩余差异定性

### B.1 仍然是拆分映射，但不是未完成

`ce-setup` 在 `spec-first` 中并未 1:1 复制为单一 skill，这个判断仍然成立。

当前真实结构是：

- `setup` 管 repo-local setup / helper tools / cleanup
- `spec-mcp-setup` 管 host-level MCP setup
- `doctor/init/clean/version-reminder` 管 repo runtime 与 CLI 状态

这是产品边界上的 intentional split，不是能力缺失。

### B.2 不应机械回退的部分

以下不应因为追平 `ce-setup` 而回退：

- `spec-mcp-setup` 的宿主识别与 MCP marker 体系
- `doctor/init/clean` 的 repo runtime 管理
- `.spec-first/` 命名空间
- `host-setup.json` 与 `spec-graph-bootstrap` 的协作链路

### B.3 当前剩余的非阻断差异

仍然存在但不影响“已完成”判定的差异：

1. 上游把更多 setup 说明集中在单 skill；当前拆成 `setup + spec-mcp-setup + CLI`
2. 上游 completion 文案引用 `/ce-update`；当前没有同名 skill，而是 `version-reminder`
3. 当前 `check-health` 相比上游新增了 `.spec-first/config.local.yaml` 缺失检查，属于本地增强

这些差异都属于 intentional divergence 或增强，不构成未完成。
