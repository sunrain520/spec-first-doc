> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、`CRG Stage-0`、`ECC`、`compound-engineering-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 文档目标

这份文档不是泛泛对比，而是给后续升级工作提供一份可执行的映射基线：

- 明确哪些 `skill` / `agent` 属于 `exact-same`、`exact-diff`、`rename-map`
- 明确哪些能力已经发生 `split-map`
- 明确哪些是上游独有能力，不能直接照搬
- 明确哪些是 `spec-first` 已经长出的独立控制面
- 为后续“逐 commit 跟核心链路、按主题评估外围能力”提供直接输入

## 核对范围与方法

本次核对覆盖：

- 当前项目 `skills/` 下全部 44 个 skill 目录
- 源项目 `plugins/compound-engineering/skills/` 下全部 41 个 skill 目录
- 当前项目 `agents/` 下全部 47 个 agent 文件
- 源项目 `plugins/compound-engineering/agents/` 下全部 49 个 agent 文件

本次核对不是抽样，而是逐项执行了两层校验：

1. 目录级校验
   - 逐个比对目录名或相对路径
2. 文件级校验
   - 对每个同名 skill 的 `SKILL.md` 做了 frontmatter `name` / `description` 读取与文件内容 hash 对比
   - 对每个同路径 agent 的 `.md` 做了 frontmatter `name` / `description` 读取与文件内容 hash 对比

判定说明：

为保证这份文档可以直接转成后续同步矩阵，映射类型必须是**互斥枚举**。后文所有条目都应只落在下面 6 类之一：

- `exact-same`：存在直接对应，且文件内容完全一致
- `exact-diff`：存在直接对应，但文件内容已有分叉
- `rename-map`：职责对应，但名称体系已改为 `ce-* -> spec-*`
- `split-map`：源项目一个能力被当前项目拆到多个入口或文件
- `source-only`：只存在于源项目，当前没有直接对应能力
- `spec-only`：只存在于当前项目

> 约束：同一个 skill / agent 在这份文档里只能归入一个映射类型，不能同时出现在多个桶中。

## 三、已验证的关键分叉点

以下不是目录噪音，而是会直接影响后续升级策略的真实分叉：

### 3.1 `spec-plan` 与 `ce-plan`

- 上游 `ce-plan` 已扩成“适用于任意多步骤任务”的通用 planning
- 当前 `spec-plan` 仍然更明确地锚定在代码库、实现路径、测试场景、repo pattern 上

后续同步建议：

- 同步结构增强、研究流程、deepening 机制、路径与输出约束
- 不要默认把“非软件任务 planning”整包搬入当前项目

### 3.2 `spec-work` 与 `ce-work`

- 上游 `ce-work` 明确支持 bare prompt 和 plan 文档双入口
- 当前 `spec-work` 更强调 plan/spec/todo 文档驱动执行

后续同步建议：

- 优先同步 workflow 质量改进、review 强制、token 优化、late reference 抽取
- 是否放宽为 bare prompt 执行，应单独做产品判断

### 3.3 `setup` / `spec-mcp-setup` / CLI 与 `ce-setup`

- 上游将环境诊断与 setup 收束为一个交互式 skill
- 当前已将其拆分为：
  - placeholder `setup`
  - 独立 `spec-mcp-setup`
  - CLI `doctor/init/clean`
  - 版本提醒和受管资产检查

后续同步建议：

- 按职责吸收，不按文件吸收
- 任何和 plugin version、cache、interactive setup 相关的更新，都应先判断落到哪个层

### 3.4 `ce-update` 与当前 CLI 健康检查

- 上游 `ce-update` 是“插件版本检查 + cache/marketplace 修复 + Claude plugin root 平台门禁”
- 当前 `version-reminder` / `doctor` 只有弱近邻关系，不构成等价映射

后续同步建议：

- 将其视为 `source-only`
- 如果未来要支持等价能力，应新立项，而不是把 `doctor` 或版本提醒误判成现成落点

### 3.5 `feature-video` 与 `ce-demo-reel`

- 两者都服务“展示变化”
- 但上游更偏 evidence capture / reel / PR visual proof
- 当前更偏视频 walkthrough

后续同步建议：

- 不要视为 1:1
- 如果未来要补“真实证据采集”，应新建独立能力，而不是把 `feature-video` 硬改成 `ce-demo-reel`

### 3.6 `reproduce-bug` 与 `ce-debug`

- 当前已有 bug reproduction 能力
- 上游 `ce-debug` 是更完整的“排障 -> 根因 -> fix”工作流

后续同步建议：

- 这是最值得补的新增能力候选
- 但不应简单覆盖 `reproduce-bug`
- 更合理的做法是：
  - 保留 `reproduce-bug` 作为 issue-grounded reproduction
  - 评估是否新增 `spec-debug` 作为完整 debug 工作流

## 五、后续升级时建议维护的同步矩阵字段

后续每一批升级，建议都按同一张矩阵推进：

| 字段 | 含义 |
|---|---|
| `主题组` | 如 review、plan、work、compound |
| `上游 commit` | 相关 commit ID 列表 |
| `源项目文件` | 上游变更涉及的关键文件 |
| `当前项目文件` | 当前仓库的实际落点 |
| `映射类型` | `exact-same`、`exact-diff`、`rename-map`、`split-map`、`source-only`、`spec-only` |
| `处理方式` | `逐 commit`、`按最终状态`、`按职责迁移`、`暂缓` |
| `优先级` | `P0 / P1 / P2 / Backlog` |
| `备注` | 是否会影响 CLI、workflow、文档叙事、运行时 |

## 六、可直接用于执行的升级顺序

建议按下面顺序推进，而不是一上来扫全量文件：

1. `review` / `spec-doc-review` / `resolve-pr-feedback`
2. `plan` / `brainstorm` / `repo-research-analyst` / `spec-flow-analyzer`
3. `work` / `compound` / `compound-refresh`
4. `work-beta`
5. 单独评估 `ce-debug`
6. 最后再看 `setup/update/sessions/slack/optimize/demo`

## 七、结论

对当前项目最重要的判断不是“有没有和上游同名文件”，而是：

- 核心 workflow skill 已经形成稳定 rename-map
- review / spec-doc-review / research agents 大量仍然是强同构区域
- `setup/update/debug/session/slack/bootstrap` 这些外围控制面已经明显分叉
- 后续升级应采用：
  - 核心链路按 commit 跟
  - 外围能力按主题评估
  - 拆分职责的区域按控制面落点迁移

这份文档应作为后续所有升级批次的映射基线使用。
