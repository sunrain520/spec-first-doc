## 目标

这份文档是在 [7.skill-agent-映射核对与升级同步指南.md](./7.skill-agent-映射核对与升级同步指南.md) 的映射基线上，进一步把**上游核心链路相关 commit** 落成可执行矩阵。

用途不是讲道理，而是直接指导后续升级：

- 哪些 commit 必须进入当前升级批次
- 哪些 commit 只适合选择性吸收
- 哪些 commit 不应进入当前主链路同步
- 每个 commit 在 `spec-first` 里的实际落点文件是什么

## 适用范围

本矩阵只覆盖当前项目的**核心链路与强同构控制面**：

- `spec-code-review`
- `spec-doc-review`
- `spec-plan`
- `spec-brainstorm`
- `spec-work`
- `spec-work-beta`
- `spec-ideate`
- `spec-compound`
- `spec-compound-refresh`
- `resolve-pr-feedback`
- 直接参与这些 workflow 的 review / spec-doc-review / research / workflow agents

不覆盖：

- `ce-update`
- `ce-debug`
- `ce-demo-reel`
- `ce-sessions`
- `ce-slack-research`
- `ce-optimize`
- 多宿主 / converter / targets / release 平台层

这些能力另立轨道，不进这份核心链路矩阵。

补充约束：

- 按当前升级目标，`tests/` 不作为本矩阵前期分析重点，因此上游测试文件不单列进“当前项目落点”
- 这不等于后续实施阶段永久忽略测试；真正迁移时仍需为每批次补验证项

## 使用规则

### 决策字段

- `MUST`
  - 进入当前核心升级批次
  - 应逐 commit 审核并迁移
- `SELECTIVE`
  - 相关但不应整包照搬
  - 只吸收其中适合 `spec-first` 当前定位的部分
- `SKIP`
  - 不进入当前核心链路同步
  - 仅保留背景信息
- `NEW-TRACK`
  - 当前没有等价能力
  - 需要单独立项，不放进本轮主链路批次

### 推荐执行顺序

1. `review` / `spec-doc-review` / `resolve-pr-feedback`
2. `plan` / `brainstorm` / `ideate`
3. `work` / `work-beta`
4. `compound` / `compound-refresh`
5. 再开 `NEW-TRACK` 项

### 共享 commit 处理规则

有些上游 commit 会同时改多个 workflow。为避免“按批次执行时漏迁一半”，本矩阵采用下面规则：

- `owner-batch`
  - 该 commit 的主归属批次
  - 负责定义该 shared commit 的语义边界、迁移范围、保留分叉与 handoff 结论
- `shared`
  - 表示该 commit 同时影响别的批次
  - 不能因为已在别的批次出现，就默认本批次无需同步对应意图

执行原则：

- 共享 commit 只指定一个 `owner-batch`
- `owner-batch` 负责定义该 commit 的语义边界、迁移意图、保留分叉和 handoff 结论，不要求包办全部真实文件落点
- `file-affinity` 批次负责最贴近本批文件组的真实文件实施与验证
- 其他批次不能重复裁决 shared commit 的语义，只能消费 owner handoff，并在自身实施时回写“哪些文件由我真正落地”
- 若一个共享 commit 同时触达多个 skill，owner-batch 必须在实施计划里拆出文件级迁移子项，并在矩阵 `notes` 中至少回写：
  - owner handoff 结论
  - 真实文件由哪个批次落地
  - 哪些批次只核查、不实施

### 5.2 批次 A 更新说明

**范围：** `spec-code-review`、`spec-doc-review`、`resolve-pr-feedback`，以及直接参与这些流程的核心 review / spec-doc-review / workflow agents。

**为什么先做：**

- 这是当前主链路里风险最高、收益也最高的一段。
- 一旦 review 门禁本身存在误判、递归、自调用、merge-base 漂移或不可信输入问题，后面 plan/work 的升级都会建立在不稳的底座上。
- 这批改动多数都是“修正协议和路由逻辑”，侵入范围集中，适合作为第一批。

**这批主要吸收什么：**

- `spec-code-review` 的 confidence rubric、autofix_class、输出模板和 base 解析加固
- `spec-code-review` 的 token / latency 优化
- reviewer / spec-doc-review subagent template 的 recursion guard
- `spec-doc-review` 的 `batch_confirm -> auto` 收敛
- `spec-doc-review` 的 auto 路由收敛、分类放宽、降 token / latency
- `resolve-pr-feedback` 的不可信输入边界和聚类逻辑
- 若干核心 reviewer agent 的防递归示例清理
- `949bdef` 在 `spec-work`、`spec-work-beta`、`spec-ideate`、`spec-compound-refresh` 的共享真实文件落点

**实施后的预期结果：**

- review 结果更稳定、更省上下文
- spec-doc-review 的自动化路由更可靠
- PR comment 处理更安全
- 上游近期在 review 层的稳定性修复可以基本对齐

### 5.3 批次 B 更新说明

**范围：** `spec-plan`、`spec-brainstorm`、`spec-ideate`，以及 `repo-research-analyst` 这种直接支撑 planning 的 research agent。

**为什么排第二：**

- plan / brainstorm 是主链路的前半段，但它们的风险通常不如 review 那样直接导致错误合并或错误通过。
- 在 review 层先稳定之后，再升级 planning / requirements 质量，实施风险更低。

**这批主要吸收什么：**

- `spec-plan` 与 `spec-doc-review` 的更强绑定
- output structure、scope sub-categorization、repo-relative path 约束
- brainstorm 中 verification 与 technical design 的边界澄清
- `spec-ideate` 的 token / latency 优化
- plan / brainstorm 的 token 优化和“晚序列引用抽取”
- 跨 workflow 共享约束：review mandatory、testing gap、delegation 前置姿态

**选择性项为什么放后面：**

- `ca78057` 的 interactive deepening mode 值得吸收思路，但不是第一优先级
- `4c7f51f`、`bd02ca7` 的 visual aids 要先做产品判断
- `320a045` 的“泛化到非软件任务”与当前定位不一致，本轮明确不纳入

**实施后的预期结果：**

- brainstorm / plan 产物更稳定、更结构化
- requirements 与 implementation plan 之间的衔接更顺
- 前置链路的 token 成本下降，但不改变当前产品的 repo-grounded 定位

### 5.4 批次 C 更新说明

**范围：** `spec-work`、`spec-work-beta`，以及和执行门禁直接相关的 testing / review 约束。

**为什么排第三：**

- work 层依赖前面的 plan / review 契约。
- 在 review 和 planning 先稳定后，再更新 work，会减少“上层规则和执行层规则不同步”的返工。

**这批主要吸收什么：**

- `6dabae6` 中“accept bare prompts”不应直接照搬

**实施后的预期结果：**

- beta 路线更接近上游近期 delegation 稳态
- 执行链的 token 消耗下降，权限行为更符合宿主预期

### 5.5 批次 D 更新说明

**实施后的预期结果：**
