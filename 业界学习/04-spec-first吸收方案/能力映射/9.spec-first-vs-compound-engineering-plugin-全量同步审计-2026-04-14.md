> Lifecycle: historical-input / external-reference. 本文保留历史 CRG/CE/ECC 方案、迁移或对比材料；其中 `src/crg`、`spec-first crg`、`graph.db`、`better-sqlite3`、`.claude-plugin`、命令数量和文件数量等旧口径可能已过期。当前 source of truth 以 `docs/archive-index.md`、`docs/README.md`、根目录 README、`docs/05-用户手册/`、`docs/contracts/`、`skills/`、`src/cli/`、`CHANGELOG.md`、`spec-mcp-setup` 和 `spec-graph-bootstrap` 为准。

## 审计结论

**结论：当前两个项目不属于“完全同步一致”。**

从代码与脚本实物核对结果看，核心 workflow 区域已经大幅同步，但仓库级功能面仍明显分叉：

- `skill` 层：当前以“重命名映射 + 直接对应但保留本地分叉 + 少量拆分映射 + spec-first 独有能力”并存为主，因此不能判定为全量功能一致。
- `agent` 层：多数上游 agent 已有直接落点，少量保留命名或模型配置差异，同时当前仓库还有面向本地产品边界的增量 agent。
- `脚本/控制面`：直接可比的 skill 内脚本共有 15 个，其中 13 个字节级一致、2 个存在差异；同时两边各自还有大量独有脚本与独立控制面。
- `仓库级能力`：上游仍保留多目标插件同步/转换平台、`scripts/release/*.ts` 发布子系统与 `dhh-rails-style` 这类当前未保留的 repo-owned skill；当前仓库则新增 `spec-first` CLI、CRG、`spec-graph-bootstrap`、`spec-graph-bootstrap`、`spec-mcp-setup`、`agent-browser` 等能力。

这意味着：**“核心链路同步度高”成立，但“两个项目已经完全同步一致”不成立。**

## 审计方法

- 核对范围：`skills/`、`agents/**/*.md`、skill 内 `scripts/`、仓库级 `bin/` / `src/` / `scripts/` 控制面。
- 核对方式：
  - 目录级：枚举两个仓库全部 skill 目录与 `agents/**/*.md` agent 文件，不抽样；`agents/research/session-history-scripts/` 这类 helper scripts 不计入 agent 数，单独视为会话能力支撑脚本。
  - 文件级：对同目录/同路径文件做 SHA-256 比对。
  - 映射级：对 `ce-* -> spec-*` 做“重命名映射”；对 `ce-setup` 做“拆分映射”；对无法一一落点的能力做“仅上游存在 / 仅当前仓库存在”判定。
- 判定口径：
  - `完全一致`：字节级一致
  - `直接对应但有差异`：存在直接对应，但内容不一致
  - `重命名映射`：能力对应，但命名/控制面已切换
  - `拆分映射`：上游单能力已被当前仓库拆到多个落点
  - `仅上游存在` / `仅当前仓库存在`：只存在于其中一个仓库

## 基于当前产品边界的再判定

上面的差异枚举是代码事实，但是否要把这些差异都视为“待补齐缺口”，取决于当前产品边界。

结合已确认的产品方向，本文补充以下前提：

- `spec-first` **不扩展多平台能力**
- `spec-first` 只维持 **Claude** 与 **Codex** 两类宿主
- 因此审计目标不应再定义成“把上游多宿主插件平台完整搬回 spec-first”

在这个前提下，主文档里的差异需要分成两类看：

1. **应保留的结构性分叉**
2. **应补齐的当前产品闭环缺口**

### 应保留的结构性分叉

| 差异域 | 是否需要同步 | 结论 | 原因 |
|---|---|---|---|
| `src/commands` + `src/sync` + `src/targets` + `src/converters` 多宿主平台 | 否 | 明确保留分叉 | 这套控制面服务于上游的多目标 plugin 转换/安装/同步平台，不符合 `spec-first` 只服务 Claude/Codex 的产品边界 |
| `scripts/release/*.ts` 发布子系统 | 否 | 明确保留分叉 | 这是上游 plugin marketplace / release 模型的一部分，当前仓库没有必要 1:1 复制 |
| repo-owned `dhh-rails-style` skill | 否 | 明确保留分叉 | 当前仓库已不再维护 repo-owned `dhh-rails-style` 源码；Rails/Ruby 场景仅在 `best-practices-researcher` 的常见映射中保留 discoverability |
| `src/crg/**/*.js`、`bootstrap`、`graph-bootstrap` | 不适用 | 作为 `spec-first` 独有能力长期保留 | 这是当前仓库已经形成的产品基础设施，不应因为上游没有而被视为负向差异 |

### 本轮已补齐的入口级闭环

如果按“让 `spec-first` 在 **Claude + Codex** 范围内形成更完整闭环”来定义同步目标，那么此前最重要的缺口现在已经补齐：

### 对“完全同步一致”的口径修正

因此，基于当前产品边界，主文档的结论应进一步收敛为：

- `spec-first` **不需要**与上游在“多宿主平台控制面”上完全一致
- `spec-first` **需要**继续维持 `skill/agent` 核心链路高同步
- `spec-first` **已经**补齐 `update / debug / sessions` 等入口级闭环，后续重点应转向持续维护 contract 与 intentional divergence 的清晰边界

换言之，当前最合理的目标不是“与上游全量同构”，而是：

**在 Claude + Codex 范围内，实现核心工作流高同步，并补齐关键控制面缺口。**

## 关键发现

1. `skill` 不是“全量一致”，而是“核心链路高同步 + 外围控制面分叉”。
2. `agent` 同步度明显高于 `skill`：当前 live 状态是 38 个“完全一致”，比旧版映射基线更接近上游。
3. 直接可比脚本多数已经同步，当前差异主要集中在：
   - `skills/spec-code-review/references/resolve-base.sh`
   - `skills/resolve-pr-feedback/scripts/get-pr-comments`
4. “是否完全同步一致”的真正阻断项不在已吸收的 `update / debug / sessions / slack-research / optimize` 入口，而在仓库级控制面与少量明确保留的 source-only 资产：
   - 上游独有：repo-owned `dhh-rails-style` skill、多目标插件同步/转换平台、`scripts/release/*.ts`
   - 当前独有：`spec-first` CLI、CRG、`spec-graph-bootstrap`、`spec-graph-bootstrap`、`spec-mcp-setup`、`agent-browser`、`rclone`、`orchestrating-swarms`
5. 因此当前最准确的架构判断是：
   - **核心 workflow prompt/agent 层：高同步**
   - **仓库产品形态与平台能力层：非同构，不完全同步**

## 最终判定

如果判定标准是“核心工作流是否大体跟上游对齐”，答案是：**是，且同步度较高。**

如果判定标准是“spec-first 与 compound-engineering-plugin 是否已经功能完全同步一致”，答案是：**否。**

阻断“完全一致”结论的不是零星文案差异，而是以下三类结构性事实：

- 仍存在 15 个“直接对应但有差异” skill / 11 个“直接对应但有差异” agent
- 这些差异已明确以命名迁移、宿主策略分叉和 `spec-first` 增量为主，并不等同于“仍有大规模未修复能力缺口”
- `agent` 层已不存在“仅上游存在”的 markdown agent；未完全一致主要来自 intentional divergence 与 spec-only 增量
- `skill` 层仍存在 1 个仅上游存在的 repo-owned skill（`dhh-rails-style`），以及多目标插件同步/转换平台、`scripts/release/*.ts` 等 source-only 控制面
- 当前仓库已经长出多项上游不存在的产品能力与 CLI/CRG 子系统

所以更准确的说法应是：**spec-first 已完成大规模核心链路同步，但没有也不应该被描述为与 compound-engineering-plugin 全量功能完全一致。**
