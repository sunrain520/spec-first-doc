# Spec-First：代码写完才发现需求理解错了，这个错误可以提前避免

**状态：** 大纲
**内容类型：** 取舍篇
**Harness 坐标：** Evidence Harness（前置质量）

## 核心论点

代码 review 只是最后一道关。需求和计划的 review 更重要，因为它们决定了 WHAT——在错误的方向上写出完美的代码，仍然是浪费。doc-review 在 plan 之前发现问题，成本最低。
