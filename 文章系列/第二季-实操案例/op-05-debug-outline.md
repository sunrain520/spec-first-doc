# Spec-First：AI 把代码改崩了，别再"你再看看"了

**状态：** 大纲
**层：** 切面篇（高价值切面，发布顺序提前到骨架前——传播性最强，人人都被 AI 改崩过）
**切面：** 调试实战 / hypothesis ledger（链路里 work/debug 那一段的放大）
**目标读者：** 被 AI 反复改崩、陷入"你再改改→又崩→再改"循环的开发者（B 型——已在用 AI 写代码、想确认 spec-first 能帮上忙；不是泛指"所有人"）

## 定位

## 结构（约 14 节）

1. 开场：每个用 AI 写代码的人都经历过——它信誓旦旦改完，你一跑还是崩，来回十几轮越改越乱（金句 + 导读）
2. 先解剖"你再看看"为什么没用：每次都在猜、改多个地方碰运气，这叫 shotgun debugging
3. 案例：一个"看起来对其实错"的 bug 登场，表面症状会把人带偏
4. debug 的第一纪律：先复现——没有能稳定触发的最小复现，一切修复都是猜
5. investigate before fixing：能把"从触发到症状"的完整因果链讲清楚（没有"莫名其妙就 X 导致 Y"的 gap）之前，不动手修
6. hypothesis ledger 登场：把每个假设记下来——假设、预测、支持证据、反驳证据、探针结果、最终根因
7. "预测"为什么关键（限定语境）：**当因果链上有不确定/非显然的环节时**，为它形成一个预测——如果假设成立，另一个代码路径上必然也为真；预测错了但"修好了"，说明你修的是症状不是根因。注意这是"测试不确定链接的工具"，不是每个假设都要走的仪式（别讲成普遍步骤）
8. 一次只改一件事：测一个假设、改一处；同时改多个"看看有没有用"就回到了 shotgun（用本案例演示改一处 vs 多处的分叉）
9. 证据必须来自复现/源码/测试/日志/运行值，不能来自"我觉得"
10. 卡住时：诊断为什么卡住，而不是更努力地试——2-3 个假设排空就停下来重新想
11. 修复纪律：先写一个能复现 bug 的失败测试，再写最小修复，只动根因（用本案例演示失败测试先行）
12. 一个真实 ledger 长什么样：把本案例的假设推演摊开给读者看（不做通用模板讲解）
13. 收尾复盘（全季固定动作）：这个 bug 如果一开始就用 ledger，少走多少弯路
14. 小结 + 结尾 CTA 四件套（**主 CTA = 转发钩子**：转给陷在"你再改改"循环的人；次级关注/阅读原文）+ 下一篇 blockquote 预告 **op-02（从 0 到 1）** 点名具体问题不剧透 + 单独一句引向 s3-08（机制深挖，给想懂原理的人，与下篇预告分开）
