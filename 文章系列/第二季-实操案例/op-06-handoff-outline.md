# Spec-First：换个 AI、换个人接手，怎么不用从头讲一遍

**状态：** 大纲
**层：** 切面篇（高价值切面）
**切面：** 跨会话与协作接力（上下文复用）
**目标读者：** 经常开新会话、换 AI、或要接手别人未完成的活的开发者/团队

## 定位

- 切面篇，不重走全链路：切"上下文复用 / 接力"这一能力。
- 这一篇是前面每篇的**价值兑现点**——前面发布的 op-01/02/05 一路攒下的 `docs/`（requirements/plan/solutions）到这里证明它不是负担，是跨会话记忆。
- 呼应整季运营主线：把"每一次任务都让下一次任务更容易"从口号，变成可操作的交接动作。
- **发布序定位（关键，本篇最易错）：** 锁定发布顺序第 5 篇——**上一篇是 op-02（从 0 到 1），下一篇是 op-03（存量改造）**。上集回顾承接 op-02，下篇预告指 op-03。**可引用的已发案例只有 op-00/01/05/02——绝不能点名尚未发布的 op-03/04 案例**（它们在本篇之后才发，读者还没看过）。
- 案例建议：钉死一个贯穿案例——直接复用 op-01 的 todo-app 标签过滤，演"两周后回来接手 / 换个 AI 继续做这个功能"，让读者在熟悉的案例上看接力动作。

**运营基调（写作约束）：**
- 主受众单一（关键）：以**个人换会话/换 AI 接力**为唯一主线开头对他说话（最大众、最高频）；"团队换人接手"收敛为后段一节延伸（参考 op-01 §14 的处理），不与个人主线平行成双主受众。
- 痛点前置：第 1 节金句即"换个会话 AI 就失忆、换个人接手又得从头口头讲一遍"的痛点画面，导读后置。
- **贴真实片段（切面篇 DNA）：** 第 4、7 节要求**贴真实读到的 plan/solutions 片段**和"换 AI 后几分钟回到状态"的前后对照，不要只描述"读一下 plan"——读者要看到接力时屏幕上真实出现的东西。
- 防机制越界（高风险对 op-06↔sessions/s3-10）：sessions/compound 只讲"接力时怎么用、带来什么价值"，不复述检索/沉淀机制，引细节篇。
- CTA 四件套 + 主次分明：导流期篇目，**主 CTA = 阅读原文去 GitHub**，次级关注/转发；下篇预告独立 blockquote。

## 结构（约 14 节）

1. 开场：金句即痛点 + 画面感（"换个会话 AI 就失忆，换个人接手又得从头口头讲一遍历史"）；一句上集回顾承接 op-02（"上一篇从 0 到 1 攒下了 requirements/plan，这篇就用它们证明：换个人、换个 AI 接着干，靠的就是这些落在文件里的东西"）；导读 blockquote 后置
2. 上下文蒸发的真相：经验全留在了关掉的对话框里，下一次（下一个会话/下一个 AI）从零开始
3. 为什么 docs/ 是解药：requirements 记 WHAT、plan 记 HOW 的边界、solutions 记踩过的坑——这些是读得到的记忆，不是脑子里会蒸发的
4. 接手一个半成品的正确顺序：先读 plan、再读 solutions、再读最近 review findings——**贴 op-01 todo-app 真实的 plan/solutions 片段**，让读者看到接手时屏幕上到底出现什么
5. sessions 能力登场：当 docs 还不够时，回头查"之前的会话里到底试了什么、哪些方案失败过、关键决策是什么"
6. sessions 给你的不是全文回放，是蒸馏过的"做过什么 / 没成什么 / 关键决策 / 相关上下文"
7. 一个真实接力动作（个人主线，贯穿案例）：用 op-01 todo-app 演——换 AI 后先让它读 plan+solutions、再用 sessions 补"上次失败的尝试"，给"几分钟回到状态"的前后对照
8. 为什么"边界 + 证据 + 沉淀"天然适合交接：每一步的输入输出都落在文件里，接手者读文件就够
9. 跨会话最容易丢的东西：失败的尝试（容易重复踩）、被否决的方案（容易重新纠结）——sessions + compound 正好接住
10. 把接力变成习惯：每次任务结束留好 handoff，下一棒就不用猜
11. 延伸一节·团队接力：同一套 docs 在团队里也成立——新人不靠老人口头讲历史，直接读 docs/plans 和 docs/solutions（作为个人主线的自然外推，不另起平行主线）
12. 收尾复盘（全季固定动作）：这个接力如果没有 docs/sessions，要多花多少"重新理解"的时间
13. 引向第三季：本篇只示范用法与价值，机制深挖在细节季（sessions 专题 / s3-10 compound）
14. 小结 + 结尾 CTA 四件套（主 CTA = 阅读原文去 GitHub）+ 下一篇 blockquote 预告 **op-03（存量改造）** 点名具体问题不剧透

## 与第三季分工（防重叠，高风险对，必写）

- **高风险对：op-06 ↔ sessions 专题 / s3-10 compound。** 本篇是"真实接力场景里跑一遍交接动作"。
- 本篇**只示范用法与价值，不复述** sessions / compound 的机制本身。
- 篇末引向细节篇（007 sessions 专题讲 sessions 检索机制、s3-10 讲 compound 沉淀机制）。
- 写作前交叉检查 sessions 专题与 s3-10 已讲机制，确保 op-06 不抢机制。
- 事实锚点（源码取证）：sessions 检索的是 agent 会话历史（~/.claude、~/.codex），返回 distilled replay references（What was tried / didn't work / Key decisions / Related context），本身不创建持久产物；它是 compound 的输入供应者之一。
