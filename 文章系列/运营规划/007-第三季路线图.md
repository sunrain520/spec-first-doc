## 一、第三季定位（跳过第二季的取舍）

读者目前只看过第一季：他们**认同了 Harness 的认知框架**（决策输入质量、六层 Harness、五个信任缺口），但**没看过第二季的主链逐个 skill 教程**。

这决定了第三季的两个约束：

1. **必须自包含、直接承接第一季认知**——不能假设读者已经会用 brainstorm/plan/work 主链。
2. **不重复第二季的"逐个 skill 怎么用"教程**——第二季还在，暂缓发布，第三季不能抢它的活，否则两季撞车。

所以第三季重新定位为「**真实落地季**」：

> 少讲「某个命令怎么用」，多讲「认同 Harness 之后，怎么在真实的项目、团队、多仓、历史场景里把它用稳」。

每篇仍服务于第一季的总命题——**让模型在更好的工程条件下做判断**——只是把战场从"认知"搬到"真实工程现场"。

**与第二季的分工：**

| | 第二季（暂缓） | 第三季（现在写） |
|---|---|---|
| 视角 | 单个 skill 怎么用（操作） | 真实场景怎么落地（治理 + 取舍） |
| 读者问题 | "这个命令做什么" | "我的项目/团队该怎么用" |
| 调性 | 教程 | 经验、边界、取舍、真实故事 |

**目标读者：** 已认同方法论的 A 型读者 + 想在团队/存量/多仓落地的进阶读者。

## 三、7 篇标题与大纲

> **标题格式约定：** 正文标题统一为 `Spec-First：xxxx`（全角冒号）。

### S3-01 · 上下文研究类（机制篇）

**建议标题：** Spec-First：AI 每次都从零开始？因为你没把"历史"喂给它

**核心论点：** 第一季讲"决策输入质量"，但最常被漏掉的一类输入是**历史上下文**——之前怎么调查的、哪些方案试过为什么失败、团队当初为何这样定、spec-first 自己又改了什么。spec-first 用三个只读 workflow 把这类"散落在会话、Slack、Release 里的历史"重新变成可消费的决策输入。

**大纲：**
1. **失败场景**：换了个会话/接手别人分支，AI（和你）都从零开始重新踩同一个坑。
2. **三个研究入口，各管一类历史**
   - `spec-sessions`：跨会话记忆——"这个仓库我之前做过什么/某问题怎么查的/哪些方案失败了"。基于当前 repo+branch，由 session-historian 检索总结。
   - `spec-slack-research`：组织记忆——团队历史决策、约束、争议，输出有出处的 research digest（不是消息列表）。依赖 Slack MCP，不可用时不阻塞。
   - `spec-release-notes`：工具演进记忆——spec-first 最近变了什么、某 skill 哪个版本变的，读 GitHub Releases，只读不改本地。
3. **共同 DNA**：只读、输出"解释型 digest"而非原始数据、为主链（brainstorm/plan/work/review）提供有出处的背景。
4. **关键判断**：研究类是"输入增强"不是"事实来源"——digest 仍需源码/测试再确认；`spec-sessions`（研究 workflow）≠ CLI `spec-first session`（多 actor worktree advisory），别混。
5. **可带走资产**：「动手前历史上下文三问」清单（我做过吗？团队定过吗？工具变过吗？）。

### S3-02 · 专项审计类（机制篇）

**建议标题：** Spec-First：代码 review 过了，为什么还是不放心

**核心论点：** 第一季的 Review Harness 讲的是主链代码 review。但真实项目里有些质量风险，通用 code-review 看不到——skill/协议本身的工程质量、移动 App 跨端一致性。spec-first 把它们做成**专门的静态审计闸**：先收集确定性事实，再让 LLM 判断风险，产出 scorecard 和改进计划，而不替代运行时验证。

**大纲：**
1. **失败场景**：code-review 全绿，但上线后发现埋点漏了、i18n 缺了、或自建的 skill 触发边界写错导致 AI 乱跑。
2. **skill-audit：把 skill 当工程协议审**——收集 source inventory / runtime drift / promise implementation / 安全信号，LLM 从 trigger 精度、scope、I/O 契约、治理投递、安全、可维护性判断风险，输出 scorecard + finding/security report + 改进计划；支持 `--runtime`、`--patch-preview`。面向构建/维护 workflow 的进阶用户。
3. **app-consistency-audit：移动 App 静态优先一致性审查**——在模拟器/真机/QA 前，比对 PRD、Figma context、source、路由、KMP/Clean Architecture、组件复用、埋点、i18n、工程质量；支持 headless/report-only/deep。
4. **共同机制**：静态优先、提升 review 输入质量、产出落在 `.spec-first/audits/` 与 `.spec-first/app-audit/`，**不替代运行时验证**。
5. **关键判断**：审计是"提前发现"不是"上线放行"；受众垂直（skill 作者 / 移动团队），但对该群体价值极高。
6. **可带走资产**：「主链 review 之外的三道闸」决策卡（什么风险该用哪道审计）。

### S3-03 · UI 交互打磨（取舍/案例篇）

**建议标题：** Spec-First：让 AI 改 UI，最难的是它"看不见"页面

**核心论点：** AI 改后端有测试兜底，改 UI 却长期是"盲改"——它看不见渲染结果。`spec-polish-beta` 用"启动 dev server → 让人实际看页面 → 基于反馈小步改 → 留截图证据"的交互循环，把肉眼可见的 UI 打磨变成有证据的受控过程。

**大纲：**
1. **失败场景**：让 AI"把这个页面调好看点"，它改了一版又一版，你来回截图描述，越改越乱。
2. **polish-beta 怎么工作**：确认目标 branch/PR（拒绝在 main/master 直接跑）→ 读 `.claude/launch.json` 或自动识别 Rails/Next/Vite/Nuxt/Astro/Remix/SvelteKit/Procfile → 解析包管理器/启动命令/端口 → 打开页面 URL → 基于用户实际反馈小步改 + 截图证据。
3. **边界**：仅适合"已经能跑的浏览器可见 UI"；非 UI 代码（后端/CLI/迁移）仍走 work/review；它是 beta，不替代 code review/QA/发布验证。
4. **承接第一季**：这是"决策输入质量"在 UI 维度的延伸——把"人眼看到的渲染"变成 AI 的输入。
5. **关键判断**：什么时候用（边看边调体验）、什么时候别用（结构性改动、main 分支）。
6. **可带走资产**：「UI polish 适用性自检」+ launch.json 最小配置示例。

> 注：S3-03 可独立成文（前端/全栈读者传播性强），若厚度不足则并回 S3-02 作为一节。

### S3-04 · 开发模式与边界（机制篇）

**建议标题：** Spec-First：你的 .spec-first，到底管多大范围

**核心论点：** 第一季把 spec-first 讲成"以仓库为边界的工程闭环"，但真实项目有单仓单项目、单仓多模块、多仓工作区三种拓扑。spec-first 的边界**按 Git 拓扑判断，不按命令参数判断**——搞错边界，需求、计划、graph facts、review 证据就会分裂。

**大纲：**
1. **失败场景**：monorepo 里给每个 module 各塞一套 `.spec-first`，结果需求/计划/证据四分五裂；或多仓工作区里 AI 把改动写错了 repo。
2. **三种模式与权威边界**（用 guide 的对照表）
   - 单仓单项目：`.spec-first` = repo root，最稳定，新手起点。
   - 单仓多模块：仍是一个 Git 工程，**不拆多套 `.spec-first`**，由 plan/task pack/work/review 按 module 边界筛选。
   - 多仓工作区：每个 child repo 有自己的 canonical facts；父目录只写 `.spec-first/workspace/*summary.json`（advisory，不是任何 child 的真相）。
3. **核心规则**：写文件/修复/测试/review autofix/commit 前，**必须有明确 target_repo**；不能让 cwd、graph 结果、workspace advisory 自动替你选 repo。命令边界：`--repo <child>` / `--all-repos`。
4. **承接第一季**：边界即"决策输入的作用域"——边界错了，再好的证据也喂错了地方。
5. **关键判断**：什么时候该拆 repo、什么时候该留在一个 repo。
6. **可带走资产**：「三种拓扑边界速查卡」。

### S3-05 · 产物治理（取舍篇）

**建议标题：** Spec-First：仓库里哪些该提交，哪些是 AI 生成的别碰

**核心论点：** 用了 spec-first 后仓库里多出一堆目录：`docs/` 的需求/计划/方案、`.claude/.codex/.agents/skills` 的 generated runtime、`.spec-first/` 的 control-plane facts。搞不清谁生成、谁读取、该不该提交、漂移了怎么修，团队协作就会乱。这正是第一季"不要修生成物，要修 source-of-truth"原则的真实落地。

**大纲：**
1. **失败场景**：有人手改了 `.claude/` 里的 generated 文件，下次 `spec-first init` 一跑全被覆盖；或把 `.spec-first/` 的 runtime facts 提交进 Git 引发冲突。
2. **三类产物，三种边界**（用 artifacts-map 的表）
   - 协作知识 `docs/brainstorms|plans|tasks|solutions` + `CHANGELOG.md`：source truth，通常提交。
   - generated runtime `.claude/ .codex/ .agents/skills`：非 source truth、不手改、漂移用 `spec-first init` 重建。
   - `.spec-first/` control-plane facts：runtime/advisory，通常不提交。
3. **核心原则**：source truth 在 `src/cli/ skills/ agents/ templates/` 和 `docs/`；项目规范写进 `AGENTS.md/CLAUDE.md/docs/contracts`，**不要塞进 generated 目录**；脚本管确定性事实，LLM 管判断。
4. **承接第一季**：这是"source-of-truth vs 生成物"边界在目录层的兑现——改错层，AI 的世界观就被污染。
5. **关键判断**：一个文件该不该进 Git，先问"它是谁生成的、能不能被重建"。
6. **可带走资产**：「产物 / Git 边界速查表」+ `.gitignore` managed block 说明。

### S3-06 · Building in Public（取舍/故事篇）

**建议标题：** Spec-First：我为什么没给它做一个"重状态机"

**核心论点：** 003 规划里一直有"流 3 Building in Public"却从未落地。这篇补上：用第一人称讲 spec-first 的一个核心设计取舍——**为什么坚持 light contract、Scripts prepare/LLM decides，而不是把 workflow 固化成重状态机**，以及双宿主（Claude Code + Codex）治理是怎么演化出来的。

**大纲：**
1. **开场**：很多人看到"workflow"就以为是一套强制流水线/状态机。我一开始也想过那么做，后来没有——这篇讲为什么。
2. **取舍一：light contract 而非重状态机**——workflow 是"轻量任务边界 + 可验证 handoff"，不是逐步指令；重状态机会让 AI 失去在边界内判断的空间，也更难维护。
3. **取舍二：Scripts prepare facts, LLM decides**——脚本只产确定性事实（readiness、inventory、drift），语义判断交给 LLM；不让脚本假装架构师。这是第一季 02 就埋下的原则，这里讲它怎么真实约束每个 skill 的设计。
4. **取舍三：双宿主治理演化**——为什么要同时支持 Claude Code 和 Codex，generated runtime mirror 与 source 的边界是怎么逼出来的（呼应 S3-05）。
5. **真实踩坑**：举 1-2 个真实演化片段（如 reviewer dispatch、upstream sync 方法论），讲当时的判断和结果。
6. **可带走判断**：好的 AI 工程系统，约束的是"输入和边界"，不是"步骤"。
7. **资产**：设计取舍备忘（light contract 的最小字段哲学）。

> 注：本篇是观点+故事调性，evidence 来自仓库真实 learning（docs/solutions、角色契约），写作前需选可公开的真实片段并降敏。

### S3-07 · 落地综述收口（综述篇）

**建议标题：** Spec-First：真实项目第一周，新项目 / 存量系统 / 小团队各怎么开

**核心论点：** 系列收口篇。把前面所有认知和能力收敛成"真实项目第一周怎么落地"的可执行综述，按三种真实起点给出最小路径，回到第一季的总命题——**不是让 AI 更聪明，而是让每次任务都让下一次任务更容易**。

**大纲：**
1. **新项目接入**：`install → doctor → init → 重启 → mcp-setup →` 第一个需求从 brainstorm（需求不清）或 plan（HOW 不清）开始。
2. **存量项目接入**：先补 runtime setup + 项目说明 + 必要 provider evidence，再进 feature；provider evidence 不 ready 时显式 degraded、补 bounded 源码读取；规范维护在 AGENTS.md/CLAUDE.md/docs/contracts。
3. **小团队协作分级**（用 best-practices 表）：一人快速交付 / 两三人协作（requirements、plan、review findings 提交）/ 多人维护同模块（用 contracts 固化规则 + compound 沉淀）/ 新成员接手（先读 plans、solutions、近期 review findings）。
4. **什么时候用 optimize / polish / audit**：optimize 要先有可重复 measurement；polish-beta 要 UI 已能跑；app-consistency-audit 用于多端需求实现前后。
5. **收口**：回到第一季——决策输入质量、可委派的工程闭环、经验复利。一句话收束全系列。
6. **资产**：「spec-first 落地第一周手册」（三种起点 × 最小路径）。
