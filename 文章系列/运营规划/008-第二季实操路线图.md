## 一、本季定位

第一季讲清了**为什么**需要 Harness（设计思想）。第二季回答**它在真实项目里怎么用、值不值得信**——用真实项目案例从头跑到尾，每步留下产物。

这是整个系列最可能建立信任、最易传播的一季：讲方法论的文章很多，但"用一个真实项目从头跑通、每步留下产物"的连续案例少见。

**目标读者：** 认同方法论的 A 型读者 → 想确认"它能用"、准备试用的 B 型读者。

**每篇 DNA（分两类）：**

- **骨架篇（op-01~04）：** 真实场景痛点 → 选地图（需求模式/拓扑）→ 走完整链路 → 每节点输入/输出 → 产物落在哪 → 可带走的判断。
- **切面篇（op-05~07）：** 真实场景痛点 → 切入链路某一段（或某项能力）→ 放大示范 → 可带走的判断 + 引向第三季对应机制篇。

两类都以"真实场景痛点开场、可带走判断收尾"，区别在骨架走全链路、切面只切一段。

## 二、7 篇架构：场景骨架 + 切面（为什么不是 7 篇同构走链路）

如果 7 篇都是"换个场景再走一遍完整链路"，到第 4、5 篇读者会疲劳——结构同构、惊喜递减。顶尖深度内容季的做法是 **主线骨架 + 不同切面**。

本季因此设计为两层：

```text
骨架层（3 篇场景案例）：覆盖需求模式 × 仓库拓扑矩阵，建立"完整链路"心智
  op-02  从 0 到 1 新产品      0-1   × 单仓单项目
  op-03  存量系统改造          10-100 × 单仓多模块
  op-04  多仓 / 团队协作        任意   × 多仓工作区

切面层（3 篇高价值切面）：不重走全链路，各自切一个读者高频真实痛点
  op-05  出错了怎么办          调试实战（链路里 work/debug 那一段的放大）
  op-06  换人/换 AI 怎么接      跨会话与协作接力（上下文复用）
  op-07  上线敢不敢信          质量门与上线信心（review/审计/测试的收口）

总览层（已就位）
  op-00  全链路走查 + 两张地图
  op-01  日常增量功能          1-10  × 单仓单项目（最简骨架，已写）
```

> **取舍说明：** 你提出的 4 个候选切面中，「调试实战」「跨会话协作接力」「质量门与上线信心」独立成篇（op-05/06/07）；「一次迭代的真实复盘」不单独占篇，而是作为**每篇案例的收尾段**反复出现（每篇都复盘"哪步跳了、哪步返工、代价多少"），让复盘成为贯穿全季的固定动作，而非一次性内容。多端不再独立成篇，并入 op-04 的一个小节。

## 三、两张地图（贯穿全季）

骨架层 3 篇建立在两张地图上（详见 op-00 总览）：

**地图一 · 三种需求模式**

| 模式 | 场景 | 需求工具 |
|---|---|---|
| 0-1 | 全新产品，方向未定 | ideate → brainstorm |
| 1-10 | 已有产品，增量功能 | brainstorm 或 spec-prd |
| 10-100 | 存量系统，增量需求 | spec-prd（brownfield） |

**地图二 · 三种仓库拓扑**

| 拓扑 | 形态 | 边界规则 |
|---|---|---|
| 单仓单项目 | 一个 repo = 一个应用 | `.spec-first` = repo root |
| 单仓多模块 | 一个 repo 多个 module | 不拆多套 `.spec-first`，按 module 边界筛选 |
| 多仓工作区 | 父目录多个 child repo | 写入前必须有明确 target_repo |

切面层 3 篇不按地图走，而是各自切一个跨场景的真实痛点。

## 五、各篇要点

### op-00 · 全链路走查（已就位）

原细节季 S2-00 上移。建立两张地图，用一个真实需求（1-10 × 单仓单项目）走完 setup → brainstorm → plan → doc-review →（write-tasks）→ work → code-review → compound 完整链路。

### op-01 · 日常增量功能（1-10 × 单仓单项目，已写）

最常见场景，最简骨架。待办应用加标签过滤，从环境就绪到经验沉淀完整跑一遍，每步给命令/产物/跳过条件，含"这不是更慢吗"的反驳与个人/团队受益。是全季的"标准模板"，后面几篇在它建立的链路心智上做变化。

### op-02 · 从 0 到 1 新产品（0-1 × 单仓单项目）

**核心问题：方向都没定，怎么起步？** 这是和 op-01 最大的差别——op-01 需求已清晰，op-02 连方向都模糊。

要点：
- `ideate` 探索多个方向候选并评估，再 `brainstorm` 收敛成 requirements brief。
- 重点讲"发散 → 收敛"：ideate 负责发散（连方向都不确定时），brainstorm 负责收敛（把模糊变可执行）。
- 案例建议：一个小而完整的新工具/新功能从一句模糊想法到能动手。
- 收尾复盘：0-1 场景哪步最不能省（需求收敛），哪步可以轻（一旦方向定了，链路其实和 op-01 趋同）。
- 与第三季 s3-02 brainstorm / s3-03 spec-prd 的分工：本篇是"0-1 真实场景里跑发散→收敛"，s3-02/03 是 brainstorm/ideate 的机制深挖；本篇不复述机制，篇末引向细节篇。

### op-03 · 存量系统改造（10-100 × 单仓多模块）

**核心问题：在一个有历史包袱的老系统上，怎么安全地改？**

要点：
- `spec-prd` 写 current-state evidence + change delta，让 plan 不用猜 WHAT。
- plan 按 module 边界拆 implementation units；review 按影响面分组。
- 重点讲"描述 delta"和"既有约束识别"——存量系统翻车的最大来源是不知道既有约束。
- 收尾复盘：为什么存量场景 doc-review 和 write-tasks 通常不能省。
- 与第三季 s3-03 spec-prd 的分工：本篇是"存量场景实战跑一遍 brownfield 链路"，s3-03 是 spec-prd 机制深挖；本篇不复述 current-state/delta 的机制定义，篇末引向 s3-03。

### op-04 · 多仓 / 团队协作（多仓工作区，含多端节）

**核心问题：多个仓库、多个人、多个端，边界怎么守不乱？**

要点：
- 父 workspace 只做 advisory，每步写入前必须有明确 `target_repo`；不能让 cwd/graph 替你选 repo。
- `--repo` / `--all-repos` 的用法边界。
- **多端小节**：多端（Web/iOS/Android/backend）是 **actors 维度，与仓库拓扑正交**——可发生在单仓多 target，也可发生在多仓；本篇借多仓案例顺带示范多端 actor 拆分，不要把"多端=多仓"混为一谈。讲多端需求怎么在 brainstorm 拆 actors、plan 列各端文件边界，给出至少一个可带走判断。
- 收尾复盘：多仓最容易出的事故（写错 repo），以及链路怎么挡住它。

### op-05 · 出错了怎么办（切面：调试实战）

**核心问题：AI 改崩了、bug 反复出现，怎么收拾？** 这是 op-01~04 都假设"顺利跑通"之后，必须补上的反面。

要点：
- 不是反复问 AI"你再看看"，而是 `debug` 的 hypothesis ledger：把失败变成可追踪的假设证据。
- 一次只改一件事、先复现再修复、根因必须有 source/test/log 支撑。
- 案例建议：拿一个真实的"看起来对其实错"的 bug 从复现走到根因。
- 与第三季 s3-08 debug 的分工：本篇是"实战跑一遍"，s3-08 是"机制深挖"，本篇末尾引向 s3-08。

### op-06 · 换人换 AI 怎么接（切面：跨会话与协作接力）

**核心问题：换了个会话、换了个 AI、接手别人的活，上下文怎么不丢？**

要点：
- `docs/` 沉淀（requirements/plan/solutions）是跨会话记忆的载体——这正是前面每篇都在攒的东西，到这篇兑现价值。
- `sessions` 类能力回顾"之前做过什么、哪些方案失败过"。
- 接手别人未完成的活：先读 plan、solutions、最近 review findings。
- 呼应运营主线：这一篇把"每次任务都让下一次更容易"从口号变成可操作的交接动作。
- 与第三季分工：本篇是"真实接力场景里跑一遍交接动作"，sessions/compound 的机制深挖在第三季细节季（sessions 专题 / s3-10 compound）；本篇只示范用法与价值，不复述机制本身，篇末引向细节篇。

### op-07 · 上线敢不敢信（切面：质量门与上线信心）

**核心问题：AI 写的代码，凭什么敢上线？** 这是整季的信任收口。

要点：
- `code-review` 的六维 + 带证据的 finding（区别于"你再检查一下"）。
- 测试、专项审计（如移动端一致性）在上线前怎么把关。
- review 按仓库已有规则（AGENTS.md/CLAUDE.md/docs/contracts）审，规范越清晰把关越准。
- 收尾：上线信心不是"相信 AI 很聪明"，是"每一步都有证据、有验证、有可追溯的把关"。
- 与第三季 s3-09 code-review 的分工：本篇是"上线前实战把关跑一遍"，s3-09 是"review 六维 + 证据契约的机制深挖"；本篇不复述六维定义本身，只示范怎么用它建立上线信心，篇末引向 s3-09。
