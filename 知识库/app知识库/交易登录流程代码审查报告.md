

## 一、完整调用链路

### 1.1 架构分层

交易登录采用 **Android UI 层 + KMP 业务层** 的桥接架构。**关键：方向是 KMP 反向驱动 Android**——App 启动时 `UITransLogin.init()`（`TradeApplication.kt:70`）向 KMP 的 `ServiceLocator.registerLoginUIHandler` 注册 `LoginUIHandler`，业务发起登录后由 **KMP 回调** `showLoginUI()` 让 Android 弹出登录界面。

```
┌──────────────────────── Android 侧（trade 模块）────────────────────────┐
│  UI 交互 / 指纹本地处理 / 状态广播                                          │
│  · UITransLogin        Android↔KMP 桥接（向 KMP 注册 LoginUIHandler）     │
│  · TransLoginHelp      POPUP 弹窗 + 指纹流程编排（611 行，最核心）          │
│  · TradeLoginActivity  PAGE 独立登录页（+ TradeLoginFragment）            │
│  · TransLoginPresenter PAGE 登录页的 MVP Presenter                        │
│  · FingerprintApi      指纹 token 本地读取 / RSA 解密 / 有效期校验          │
│  · TransLoginManager   登录请求发起(直驱) + 登录后处理 + 副作用接口编排      │
│  · TransLoginErrTool   登录错误码处理                                      │
└──────────────┬────────────────────────────────────┬───────────────────────┘
   注册/回调驱动 │ LoginUIHandler                      │ IUserPerformLogin
   showLoginUI  │ performLogin(suspend)               │ 回调桥接(密码/token)
┌──────────────▼────────────────────────────────────▼───────────────────────┐
│                    KMP 层（biz-common，未检出）                            │
│  · TradeLoginHelper.startLoginForRequest()  ★真正的登录 HTTP 请求         │
│  · TradeLoginHelper.checkLogin() / isLoginSync()   登录状态查询/判断       │
│  · PermissionManager.grabPermission()       LV2 行情权限切换              │
└───────────────────────────────────────────────────────────────────────────┘
```

**两套并行登录机制**（登录态判断统一走 KMP，**发起登录请求**分两套；详见 §1.2）：

| 机制 | 发起路径 | HTTP 由谁调用 | 引入时间 |
|------|---------|--------------|---------|
| **A. KMP 驱动**（主） | 用户输入 → `userPerformLogin.onResult(isPwd,result)` → KMP `performLogin` 的 `continuation.resume(UserInputData)` | **KMP 内部** `startLoginForRequest()` | 2024/11 改版（`LoginHelper.kt`/`UITransLogin.kt`） |
| **B. Android 直驱**（遗留，仍在生产运行） | `TransLoginPresenter.pwdLogin/touchLogin` → `TransLoginManager.requestPwdLogin/requestTouchLogin` → `requestLogin` | **Android 直接调** `TradeLoginHelper.startLoginForRequest()` | 2016/10（`TransLoginPresenter`） |

### 1.2 两套机制如何路由（统一 KMP 入口 + 两套发起并存，无开关）

**结论：两套代码并存，没有任何运行时开关 / 灰度 / 配置项；线上 release 两套都在跑，按场景分工。** 关键认知（容易误判的点）：

> **所有交易登录入口都先经过 KMP 统一判断登录态**，不存在"绕过 KMP 直接打开登录页"的入口。两参 `start` 看似直驱，实则内部先调 `executeIfUnLogin` → KMP：

```java
// TradeLoginActivity.java:37 —— 两参 start 并非直接开页，先进 KMP
public static void start(Context context, Intent target) {
    LoginHelperKt.executeIfUnLogin(context, target, () -> {   // ← 先进 KMP 判断登录态
        TradeLoginActivity.start(context, target, null);       //   KMP 判定未登录后才开页
        return null;
    });
}
```

**线上实际分工**：登录态判断 100% 由 KMP 统一负责；真正发登录请求时按 UI 形态分两套：

```
                  所有交易登录入口
                        │
                        ▼
   KMP TradeLoginHelper（统一登录态判断，2024/11 架构，release 必经）
      isLoginSync / isUnlockSync / executeIfUnLogin / executeIfUnlocked
                        │
            ┌───────────┴────────────┐
            ▼                         ▼
       POPUP 弹窗                 PAGE 独立登录页
   （解锁类：下单/条件单/改密）   （TransSetInitPasswordOrLoginFragment
       机制 A                      → TradeLoginFragment，isIndependentPage=true）
   KMP 全程驱动，KMP 内部发 HTTP   userPerformLogin == null
                                  → 机制 B：presenter.pwdLogin 直接调 startLoginForRequest
```

**运行时分叉点**：UI 落到 PAGE 后，`TradeLoginFragment.transLogin()` 用一个静态字段是否为 null 决定 A/B：

```java
// TradeLoginFragment.transLogin(pwd:String?) —— PAGE 内的 A/B 分叉（TradeLoginFragment.kt:373-386）
if (userPerformLogin != null) {   // ← 字段非空 = 机制 A：KMP 显式注入了回调
    notifyKMPToLogin(true, pwd)    //   把密码回传给 KMP，KMP 内部发 HTTP
} else {                          // ← 字段为空 = 机制 B：原生 Presenter 直驱
    presenter.pwdLogin(...)        //   Presenter 直接调登录接口
}
```

**各场景线上走向**：

| 用户场景 | 入口 | 是否注入 KMP 回调 | 线上走哪套 | UI |
|---------|------|------------------|-----------|-----|
| 下单/条件单/改密 **解锁** | `executeIfUnlocked()` | 是（`UITransLogin` 构造 `userPerformLoginImpl`） | **机制 A** | POPUP 弹窗 |
| KMP 决定走独立页且注入回调 | `showLoginUI(PAGE)` → `start(ctx,intent,callback)`（三参） | 是 | **机制 A** | PAGE |
| 点交易页未登录 → 跳登录页（**最常见**） | 业务页 `start(ctx,target)`（两参）→ `executeIfUnLogin` → KMP → 开 PAGE | **否**（两参内部传 `null`） | **机制 B** | PAGE |

> 两参入口（`TransLoginInterceptor`、`PlaceOrderFragment`、`TransCommonActivity`、`PageHelper` 等绝大多数业务页）经 `executeIfUnLogin` 进 KMP，KMP 判定未登录后打开 PAGE，页面内 `userPerformLogin==null` → 落到**机制 B**。这是线上最高频的登录路径。

**架构判断**：这是一次**未完成的迁移**——2024/11 引入"KMP 统一登录态判断 + KMP 驱动 UI"（机制 A），但 2016 年的原生 Presenter 直驱（机制 B）未下线，作为 PAGE 独立登录页的发起方继续在生产运行。两套靠共用 UI 上的静态字段 `userPerformLogin` 做运行时分叉。**结论：release 包内两套登录发起逻辑都活跃，机制 A 与 I-08/I-11 相关问题、机制 B 与 I-03/I-13 相关问题在生产环境都会触发。** 该静态分流设计直接引出 **I-11**（静态字段跨入口残留可致误路由，见 §3）。

**为何独立登录页不走 KMP 编排？**（部分为推断，KMP 源码未检出）

需先澄清：独立登录页**并非完全不走 KMP**——登录态判断（`isLoginSync`）和发请求（`requestLogin → startLoginForRequest`）仍调 KMP，它只是**绕过了 KMP 的 UI 协程编排层**（`showLoginUI → performLogin(suspend) → continuation.resume`）。原因：

1. **语义不匹配**：机制 A 的协程"挂起-恢复"模型适配"**页内解锁**"——用户正下单，KMP 挂起业务流、等解锁后原地 `resume` 继续。而独立登录页是"**导航式登录**"——跳转到专门的 Activity，没有"被挂起的业务流"要恢复，让协程跨整个 Activity 生命周期挂起反而是负担。
2. **稳健性**：协程挂起着等一个长生命周期 Activity（含返回、旋转、设密码子流程）正是 I-01/I-08/I-11 这类 bug 的温床；独立页交给 Android 自管生命周期、命令式直调 KMP 函数，对长驻页面更稳。
3. **遗留**：机制 B 是 2016 年的老入口，2024 改版只做了兼容（`userPerformLogin != null` 时走 A），未强制迁移。

> 即两套并存不是单纯冗余，而对应**两类语义不同的入口**（页内解锁 vs 登录页导航）。收敛前提是 KMP 编排层能稳妥驱动完整 Activity 生命周期——而这正是当前 I-01/I-08/I-11 暴露的薄弱点。

### 1.3 后端接口清单

| # | 接口语义 | 调用入口 | 精确路径 | 触发时机 |
|---|---------|---------|---------|---------|
| 1 | **交易登录**（密码/指纹统一） | `TradeLoginHelper.startLoginForRequest()`（KMP） | KMP 内部封装 | 用户输入密码 / 指纹验证通过 |
| 2 | **登录状态查询** | `TradeLoginHelper.checkLogin()`（KMP） | KMP 内部封装 | App 启动 / `postInitLoginStatus()` |
| 3 | **LV2 行情权限切换** | `PermissionManager.grabPermission()`（KMP，经 `Router.switchLV2`/`SwitchLV2Util`） | KMP 内部封装 | 登录成功后，非 LV2 行情时 |
| 4 | **指纹设置保存** | `CommAPI.platform().save()` | 平台域 retrofit | 开启/关闭指纹登录 |
| 5 | **登出交易系统** | `ApiService.transLogout()` | `POST /stock/logout`（金融域） | 退出 / 被挤下线 |

```

        │  ② 机制A 驱动入口: 业务调 KMP 登录 → KMP 回调已注册的 LoginUIHandler
        ▼

        │                                                 │
```
